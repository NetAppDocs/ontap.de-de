= Erstellen einer ONTAP S3 Bucket-Lebenszyklusverwaltungsregel
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Ab ONTAP 9.13.1 können Lebenszyklusmanagementregeln erstellt werden, um die Lebenszyklen von Objekten in Ihren S3-Buckets zu verwalten. Löschregeln für bestimmte Objekte in einem Bucket lassen sich definieren, und durch diese Regeln können diese Bucket-Objekte ablaufen. So werden Aufbewahrungsanforderungen erfüllt und der gesamte S3-Objektspeicher effizient verwaltet.


NOTE: Wenn die Objektsperrung für Ihre Bucket-Objekte aktiviert ist, werden die Lebenszyklusverwaltungsregeln für den Objektablauf nicht auf gesperrte Objekte angewendet. Informationen zur Objektsperrung finden sich unter link:../s3-config/create-bucket-task.html["Einen Bucket erstellen"].

.Bevor Sie beginnen
* Eine S3-fähige SVM mit einem S3-Server und einem Bucket muss bereits existieren. Siehe link:create-svm-s3-task.html["Eine SVM für S3 erstellen"] für weitere Informationen.
* Bucket-Lifecycle-Management-Regeln werden nicht unterstützt, wenn S3 in Multiprotokoll-NAS-Volumes oder wenn S3 in MetroCluster Konfigurationen verwendet wird.


.Über diese Aufgabe
Beim Erstellen Ihrer Lebenszyklusverwaltungsregeln können Sie die folgenden Löschaktionen auf Ihre Bucket-Objekte anwenden:

* Löschen aktueller Versionen, diese Aktion führt zum Ablauf von Objekten, die durch die Regel identifiziert werden. Wenn die Versionierung für den Bucket aktiviert ist, macht S3 alle abgelaufenen Objekte nicht mehr verfügbar. Wenn die Versionierung nicht aktiviert ist, löscht diese Regel die Objekte dauerhaft. Die CLI-Aktion ist  `Expiration`.
* Löschung nicht aktueller Versionen – Diese Aktion legt fest, wann S3 nicht aktuelle Objekte endgültig entfernen kann. Die CLI-Aktion ist  `NoncurrentVersionExpiration`.
+

NOTE: Eine nicht aktuelle Version basiert auf dem Erstellungs- oder Änderungszeitpunkt der aktuellen Version. Die verzögerte Entfernung nicht aktueller Objekte kann hilfreich sein, wenn ein Objekt versehentlich gelöscht oder überschrieben wird. Beispielsweise kann eine Ablaufregel so konfiguriert werden, dass nicht aktuelle Versionen fünf Tage nach dem Wechsel in den nicht aktuellen Status gelöscht werden. Angenommen, am 1.1.2014 um 10:30 Uhr UTC wird ein Objekt namens `photo.gif` (Versions-ID `111111`) erstellt. Am 2.1.2014 um 11:30 Uhr UTC wird `photo.gif` (Versions-ID `111111`) versehentlich gelöscht, wodurch eine Löschmarkierung mit einer neuen Versions-ID (wie Versions-ID `4857693`) erstellt wird. Nun stehen fünf Tage zur Verfügung, um die ursprüngliche Version von `photo.gif` (Versions-ID `111111`) wiederherzustellen, bevor die Löschung endgültig ist. Am 8.1.2014 um 00:00 Uhr UTC wird die Ablaufregel für die Löschung ausgeführt und löscht `photo.gif` (Versions-ID `111111`) endgültig, fünf Tage nachdem sie zu einer nicht aktuellen Version wurde.

* Löschen abgelaufener Löschmarkierungen - Diese Aktion löscht abgelaufene Löschmarkierungen von Objekten. In Buckets mit aktivierter Versionierung werden Objekte mit einer Löschmarkierung zu den aktuellen Versionen der Objekte. Die Objekte werden nicht gelöscht, und es kann keine Aktion an ihnen durchgeführt werden. Diese Objekte sind abgelaufen, wenn keine aktuellen Versionen mehr mit ihnen verknüpft sind. Die CLI-Aktion ist  `Expiration`.
* Löschen unvollständiger Multipart-Uploads – Diese Aktion legt eine maximale Zeit (in Tagen) fest, die Multipart-Uploads im Status „laufend“ verbleiben dürfen. Danach werden sie gelöscht. Die CLI-Aktion ist  `AbortIncompleteMultipartUpload`.


Die Vorgehensweise hängt von der verwendeten Schnittstelle ab. Mit ONTAP 9.13.1 ist die Nutzung der CLI erforderlich. Ab ONTAP 9.14.1 ist auch die Verwendung von System Manager möglich.



[[manage-lifecycle-management-rules-with-the-cli]]
== Lebenszyklusmanagementregeln mit der CLI verwalten

Ab ONTAP 9.13.1 kann die ONTAP CLI verwendet werden, um Lebenszyklusverwaltungsregeln zu erstellen, mit denen Objekte in S3-Buckets ablaufen.

.Bevor Sie beginnen
Für die CLI sind beim Erstellen einer Bucket-Lebenszyklusverwaltungsregel die erforderlichen Felder für jeden Ablaufaktionstyp zu definieren. Diese Felder können nach der erstmaligen Erstellung geändert werden. Die folgende Tabelle zeigt die eindeutigen Felder für jeden Aktionstyp.

[cols="30,70"]
|===


| Aktionstyp | Einzigartige Felder 


 a| 
NonCurrentVersionExpiration
 a| 
* `-non-curr-days` - Anzahl der Tage, nach denen nicht mehr aktuelle Versionen gelöscht werden
* `-new-non-curr-versions` - Anzahl der zuletzt beibehaltenen nicht aktuellen Versionen




 a| 
Ablauf
 a| 
* `-obj-age-days` - Anzahl der Tage seit der Erstellung, nach denen die aktuelle Version der Objekte gelöscht werden kann
* `-obj-exp-date` - Genaues Datum, an dem die Objekte ablaufen sollen
* `-expired-obj-del-markers` - Objektlöschmarkierungen bereinigen




 a| 
AbortIncompleteMultipartUpload
 a| 
* `-after-initiation-days` - Anzahl der Tage nach der Initiierung, nach denen der Upload abgebrochen werden kann


|===
Damit die Bucket-Lebenszyklusverwaltungsregel nur auf eine bestimmte Teilmenge von Objekten angewendet wird, müssen Administratoren beim Erstellen der Regel jeden Filter festlegen. Wenn diese Filter beim Erstellen der Regel nicht festgelegt werden, wird die Regel auf alle Objekte im Bucket angewendet.

Alle Filter können nach ihrer Erstellung geändert werden, _mit Ausnahme_ der folgenden: +

* `-prefix`
* `-tags`
* `-obj-size-greater-than`
* `-obj-size-less-than`


.Schritte
. Verwenden Sie den  `vserver object-store-server bucket lifecycle-management-rule create` Befehl mit den erforderlichen Feldern für Ihren Ablaufaktionstyp, um Ihre Bucket-Lebenszyklusverwaltungsregel zu erstellen.


.Beispiel
Der folgende Befehl erstellt eine NonCurrentVersionExpiration Bucket-Lebenszyklusverwaltungsregel:

[listing]
----
vserver object-store-server bucket lifecycle-management-rule create -vserver <svm_name> -bucket <bucket_name> -rule-id <rule_name> -action NonCurrentVersionExpiration -index <lifecycle_rule_index_integer> -is-enabled {true|false} -prefix <object_name> -tags <text> -obj-size-greater-than {<integer>[KB|MB|GB|TB|PB]} -obj-size-less-than {<integer>[KB|MB|GB|TB|PB]} -new-non-curr-versions <integer> -non-curr-days <integer>
----
.Beispiel
Der folgende Befehl erstellt eine Lebenszyklusverwaltungsregel für den Expiration-Bucket:

[listing]
----
vserver object-store-server bucket lifecycle-management-rule create -vserver <svm_name> -bucket <bucket_name> -rule-id <rule_name> -action Expiration -index <lifecycle_rule_index_integer> -is-enabled {true|false} -prefix <object_name> -tags <text> -obj-size-greater-than {<integer>[KB|MB|GB|TB|PB]} -obj-size-less-than {<integer>[KB|MB|GB|TB|PB]} -obj-age-days <integer> -obj-exp-date <"MM/DD/YYYY HH:MM:SS"> -expired-obj-del-marker {true|false}
----
.Beispiel
Der folgende Befehl erstellt eine AbortIncompleteMultipartUpload Bucket-Lebenszyklusverwaltungsregel:

[listing]
----
vserver object-store-server bucket lifecycle-management-rule create -vserver <svm_name> -bucket <bucket_name> -rule-id <rule_name> -action AbortIncompleteMultipartUpload -index <lifecycle_rule_index_integer> -is-enabled {true|false} -prefix <object_name> -tags <text> -obj-size-greater-than {<integer>[KB|MB|GB|TB|PB]} -obj-size-less-than {<integer>[KB|MB|GB|TB|PB]} -after-initiation-days <integer>
----


[[manage-lifecycle-management-rules-with-system-manager]]
== Lebenszyklusmanagementregeln mit System Manager verwalten

Ab ONTAP 9.14.1 ist es möglich, S3-Objekte mithilfe von System Manager ablaufen zu lassen. Es besteht die Möglichkeit, Lebenszyklusverwaltungsregeln für S3-Objekte hinzuzufügen, zu bearbeiten und zu löschen. Zusätzlich kann eine für einen Bucket erstellte Lebenszyklusregel importiert und für die Objekte in einem anderen Bucket verwendet werden. Eine aktive Regel kann deaktiviert und später wieder aktiviert werden.



[[add-a-lifecycle-management-rule]]
=== Eine Lebenszyklusverwaltungsregel hinzufügen

. *Speicher > Buckets* auswählen.
. Wählen Sie den Bucket aus, für den Sie die Ablaufregel festlegen möchten.
. Klicken Sie auf das image:icon_kabob.gif["Symbol für Menüoptionen"] Symbol und wählen Sie *Lebenszyklusregeln verwalten*.
. *Hinzufügen > Lebenszyklusregel* auswählen.
. Auf der Seite „Lebenszyklusregel hinzufügen“ den Namen der Regel hinzufügen.
. Definieren Sie den Geltungsbereich der Regel, ob sie für alle Objekte im Bucket oder nur für bestimmte Objekte gelten soll. Wenn Sie Objekte angeben möchten, fügen Sie mindestens eines der folgenden Filterkriterien hinzu:
+
.. Präfix: Geben Sie ein Präfix für die Objektschlüsselnamen an, auf die die Regel angewendet werden soll. In der Regel ist dies der Pfad oder Ordner des Objekts. Pro Regel kann ein Präfix eingegeben werden. Sofern kein gültiges Präfix angegeben wird, gilt die Regel für alle Objekte in einem Bucket.
.. Tags: Es können bis zu drei Schlüssel-Wert-Paare (Tags) für die Objekte angegeben werden, auf die die Regel angewendet werden soll. Nur gültige Schlüssel werden zum Filtern verwendet. Der Wert ist optional. Falls Werte hinzugefügt werden, sollten nur gültige Werte für die entsprechenden Schlüssel angegeben werden.
.. Größe: Sie können den Bereich zwischen minimaler und maximaler Objektgröße einschränken. Sie können einen oder beide Werte eingeben. Die Standardeinheit ist MiB.


. Aktion angeben:
+
.. *Aktuelle Objektversionen ablaufen lassen*: Es kann eine Regel festgelegt werden, nach der alle aktuellen Objekte nach einer bestimmten Anzahl von Tagen seit ihrer Erstellung oder an einem bestimmten Datum dauerhaft nicht mehr verfügbar sind. Diese Option ist nicht verfügbar, wenn die Option *Löschmarkierungen abgelaufener Objekte löschen* ausgewählt ist.
.. *Nicht mehr aktuelle Versionen endgültig löschen*: Geben Sie die Anzahl der Tage an, nach denen die nicht mehr aktuelle Version gelöscht wird, und die Anzahl der zu behaltenden Versionen.
.. *Abgelaufene Objektlöschmarkierungen löschen*: Diese Aktion löscht Objekte mit abgelaufenen Löschmarkierungen, also Löschmarkierungen ohne zugehöriges aktuelles Objekt.
+

NOTE: Diese Option ist nicht verfügbar, wenn die Option *Aktuelle Version von Objekten ablaufen lassen* ausgewählt wird, die alle Objekte nach Ablauf der Aufbewahrungsfrist automatisch löscht. Diese Option ist ebenfalls nicht verfügbar, wenn Objekttags zum Filtern verwendet werden.

.. *Unvollständige Multipart-Uploads löschen*: Legen Sie die Anzahl der Tage fest, nach denen unvollständige Multipart-Uploads gelöscht werden. Wenn laufende Multipart-Uploads innerhalb des angegebenen Aufbewahrungszeitraums fehlschlagen, können die unvollständigen Multipart-Uploads gelöscht werden. Diese Option ist nicht verfügbar, wenn Objekt-Tags zum Filtern verwendet werden.
.. Auf *Speichern* klicken.






[[import-a-lifecycle-rule]]
=== Import einer Lebenszyklusregel

. *Speicher > Buckets* auswählen.
. Wählen Sie den Bucket aus, für den Sie die Ablaufregel importieren möchten.
. Klicken Sie auf das image:icon_kabob.gif["Symbol für Menüoptionen"] Symbol und wählen Sie *Lebenszyklusregeln verwalten*.
. *Hinzufügen > Regel importieren* auswählen.
. Wählen Sie den Bucket aus, aus dem Sie die Regel importieren möchten. Die für den ausgewählten Bucket definierten Lebenszyklusverwaltungsregeln werden angezeigt.
. Wählen Sie die Regel aus, die Sie importieren möchten. Es besteht die Möglichkeit, jeweils eine Regel auszuwählen, wobei standardmäßig die erste Regel ausgewählt ist.
. Klicken Sie auf *Import*.




[[edit-delete-or-disable-a-rule]]
=== Eine Regel bearbeiten, löschen oder deaktivieren

Sie können nur die mit der Regel verknüpften Lebenszyklusverwaltungsaktionen bearbeiten. Wenn die Regel mit Objekt-Tags gefiltert wurde, sind die Optionen *Abgelaufene Objektlöschmarkierungen löschen* und *Unvollständige Multipart-Uploads löschen* nicht verfügbar.

Wenn eine Regel gelöscht wird, gilt diese Regel nicht mehr für zuvor verknüpfte Objekte.

. *Speicher > Buckets* auswählen.
. Wählen Sie den Bucket aus, für den Sie die Lebenszyklusverwaltungsregel bearbeiten, löschen oder deaktivieren möchten.
. Klicken Sie auf das image:icon_kabob.gif["Symbol für Menüoptionen"] Symbol und wählen Sie *Lebenszyklusregeln verwalten*.
. Wählen Sie die gewünschte Regel aus. Es ist möglich, jeweils eine Regel zu bearbeiten und zu deaktivieren. Mehrere Regeln können gleichzeitig gelöscht werden.
. *Bearbeiten*, *Löschen* oder *Deaktivieren* auswählen und den Vorgang abschließen.
