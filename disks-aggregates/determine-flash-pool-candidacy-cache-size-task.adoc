= Bestimmen Sie die ONTAP Flash Pool-Eignung und die optimale Cache-Größe
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Bevor eine bestehende lokale Tier in eine Flash Pool lokale Tier umgewandelt wird, kann festgestellt werden, ob die lokale Tier I/O-gebunden ist und welche Flash Pool Cache-Größe für die jeweilige Workload und das Budget am besten geeignet ist. Es kann außerdem überprüft werden, ob der Cache einer bestehenden Flash Pool lokalen Tier korrekt dimensioniert ist.


NOTE: Vor ONTAP 9.7 verwendet System Manager den Begriff _aggregate_, um eine _lokale Ebene_ zu beschreiben. Unabhängig von der ONTAP Version verwendet die ONTAP CLI den Begriff _aggregate_. Weitere Informationen zu lokalen Ebenen finden sich unter link:../disks-aggregates/index.html["Festplatten und lokale Ebenen"].

.Bevor Sie beginnen
Es sollte ungefähr bekannt sein, wann die lokale Tier, die analysiert wird, ihre Spitzenlast erreicht.

.Schritte
. Erweiterter Modus:
+
`set advanced`

. Wenn festgestellt werden soll, ob eine bestehende lokale Ebene für die Umwandlung in eine Flash Pool lokale Ebene geeignet ist, sollte ermittelt werden, wie stark die Festplatten in der lokalen Ebene während einer Spitzenlastperiode ausgelastet sind und wie sich dies auf die Latenz auswirkt:
+
`statistics show-periodic -object disk:raid_group -instance _raid_group_name_ -counter disk_busy|user_read_latency -interval 1 -iterations 60`

+
Sie können entscheiden, ob es für diese lokale Ebene sinnvoll ist, die Latenz durch Hinzufügen eines Flash Pool Cache zu reduzieren.

+
Der folgende Befehl zeigt die Statistiken für die erste RAID-Gruppe der lokalen Tier "`aggr1`":

+
`statistics show-periodic -object disk:raid_group -instance /aggr1/plex0/rg0 -counter disk_busy|user_read_latency -interval 1 -iterations 60`

. Automatisierter Workload Analyzer (AWA) starten:
+
`storage automated-working-set-analyzer start -node _node_name_ -aggregate _aggr_name_`

+
AWA beginnt mit der Erfassung von Workload-Daten für die Volumes, die der angegebenen lokalen Tier zugeordnet sind.

. Erweiterten Modus beenden:
+
`set admin`

+
AWA sollte so lange ausgeführt werden, bis ein oder mehrere Intervalle mit Spitzenlast aufgetreten sind. AWA sammelt Workload-Statistiken für die Volumes, die der angegebenen lokalen Ebene zugeordnet sind, und analysiert Daten für einen Zeitraum von bis zu einer rollierenden Woche. Wenn AWA länger als eine Woche ausgeführt wird, werden nur Daten aus der jeweils letzten Woche berücksichtigt. Die Schätzungen der Cache-Größe basieren auf den höchsten Lasten, die während des Datenerfassungszeitraums beobachtet wurden; die Last muss nicht während des gesamten Datenerfassungszeitraums hoch sein.

. Erweiterter Modus:
+
`set advanced`

. Anzeige der Arbeitslastanalyse:
+
`storage automated-working-set-analyzer show -node _node_name_ -instance`

. AWA anhalten:
+
`storage automated-working-set-analyzer stop _node_name_`

+
Sämtliche Arbeitslastdaten werden gelöscht und stehen nicht mehr zur Analyse zur Verfügung.

. Erweiterten Modus beenden:
+
`set admin`
