SnapMirror active sync bietet asymmetrischen Schutz und ab ONTAP 9.15.1 auch symmetrischen Aktiv/Aktiv-Schutz.



[[configure-asymmetric-protection]]
== Asymmetrischen Schutz konfigurieren

Die Konfiguration des asymmetrischen Schutzes für SAN-Konfigurationen mit SnapMirror active sync umfasst die Auswahl von LUNs im ONTAP Quell-Cluster und deren Hinzufügen zu einer Konsistenzgruppe.

.Bevor Sie beginnen
* Sie benötigen eine SnapMirror Synchronous-Lizenz.
* Sie müssen Administrator eines Clusters oder einer Storage-VM sein.
* Alle Bestandteile einer Konsistenzgruppe müssen sich in einer einzigen Storage VM (SVM) befinden.
+
** LUNs können auf verschiedenen Volumes liegen.


* Quell- und Ziel-Cluster dürfen nicht identisch sein.
* Es kann keine SnapMirror Aktiv-Sync-Konsistenzgruppenbeziehung zwischen ASA Clustern und Nicht-ASA-Clustern hergestellt werden.
* Der Standard-IPspace ist für SnapMirror active sync für Cluster-Peer-Beziehungen erforderlich. Benutzerdefinierte IPspaces werden nicht unterstützt.
* Der Name der Konsistenzgruppe muss eindeutig sein.
* Die Volumes auf dem sekundären (Ziel-Cluster) müssen vom Typ DP sein.
* Die primäre und die sekundäre SVM müssen in einer Peering-Beziehung stehen.


.Schritte
Eine Konsistenzgruppe lässt sich mithilfe der ONTAP CLI oder des System Manager konfigurieren.

Ab ONTAP 9.10.1 bietet ONTAP im System Manager einen Endpunkt und ein Menü für Konsistenzgruppen mit zusätzlichen Verwaltungsfunktionen. Wenn ONTAP 9.10.1 oder höher verwendet wird, siehe link:../consistency-groups/configure-task.html["Eine Konsistenzgruppe konfigurieren"] dann link:../consistency-groups/protect-task.html["Schutz konfigurieren"] zur Erstellung einer SnapMirror Aktiv Sync-Beziehung.


IMPORTANT: Von ONTAP 9.8 bis ONTAP 9.14.1 wird SnapMirror active sync als SnapMirror Business Continuity (SM-BC) bezeichnet.

[role="tabbed-block"]
====
.System Manager
--
. Im primären Cluster zu *Schutz > Übersicht > Schutz für Geschäftskontinuität > LUNs schützen* navigieren.
. Wählen Sie die LUNs aus, die Sie schützen möchten, und fügen Sie sie einer Schutzgruppe hinzu.
. Den Ziel-Cluster und die SVM auswählen.
. *Beziehung initialisieren* ist standardmäßig ausgewählt. Mit *Speichern* beginnt der Schutz.
. *Dashboard > Leistung* zeigt die IOPS-Aktivität für die LUNs an.
. Auf dem Ziel-Cluster kann im System Manager überprüft werden, ob die Beziehung für den Schutz der Geschäftskontinuität synchron ist: *Schutz > Beziehungen*.


--
.CLI
--
. Eine Konsistenzgruppenbeziehung vom Ziel-Cluster aus erstellen.
`destination::> snapmirror create -source-path _source-path_ -destination-path _destination-path_ -cg-item-mappings _volume-paths_ -policy _policy-name_`
+
Sie können bis zu 12 Teil-Volumes mit dem  `cg-item-mappings`-Parameter im  `snapmirror create`-Befehl zuordnen.

+
Das folgende Beispiel erstellt zwei Konsistenzgruppen: `cg_src_ on the source with `vol1` und `vol2` und eine gespiegelte Ziel-Konsistenzgruppe, `cg_dst`.

+
`destination::> snapmirror create -source-path vs1_src:/cg/cg_src -destination-path vs1_dst:/cg/cg_dst -cg-item-mappings vol_src1:@vol_dst1,vol_src2:@vol_dst2 -policy AutomatedFailOverDuplex`

. Vom Ziel-Cluster aus die Konsistenzgruppe initialisieren.
+
`destination::>snapmirror initialize -destination-path _destination-consistency-group_`

. Bestätigen Sie, dass die Initialisierung erfolgreich abgeschlossen wurde. Der Status sollte  `InSync` sein.
+
`snapmirror show`

. Auf jedem Cluster eine Initiatorgruppe erstellen, damit LUNs dem Initiator auf dem Anwendungshost zugeordnet werden können.
`lun igroup create -igroup _name_ -protocol _fcp|iscsi_ -ostype _os_ -initiator _initiator_name_`
+
Weitere Informationen zu  `lun igroup create` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/lun-igroup-create.html["ONTAP-Befehlsreferenz"^].

. Auf jedem Cluster die LUNs der Initiatorgruppe zuordnen:
+
`lun map -path _path_name_ -igroup _igroup_name_`

. Die erfolgreiche Ausführung der LUN-Zuordnung lässt sich mit dem  `lun map` Befehl überprüfen. Anschließend ist es möglich, die neuen LUNs auf dem Anwendungshost zu erkennen.


--

[[configure-symmetric-activeactive-protection]]
== Symmetrischer Aktiv/Aktiv-Schutz konfigurieren

Sie können einen symmetrischen Schutz für SAN-Konfigurationen mithilfe des System Manager oder der ONTAP CLI einrichten. In beiden Schnittstellen sind unterschiedliche Schritte für xref:index.html#key-concepts[einheitliche und nicht einheitliche Konfigurationen] erforderlich.

.Bevor Sie beginnen
* Beide Cluster müssen ONTAP 9.15.1 oder höher ausführen.
* Symmetrische Aktiv/Aktiv Konfigurationen erfordern die  `AutomatedFailoverDuplex` Datensicherungsstrategie. Alternativ ist es möglich, xref:../data-protection/create-custom-replication-policy-concept.html[eine benutzerdefinierte SnapMirror Richtlinie erstellen] sofern die  `-type`  `automated-failover-duplex` erfüllt ist.
* In ONTAP 9.15.1 wird symmetrisches Aktiv/Aktiv nur auf Zwei-Node-Clustern unterstützt.
* Ab ONTAP 9.16.1 GA unterstützt SnapMirror Active Sync symmetrische Aktiv/Aktiv-Konfigurationen auf Vier-Node-Clustern.
+
** Um SnapMirror active sync auf einem Vier-Node-Cluster zu verwenden, muss ONTAP 9.16.1 GA oder neuer ausgeführt werden.
** Vor der Bereitstellung einer Vier-Node-Konfiguration ist Folgendes zu beachten xref:../peering/create-cluster-relationship-93-later-task.adoc[eine Cluster Peer-Beziehung erstellen].
** Die Informationen xref:limits-reference.adoc[Grenzen] für Vier-Node-Cluster werden angezeigt.
** Wenn Sie zu einem Zwei-Node-Cluster zurückkehren, müssen Sie die SnapMirror Active Sync-Beziehungen aus dem Cluster entfernen, bevor Sie zurückkehren.
** Die Vier-Node-Konfiguration kann verwendet werden, um Speicher und Controller zu aktualisieren. Dieser Vorgang ist unterbrechungsfrei und erweitert den Cluster, während Volumes in die neuen Nodes verschoben werden. Weitere Informationen finden sich unter link:upgrade-revert-task.html#refresh-a-cluster["einen Cluster aktualisieren"].


* Ab ONTAP 9.17.1 kann der symmetrische Aktiv/Aktiv-Schutz auf NVMe-Namespaces nur konfiguriert werden, wenn beide Cluster ONTAP 9.17.1 oder höher ausführen.




[[configure-symmetric-activeactive-protection-using-a-scsi-snapmirror-active-sync-configuration]]
== Symmetrischer Aktiv/Aktiv-Schutz mit einer SCSI SnapMirror active sync Konfiguration

.Schritte
Mit dem System Manager oder der ONTAP CLI lässt sich ein symmetrischer Aktiv/Aktiv-Schutz mithilfe von SCSI-Protokoll-Hostzuordnungen konfigurieren.

[role="tabbed-block"]
====
.System Manager
--
.Schritte für eine einheitliche Konfiguration
. Auf der Hauptseite, link:../consistency-groups/configure-task.html#create-a-consistency-group-with-new-luns-or-volumes["Erstellen einer Konsistenzgruppe unter Verwendung neuer LUNs"].
+
.. Beim Erstellen der Konsistenzgruppe Host-Initiatoren angeben, um Initiatorgruppen zu erstellen.
.. Das Kontrollkästchen zum **Aktivieren von SnapMirror** auswählen und anschließend die  `AutomatedFailoverDuplex` Richtlinie wählen.
.. Im daraufhin angezeigten Dialogfeld das Kontrollkästchen **Initiatorgruppen replizieren** aktivieren, um Initiatorgruppen zu replizieren. In **Proximity-Einstellungen bearbeiten** proximale SVMs für die Hosts festlegen.
.. **Speichern** auswählen.




.Schritte für eine nicht einheitliche Konfiguration
. Auf der Hauptseite, link:../consistency-groups/configure-task.html#create-a-consistency-group-with-new-luns-or-volumes["Erstellen einer Konsistenzgruppe unter Verwendung neuer LUNs"].
+
.. Beim Erstellen der Konsistenzgruppe Host-Initiatoren angeben, um Initiatorgruppen zu erstellen.
.. Das Kontrollkästchen zum **Aktivieren von SnapMirror** auswählen und anschließend die  `AutomatedFailoverDuplex` Richtlinie wählen.
.. **Speichern** auswählen, um die LUNs, Konsistenzgruppe, Initiatorgruppe, SnapMirror Beziehung und Initiatorgruppen-Zuordnung zu erstellen.


. Am sekundären Standort wird eine Initiatorgruppe erstellt und die LUNs werden zugeordnet.
+
.. Zu **Hosts** > **SAN Initiatorgruppen** navigieren.
.. **+Hinzufügen** auswählen, um eine neue Initiatorgruppe zu erstellen.
.. Einen **Namen** angeben, das **Host-Betriebssystem** auswählen und anschließend die **Mitglieder der Initiatorgruppe** wählen.
.. **Speichern** auswählen, um die Beziehung zu initialisieren.


. Die neue Initiatorgruppe wird den Ziel-LUNs zugeordnet.
+
.. Zu **Speicher** > **LUNs** navigieren.
.. Alle LUNs auswählen, die der Initiatorgruppe zugeordnet werden sollen.
.. **Mehr** und dann **Initiatorgruppen zuordnen** auswählen.




--
.CLI
--
.Schritte für eine einheitliche Konfiguration
. Eine neue SnapMirror Beziehung erstellen, die alle Volumes in der Anwendung gruppiert. Dabei ist die  `AutomatedFailOverDuplex` Richtlinie zur Einrichtung der bidirektionalen synchronen Replikation zu bestimmen.
+
`snapmirror create -source-path <source_path> -destination-path <destination_path> -cg-item-mappings <source_volume:@destination_volume> -policy AutomatedFailOverDuplex`

+
Beispiel: Das folgende Beispiel erstellt zwei Konsistenzgruppen: cg_src auf der Quelle mit vol1 und vol2 sowie eine gespiegelte Konsistenzgruppe auf dem Ziel, cg_dst.

+
[listing]
----
destination::> snapmirror create -source-path vs1_src:/cg/cg_src -destination-path vs1_dst:/cg/cg_dst -cg-item-mappings vol_src1:@vol_dst1,vol_src2:@vol_dst2 -policy AutomatedFailOverDuplex
----
. Die SnapMirror Beziehung initialisieren:
`snapmirror initialize -destination-path <destination-consistency-group>`
. Der erfolgreiche Abschluss des Vorgangs lässt sich bestätigen, indem abgewartet wird, bis das  `Mirrored State` als  `SnapMirrored` und das  `Relationship Status` als  `Insync` angezeigt wird.
+
`snapmirror show -destination-path <destination_path>`

. Auf Ihrem Host wird die Host-Konnektivität mit Zugriff auf jeden Cluster entsprechend Ihren Anforderungen konfiguriert.
. Die Initiatorgruppe-Konfiguration wird eingerichtet. Die bevorzugten Pfade für Initiatoren im lokalen Cluster werden festgelegt. Die Option, die Konfiguration für inverse Affinität auf den Peer-Cluster zu replizieren, wird angegeben.
+
`SiteA::> igroup create -vserver <svm_name> -ostype <os_type> -igroup <igroup_name> -replication-peer <peer_svm_name> -initiator <host>`

+

NOTE: Ab ONTAP 9.16.1 ist der  `-proximal-vserver local` Parameter in diesem Befehl zu verwenden.

+
`SiteA::> igroup add -vserver <svm_name> -igroup <igroup_name> -ostype <os_type> -initiator <host>`

+

NOTE: Ab ONTAP 9.16.1 ist der  `-proximal-vserver peer` Parameter in diesem Befehl zu verwenden.

. Vom Host aus werden die Pfade ermittelt und es wird überprüft, ob die Hosts über einen aktiven/optimierten Pfad zum Storage-LUN vom bevorzugten Cluster verfügen.
. Die Anwendung wird bereitgestellt und die VM-Workloads werden auf die Cluster verteilt, sodass der erforderliche Lastausgleich erreicht wird.


.Schritte für eine nicht einheitliche Konfiguration
. Eine neue SnapMirror Beziehung erstellen, die alle Volumes in der Anwendung gruppiert. Dabei ist die  `AutomatedFailOverDuplex` Richtlinie zur Einrichtung der bidirektionalen synchronen Replikation zu bestimmen.
+
`snapmirror create -source-path <source_path> -destination-path <destination_path> -cg-item-mappings <source_volume:@destination_volume> -policy AutomatedFailOverDuplex`

+
Beispiel: Das folgende Beispiel erstellt zwei Konsistenzgruppen: cg_src auf der Quelle mit vol1 und vol2 sowie eine gespiegelte Konsistenzgruppe auf dem Ziel, cg_dst.

+
[listing]
----
destination::> snapmirror create -source-path vs1_src:/cg/cg_src -destination-path vs1_dst:/cg/cg_dst -cg-item-mappings vol_src1:@vol_dst1,vol_src2:@vol_dst2 -policy AutomatedFailOverDuplex
----
. Die SnapMirror Beziehung initialisieren:
`snapmirror initialize -destination-path <destination-consistency-group>`
. Der erfolgreiche Abschluss des Vorgangs lässt sich bestätigen, indem abgewartet wird, bis das  `Mirrored State` als  `SnapMirrored` und das  `Relationship Status` als  `Insync` angezeigt wird.
+
`snapmirror show -destination-path <destination_path>`

. Auf Ihrem Host wird die Host-Konnektivität mit Zugriff auf jeden Cluster entsprechend Ihren Anforderungen konfiguriert.
. Die Initiatorgruppen-Konfigurationen sind sowohl auf dem Quell-Cluster als auch auf dem Ziel-Cluster zu erstellen.
+
`# primary site
SiteA::> igroup create -vserver <svm_name> -igroup <igroup_name> -initiator <host_1_name_>`

+
`# secondary site
SiteB::> igroup create -vserver <svm_name> -igroup <igroup_name> -initiator <host_2_name>`

. Vom Host aus werden die Pfade ermittelt und es wird überprüft, ob die Hosts über einen aktiven/optimierten Pfad zum Storage-LUN vom bevorzugten Cluster verfügen.
. Die Anwendung wird bereitgestellt und die VM-Workloads werden auf die Cluster verteilt, sodass der erforderliche Lastausgleich erreicht wird.

[[configure-symmetric-activeactive-protection-using-an-nvme-snapmirror-active-sync-configuration]]
== Symmetrischer Aktiv/Aktiv-Schutz mit einer NVMe SnapMirror active sync Konfiguration

.Bevor Sie beginnen
Zusätzlich zu den Anforderungen für die Konfiguration des symmetrischen Aktiv/Aktiv-Schutzes ist es wichtig, die unterstützten und nicht unterstützten Konfigurationen bei der Verwendung des NVMe-Protokolls zu kennen.

* Konsistenzgruppen können ein oder mehrere Subsysteme enthalten.
* Volumes innerhalb der Konsistenzgruppe können Namespace-Zuordnungen aus mehreren Subsystemen aufweisen.
* Subsysteme dürfen keine Namespace-Maps haben, die zu mehr als einer Konsistenzgruppe gehören.
* Subsysteme können nicht einige Namespace-Maps haben, die zu einer Konsistenzgruppe gehören, und einige Namespace-Maps, die nicht zu einer Konsistenzgruppe gehören.
* Subsysteme müssen Namespace-Maps haben, die Teil derselben Konsistenzgruppe sind.


.Schritte
Ab ONTAP 9.17.1 ist es möglich, mit System Manager oder der ONTAP CLI eine Konsistenzgruppe zu erstellen und einen symmetrischen Aktiv/Aktiv-Schutz mithilfe von NVMe-Protokoll-Hostzuordnungen zu konfigurieren.

[role="tabbed-block"]
====
.System Manager
--
. Auf der Hauptseite, link:../consistency-groups/configure-task.html#create-a-consistency-group-with-new-luns-or-volumes["Erstellen einer Konsistenzgruppe mithilfe neuer Volumes oder NVMe-Namespaces"].
. *+Hinzufügen* auswählen und *Neue NVMe-Namespaces verwenden* wählen.
. Den Namen der Konsistenzgruppe eingeben.
. *Mehr* auswählen.
. Im Abschnitt *Schutz* *Aktivieren SnapMirror* auswählen und dann die  `AutomatedFailoverDuplex` Richtlinie wählen.
. Im Abschnitt *Host-Zuordnung* entweder *Vorhandenes NVMe-Subsystem* oder *Neues NVMe-Subsystem* auswählen.
. *In der Nähe von* auswählen, um die proximale SVM zu ändern. Die Quell-SVM ist standardmäßig ausgewählt.
. Gegebenenfalls kann ein weiteres NVMe Subsystem hinzugefügt werden.


--
.CLI
--
. Eine neue SnapMirror Beziehung erstellen, die alle Volumes gruppiert, die sämtliche von der Anwendung verwendeten NVMe-Namespaces enthalten. Dabei ist die  `AutomatedFailOverDuplex` Richtlinie zur Einrichtung der bidirektionalen synchronen Replikation zu bestimmen.
+
`snapmirror create -source-path <source_path> -destination-path <destination_path> -cg-item-mappings <source_volume:@destination_volume> -policy AutomatedFailOverDuplex`

+
Beispiel:

+
[listing]
----
DST::> snapmirror create -source-path vs_src:/cg/cg_src_1 -destination-path vs_dst:/cg/cg_dst_1 -cg-item-mappings vs_src_vol1:@vs_dst_vol1,vs_src_vol2:@vs_dst_vol2 -policy AutomatedFailOverDuplex
----
. Die SnapMirror Beziehung initialisieren:
`snapmirror initialize -destination-path <destination-consistency-group>`
+
Beispiel:

+
[listing]
----
DST::> snapmirror initialize -destination-path vs1:/cg/cg_dst_1
----
. Der erfolgreiche Abschluss des Vorgangs lässt sich bestätigen, indem abgewartet wird, bis das  `Mirrored State` als  `SnapMirrored` und das  `Relationship Status` als  `Insync` angezeigt wird.
+
`snapmirror show -destination-path <destination_path>`

+
Die NVMe-Subsysteme, die den NVMe-Namespaces in den primären Volumes zugeordnet sind, werden automatisch auf den sekundären Cluster repliziert.

. Auf Ihrem Host wird die Host-Konnektivität mit Zugriff auf jeden Cluster entsprechend Ihren Anforderungen konfiguriert.
. Geben Sie die SVM an, die sich in der Nähe jedes Ihrer Hosts befindet. Dadurch erhalten die Hosts Zugriff auf den NVMe-Namensraum über einen Pfad vom bevorzugten Cluster. Dies kann die SVM im primären Cluster oder die SVM im DR Cluster sein.
+
Der folgende Befehl gibt an, dass SVM VS_A dem Host H1 nahe ist und VS_A als proximale SVM festgelegt ist:

+
`SiteA::> vserver nvme subsystem host add -subsystem ss1 -host-nqn <H1_NQN> -proximal-vservers <VS_A>`

+
Der folgende Befehl zeigt an, dass SVM VS_B dem Host H2 nahe ist und VS_B als proximale SVM festgelegt wird:

+
`SiteB::> vserver nvme subsystem host add -subsystem ss1 -host-nqn <H2_NQN> -proximal-vservers <VS_B>`

. Vom Host aus lassen sich die Pfade ermitteln und es kann überprüft werden, ob die Hosts über einen Aktiv/optimiert-Pfad zum Speicher vom bevorzugten Cluster verfügen.
. Die Anwendung wird bereitgestellt und die VM-Workloads werden auf die Cluster verteilt, sodass der erforderliche Lastausgleich erreicht wird.
