= Eine bestehende ONTAP SnapMirror SAN-Beziehung in eine SnapMirror Active Sync SAN-Beziehung umwandeln
:hardbreaks:
:allow-uri-read: 
:nofooter: 
:icons: font
:linkattrs: 
:imagesdir: ../media/


[role="lead"]
Wenn SnapMirror-Schutz konfiguriert wurde, kann eine SAN-Beziehung in SnapMirror active sync für SAN umgewandelt werden. Ab ONTAP 9.15.1 kann die Beziehung so umgestellt werden, dass symmetrischer Aktiv/Aktiv-Schutz verwendet wird.

Informationen zur Umwandlung einer SVM-DR-Beziehung in eine SnapMirror active sync for NAS-Beziehung sind unter link:protect-nas-task.html#convert-an-svm-dr-relationship-to-a-snapmirror-active-sync-relationship["Konvertieren einer SVM-DR-Beziehung in eine SnapMirror active sync-Beziehung"] zu finden.



[[convert-an-existing-iscsi-or-fc-snapmirror-relationship-to-an-asymmetric-snapmirror-active-sync-relationship]]
== Eine bestehende iSCSI oder FC SnapMirror Beziehung in eine asymmetrische SnapMirror Active Sync Beziehung umwandeln

Wenn bereits eine iSCSI- oder FC-SnapMirror-synchron Beziehung zwischen einem Quell- und einem Ziel-Cluster besteht, kann diese in eine asymmetrische SnapMirror Active Sync-Beziehung umgewandelt werden. Dadurch ist es möglich, die gespiegelten Volumes einer Konsistenzgruppe zuzuordnen, was einen RPO von null über einen Workload mit mehreren Volumes sicherstellt. Zudem können vorhandene SnapMirror-Snapshots beibehalten werden, falls auf einen Zeitpunkt vor dem Einrichten der SnapMirror Active Sync-Beziehung zurückgegriffen werden muss.

.Über diese Aufgabe
* Sie sind als Cluster- und SVM-Administrator auf dem primären und sekundären Cluster erforderlich.
* Sie können eine Zero-RPO- in eine Zero-RTO-Synchronisierung nicht durch Ändern der SnapMirror-Richtlinie umwandeln.
* Es ist sicherzustellen, dass die LUNs nicht zugeordnet sind, bevor der  `snapmirror create` Befehl ausgeführt wird.
+
Wenn vorhandene LUNs auf dem sekundären Volume zugeordnet sind und die `AutomatedFailover` Richtlinie konfiguriert ist, löst der `snapmirror create` Befehl einen Fehler aus.



.Bevor Sie beginnen
* Eine SnapMirror synchron Beziehung mit einem RPO von Null muss zwischen dem primären und dem sekundären Cluster bestehen.
* Alle LUNs auf dem Ziel-Volume müssen unzugeordnet werden, bevor die Zero RTO SnapMirror Beziehung erstellt werden kann.
* link:mediator-install-task.html["ONTAP Mediator"]muss für SnapMirror active sync konfiguriert sein.


.Schritte
. Im sekundären Cluster eine SnapMirror-Aktualisierung der bestehenden Beziehung durchführen:
+
`SiteB::>snapmirror update -destination-path vs1_dst:vol1`

. Überprüfen Sie, ob das SnapMirror Update erfolgreich abgeschlossen wurde:
+
`SiteB::>snapmirror show`

. Jede der synchronen Beziehungen mit RPO Null unterbrechen:
+
`SiteB::>snapmirror quiesce -destination-path vs1_dst:vol1`

+
`SiteB::>snapmirror quiesce -destination-path vs1_dst:vol2`

. Jede der synchronen Beziehungen mit RPO Null löschen:
+
`SiteB::>snapmirror delete -destination-path vs1_dst:vol1`

+
`SiteB::>snapmirror delete -destination-path vs1_dst:vol2`

. Die SnapMirror Beziehung der Quelle wird freigegeben, die gemeinsamen Snapshots bleiben jedoch erhalten:
+
`SiteA::>snapmirror release -relationship-info-only true -destination-path vs1_dst:vol1`

+
`SiteA::>snapmirror release -relationship-info-only true -destination-path vs1_dst:vol2`

. Erstellung einer SnapMirror synchronen Beziehung mit null RTO:
+
`SiteB::> snapmirror create -source-path vs1_src:/cg/cg_src -destination-path vs1_dst:/cg/cg_dst -cg-item-mappings vol1:@vol1,vol2:@vol2 -policy AutomatedFailover`

. Die Konsistenzgruppe erneut synchronisieren:
+
`SiteB::> snapmirror resync -destination-path vs1_dst:/cg/cg_dst`

. Die Host-LUN-I/O-Pfade werden erneut gescannt, um alle Pfade zu den LUNs wiederherzustellen.




[[convert-an-existing-iscsi-or-fc-snapmirror-relationship-to-symmetric-activeactive]]
== Eine bestehende iSCSI oder FC SnapMirror Beziehung in eine symmetrische Aktiv/Aktiv umwandeln

Ab ONTAP 9.15.1 ist es möglich, eine bestehende iSCSI oder FC SnapMirror Beziehung in eine SnapMirror active sync symmetrische Aktiv/Aktiv Beziehung zu konvertieren.

.Bevor Sie beginnen
* ONTAP 9.15.1 oder höher muss ausgeführt werden.
* Eine SnapMirror synchrone Beziehung mit null RPO muss zwischen dem primären und dem sekundären Cluster bestehen.
* Alle LUNs auf dem Ziel-Volume müssen unzugeordnet werden, bevor die Zero RTO SnapMirror Beziehung erstellt werden kann.
* SnapMirror active sync unterstützt ausschließlich SAN-Protokolle (nicht NFS/CIFS). Es ist sicherzustellen, dass kein Bestandteil der Konsistenzgruppe für den NAS-Zugriff eingebunden ist.
* link:mediator-install-task.html["ONTAP Mediator"]muss für SnapMirror active sync konfiguriert sein.


.Schritte
. Im sekundären Cluster eine SnapMirror-Aktualisierung der bestehenden Beziehung durchführen:
+
`SiteB::>snapmirror update -destination-path vs1_dst:vol1`

. Überprüfen Sie, ob das SnapMirror Update erfolgreich abgeschlossen wurde:
+
`SiteB::>snapmirror show`

. Jede der synchronen Beziehungen mit RPO Null unterbrechen:
+
`SiteB::>snapmirror quiesce -destination-path vs1_dst:vol1`

+
`SiteB::>snapmirror quiesce -destination-path vs1_dst:vol2`

. Jede der synchronen Beziehungen mit RPO Null löschen:
+
`SiteB::>snapmirror delete -destination-path vs1_dst:vol1`

+
`SiteB::>snapmirror delete -destination-path vs1_dst:vol2`

. Die SnapMirror Beziehung der Quelle wird freigegeben, die gemeinsamen Snapshots bleiben jedoch erhalten:
+
`SiteA::>snapmirror release -relationship-info-only true -destination-path vs1_dst:vol1`

+
`SiteA::>snapmirror release -relationship-info-only true -destination-path vs1_dst:vol2`

. Eine Zero-RTO-SnapMirror synchron Beziehung mit der AutomatedFailoverDuplex Richtlinie erstellen:
+
`SiteB::> snapmirror create -source-path vs1_src:/cg/cg_src -destination-path vs1_dst:/cg/cg_dst -cg-item-mappings vol1:@vol1,vol2:@vol2 -policy AutomatedFailoverDuplex`

. Wenn sich die vorhandenen Hosts lokal im primären Cluster befinden, sollte der Host dem sekundären Cluster hinzugefügt und die Konnektivität mit entsprechendem Zugriff auf jeden Cluster hergestellt werden.
. Am sekundären Standort die LUN-Zuordnungen auf den Initiatorgruppen löschen, die mit Remote-Hosts verknüpft sind.
+

NOTE: Es ist sicherzustellen, dass die Initiatorgruppe keine Zuordnungen für nicht replizierte LUNs enthält.

+
`SiteB::> lun mapping delete -vserver <svm_name> -igroup <igroup> -path <>`

. Am primären Standort wird die Initiatorkonfiguration für bestehende Hosts angepasst, sodass der proximale Pfad für Initiatoren im lokalen Cluster festgelegt ist.
+
`SiteA::> igroup initiator add-proximal-vserver -vserver <svm_name> -initiator <host> -proximal-vserver <server>`

. Fügen Sie eine neue Initiatorgruppe (igroup) und einen Initiator für die neuen Hosts hinzu und legen Sie die Host-Nähe für die Host-Affinität auf den lokalen Standort fest. Aktivieren Sie die igroup Replikation, um die Konfiguration zu replizieren und die Host-Lokalität auf dem Remote-Cluster umzukehren.
+
``
SiteA::> igroup modify -vserver vsA -igroup ig1 -replication-peer vsB
SiteA::> igroup initiator add-proximal-vserver -vserver vsA -initiator host2 -proximal-vserver vsB
``

. Ermitteln Sie die Pfade auf den Hosts und überprüfen Sie, ob die Hosts über einen Aktiv/Optimiert-Pfad zum Speicher-LUN vom bevorzugten Cluster verfügen.
. Die Anwendung wird bereitgestellt und die VM-Workloads werden auf die Cluster verteilt.
. Die Konsistenzgruppe erneut synchronisieren:
+
`SiteB::> snapmirror resync -destination-path vs1_dst:/cg/cg_dst`

. Die Host-LUN-I/O-Pfade werden erneut gescannt, um alle Pfade zu den LUNs wiederherzustellen.
