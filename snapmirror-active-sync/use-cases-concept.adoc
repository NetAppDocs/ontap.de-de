= Anwendungsfälle für ONTAP SnapMirror Active Sync
:hardbreaks:
:allow-uri-read: 
:nofooter: 
:icons: font
:linkattrs: 
:imagesdir: ../media/


[role="lead"]
Die Anforderungen eines global vernetzten Geschäftsumfelds erfordern eine schnelle Wiederherstellung geschäftskritischer Applikationsdaten ohne Datenverlust im Falle von Störungen wie Cyberangriffen, Stromausfällen oder Naturkatastrophen. Diese Anforderungen sind in Bereichen wie dem Finanzsektor und bei Unternehmen, die regulatorischen Vorgaben wie der General Data Protection Regulation (GDPR) unterliegen, besonders hoch.

SnapMirror active sync bietet folgende Anwendungsfälle:

.Anwendungsbereitstellung für ein Recovery Time Objective (RTO) von null
In einer SnapMirror active sync-Umgebung gibt es einen primären und einen sekundären Cluster. Eine LUN im primären Cluster  `L1P` hat ein Spiegelbild  `L1S` im sekundären Cluster; beide LUNs verwenden dieselbe serielle ID und werden dem Host als Lese-/Schreib-LUNs gemeldet. In asymmetrischen Konfigurationen werden Lese- und Schreibvorgänge jedoch nur auf der primären LUN ausgeführt  `L1P`. Schreibvorgänge auf das Spiegelbild  `L1S` werden über einen Proxy abgewickelt.

.Anwendungsbereitstellung für null RTO oder transparentes Anwendungs-Failover (TAF)
TAF basiert auf dem softwarebasierten Pfad-Failover der Host-MPIO-Software, um einen unterbrechungsfreien Zugriff auf den Speicher zu ermöglichen. Beide LUN Kopien, beispielsweise die primäre (L1P) und die Spiegelkopie (L1S), besitzen dieselbe Identität (Seriennummer) und werden dem Host als lese- und schreibbar gemeldet. In asymmetrischen Konfigurationen werden Lese- und Schreibvorgänge jedoch ausschließlich vom primären Volume ausgeführt. E/A-Vorgänge, die an die Spiegelkopie gerichtet sind, werden an die primäre Kopie weitergeleitet. Der bevorzugte Pfad des Hosts zu L1 ist VS1:N1, basierend auf dem ALUA-Zugriffsstatus Aktiv Optimiert (A/O). ONTAP Mediator ist für die Bereitstellung erforderlich, hauptsächlich um das Failover (geplant oder ungeplant) im Falle eines Speicherausfalls auf dem primären Volume durchzuführen.

image:snapmirror-active-sync-alua-active-optimized.png["SnapMirror active sync ALUA aktive optimierte Pfade"]

TAF arbeitet in zwei Modi: Automatisches Failover und Automatisches Failover Duplex. Bei Automatischem Failover werden Lese- und Schreibvorgänge ausschließlich vom primären Volume verarbeitet, daher werden IOs, die an die Spiegelkopie (die selbst keine Schreibvorgänge ausführen kann) gerichtet sind, an die primäre Kopie weitergeleitet. Bei Automatischem Failover Duplex können sowohl die primäre als auch die sekundäre Kopie IOs verarbeiten, sodass kein Proxy erforderlich ist.

image:snapmirror-active-sync-automatedfailoverduplex-scenario.png["SnapMirror active sync AutomatedFailoverDuplex Szenario"]

Wenn Sie NVMe für den Hostzugriff mit ONTAP 9.17.1 verwenden, wird nur die AutomatedFailoverDuplex Richtlinie unterstützt.

SnapMirror active sync verwendet ALUA, einen Mechanismus, der es einem Anwendungshost ermöglicht, Multipathing-Software mit Pfaden zu nutzen, die mit Prioritäten und Zugriffsverfügbarkeit für die Kommunikation des Anwendungshosts mit dem Storage-Array angekündigt werden. ALUA kennzeichnet aktive optimierte Pfade zu den Controllern, denen die LUN gehört, und andere als aktive nicht optimierte Pfade, die nur verwendet werden, wenn der primäre Pfad ausfällt.

SnapMirror active sync mit NVMe-Protokoll verwendet Asymmetric Namespace Access (ANA), wodurch Anwendungshosts optimierte und nicht optimierte Pfade zu geschützten NVMe-Namespaces ermitteln können. Das ONTAP NVMe Target veröffentlicht die entsprechenden Pfadzustände, sodass Anwendungshosts den optimalen Pfad für einen geschützten NVMe-Namespace nutzen können.

.Clusteranwendungen
Clusterbasierte Applikationen, einschließlich VMware Metro Storage Cluster, Oracle RAC und Windows Failover Clustering mit SQL, erfordern simultanen Zugriff, sodass die VMs ohne Leistungseinbußen auf einen anderen Standort überführt werden können. SnapMirror active sync Symmetric Aktiv/Aktiv bedient IO lokal mit bidirektionaler Replizierung, um die Anforderungen clusterbasierter Applikationen zu erfüllen. Ab ONTAP 9.16.1 wird Symmetric Aktiv/Aktiv in einer Konfiguration mit vier Node Clustern unterstützt, wodurch die Zwei-Node-Cluster-Beschränkung in ONTAP 9.15.1 erweitert wird.

.Katastrophenszenario
Mehrere Volumes einer Anwendung lassen sich synchron zwischen geografisch verteilten Standorten replizieren. Bei einer Störung des primären Clusters kann automatisch auf die sekundäre Kopie umgeschaltet werden, wodurch die Geschäftskontinuität für Tier-1-Applikationen gewährleistet wird. Wenn der Standort, an dem sich das primäre Cluster befindet, von einer Katastrophe betroffen ist, markiert die Multipathing-Software des Hosts alle Pfade durch das Cluster als ausgefallen und verwendet die Pfade des sekundären Clusters. Das Ergebnis ist ein unterbrechungsfreies Failover auf die Spiegelkopie, ermöglicht durch ONTAP Mediator.

.Erweiterte Anwendungsunterstützung
SnapMirror active sync bietet Flexibilität mit einfach zu verwendender, applikationsspezifischer Granularität und automatischem Failover. SnapMirror active sync verwendet die bewährte SnapMirror synchrone Replizierung über ein IP-Netzwerk, um Daten mit hoher Geschwindigkeit über LAN oder WAN zu replizieren, sodass eine hohe Datenverfügbarkeit und schnelle Datenreplizierung für geschäftskritische Applikationen wie Oracle, Microsoft SQL Server und andere sowohl in virtuellen als auch in physischen Umgebungen erreicht werden.

SnapMirror active sync ermöglicht, dass geschäftskritische Business-Services auch bei einem vollständigen Standortausfall mit TAF auf die sekundäre Kopie weiter betrieben werden können. Für dieses Failover sind weder manuelle Eingriffe noch zusätzliche Skripte erforderlich.
