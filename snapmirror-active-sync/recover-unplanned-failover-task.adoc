= Wiederherstellung nach automatischen ungeplanten ONTAP Cluster Failover-Operationen
:hardbreaks:
:allow-uri-read: 
:nofooter: 
:icons: font
:linkattrs: 
:imagesdir: ../media/


[role="lead"]
Ein automatischer ungeplanter Failover (AUFO) erfolgt, wenn der primäre Cluster ausfällt oder isoliert ist. Der ONTAP Mediator erkennt, wenn ein Failover auftritt, und führt einen automatischen ungeplanten Failover zum sekundären Cluster durch. Der sekundäre Cluster wird zum primären Cluster konvertiert und beginnt mit der Bedienung der Clients. Dieser Vorgang erfolgt nur mit Unterstützung des ONTAP Mediators.


NOTE: Nach dem automatischen ungeplanten Failover ist es wichtig, die Hostpfade erneut zu scannen, damit kein Verlust von E/A-Pfaden auftritt.



[[reestablish-the-protection-relationship-after-an-unplanned-failover]]
== Die Schutzbeziehung nach einem ungeplanten Failover wiederherstellen

Die Schutzbeziehung kann mithilfe von System Manager oder der ONTAP CLI wiederhergestellt werden.

[role="tabbed-block"]
====
.System Manager
--
.Schritte

IMPORTANT: Von ONTAP 9.8 bis ONTAP 9.14.1 wird SnapMirror active sync als SnapMirror Business Continuity (SM-BC) bezeichnet.

. Zu *Schutz > Beziehungen* navigieren und abwarten, bis der Beziehungsstatus "InSync" angezeigt wird.
. Um den Betrieb auf dem ursprünglichen Quell-Cluster fortzusetzen, image:icon_kabob.gif["Symbol für Menüoptionen"] und *Failover* auswählen.


--
.CLI
--
Der Status des automatischen ungeplanten Failovers kann mit dem  `snapmirror failover show` Befehl überwacht werden.

Beispiel:

....
ClusterB::> snapmirror failover show -instance
Start Time: 9/23/2020 22:03:29
         Source Path: vs1:/cg/scg3
    Destination Path: vs3:/cg/dcg3
     Failover Status: completed
        Error Reason:
            End Time: 9/23/2020 22:03:30
Primary Data Cluster: cluster-2
Last Progress Update: -
       Failover Type: unplanned
  Error Reason codes: -
....
In der link:https://docs.netapp.com/us-en/ontap-ems-9131/smbc-aufo-events.html["EMS-Referenz"^] finden sich Informationen zu Ereignismeldungen und Korrekturmaßnahmen.

--
====


[[resume-protection-in-a-fan-out-configuration-after-failover]]
== Wiederaufnahme des Schutzes in einer Fan-Out-Konfiguration nach einem Failover

Ab ONTAP 9.15.1 unterstützt SnapMirror active sync die automatische Neukonfiguration im Fan-Out-Bein nach einem Failover-Ereignis in einer SAN-Beziehung. In einer NAS-Konfiguration ist die Schutzbeziehung nach einem SnapMirror active sync Failover manuell neu zu konfigurieren. Das asynchrone Fan-Out-Bein kann eine Konsistenzgruppenbeziehung oder eine unabhängige Volume-Beziehung sein. Weitere Informationen finden sich unter link:interoperability-reference.html#fan-out-configurations["Fan-Out Konfigurationen"].

Wenn ONTAP 9.14.1 oder eine frühere Version verwendet wird und ein Failover auf dem sekundären Cluster in der SnapMirror Aktiv/Aktiv-Sync-Beziehung auftritt, wird das SnapMirror asynchrone Ziel-Cluster fehlerhaft. Der Schutz muss manuell wiederhergestellt werden, indem die Beziehung mit dem SnapMirror asynchronen Endpunkt gelöscht und neu erstellt wird.

.Schritte
. Es wird überprüft, ob das Failover erfolgreich abgeschlossen wurde:
`snapmirror failover show`
. Am SnapMirror asynchronen Endpunkt den Fan-Out-Endpunkt löschen:
`snapmirror delete -destination-path _destination_path_`
. Am dritten Standort wird eine SnapMirror asynchrone Beziehung zwischen dem neuen SnapMirror Active Sync-Primärvolume und dem asynchronen Fan-Out-Zielvolume erstellt:
`snapmirror create -source-path _source_path_ -destination-path _destination_path_ -policy MirrorAllSnapshots -schedule _schedule_`
. Die Beziehung neu synchronisieren:
`snapmirror resync -destination-path _destination_path_`
. Beziehungsstatus und Zustand der Beziehung prüfen:
`snapmirror show`
