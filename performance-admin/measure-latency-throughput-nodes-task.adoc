= ONTAP Durchsatz und Latenz zwischen den Knoten prüfen
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Sie können den  `network test-path`Befehl verwenden, um Netzwerkengpässe zu identifizieren oder Netzwerkpfade zwischen Knoten vorab zu qualifizieren. Der Befehl kann zwischen Intercluster-Knoten oder Intracluster-Knoten ausgeführt werden.

.Bevor Sie beginnen
* Sie müssen Cluster-Administrator sein, um diese Aufgabe ausführen zu können.
* Für diese Aufgabe sind Befehle auf dem erweiterten Berechtigungslevel erforderlich.
* Für einen Intercluster-Pfad müssen Quell- und Zielcluster per Peering verbunden sein.


.Über diese Aufgabe
Gelegentlich entspricht die Netzwerkleistung zwischen den Knoten möglicherweise nicht den Erwartungen an Ihre Pfadkonfiguration. Eine Übertragungsrate von 1 Gbit/s für die großen Datentransfers, wie sie bei SnapMirror Replikationsvorgängen auftreten, wäre nicht mit einer 10 GbE Verbindung zwischen Quell- und Ziel-Cluster vereinbar.

Sie können den  `network test-path`Befehl verwenden, um Durchsatz und Latenz zwischen Knoten zu messen. Der Befehl kann zwischen Intercluster-Knoten oder Intracluster-Knoten ausgeführt werden.

[NOTE]
====
Der Test sättigt den Netzwerkpfad mit Daten, daher sollte der Befehl nur ausgeführt werden, wenn das System nicht ausgelastet ist und der Netzwerkverkehr zwischen den Knoten nicht übermäßig hoch ist. Der Test wird nach zehn Sekunden abgebrochen. Der Befehl kann nur zwischen ONTAP 9 Knoten ausgeführt werden.

====
Die  `session-type` Option kennzeichnet die Art des Vorgangs, der über den Netzwerkpfad ausgeführt wird, zum Beispiel „AsyncMirrorRemote“ für SnapMirror Replikation an ein Remote-Ziel. Der Typ bestimmt die im Test verwendete Datenmenge. Die folgende Tabelle definiert die Sitzungstypen:

|===


| Sitzungstyp | Beschreibung 


 a| 
AsyncMirrorLocal
 a| 
Einstellungen, die von SnapMirror zwischen Knoten im selben Cluster verwendet werden



 a| 
AsyncMirrorRemote
 a| 
Einstellungen, die von SnapMirror zwischen Knoten in verschiedenen Clustern verwendet werden (Standardtyp)



 a| 
RemoteDataTransfer
 a| 
Einstellungen, die ONTAP für den Remote-Datenzugriff zwischen Knoten im selben Cluster verwendet (z. B. eine NFS-Anfrage an einen Knoten für eine Datei, die in einem Volume auf einem anderen Knoten gespeichert ist)

|===
.Schritte
. Wechsel zum erweiterten Berechtigungslevel:
+
`set -privilege advanced`

. Durchsatz und Latenz zwischen den Knoten:
+
`network test-path -source-node _source_nodename_ |local -destination-cluster _destination_clustername_ -destination-node _destination_nodename_ -session-type Default|AsyncMirrorLocal|AsyncMirrorRemote|SyncMirrorRemote|RemoteDataTransfer`

+
Der Quellknoten muss sich im lokalen Cluster befinden. Der Zielknoten kann sich im lokalen Cluster oder in einem verbundenen Cluster befinden. Ein Wert von „local“ für `-source-node` gibt den Knoten an, auf dem Sie den Befehl ausführen.

+
Der folgende Befehl misst Durchsatz und Latenz für Replikationsvorgänge vom Typ SnapMirror zwischen  `node1` auf dem lokalen Cluster und  `node3` auf  `cluster2`:

+
[listing]
----
cluster1::> network test-path -source-node node1 -destination-cluster cluster2 -destination-node node3 -session-type AsyncMirrorRemote
----
+
Beispielausgabe (die Ausgabedetails können je nach Ihrer ONTAP-Version variieren):

+
[listing]
----
Test Duration:      10.88 secs
Send Throughput:    18.23 MB/sec
Receive Throughput: 18.23 MB/sec
MB sent:            198.31
MB received:        198.31
Avg latency in ms:  2301.47
----
+
Weitere Informationen zu  `network test-path` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/network-test-path.html["ONTAP-Befehlsreferenz"^].

. Zurück zu als Administrator angemeldet:
+
`set -privilege admin`



.Nachdem Sie fertig sind
Falls die Leistung nicht den Erwartungen an die Pfadkonfiguration entspricht, empfiehlt es sich, die Leistungsstatistiken der Knoten zu überprüfen, verfügbare Tools zu nutzen, um das Problem im Netzwerk einzugrenzen, die Switch-Einstellungen zu prüfen und Ähnliches.
