= Warteschlangentiefen für ONTAP SAN Hosts anpassen
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Möglicherweise müssen Sie die Warteschlangenlängen auf Ihrem Host anpassen, um die maximalen Werte für ITNs pro Node und FC Port Fan-In zu erreichen. link:calculate-queue-depth-task.html["die optimale Warteschlangenlänge berechnen"] für Ihre Umgebung.



[[aix-hosts]]
== AIX-Hosts

Die Warteschlangentiefe auf AIX-Hosts kann mit dem  `chdev` Befehl geändert werden. Änderungen, die mit dem  `chdev` Befehl vorgenommen werden, bleiben nach einem Neustart erhalten.

Beispiele:

* Um die Warteschlangenlänge für das Gerät hdisk7 zu ändern, wird folgender Befehl verwendet:
+
`chdev -l hdisk7 -a queue_depth=32`

* Zum Ändern der Warteschlangenlänge für den fcs0 HBA wird folgender Befehl verwendet:
+
`chdev -l fcs0 -a num_cmd_elems=128`

+
Der Standardwert für `num_cmd_elems` ist 200. Der Maximalwert ist 2.048.

+
[NOTE]
====
Es kann erforderlich sein, den HBA offline zu nehmen, um  `num_cmd_elems` zu ändern und ihn dann mit den Befehlen  `rmdev -l fcs0 -R` und  `makdev -l fcs0 -P` wieder online zu bringen.

====




[[hp-ux-hosts]]
== HP-UX Hosts

Die LUN- oder Gerätewarteschlangentiefe auf HP-UX-Hosts kann mit dem Kernel-Parameter `scsi_max_qdepth` geändert werden. Die HBA-Warteschlangentiefe kann mit dem Kernel-Parameter `max_fcp_reqs` geändert werden.

* Der Standardwert für `scsi_max_qdepth` ist 8. Der Maximalwert ist 255.
+
`scsi_max_qdepth`Die Einstellung kann im laufenden System dynamisch mit der  `-u` Option des  `kmtune` Befehls geändert werden. Die Änderung gilt für alle Geräte im System. Zum Beispiel kann mit dem folgenden Befehl die LUN-Warteschlangentiefe auf 64 erhöht werden:

+
`kmtune -u -s scsi_max_qdepth=64`

+
Die Warteschlangenlänge für einzelne Gerätedateien kann mithilfe des  `scsictl` Befehls geändert werden. Änderungen mit dem  `scsictl` Befehl bleiben nach einem Systemneustart nicht erhalten. Die Warteschlangenlänge für eine bestimmte Gerätedatei kann mit folgendem Befehl angezeigt und geändert werden:

+
`scsictl -a /dev/rdsk/c2t2d0`

+
`scsictl -m queue_depth=16 /dev/rdsk/c2t2d0`

* Der Standardwert für `max_fcp_reqs` ist 512. Der Maximalwert ist 1024.
+
Der Kernel muss neu kompiliert und das System neu gestartet werden, damit die Änderungen  `max_fcp_reqs` wirksam werden. Um beispielsweise die HBA-Warteschlangentiefe auf 256 zu ändern, kann folgender Befehl verwendet werden:

+
`kmtune -u -s max_fcp_reqs=256`





[[solaris-hosts]]
== Solaris Hosts

Sie können die LUN- und HBA-Warteschlangentiefe für Ihre Solaris Hosts festlegen.

* Für die LUN-Warteschlangentiefe gilt: Die Anzahl der auf einem Host verwendeten LUNs multipliziert mit der Drosselung pro LUN (lun-queue-depth) muss kleiner oder gleich dem Wert tgt-queue-depth auf dem Host sein.
* Für die Warteschlangentiefe in einem Sun-Stack: Die nativen Treiber erlauben keine Einstellungen pro LUN oder pro Ziel `max_throttle` auf HBA-Ebene. Die empfohlene Methode zur Festlegung des `max_throttle` Werts für native Treiber ist auf Geräte-Typ-Ebene (VID_PID) in den `/kernel/drv/sd.conf` und `/kernel/drv/ssd.conf` Dateien. Das Host Utility setzt diesen Wert auf 64 für MPxIO Konfigurationen und auf 8 für Veritas DMP Konfigurationen.


.Schritte
. `# cd/kernel/drv`
. `# vi lpfc.conf`
. Suchen nach `/tft-queue (/tgt-queue)`
+
`tgt-queue-depth=32`

+
[NOTE]
====
Der Standardwert ist bei der Installation auf 32 festgelegt.

====
. Der gewünschte Wert wird entsprechend der Konfiguration Ihrer Umgebung festgelegt.
. Die Datei speichern.
. Den Host mit dem  `+sync; sync; sync; reboot -- -r+` Befehl neu starten.




[[vmware-hosts-for-a-qlogic-hba]]
== VMware Hosts für einen QLogic HBA

Mit dem  `esxcfg-module` Befehl lassen sich die HBA Zeitüberschreitungseinstellungen ändern. Eine manuelle Aktualisierung der  `esx.conf` Datei wird nicht empfohlen.

.Schritte
. Melden Sie sich an der Service-Konsole als Root-Benutzer an.
. Mit dem  `#vmkload_mod -l` Befehl lässt sich überprüfen, welches Qlogic HBA Modul aktuell geladen ist.
. Für eine einzelne Instanz einer Qlogic HBA wird folgender Befehl ausgeführt:
+
`#esxcfg-module -s ql2xmaxqdepth=64 qla2300_707`

+
[NOTE]
====
Dieses Beispiel verwendet das Modul qla2300_707. Das passende Modul ist entsprechend der Ausgabe von  `vmkload_mod -l` zu wählen.

====
. Die Änderungen werden mit folgendem Befehl gespeichert:
+
`#/usr/sbin/esxcfg-boot -b`

. Starten Sie den Server mit folgendem Befehl neu:
+
`#reboot`

. Die Änderungen werden mit den folgenden Befehlen bestätigt:
+
.. `#esxcfg-module -g qla2300_707`
.. `qla2300_707 enabled = 1 options = 'ql2xmaxqdepth=64'`






[[vmware-hosts-for-an-emulex-hba]]
== VMware Hosts für einen Emulex HBA

Mit dem  `esxcfg-module` Befehl lassen sich die HBA Zeitüberschreitungseinstellungen ändern. Eine manuelle Aktualisierung der  `esx.conf` Datei wird nicht empfohlen.

.Schritte
. Melden Sie sich an der Service-Konsole als Root-Benutzer an.
. Mit dem  `#vmkload_mod -l grep lpfc`Befehl lässt sich überprüfen, welcher Emulex HBA aktuell geladen ist.
. Für eine einzelne Instanz einer Emulex HBA ist folgender Befehl einzugeben:
+
`#esxcfg-module -s lpfc0_lun_queue_depth=16 lpfcdd_7xx`

+
[NOTE]
====
Je nach HBA-Modell kann das Modul entweder lpfcdd_7xx oder lpfcdd_732 sein. Der obige Befehl verwendet das Modul lpfcdd_7xx. Das passende Modul sollte entsprechend dem Ergebnis von  `vmkload_mod -l` verwendet werden.

====
+
Durch Ausführen dieses Befehls wird die LUN-Warteschlangentiefe für den durch lpfc0 repräsentierten HBA auf 16 gesetzt.

. Für mehrere Instanzen einer Emulex HBA ist folgender Befehl auszuführen:
+
`a esxcfg-module -s "lpfc0_lun_queue_depth=16 lpfc1_lun_queue_depth=16" lpfcdd_7xx`

+
Die LUN-Warteschlangenlänge für lpfc0 und die LUN-Warteschlangenlänge für lpfc1 ist auf 16 eingestellt.

. Führen Sie den folgenden Befehl aus:
+
`#esxcfg-boot -b`

. Neustart mit  `#reboot`.




[[windows-hosts-for-an-emulex-hba]]
== Windows Hosts für einen Emulex HBA

Auf Windows-Hosts kann das  `LPUTILNT` Dienstprogramm verwendet werden, um die Warteschlangenlänge für Emulex HBAs zu aktualisieren.

.Schritte
. Das Dienstprogramm  `LPUTILNT` im Verzeichnis  `C:\WINNT\system32` ausführen.
. Im Menü auf der rechten Seite *Laufwerksparameter* auswählen.
. Scrollen Sie nach unten und doppelklicken Sie auf *QueueDepth*.
+
[NOTE]
====
Wenn Sie *QueueDepth* größer als 150 einstellen, muss auch der folgende Wert in der Windows-Registrierung entsprechend erhöht werden:

`HKEY_LOCAL_MACHINE\System\CurrentControlSet\Services\lpxnds\Parameters\Device\NumberOfRequests`

====




[[windows-hosts-for-a-qlogic-hba]]
== Windows Hosts für einen Qlogic HBA

Auf Windows-Hosts können Sie das  `SANsurfer` HBA-Manager-Dienstprogramm verwenden, um die Warteschlangenlängen für Qlogic HBAs zu aktualisieren.

.Schritte
. Das  `SANsurfer`HBA Manager Dienstprogramm ausführen.
. Klicken Sie auf *HBA port* > *Einstellungen*.
. Im Listenfeld auf *Erweiterte HBA-Porteinstellungen* klicken.
. Der  `Execution Throttle`Parameter wird aktualisiert.




[[linux-hosts-for-emulex-hba]]
== Linux Hosts für Emulex HBA

Sie können die Warteschlangenlängen eines Emulex HBA auf einem Linux-Host aktualisieren. Damit die Aktualisierungen nach einem Neustart erhalten bleiben, ist das Erstellen eines neuen RAM-Disk-Images und ein anschließender Neustart des Hosts erforderlich.

.Schritte
. Die zu ändernden Parameter für die Warteschlangentiefe identifizieren:
+
`modinfo lpfc|grep queue_depth`

+
Es wird eine Liste der Warteschlangenlängenparameter mit ihrer Beschreibung angezeigt. Abhängig von Ihrer Betriebssystemversion können Sie einen oder mehrere der folgenden Warteschlangenlängenparameter ändern:

+
** `lpfc_lun_queue_depth`: Maximale Anzahl von FC-Befehlen, die einer bestimmten LUN (uint) zugeordnet werden können
** `lpfc_hba_queue_depth`: Maximale Anzahl von FC-Befehlen, die an einen lpfc HBA (uint) in die Warteschlange gestellt werden können
** `lpfc_tgt_queue_depth`: Maximale Anzahl von FC-Befehlen, die an einen bestimmten Zielport in die Warteschlange gestellt werden können (uint)
+
Der  `lpfc_tgt_queue_depth`Parameter ist nur für Red Hat Enterprise Linux 7.x Systeme, SUSE Linux Enterprise Server 11 SP4 Systeme und 12.x Systeme anwendbar.



. Die Warteschlangenlängen werden aktualisiert, indem die Warteschlangenlängenparameter zur  `/etc/modprobe.conf` Datei für ein Red Hat Enterprise Linux 5.x System und zur  `/etc/modprobe.d/scsi.conf` Datei für ein Red Hat Enterprise Linux 6.x oder 7.x System oder ein SUSE Linux Enterprise Server 11.x oder 12.x System hinzugefügt werden.
+
Je nach Version Ihres Betriebssystems lassen sich einer oder mehrere der folgenden Befehle hinzufügen:

+
** `options lpfc lpfc_hba_queue_depth=new_queue_depth`
** `options lpfc lpfc_lun_queue_depth=new_queue_depth`
** `options lpfc_tgt_queue_depth=new_queue_depth`


. Ein neues RAM-Disk-Image wird erstellt, anschließend wird der Host neu gestartet, damit die Aktualisierungen auch nach Neustarts erhalten bleiben.
+
Weitere Informationen enthält die link:../system-admin/index.html["Systemadministration"] für Ihre Version des Linux-Betriebssystems.

. Es sollte sichergestellt sein, dass die Werte für die Warteschlangenlänge für jeden der geänderten Warteschlangenlängen-Parameter aktualisiert wurden:
+


+
[listing]
----
root@localhost ~]#cat /sys/class/scsi_host/host5/lpfc_lun_queue_depth
      30
----
+
Der aktuelle Wert der Warteschlangenlänge wird angezeigt.





[[linux-hosts-for-qlogic-hba]]
== Linux Hosts für QLogic HBA

Sie können die Warteschlangentiefe eines QLogic-Treibers auf einem Linux-Host aktualisieren. Damit die Aktualisierungen nach einem Neustart erhalten bleiben, ist das Erstellen eines neuen RAM-Disk-Images und ein anschließender Neustart des Hosts erforderlich. Die Warteschlangentiefe des QLogic HBA kann über die QLogic HBA-Management-GUI oder die Befehlszeilenschnittstelle (CLI) angepasst werden.

Diese Aufgabe zeigt, wie die QLogic HBA CLI verwendet wird, um die QLogic HBA Warteschlangentiefe zu ändern

.Schritte
. Den zu ändernden Parameter für die Gerätewarteschlangenlänge identifizieren:
+
`modinfo qla2xxx | grep ql2xmaxqdepth`

+
Sie können nur den  `ql2xmaxqdepth` Parameter „Warteschlangentiefe“ ändern, der die maximale Warteschlangentiefe angibt, die für jede LUN festgelegt werden kann. Der Standardwert beträgt 64 für RHEL 7.5 und später. Der Standardwert beträgt 32 für RHEL 7.4 und früher.

+
[listing]
----

parm:       ql2xmaxqdepth:Maximum queue depth to set for each LUN. Default is 64. (int)
----
. Der Wert für die Gerätewarteschlangenlänge wird aktualisiert:
+
** Wenn die Änderungen dauerhaft gespeichert werden sollen, sind die folgenden Schritte erforderlich:
+
... Die Warteschlangentiefen werden aktualisiert, indem der Parameter „queue depth“ zur  `/etc/modprobe.conf` Datei für ein Red Hat Enterprise Linux 5.x System und zur  `/etc/modprobe.d/scsi.conf` Datei für ein Red Hat Enterprise Linux 6.x oder 7.x System oder ein SUSE Linux Enterprise Server 11.x oder 12.x System hinzugefügt wird: `options qla2xxx ql2xmaxqdepth=new_queue_depth`
... Ein neues RAM-Disk-Image wird erstellt, anschließend wird der Host neu gestartet, damit die Aktualisierungen auch nach Neustarts erhalten bleiben.

+
`echo new_queue_depth > /sys/module/qla2xxx/parameters/ql2xmaxqdepth`
