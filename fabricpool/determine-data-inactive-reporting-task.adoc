= Inaktive ONTAP Daten mit der Berichterstattung über inaktive Daten analysieren
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Die Meldung inaktiver Daten (IDR) ist ein hervorragendes Werkzeug, um die Menge inaktiver (kalter) Daten zu ermitteln, die aus einer lokalen Ebene ausgelagert werden können.

Standardmäßig verwendet IDR eine 31-tägige Abkühlphase, um zu bestimmen, welche Daten als inaktiv gelten. Die Menge der kalten Daten, die ausgelagert werden, hängt von den auf den Volumes festgelegten Tiering-Richtlinien ab. In Versionen vor ONTAP 9.8 verwendete IDR eine feste 31-tägige Abkühlphase.

.Bevor Sie beginnen
**ONTAP 9.4**

* IDR ist standardmäßig auf allen FabricPool Aggregaten aktiviert.
* IDR kann auf Nicht-FabricPool SSD-Aggregaten aktiviert werden.


**ONTAP 9.6**

* IDR ist standardmäßig auf allen Nicht-FabricPool SSD-Aggregaten aktiviert.
* IDR kann auf Nicht-FabricPool HDD-Aggregaten aktiviert werden.


**ONTAP 9.8**

* Die IDR-Kühlperiode kann über die Einstellung volume-tiering-minimum-cooling-days angepasst werden.


.Über diese Aufgabe
* IDR meldet inaktive Daten, die ausgelagert werden können, nicht jedoch inaktive Daten, die bereits ausgelagert wurden. Wenn FabricPool aktiviert ist und inaktive Daten ausgelagert wurden, werden sie nicht mehr von IDR gemeldet.


[NOTE]
====
Obwohl IDR auf allen SSD-Aggregaten (ONTAP 9.6 und höher) standardmäßig aktiviert ist, wird IDR automatisch deaktiviert und die Kühltage auf null zurückgesetzt, sobald eine Client-Workload 90 % der Systemressourcen benötigt. In diesem Fall wird IDR nicht automatisch wieder aktiviert.

Um zu vermeiden, dass der automatische Prozess IDR abschaltet, um Ressourcen für andere Workloads freizugeben, aktivieren Sie -is-inactive-data-reporting-enabled manuell auf true.

====
* Inaktive Datenberichterstattung, wenn FabricPool nicht aktiviert werden kann, einschließlich der folgenden Fälle:
+
** Wurzelaggregate
** Aggregate, die Volumes mit einer anderen Speicherplatzzusage als „keine“ enthalten. (Thick-Provisioned Volumes werden nicht unterstützt.)
** MetroCluster Aggregate mit ONTAP Versionen vor 9.7
** Flash Pool (Hybridaggregate oder SnapLock Aggregate)






[[procedure]]
== Verfahren

Mit ONTAP System Manager oder der ONTAP CLI lässt sich ermitteln, wie viele Daten inaktiv sind.

[role="tabbed-block"]
====
.System Manager
--
. Wählen Sie eine der folgenden Optionen:
+
** Wenn bereits HDD-Aggregate vorhanden sind, zu *Speicher > Ebenen* navigieren und image:icon_kabob.gif["Symbol für Menüoptionen"] für das Aggregat auswählen, für das die Berichterstattung über inaktive Daten aktiviert werden soll.
** Wenn keine Cloud-Ebenen konfiguriert sind, auf *Dashboard* wechseln und unter *Kapazität* auf den Link *Aktivieren der Berichterstattung über inaktive Daten* klicken.




--
.CLI
--
.Aktivierung der Meldung inaktiver Daten über die CLI:
. Wenn das Aggregat, für das Sie die Berichterstattung über inaktive Daten anzeigen möchten, nicht in FabricPool verwendet wird, kann die Berichterstattung über inaktive Daten für das Aggregat mit dem  `storage aggregate modify` Befehl und dem  `-is-inactive-data-reporting-enabled` `true` Parameter aktiviert werden.
+
[listing]
----
cluster1::> storage aggregate modify -aggregate aggr1 -is-inactive-data-reporting-enabled true
----
+
Die Funktion zur Meldung inaktiver Daten muss explizit für ein Aggregat aktiviert werden, das nicht für FabricPool verwendet wird.

+
Sie können die Meldung inaktiver Daten für ein FabricPool-fähiges Aggregat nicht aktivieren und müssen dies auch nicht, da das Aggregat bereits die Meldung inaktiver Daten enthält. Der  `-is-inactive-data-reporting-enabled` Parameter funktioniert bei FabricPool-fähigen Aggregaten nicht.

+
Der  `-fields is-inactive-data-reporting-enabled`Parameter des  `storage aggregate show`Befehls zeigt, ob die Meldung inaktiver Daten für ein Aggregat aktiviert ist.

. Um anzuzeigen, wie viele Daten auf einem Volume inaktiv sind, kann der  `volume show` Befehl mit dem  `-fields performance-tier-inactive-user-data,performance-tier-inactive-user-data-percent` Parameter verwendet werden.
+
[listing]
----
cluster1::> volume show -fields performance-tier-inactive-user-data,performance-tier-inactive-user-data-percent

vserver volume performance-tier-inactive-user-data performance-tier-inactive-user-data-percent
------- ------ ----------------------------------- -------------------------------------------
vsim1   vol0   0B                                  0%
vs1     vs1rv1 0B                                  0%
vs1     vv1    10.34MB                             0%
vs1     vv2    10.38MB                             0%
4 entries were displayed.
----
+
** Das  `performance-tier-inactive-user-data` Feld zeigt an, wie viele der im Aggregat gespeicherten Benutzerdaten inaktiv sind.
** Das  `performance-tier-inactive-user-data-percent` Feld zeigt an, welcher Prozentsatz der Daten im gesamten aktiven Dateisystem und in den Snapshots inaktiv ist.
** Für ein Aggregat, das nicht für FabricPool verwendet wird, nutzt die Meldung inaktiver Daten die Tiering-Richtlinie, um zu entscheiden, wie viele Daten als kalt gemeldet werden.
+
*** Für die  `none`Tiering-Policy werden 31 Tage verwendet.
*** Für die  `snapshot-only` und  `auto` verwendet die Meldung inaktiver Daten  `tiering-minimum-cooling-days`.
*** Für die  `ALL`Richtlinie wird bei der Meldung inaktiver Daten angenommen, dass die Daten innerhalb eines Tages in eine andere Ebene verschoben werden.
+
Bis zum Erreichen der Periode zeigt die Ausgabe "`-`" für die Menge der inaktiven Daten anstelle eines Wertes.



** Bei einem Volume, das Teil von FabricPool ist, hängt das, was ONTAP als inaktiv meldet, von der auf dem Volume festgelegten Tiering-Richtlinie ab.
+
*** Für die  `none`Tiering-Policy meldet ONTAP die Menge des gesamten Volumens, die seit mindestens 31 Tagen inaktiv ist. Der  `-tiering-minimum-cooling-days`Parameter kann nicht mit der  `none`Tiering-Policy verwendet werden.
*** Für die  `ALL`,  `snapshot-only` und  `auto` Tiering-Richtlinien wird die Meldung inaktiver Daten nicht unterstützt.
