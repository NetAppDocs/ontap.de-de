= Ein Volume auf einer FabricPool ONTAP lokalen Ebene erstellen
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Volumes können zu FabricPool hinzugefügt werden, indem neue Volumes direkt in der FabricPool-aktivierten lokalen Ebene erstellt oder vorhandene Volumes von einer anderen lokalen Ebene in die FabricPool-aktivierte lokale Ebene verschoben werden.


NOTE: Vor ONTAP 9.7 verwendet System Manager den Begriff _aggregate_, um eine _lokale Ebene_ zu beschreiben. Unabhängig von der ONTAP Version verwendet die ONTAP CLI den Begriff _aggregate_. Weitere Informationen zu lokalen Ebenen finden sich unter link:../disks-aggregates/index.html["Festplatten und lokale Ebenen"].

Wenn ein Volume für FabricPool erstellt wird, besteht die Möglichkeit, eine Tiering-Richtlinie anzugeben. Wird keine Tiering-Richtlinie angegeben, verwendet das erstellte Volume die Standard `snapshot-only` Tiering-Richtlinie. Für ein Volume mit der `snapshot-only` oder `auto` Tiering-Richtlinie kann auch die minimale Abkühlperiode für das Tiering angegeben werden.

.Bevor Sie beginnen
* Das Festlegen eines Volumes zur Verwendung der  `auto` Tiering-Policy oder das Angeben der minimalen Kühlperiode für das Tiering erfordert ONTAP 9.4 oder höher.
* Die Verwendung von FlexGroup Volumes erfordert ONTAP 9.5 oder höher.
* Das Festlegen eines Volumes zur Verwendung der  `all` Tiering-Policy erfordert ONTAP 9.6 oder höher.
* Das Festlegen eines Volumes zur Verwendung des  `-cloud-retrieval-policy` Parameters erfordert ONTAP 9.8 oder höher.


.Schritte
. Ein neues Volume für FabricPool kann mit dem  `volume create` Befehl erstellt werden.
+
** Der  `-tiering-policy`optionale Parameter ermöglicht die Angabe der Tiering-Richtlinie für das Volume.
+
Eine der folgenden Tiering-Richtlinien kann angegeben werden:

+
*** `snapshot-only` (Standard)
*** `auto`
*** `all`
*** `backup` (veraltet)
*** `none`
+
link:tiering-policies-concept.html["FabricPool Volume-Tiering-Richtlinien"]



** Der  `-cloud-retrieval-policy`optionale Parameter ermöglicht Cluster-Administratoren mit erweitertem Berechtigungsniveau, das durch die Tiering-Richtlinie gesteuerte Standardverhalten bei der Cloud-Migration oder beim Abrufen zu überschreiben.
+
Sie können eine der folgenden Cloud-Abrufrichtlinien festlegen:

+
*** `default`
+
Die Tiering-Richtlinie legt fest, welche Daten zurückgeholt werden, daher gibt es keine Änderung beim Abruf von Cloud-Daten mit  `default` cloud-retrieval-policy. Das Verhalten ist somit identisch mit dem in Versionen vor ONTAP 9.8:

+
**** Wenn die Tiering-Richtlinie  `none` oder  `snapshot-only` ist, bedeutet „`default`“, dass alle von Clients gelesenen Daten vom Cloud-Tier in den Performance-Tier übertragen werden.
**** Wenn die Tiering-Richtlinie  `auto` ist, werden alle clientseitig gesteuerten zufälligen Lesevorgänge abgerufen, jedoch keine sequenziellen Lesevorgänge.
**** Wenn die Tiering-Richtlinie  `all`ist, werden keine clientseitig gesteuerten Daten aus dem Cloud-Tier abgerufen.


*** `on-read`
+
Alle clientseitigen Datenlesevorgänge werden von der Cloud-Tier in die Performance-Tier übertragen.

*** `never`
+
Es werden keine clientgesteuerten Daten von der Cloud-Ebene in die Performance-Ebene übertragen.

*** `promote`
+
**** Bei der Tiering-Richtlinie `none`werden alle Cloud-Daten vom Cloud-Tier in den Performance-Tier verschoben.
**** Für die Tiering-Richtlinie `snapshot-only` werden alle aktiven Dateisystemdaten vom Cloud-Tier auf den Performance-Tier verschoben.




** Der  `-tiering-minimum-cooling-days` optionale Parameter auf der erweiterten Berechtigungsstufe ermöglicht die Angabe der minimalen Abkühlperiode für ein Volume, das die  `snapshot-only` oder  `auto` Tiering-Richtlinie verwendet.
+
Ab ONTAP 9.8 kann ein Wert zwischen 2 und 183 für die Mindestanzahl an Kühltagen pro Stufe angegeben werden. Wenn eine Version von ONTAP vor 9.8 verwendet wird, kann ein Wert zwischen 2 und 63 für die Mindestanzahl an Kühltagen für das Tiering angegeben werden.





.Beispiel für die Erstellung eines Volumens für FabricPool
Das folgende Beispiel erstellt ein Volume mit dem Namen "`myvol1`" im "`myFabricPool`" FabricPool-fähigen lokalen Tier. Die Tiering-Richtlinie ist auf `auto` gesetzt und die minimale Tiering-Abkühlperiode beträgt 45 Tage:

[listing]
----
cluster1::*> volume create -vserver myVS -aggregate myFabricPool
-volume myvol1 -tiering-policy auto -tiering-minimum-cooling-days 45
----
