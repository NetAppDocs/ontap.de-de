= Überblick über die Storage Virtualisierung
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Sie verwenden _Storage Virtual Machines (SVMs)_, um Daten für Clients und Hosts bereitzustellen. Wie eine virtuelle Maschine, die auf einem Hypervisor ausgeführt wird, ist eine SVM eine logische Entität, die physische Ressourcen abstrahiert. Der Zugriff auf Daten über die SVM ist nicht an einen Speicherort gebunden. Der Netzwerkzugriff auf die SVM ist nicht an einen physischen Port gebunden.


NOTE: SVMs wurden früher als „vservers“ bezeichnet. Die ONTAP Befehlszeile verwendet weiterhin den Begriff „vserver“.

Eine SVM stellt Daten für Clients und Hosts von einem oder mehreren Volumes über eine oder mehrere logische Netzwerkschnittstellen (LIFs) bereit. Volumes können jedem Datenaggregat im Cluster zugewiesen werden. LIFs können von jedem physischen oder logischen Port gehostet werden. Sowohl Volumes als auch LIFs können verschoben werden, ohne den Datenbetrieb zu unterbrechen, sei es bei Hardware-Upgrades, dem Hinzufügen von Knoten, der Leistungsoptimierung oder der Optimierung der Kapazität über Aggregate hinweg.

Dasselbe SVM kann einen LIF für NAS-Datenverkehr und einen LIF für SAN-Datenverkehr haben. Clients und Hosts benötigen nur die Adresse des LIF (IP-Adresse für NFS, SMB oder iSCSI; WWPN für FC), um auf das SVM zuzugreifen. LIFs behalten ihre Adressen, wenn sie verschoben werden. Ports können mehrere LIFs hosten. Jedes SVM verfügt über eigene Sicherheit, Administration und Namensraum.

Zusätzlich zu den Daten-SVMs setzt ONTAP spezielle SVMs für die Administration ein:

* Bei der Einrichtung des Clusters wird eine _admin SVM_ erstellt.
* Eine _Node SVM_ wird erstellt, wenn ein Knoten einem neuen oder bestehenden Cluster beitritt.
* Eine _System-SVM_ wird für die Kommunikation auf Clusterebene in einem IPspace automatisch erstellt.


Diese SVMs können nicht zur Datenbereitstellung verwendet werden. Es gibt außerdem spezielle LIFs für den Datenverkehr innerhalb und zwischen Clustern sowie für die Cluster- und Knotenverwaltung.



[[why-ontap-is-like-middleware]]
== Warum ONTAP wie Middleware ist

Die logischen Objekte, die ONTAP für Speicherverwaltungsaufgaben verwendet, erfüllen die bekannten Ziele eines gut konzipierten Middleware-Pakets: Sie schützen den Administrator vor Implementierungsdetails auf niedriger Ebene und isolieren die Konfiguration von Änderungen an physischen Eigenschaften wie Knoten und Ports. Der Grundgedanke ist, dass der Administrator Volumes und LIFs einfach verschieben kann, indem nur wenige Felder anstatt der gesamten Speicherinfrastruktur neu konfiguriert werden.
