= Snapshots
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Traditionell dienten ONTAP Replikationstechnologien der Notfallwiederherstellung (DR) und der Datenarchivierung. Mit dem Aufkommen von Cloud-Services wurde die ONTAP Replikation an den Datentransfer zwischen Endpunkten im NetApp data fabric angepasst. Die Grundlage für all diese Verwendungen ist die ONTAP Snapshot-Technologie.

Ein Snapshot (früher Snapshot-Kopie) ist ein schreibgeschütztes, zeitpunktgenaues Image eines Volumes. Nach der Erstellung eines Snapshots verweisen das aktive Dateisystem und der Snapshot auf dieselben Festplattenblöcke, daher benötigt der Snapshot keinen zusätzlichen Speicherplatz. Im Laufe der Zeit verbraucht das Image nur minimalen Speicherplatz und verursacht einen vernachlässigbaren Performance-Overhead, da nur Änderungen an Dateien seit dem letzten Snapshot aufgezeichnet werden.

Snapshots verdanken ihre Effizienz der Kerntechnologie von ONTAP zur Virtualisierung, dem Write Anywhere File Layout (WAFL). Wie eine Datenbank verwendet WAFL Metadaten, um auf die tatsächlichen Datenblöcke auf der Festplatte zu verweisen. Im Gegensatz zu einer Datenbank überschreibt WAFL jedoch keine vorhandenen Blöcke. WAFL schreibt aktualisierte Daten in einen neuen Block und passt die Metadaten an.

Snapshots sind effizient, da ONTAP beim Erstellen eines Snapshots anstelle des Kopierens von Datenblöcken auf Metadaten verweist. Dadurch entfallen sowohl die „Suchzeit“, die bei anderen Systemen beim Auffinden der zu kopierenden Blöcke anfällt, als auch die Kosten für das Erstellen der Kopie selbst.

Mithilfe eines Snapshots können einzelne Dateien oder LUNs wiederhergestellt oder der gesamte Inhalt eines Volumes zurückgespielt werden. ONTAP vergleicht Zeigerinformationen im Snapshot mit den Daten auf der Festplatte, um das fehlende oder beschädigte Objekt ohne Ausfallzeiten oder nennenswerte Leistungseinbußen zu rekonstruieren.

Eine _Snapshot-Richtlinie_ definiert, wie das System Snapshots von Volumes erstellt. Die Richtlinie legt fest, wann die Snapshots erstellt werden, wie viele Kopien aufzubewahren sind, wie sie benannt und wie sie für die Replikation gekennzeichnet werden. Beispielsweise kann ein System täglich um 00:10 Uhr einen Snapshot erstellen, die zwei neuesten Kopien aufbewahren, sie "`daily`" nennen (ergänzt um einen Zeitstempel) und sie für die Replikation mit "`daily`" kennzeichnen.
