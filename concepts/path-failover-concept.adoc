= Erfahren Sie mehr über das ONTAP Path-Failover-Management
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Es gibt wichtige Unterschiede in der Art und Weise, wie ONTAP das Pfad-Failover in NAS- und SAN-Topologien handhabt. Ein NAS-LIF migriert nach einem Verbindungsausfall automatisch zu einem anderen Netzwerkport. Ein SAN-LIF migriert nicht (es sei denn, Sie verschieben ihn nach dem Ausfall manuell). Stattdessen leitet die Multipathing-Technologie auf dem Host den Datenverkehr zu einem anderen LIF um, auf demselben SVM, aber mit Zugriff auf einen anderen Netzwerkport.



[[nas-path-failover]]
== NAS Pfad-Failover

Eine NAS-LIF migriert nach einem Verbindungsausfall auf ihrem aktuellen Port automatisch zu einem verbleibenden Netzwerkport. Der Port, zu dem die LIF migriert, muss Mitglied der _Failover-Gruppe_ für die LIF sein. Die _Failover-Gruppenrichtlinie_ grenzt die Failover-Ziele für eine Daten-LIF auf Ports des Node ein, der Eigentümer der Daten ist, sowie auf dessen HA-Partner.

Ab ONTAP 9.19.1 erweitert die systemdefinierte Failover-Gruppenrichtlinie von ONTAP die Failover-Ziele für NAS-Daten-LIFs dynamisch, sodass potenziell alle verfügbaren Ports in der Broadcast-Domäne auf jedem verfügbaren Knoten im Cluster einbezogen werden können. Sie können die systemdefinierte Richtlinie auf jede Failover-Gruppe anwenden. Vor ONTAP 9.19.1 konnten LIFs mit dieser Richtlinie nur auf einen anderen Knoten im Cluster ausweichen; wenn dieser Knoten nicht im Quorum war, fand kein Failover statt, selbst wenn andere Knoten verfügbar waren.

Zur administrativen Vereinfachung erstellt ONTAP eine Failover-Gruppe für jede Broadcast-Domäne in der Netzwerkarchitektur. Broadcast-Domänen gruppieren Ports, die zum selben Layer-2-Netzwerk gehören. Wenn beispielsweise VLANs verwendet werden, um den Datenverkehr nach Abteilungen (Engineering, Marketing, Finance usw.) zu trennen, definiert jedes VLAN eine separate Broadcast-Domäne. Die mit der Broadcast-Domäne verknüpfte Failover-Gruppe wird automatisch aktualisiert, sobald ein Port der Broadcast-Domäne hinzugefügt oder entfernt wird.

Es ist fast immer eine gute Idee, eine Broadcast-Domäne zur Definition einer Failover-Gruppe zu verwenden, um sicherzustellen, dass die Failover-Gruppe aktuell bleibt. Gelegentlich kann es jedoch sinnvoll sein, eine Failover-Gruppe zu definieren, die keiner Broadcast-Domäne zugeordnet ist. Beispielsweise kann es gewünscht sein, dass LIFs nur auf Ports in einer Teilmenge der in der Broadcast-Domäne definierten Ports übergehen.

 a| 
*_Subnetze_*

Ein Subnetz reserviert einen Block von IP-Adressen in einer Broadcast-Domäne. Diese Adressen gehören zum selben Layer-3-Netzwerk und werden Ports in der Broadcast-Domäne zugewiesen, wenn eine LIF erstellt wird. Es ist in der Regel einfacher und weniger fehleranfällig, beim Definieren einer LIF-Adresse einen Subnetznamen anzugeben als eine IP-Adresse und eine Netzwerkmaske.

[[san-path-failover]]
== SAN Pfad-Failover

Ein SAN-Host nutzt ALUA (Asymmetric Logical Unit Access) und MPIO (Multipath I/O), um den Datenverkehr nach einem Verbindungsausfall auf eine verbleibende LIF umzuleiten. Vordefinierte Pfade bestimmen die möglichen Routen zur LUN, die von der SVM bereitgestellt wird.

In einer SAN-Umgebung werden Hosts als _Initiatoren_ von Anforderungen an LUN-_Ziele_ betrachtet. MPIO ermöglicht mehrere Pfade von Initiatoren zu Zielen. ALUA identifiziert die direktesten Pfade, die als _optimierte Pfade_ bezeichnet werden.

In der Regel werden mehrere optimierte Pfade zu LIFs auf dem Knoten, dem die LUN gehört, und mehrere nicht optimierte Pfade zu LIFs auf dessen HA-Partner konfiguriert. Fällt ein Port auf dem Eigentümerknoten aus, wird der Datenverkehr vom Host zu den verbleibenden Ports geleitet. Fallen alle Ports aus, wird der Datenverkehr vom Host über die nicht optimierten Pfade geleitet.

ONTAP Selective LUN Map (SLM) beschränkt standardmäßig die Anzahl der Pfade vom Host zu einer LUN. Eine neu erstellte LUN ist nur über Pfade zu dem Knoten erreichbar, dem die LUN gehört, oder zu dessen HA Partner. Der Zugriff auf eine LUN kann auch eingeschränkt werden, indem LIFs in einem Port Set für den Initiator konfiguriert werden.

|===


 a| 
*Verschieben von Volumes in SAN-Umgebungen*

Standardmäßig begrenzt ONTAP _Selective LUN Map (SLM)_ die Anzahl der Pfade zu einer LUN von einem SAN-Host. Auf eine neu erstellte LUN kann nur über Pfade zu dem Node, der die LUN besitzt, oder dessen HA-Partner zugegriffen werden, den _Reporting Nodes_ für die LUN.

Das bedeutet, dass beim Verschieben eines Volumes auf einen Knoten eines anderen HA-Paares die Reporting-Knoten des Ziel-HA-Paares zur LUN-Zuordnung hinzugefügt werden müssen. Danach lassen sich die neuen Pfade in der MPIO-Konfiguration angeben. Nach Abschluss der Volume-Verschiebung können die Reporting-Knoten des Quell-HA-Paares aus der Zuordnung entfernt werden.
