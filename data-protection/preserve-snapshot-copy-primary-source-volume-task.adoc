= Sicherstellen eines gemeinsamen Snapshots in einer ONTAP Mirror Vault-Bereitstellung
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Sie können den  `snapmirror snapshot-owner create` Befehl verwenden, um einen gekennzeichneten Snapshot auf dem sekundären System in einer Mirror-Vault-Bereitstellung zu erhalten. Dadurch wird sichergestellt, dass ein gemeinsamer Snapshot für die Aktualisierung der Vault-Beziehung vorhanden ist.

.Über diese Aufgabe
Bei Verwendung einer kombinierten Mirror-Vault-Fan-Out- oder Kaskadenbereitstellung ist zu beachten, dass Aktualisierungen fehlschlagen, wenn auf den Quell- und Ziel-Volumes kein gemeinsamer Snapshot vorhanden ist.

Dies stellt bei der Spiegelbeziehung in einer Mirror-Vault-Fan-Out- oder Kaskadenbereitstellung niemals ein Problem dar, da SnapMirror immer einen Snapshot des Quellvolumes erstellt, bevor die Aktualisierung durchgeführt wird.

Dies könnte jedoch ein Problem für die Vault-Beziehung darstellen, da SnapMirror beim Aktualisieren einer Vault-Beziehung keinen Snapshot des Quell-Volumes erstellt. Es ist erforderlich, das  `snapmirror snapshot-owner create` zu verwenden, um sicherzustellen, dass mindestens ein gemeinsamer Snapshot sowohl auf dem Quell- als auch auf dem Ziel-Volume der Vault-Beziehung vorhanden ist. link:supported-deployment-config-concept.html["Weitere Informationen zu Fan-out- und Kaskadierungsbereitstellungen für den Datenschutz"].

.Schritte
. Weisen Sie auf dem Quell-Volume dem gekennzeichneten Snapshot, den Sie beibehalten möchten, einen Besitzer zu:
+
`snapmirror snapshot-owner create -vserver <SVM> -volume <volume> -snapshot <snapshot> -owner <owner>`

+
Im folgenden Beispiel wird  `ApplicationA` als Eigentümer des  `snap1` Snapshots zugewiesen:

+
[listing]
----
clust1::> snapmirror snapshot-owner create -vserver vs1 -volume vol1
-snapshot snap1 -owner ApplicationA
----
+
Weitere Informationen zu  `snapmirror snapshot-owner create` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-snapshot-owner-create.html["ONTAP-Befehlsreferenz"^].

. Die Spiegelbeziehung wird aktualisiert, wie in link:update-replication-relationship-manual-task.html["Manuelles Aktualisieren einer Replikationsbeziehung"] beschrieben.
+
Alternativ ist es möglich, auf die planmäßige Aktualisierung der Spiegelbeziehung zu warten.

. Den beschrifteten Snapshot an das Tresorziel übertragen:
+
`snapmirror update -source-path <SVM:volume>|<cluster://SVM/volume>, ... -destination-path <SVM:volume>|<cluster://SVM/volume>, ... -source-snapshot snapshot`

+
Das folgende Beispiel überträgt den  `snap1` Snapshot::
+
--
[listing]
----
clust1::> snapmirror update -vserver vs1 -volume vol1
-source-snapshot snap1
----
Der gekennzeichnete Snapshot bleibt erhalten, wenn die Vault-Beziehung aktualisiert wird.

Weitere Informationen zu  `snapmirror update` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-update.html["ONTAP-Befehlsreferenz"^].

--


. Auf dem Quellvolume den Besitzer aus dem gekennzeichneten Snapshot entfernen:
+
`snapmirror snapshot-owner delete -vserver _SVM_ -volume _volume_ -snapshot _snapshot_ -owner _owner_`

+
Die folgenden Beispiele entfernen  `ApplicationA` als Eigentümer des  `snap1` Snapshots:

+
[listing]
----
clust1::> snapmirror snapshot-owner delete -vserver vs1 -volume vol1
-snapshot snap1 -owner ApplicationA
----
+
Weitere Informationen zu  `snapmirror snapshot-owner delete` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-snapshot-owner-delete.html["ONTAP-Befehlsreferenz"^].
