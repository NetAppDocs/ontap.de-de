= Das ursprüngliche ONTAP SnapMirror Quellvolume wird reaktiviert
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Sie können die ursprüngliche Datensicherungsbeziehung zwischen Quell- und Zielvolume wiederherstellen, wenn keine Daten mehr vom Zielvolume bereitgestellt werden müssen.

.Über diese Aufgabe
* Das folgende Verfahren setzt voraus, dass die Baseline im ursprünglichen Quellvolume intakt ist. Falls die Baseline nicht intakt ist, muss die Beziehung zwischen dem Volume, von dem Daten bereitgestellt werden, und dem ursprünglichen Quellvolume erstellt und initialisiert werden, bevor das Verfahren durchgeführt wird.
* Die Hintergrundvorbereitung und die Data-Warehousing-Phase einer XDP SnapMirror Beziehung können viel Zeit in Anspruch nehmen. Es ist nicht ungewöhnlich, dass die SnapMirror Beziehung den Status „preparing“ über einen längeren Zeitraum meldet.


.Schritte
. Die ursprüngliche Datensicherungsbeziehung umkehren:
+
`snapmirror resync -source-path _SVM:volume_ -destination-path _SVM:volume_`

+
Weitere Informationen zu  `snapmirror resync` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-resync.html["ONTAP-Befehlsreferenz"^].

+
[NOTE]
====
Dieser Befehl muss von der ursprünglichen Quell-SVM oder dem ursprünglichen Quell-Cluster ausgeführt werden. Obwohl für die Resynchronisierung kein Basistransfer erforderlich ist, kann sie zeitaufwändig sein. Die Resynchronisierung kann außerhalb der Spitzenzeiten durchgeführt werden. Der Befehl schlägt fehl, wenn auf der Quelle und dem Ziel kein gemeinsamer Snapshot vorhanden ist. Mit  `snapmirror initialize` kann die Beziehung neu initialisiert werden. Weitere Informationen zu  `snapmirror initialize` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-initialize.html["ONTAP-Befehlsreferenz"^].

====
+
Das folgende Beispiel kehrt die Beziehung zwischen dem ursprünglichen Quellvolume, `volA` auf `svm1`, und dem Volume, von dem Daten bereitgestellt werden, `volA_dst` auf `svm_backup` um:

+
[listing]
----
cluster_src::> snapmirror resync -source-path svm_backup:volA_dst -destination-path svm1:volA
----
. Wenn Sie bereit sind, den Datenzugriff auf die ursprüngliche Quelle wiederherzustellen, beenden Sie den Zugriff auf das ursprüngliche Zielvolume. Eine Möglichkeit hierfür ist das Stoppen der ursprünglichen Ziel-SVM:
+
`vserver stop -vserver _SVM_`

+
[NOTE]
====
Dieser Befehl muss von der ursprünglichen Ziel-SVM oder dem ursprünglichen Ziel-Cluster ausgeführt werden. Dieser Befehl unterbindet den Benutzerzugriff auf die gesamte ursprüngliche Ziel-SVM. Es besteht die Möglichkeit, den Zugriff auf das ursprüngliche Ziel-Volume mit anderen Methoden zu unterbinden.

====
+
Das folgende Beispiel stoppt die ursprüngliche Ziel-SVM:

+
[listing]
----
cluster_dst::> vserver stop svm_backup
----
+
Weitere Informationen zu  `vserver stop` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/vserver-stop.html["ONTAP-Befehlsreferenz"^].

. Die umgekehrte Beziehung aktualisieren:
+
`snapmirror update -source-path _SVM:volume_ -destination-path _SVM:volume_`

+
[NOTE]
====
Dieser Befehl muss von der ursprünglichen Quell-SVM oder dem ursprünglichen Quell-Cluster ausgeführt werden.

====
+
Das folgende Beispiel aktualisiert die Beziehung zwischen dem Volume, von dem Daten bereitgestellt werden, `volA_dst` auf `svm_backup`, und dem ursprünglichen Quellvolume, `volA` auf `svm1`:

+
[listing]
----
cluster_src::> snapmirror update -source-path svm_backup:volA_dst -destination-path svm1:volA
----
+
Weitere Informationen zu  `snapmirror update` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-update.html["ONTAP-Befehlsreferenz"^].

. Vom ursprünglichen Quell-SVM oder dem ursprünglichen Quell-Cluster geplante Übertragungen für die umgekehrte Beziehung anhalten:
+
`snapmirror quiesce -source-path _SVM:volume_ -destination-path _SVM:volume_`

+
[NOTE]
====
Dieser Befehl muss von der ursprünglichen Quell-SVM oder dem ursprünglichen Quell-Cluster ausgeführt werden.

====
+
Das folgende Beispiel stoppt geplante Übertragungen zwischen dem ursprünglichen Zielvolume, `volA_dst` auf `svm_backup`, und dem ursprünglichen Quellvolume, `volA` auf `svm1`:

+
[listing]
----
cluster_src::> snapmirror quiesce -source-path svm_backup:volA_dst -destination-path svm1:volA
----
+
Weitere Informationen zu  `snapmirror quiesce` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-quiesce.html["ONTAP-Befehlsreferenz"^].

. Wenn das letzte Update abgeschlossen ist und die Beziehung den Status „Quiesced“ anzeigt, kann der folgende Befehl auf der ursprünglichen Quell-SVM oder dem ursprünglichen Quell-Cluster ausgeführt werden, um die umgekehrte Beziehung aufzuheben:
+
`snapmirror break -source-path _SVM:volume_ -destination-path _SVM:volume_`

+
[NOTE]
====
Dieser Befehl muss von der ursprünglichen Quell-SVM oder dem Quell-Cluster ausgeführt werden.

====
+
Das folgende Beispiel unterbricht die Beziehung zwischen dem ursprünglichen Zielvolume, `volA_dst` auf `svm_backup`, und dem ursprünglichen Quellvolume, `volA` auf `svm1`:

+
[listing]
----
cluster_scr::> snapmirror break -source-path svm_backup:volA_dst -destination-path svm1:volA
----
+
Weitere Informationen zu  `snapmirror break` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-break.html["ONTAP-Befehlsreferenz"^].

. Aus der ursprünglichen Quell-SVM oder dem ursprünglichen Quell-Cluster die umgekehrte Datensicherungsbeziehung löschen:
+
`snapmirror delete -source-path _SVM:volume_ -destination-path _SVM:volume_`

+
[NOTE]
====
Dieser Befehl muss von der ursprünglichen Quell-SVM oder dem ursprünglichen Quell-Cluster ausgeführt werden.

====
+
Das folgende Beispiel löscht die umgekehrte Beziehung zwischen dem ursprünglichen Quellvolume, `volA` auf `svm1`, und dem Volume, von dem Daten bereitgestellt werden, `volA_dst` auf `svm_backup`:

+
[listing]
----
cluster_src::> snapmirror delete -source-path svm_backup:volA_dst -destination-path svm1:volA
----
+
Weitere Informationen zu  `snapmirror delete` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-delete.html["ONTAP-Befehlsreferenz"^].

. Die umgekehrte Beziehung vom ursprünglichen Ziel-SVM oder vom ursprünglichen Ziel-Cluster aufheben.
+
`snapmirror release -source-path _SVM:volume_ -destination-path _SVM:volume_`

+
[NOTE]
====
Dieser Befehl muss von der ursprünglichen Ziel-SVM oder dem ursprünglichen Ziel-Cluster ausgeführt werden.

====
+
Das folgende Beispiel hebt die umgekehrte Beziehung zwischen dem ursprünglichen Zielvolume, `volA_dst` auf `svm_backup`, und dem ursprünglichen Quellvolume, `volA` auf `svm1`, auf:

+
[listing]
----
cluster_dst::> snapmirror release -source-path svm_backup:volA_dst -destination-path svm1:volA
----
+
Weitere Informationen zu  `snapmirror release` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-release.html["ONTAP-Befehlsreferenz"^].

. Die ursprüngliche Datensicherungsbeziehung vom ursprünglichen Ziel-Cluster wiederherstellen:
+
`snapmirror resync -source-path _SVM:volume_ -destination-path _SVM:volume_`

+
Das folgende Beispiel stellt die Beziehung zwischen dem ursprünglichen Quellvolume, `volA` auf  `svm1`, und dem ursprünglichen Zielvolume, `volA_dst` auf  `svm_backup`, wieder her:

+
[listing]
----
cluster_dst::> snapmirror resync -source-path svm1:volA -destination-path svm_backup:volA_dst
----
+
Weitere Informationen zu  `snapmirror resync` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-resync.html["ONTAP-Befehlsreferenz"^].

. Gegebenenfalls die ursprüngliche Ziel-SVM starten:
+
`vserver start -vserver _SVM_`

+
Das folgende Beispiel startet die ursprüngliche Ziel-SVM:

+
[listing]
----
cluster_dst::> vserver start svm_backup
----
+
Weitere Informationen zu  `vserver start` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/vserver-start.html["ONTAP-Befehlsreferenz"^].



.Nachdem Sie fertig sind
Mit dem  `snapmirror show`Befehl lässt sich überprüfen, ob die SnapMirror Beziehung erstellt wurde.

Weitere Informationen zu  `snapmirror show` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-show.html["ONTAP-Befehlsreferenz"^].
