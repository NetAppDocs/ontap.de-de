= Wiederherstellung von Volumeninhalten aus einem ONTAP SnapMirror Ziel
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Sie können den Inhalt eines gesamten Volumes aus einem Snapshot in einem SnapMirror Ziel-Volume wiederherstellen. Der Inhalt des Volumes kann auf das ursprüngliche Quell-Volume oder auf ein anderes Volume wiederhergestellt werden.

.Über diese Aufgabe
Dieses Verfahren gilt für FAS, AFF und ASA Systeme. Bei einem ASA r2 System (ASA A1K, ASA A90, ASA A70, ASA A50, ASA A30, ASA A20 oder ASA C30) ist link:https://docs.netapp.com/us-en/asa-r2/data-protection/restore-data.html["diese Schritte"^] zum Wiederherstellen von Daten zu verwenden. ASA r2 Systeme bieten eine vereinfachte ONTAP Erfahrung speziell für ausschließlich SAN-basierte Kunden.

Das Zielvolume für den Wiederherstellungsvorgang muss eines der folgenden sein:

* Ein lesen/schreiben Volume, in diesem Fall führt SnapMirror eine inkrementelle Wiederherstellung durch, vorausgesetzt, dass Quell- und Ziel-Volume über einen gemeinsamen Snapshot verfügen (was typischerweise der Fall ist, wenn auf das ursprüngliche Quell-Volume wiederhergestellt wird).
+
[NOTE]
====
Der Befehl schlägt fehl, wenn kein gemeinsamer Snapshot vorhanden ist. Der Inhalt eines Volumes kann nicht auf einem leeren lesen/schreiben-Volume wiederhergestellt werden.

====
* Ein leeres Datensicherungsvolume, in diesem Fall führt SnapMirror eine _Baseline-Wiederherstellung_ durch, bei der der angegebene Snapshot und alle von ihm referenzierten Datenblöcke auf das Quellvolume übertragen werden.


Das Wiederherstellen des Inhalts eines Volumes ist ein unterbrechender Vorgang. SMB-Datenverkehr darf auf dem SnapVault primären Volume nicht ausgeführt werden, wenn ein Wiederherstellungsvorgang läuft.

Wenn auf dem Zielvolume für die Wiederherstellung die Komprimierung aktiviert ist und auf dem Quellvolume die Komprimierung nicht aktiviert ist, sollte die Komprimierung auf dem Zielvolume deaktiviert werden. Die Komprimierung muss nach Abschluss der Wiederherstellung wieder aktiviert werden.

Alle für das Zielvolume definierten Kontingentregeln werden vor der Wiederherstellung deaktiviert. Mit dem  `volume quota modify` Befehl lassen sich die Kontingentregeln nach Abschluss der Wiederherstellung wieder aktivieren.

Wenn Daten in einem Volume verloren gehen oder beschädigt werden, kann eine Wiederherstellung durch das Zurücksetzen auf einen früheren Snapshot erfolgen.

Dieses Verfahren ersetzt die aktuellen Daten auf dem Quellvolume durch Daten aus einer früheren Snapshot-Version. Diese Aufgabe ist auf dem Ziel-Cluster durchzuführen.

.Schritte
Der Inhalt eines Volumes kann mithilfe von System Manager oder der ONTAP CLI wiederhergestellt werden.

[role="tabbed-block"]
====
.System Manager
--
. *Schutz > Beziehungen* auswählen und anschließend den Namen des Quellvolumes anklicken.
. Klicken Sie image:icon_kabob.gif["Symbol für Menüoptionen"] und wählen Sie dann *Wiederherstellen*.
. Unter *Quelle* ist standardmäßig das Quellvolume ausgewählt. *Anderes Volume* kann ausgewählt werden, wenn ein anderes Volume als das Quellvolume gewählt werden soll.
. Unter *Ziel* den Snapshot auswählen, der wiederhergestellt werden soll.
. Wenn sich Quelle und Ziel auf unterschiedlichen Clustern befinden, im Remote-Cluster unter *Schutz > Beziehungen* den Wiederherstellungsfortschritt überwachen.


--
.CLI
--
. Die Snapshots im Ziel-Volume auflisten:
+
[source, cli]
----
volume snapshot show -vserver <SVM> -volume <volume>
----
+
Das folgende Beispiel zeigt die Momentaufnahmen auf dem  `vserverB:secondary1` Ziel-Cluster:

+
[listing]
----

cluster_dst::> volume snapshot show -vserver vserverB -volume secondary1

Vserver     Volume      Snapshot                State    Size  Total% Used%
-------     ------      ---------- ----------- ------   -----  ------ -----
vserverB    secondary1  hourly.2013-01-25_0005  valid   224KB     0%    0%
                        daily.2013-01-25_0010   valid   92KB      0%    0%
                        hourly.2013-01-25_0105  valid   228KB     0%    0%
                        hourly.2013-01-25_0205  valid   236KB     0%    0%
                        hourly.2013-01-25_0305  valid   244KB     0%    0%
                        hourly.2013-01-25_0405  valid   244KB     0%    0%
                        hourly.2013-01-25_0505  valid   244KB     0%    0%

7 entries were displayed.
----
. Den Inhalt eines Volumes aus einem Snapshot in einem SnapMirror Ziel-Volume wiederherstellen:
+
`snapmirror restore -source-path <SVM:volume>|<cluster://SVM/volume> -destination-path <SVM:volume>|<cluster://SVM/volume> -source-snapshot <snapshot>`

+

NOTE: Dieser Befehl muss von der ursprünglichen Quell-SVM oder dem ursprünglichen Quell-Cluster ausgeführt werden.

+
Der folgende Befehl stellt den Inhalt des ursprünglichen Quellvolumes `primary1` aus dem Snapshot `daily.2013-01-25_0010` im ursprünglichen Zielvolume `secondary1` wieder her:

+
[listing]
----
cluster_src::> snapmirror restore -source-path vserverB:secondary1 -destination-path vserverA:primary1 -source-snapshot daily.2013-01-25_0010

Warning: All data newer than snapshot daily.2013-01-25_0010 on volume vserverA:primary1 will be deleted.

Do you want to continue? {y|n}: y

[Job 34] Job is queued: snapmirror restore from source vserverB:secondary1 for the snapshot daily.2013-01-25_0010.
----
. Das wiederhergestellte Volume erneut einbinden und alle Anwendungen, die das Volume verwenden, neu starten.
