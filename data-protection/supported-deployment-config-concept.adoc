= Informationen zu ONTAP Datenschutz Fan-Out und Kaskadenbereitstellungen
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Eine _fan-out_ Bereitstellung kann verwendet werden, um den Datenschutz auf mehrere sekundäre Systeme auszuweiten. Eine _cascade_ Bereitstellung kann verwendet werden, um den Datenschutz auf tertiäre Systeme auszuweiten.

Sowohl Fan-Out- als auch Kaskadenbereitstellungen unterstützen beliebige Kombinationen aus SnapMirror DR, SnapVault oder einheitlicher Replikation. Ab ONTAP 9.5 unterstützen SnapMirror synchron Beziehungen Fan-Out-Bereitstellungen mit einer oder mehreren SnapMirror asynchron Beziehungen. Nur eine Beziehung in der Fan-Out-Konfiguration darf eine SnapMirror synchron Beziehung sein, alle anderen Beziehungen vom Quellvolume müssen SnapMirror asynchron Beziehungen sein. SnapMirror synchron Beziehungen unterstützen auch Kaskadenbereitstellungen (ab ONTAP 9.6); die Beziehung vom Zielvolume der SnapMirror synchron Beziehung muss jedoch eine SnapMirror asynchron Beziehung sein. xref:../snapmirror-active-sync/recover-unplanned-failover-task.html[SnapMirror Active Sync] (unterstützt ab ONTAP 9.13.1) unterstützt ebenfalls Fan-Out-Konfigurationen.


NOTE: Mithilfe einer Fan-in-Bereitstellung lassen sich Datensicherungsbeziehungen zwischen mehreren primären Systemen und einem einzelnen sekundären System erstellen. Jede Beziehung muss ein anderes Volume auf dem sekundären System verwenden.


NOTE: Es sollte beachtet werden, dass Volumes, die Teil einer Fan-Out- oder Kaskadenkonfiguration sind, länger für die Resynchronisierung benötigen können. Es ist nicht ungewöhnlich, dass die SnapMirror Beziehung den Status „preparing“ über einen längeren Zeitraum meldet.



[[how-fan-out-deployments-work]]
== Funktionsweise von Fan-Out-Bereitstellungen

SnapMirror unterstützt _multiple-mirrors_ und _mirror-vault_ Fan-Out-Bereitstellungen.

Eine Multiple-Mirrors-Fan-Out-Bereitstellung besteht aus einem Quellvolume, das eine Spiegelbeziehung zu mehreren sekundären Volumes aufweist.

image:sm-mirror-mirror-fanout.png["Datenschutz-Bereitstellung: Mehrere Mirror-Fanout-Konfigurationen"]

Eine Mirror Vault Fan Out Bereitstellung besteht aus einem Quellvolume, das eine Spiegelbeziehung zu einem sekundären Volume und eine SnapVault Beziehung zu einem anderen sekundären Volume hat.

image:sm-mirror-vault-fanout.png["Datensicherungsbereitstellung: Mirror Vault Fanout"]

Ab ONTAP 9.5 sind Fan-out-Bereitstellungen mit SnapMirror synchronen Beziehungen möglich; jedoch darf nur eine Beziehung in der Fan-out-Konfiguration eine SnapMirror synchron Beziehung sein, alle anderen Beziehungen vom Quellvolume müssen SnapMirror asynchron Beziehungen sein.

image:ssm-fanout.gif["Bereitstellung von Datenschutzmaßnahmen: Fan-out-Bereitstellungen mit SnapMirror synchronen Beziehungen"]



[[how-cascade-deployments-work]]
== Wie Kaskadenbereitstellungen funktionieren

SnapMirror unterstützt kaskadierende Bereitstellungen von _mirror-mirror_, _mirror-vault_, _vault-mirror_ und _vault-vault_.

Eine Mirror-Mirror-Kaskadenbereitstellung besteht aus einer Kette von Beziehungen, in der ein Quellvolume auf ein sekundäres Volume gespiegelt wird und das sekundäre Volume auf ein tertiäres Volume gespiegelt wird. Wenn das sekundäre Volume nicht verfügbar ist, lässt sich die Beziehung zwischen dem primären und dem tertiären Volume synchronisieren, ohne einen neuen Basistransfer durchzuführen.

In einer Beziehung von kaskadierten Volumes werden Langzeitaufbewahrungs-Snapshots nur auf dem finalen SnapMirror Ziel-Volume der Kaskade in allen Versionen von ONTAP 9 unterstützt. Die Aktivierung von Langzeitaufbewahrungs-Snapshots auf einem beliebigen Zwischen-Volume in der Kaskade führt zu fehlenden Backups und Snapshots. Wenn eine nicht unterstützte Konfiguration vorliegt, bei der Langzeitaufbewahrungs-Snapshots auf einem beliebigen Zwischen-Volume einer Kaskade aktiviert wurden, sollte der technische Support kontaktiert und auf die link:https://kb.netapp.com/on-prem/ontap/DP/SnapMirror/SnapMirror-KBs/Cascading_a_volume_with_Long-Term_Retention_(LTR)_snapshots_enabled_is_not_supported["NetApp Knowledge Base: Das Kaskadieren eines Volumes mit aktivierten Langzeitaufbewahrungs-Snapshots (LTR) wird nicht unterstützt."^] verwiesen werden.

Die folgenden ONTAP Versionen erlauben es nicht, Langzeit-Snapshots auf einem beliebigen Volume in einer Kaskade zu aktivieren, außer auf dem endgültigen SnapMirror Zielvolume.

* 9.15.1 und später
* 9.14.1P2 und P4 bis P14
* 9.13.1P9 bis P17
* 9.12.1 P12 bis P19
* 9.11.1P15 bis P20
* 9.10.1P18 bis P20
* 9.9.1P20


Weitere Informationen zu link:long-term-retention-snapshots-concept.html["Langzeitaufbewahrungs-Snapshots"].

Ab ONTAP 9.6 werden SnapMirror synchron Beziehungen in einer Mirror-Mirror-Kaskadenbereitstellung unterstützt. Nur die primären und sekundären Volumes können sich in einer SnapMirror synchron Beziehung befinden. Die Beziehung zwischen den sekundären und tertiären Volumes muss asynchron sein.

image:sm-mirror-mirror-cascade.png["SnapMirror Bereitstellung: Quelle zu Spiegel-Spiegel-Kaskadenkette"]

Eine Mirror-Vault-Kaskadenimplementierung besteht aus einer Kette von Beziehungen, in der ein Quellvolume auf ein sekundäres Volume gespiegelt wird und das sekundäre Volume auf ein tertiäres Volume übertragen wird.

image:sm-mirror-vault-cascade.png["SnapMirror Deployment: Quelle zu Mirror-Vault Kaskadenkette"]

Vault Mirror und Vault Vault Kaskadenbereitstellungen werden ebenfalls unterstützt:

* Eine Vault-Mirror-Kaskadenimplementierung besteht aus einer Kette von Beziehungen, in der ein Quellvolume auf ein sekundäres Volume gesichert wird und das sekundäre Volume auf ein Tertiärvolume gespiegelt wird.
* Eine Vault-Vault-Kaskadenbereitstellung besteht aus einer Kette von Beziehungen, in der ein Quellvolume in ein sekundäres Volume repliziert wird und das sekundäre Volume in ein tertiäres Volume repliziert wird.
