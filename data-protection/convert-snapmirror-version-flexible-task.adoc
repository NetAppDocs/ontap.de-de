= Konvertieren einer bestehenden ONTAP SnapMirror DP-Beziehung in XDP
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Wenn Sie auf ONTAP 9.12.1 oder höher aktualisieren, müssen DP-Typ-Beziehungen vor dem Upgrade in XDP konvertiert werden. ONTAP 9.12.1 und höher unterstützt keine DP-Typ-Beziehungen. Eine bestehende DP-Typ-Beziehung kann einfach in XDP konvertiert werden, um die versionsflexible SnapMirror Funktionalität zu nutzen.

Vor dem Upgrade auf ONTAP 9.12.1 müssen bestehende DP-Beziehungen in XDP konvertiert werden, bevor ein Upgrade auf ONTAP 9.12.1 und spätere Versionen möglich ist.

.Über diese Aufgabe
* SnapMirror konvertiert bestehende DP-Beziehungen nicht automatisch in XDP-Beziehungen. Für die Konvertierung der Beziehung ist es erforderlich, die bestehende Beziehung zu unterbrechen und zu löschen, eine neue XDP-Beziehung zu erstellen und die Beziehung erneut zu synchronisieren.
* Bei der Planung Ihrer Umstellung sollten Sie beachten, dass die Vorbereitung im Hintergrund und die Datenspeicherungsphase einer XDP SnapMirror Beziehung viel Zeit in Anspruch nehmen können. Es ist nicht ungewöhnlich, dass die SnapMirror Beziehung den Status „preparing“ über einen längeren Zeitraum meldet.


[NOTE]
====
Nach der Konvertierung eines SnapMirror Beziehungstyps von DP zu XDP werden speicherbezogene Einstellungen, wie automatische Größenanpassung und Speicherplatzzusage, nicht mehr an das Ziel repliziert.

====
.Schritte
. Im Ziel-Cluster ist sicherzustellen, dass die SnapMirror Beziehung vom Typ DP ist, der Spiegelungsstatus SnapMirrored ist, der Beziehungsstatus auf Idle steht und die Beziehung fehlerfrei ist:
+
[source, cli]
----
snapmirror show -destination-path <SVM:volume>
----
+

+
[NOTE]
====

+
[source, cli]

----

. Um sicherzustellen, dass geplante Aktualisierungen während der Konvertierung nicht ausgeführt werden, die bestehende DP-Typ-Beziehung in einen inaktiven Zustand versetzen:
+
[source, cli]
----
snapmirror quiesce -source-path <SVM:volume> -destination-path <SVM:volume>
----
+
[NOTE]
====
Dieser Befehl muss von der Ziel-SVM oder dem Ziel-Cluster ausgeführt werden.

====
+
Das folgende Beispiel versetzt die Beziehung zwischen dem Quellvolume `volA` auf `svm1` und dem Zielvolume `volA_dst` auf `svm_backup` in den Ruhezustand:

+
[listing]
----
cluster_dst::> snapmirror quiesce -destination-path svm_backup:volA_dst
----
+
Weitere Informationen zu  `snapmirror quiesce` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-quiesce.html["ONTAP-Befehlsreferenz"^].

. Die bestehende DP-Beziehung auflösen:
+
[source, cli]
----
snapmirror break -destination-path <SVM:volume>
----
+
[NOTE]
====
Dieser Befehl muss von der Ziel-SVM oder dem Ziel-Cluster ausgeführt werden.

====
+
Das folgende Beispiel unterbricht die Beziehung zwischen dem Quellvolume `volA` auf `svm1` und dem Zielvolume `volA_dst` auf `svm_backup`:

+
[listing]
----
cluster_dst::> snapmirror break -destination-path svm_backup:volA_dst
----
+
Weitere Informationen zu  `snapmirror break` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-break.html["ONTAP-Befehlsreferenz"^].

. Wenn die automatische Löschung von Snapshots auf dem Zielvolume aktiviert ist, sollte sie deaktiviert werden:
+
[source, cli]
----
volume snapshot autodelete modify -vserver _SVM_ -volume _volume_ -enabled false
----
+
Das folgende Beispiel deaktiviert die automatische Löschung von Snapshots auf dem Zielvolume `volA_dst`:

+
[listing]
----
cluster_dst::> volume snapshot autodelete modify -vserver svm_backup -volume volA_dst -enabled false
----
. Die bestehende DP-Beziehung löschen:
+
[source, cli]
----
snapmirror delete -destination-path <SVM:volume>
----
+
Weitere Informationen zu  `snapmirror-delete` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-delete.html["ONTAP-Befehlsreferenz"^].

+
[NOTE]
====
Dieser Befehl muss von der Ziel-SVM oder dem Ziel-Cluster ausgeführt werden.

====
+
Das folgende Beispiel löscht die Beziehung zwischen dem Quellvolume `volA` auf `svm1` und dem Zielvolume `volA_dst` auf `svm_backup`:

+
[listing]
----
cluster_dst::> snapmirror delete -destination-path svm_backup:volA_dst
----
. Die ursprüngliche SVM Disaster Recovery Beziehung auf dem Quell-Cluster freigeben:
+
[source, cli]
----
snapmirror release -destination-path <SVM:volume> -relationship-info-only true
----
+
Das folgende Beispiel gibt die SVM Disaster Recovery Beziehung frei:

+
[listing]
----
cluster_src::> snapmirror release -destination-path svm_backup:volA_dst -relationship-info-only true
----
+
Weitere Informationen zu  `snapmirror release` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-release.html["ONTAP-Befehlsreferenz"^].

. Sie können die Ausgabe, die Sie vom  `snapmirror show` Befehl behalten haben, verwenden, um die neue XDP Typ Beziehung zu erstellen:
+
[source, cli]
----
snapmirror create -source-path <SVM:volume> -destination-path <SVM:volume>  -type XDP -schedule <schedule> -policy <policy>
----
+
Die neue Beziehung muss dasselbe Quell- und Zielvolume verwenden. Weitere Informationen zu den in diesem Verfahren beschriebenen Befehlen befinden sich in der link:https://docs.netapp.com/us-en/ontap-cli/["ONTAP-Befehlsreferenz"^].

+
[NOTE]
====
Dieser Befehl muss von der Ziel-SVM oder dem Ziel-Cluster ausgeführt werden.

====
+
Das folgende Beispiel erstellt eine SnapMirror Disaster-Recovery-Beziehung zwischen dem Quell-Volume `volA` auf `svm1` und dem Ziel-Volume `volA_dst` auf `svm_backup` unter Verwendung der Standard `MirrorAllSnapshots`-Richtlinie:

+
[listing]
----
cluster_dst::> snapmirror create -source-path svm1:volA -destination-path svm_backup:volA_dst
-type XDP -schedule my_daily -policy MirrorAllSnapshots
----
. Die Quell- und Ziel-Volumes werden erneut synchronisiert:
+
[source, cli]
----
snapmirror resync -source-path <SVM:volume> -destination-path <SVM:volume>
----
+
Um die Resynchronisierungszeit zu verbessern, können Sie die  `-quick-resync` Option verwenden, sollten sich aber darüber im Klaren sein, dass Einsparungen bei der Speichereffizienz verloren gehen können.

+
[NOTE]
====
Dieser Befehl muss von der Ziel-SVM oder dem Ziel-Cluster ausgeführt werden. Obwohl für die Resynchronisierung kein Basistransfer erforderlich ist, kann sie zeitaufwändig sein. Die Resynchronisierung kann außerhalb der Spitzenzeiten durchgeführt werden.

====
+
Das folgende Beispiel synchronisiert die Beziehung zwischen dem Quellvolume `volA` auf `svm1` und dem Zielvolume `volA_dst` auf `svm_backup`:

+
[listing]
----
cluster_dst::> snapmirror resync -source-path svm1:volA -destination-path svm_backup:volA_dst
----
+
Weitere Informationen zu  `snapmirror resync` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-resync.html#parameters.html["ONTAP-Befehlsreferenz"^].

. Falls die automatische Löschung von Snapshots deaktiviert wurde, sollte sie wieder aktiviert werden:
+
[source, cli]
----
volume snapshot autodelete modify -vserver <SVM> -volume <volume> -enabled true
----


.Nachdem Sie fertig sind
. Mit dem  `snapmirror show`Befehl lässt sich überprüfen, ob die SnapMirror Beziehung erstellt wurde.
+
Weitere Informationen zu  `snapmirror show` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-show.html["ONTAP-Befehlsreferenz"^].

. Sobald das SnapMirror XDP-Zielvolume mit der Aktualisierung von Snapshots gemäß der SnapMirror Richtlinie beginnt, kann mit der Ausgabe des  `snapmirror list-destinations` Befehls aus dem Quell-Cluster die neue SnapMirror XDP-Beziehung angezeigt werden.


.Zusätzliche Informationen zu DP-Beziehungen
Ab ONTAP 9.3 ist der XDP Modus standardmäßig aktiviert, und alle Aufrufe des DP Modus in der Befehlszeile oder in neuen oder bestehenden Skripten werden automatisch in den XDP Modus konvertiert.

Bestehende Beziehungen bleiben unberührt. Wenn eine Beziehung bereits vom Typ DP ist, bleibt sie vom Typ DP. Ab ONTAP 9.5 ist MirrorAndVault die Standardrichtlinie, wenn kein Datenschutzmodus angegeben ist oder wenn der XDP-Modus als Beziehungstyp festgelegt wurde. Die folgende Tabelle zeigt das erwartete Verhalten.

[cols="3*"]
|===


| Wenn Sie angeben... | Der Typ ist... | Die Standardrichtlinie (wenn Sie keine Richtlinie angeben) ist... 


 a| 
DP
 a| 
XDP
 a| 
MirrorAllSnapshots (SnapMirror DR)



 a| 
Nichts
 a| 
XDP
 a| 
MirrorAndVault (einheitliche Replikation)



 a| 
XDP
 a| 
XDP
 a| 
MirrorAndVault (einheitliche Replikation)

|===
Wie die Tabelle zeigt, gewährleisten die unter verschiedenen Umständen XDP zugewiesenen Standardrichtlinien, dass die Konvertierung die funktionale Äquivalenz der vorherigen Typen beibehält. Es können selbstverständlich auch andere Richtlinien verwendet werden, einschließlich Richtlinien für die einheitliche Replikation.

[cols="3*"]
|===


| Wenn Sie angeben... | Und die Richtlinie lautet... | Das Ergebnis ist... 


 a| 
DP
 a| 
MirrorAllSnapshots
 a| 
SnapMirror DR



 a| 
XDPDefault
 a| 
SnapVault



 a| 
MirrorAndVault
 a| 
Einheitliche Replikation



 a| 
XDP
 a| 
MirrorAllSnapshots
 a| 
SnapMirror DR



 a| 
XDPDefault
 a| 
SnapVault



 a| 
MirrorAndVault
 a| 
Einheitliche Replikation

|===
Die einzigen Ausnahmen von der Umwandlung sind wie folgt:

* SVM-Datensicherungsbeziehungen sind in ONTAP 9.3 und früher weiterhin standardmäßig im DP-Modus.
+
Ab ONTAP 9.4 sind SVM-Datensicherungsbeziehungen standardmäßig im XDP-Modus.

* Die Load-Sharing-Datenschutzbeziehungen des Root-Volumes verbleiben weiterhin standardmäßig im DP-Modus.
* SnapLock Datenschutzbeziehungen verbleiben in ONTAP 9.4 und früher weiterhin standardmäßig im DP-Modus.
+
Ab ONTAP 9.5 werden SnapLock Datenschutzbeziehungen standardmäßig im XDP-Modus hergestellt.

* Explizite Aufrufe von DP verwenden weiterhin standardmäßig den DP-Modus, wenn die folgende clusterweite Option festgelegt ist:
+
[listing]
----
options replication.create_data_protection_rels.enable on
----
+
Diese Option wird ignoriert, wenn DP nicht explizit aufgerufen wird.
