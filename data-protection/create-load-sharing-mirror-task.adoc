= ONTAP Load-Sharing Mirror-Beziehungen erstellen und initialisieren
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Für jedes SVM Root Volume, das NAS-Daten im Cluster bereitstellt, sollte ein Load-Sharing Mirror (LSM) erstellt werden. Bei Clustern, die aus zwei oder mehr HA-Paaren bestehen, empfiehlt sich die Verwendung von Load-Sharing Mirrors der SVM Root Volumes, damit der Namespace für Clients auch dann zugänglich bleibt, wenn beide Nodes eines HA-Paares ausfallen. Load-Sharing Mirrors sind für Cluster, die nur aus einem HA-Paar bestehen, nicht geeignet.

.Bevor Sie beginnen
Ab ONTAP 9.16.1 kann beim Erstellen einer Load-Sharing-Mirror-Beziehung für die Ziel-SVM kein Speicherlimit aktiviert sein.

.Über diese Aufgabe
Wenn Sie ein LSM auf demselben Knoten erstellen und dieser Knoten nicht verfügbar ist, entsteht ein Single Point of Failure, und es gibt keine zweite Kopie, um die Daten für Clients weiterhin zugänglich zu halten. Wenn das LSM jedoch auf einem anderen Knoten als dem mit dem Root Volume oder auf einem anderen HA-Paar erstellt wird, sind die Daten auch im Falle eines Ausfalls weiterhin zugänglich.

Beispielsweise in einem Vier-Node-Cluster mit einem Root-Volume auf drei Knoten:

* Für das Root-Volume auf HA 1 Knoten 1 das LSM auf HA 2 Knoten 1 oder HA 2 Knoten 2 erstellen.
* Für das Root-Volume auf HA 1 Knoten 2 ist das LSM auf HA 2 Knoten 1 oder HA 2 Knoten 2 zu erstellen.
* Für das Root-Volume auf HA 2 Knoten 1 ist das LSM auf HA 1 Knoten 1 oder HA 1 Knoten 2 zu erstellen.


.Schritte
. Ein Ziel-Volume für den LSM erstellen:
+
[source, cli]
----
volume create -vserver <SVM> -volume <volume> -aggregate <aggregate> -type DP -size <size>
----
+
Das Zielvolume sollte mindestens so groß sein wie das Stammvolume.

+
Es ist eine Best Practice, das Root- und das Ziel-Volume mit Suffixen wie  `_root` und  `_m1` zu benennen.

+
Weitere Informationen zu  `volume create` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/volume-create.html["ONTAP-Befehlsreferenz"^].

+
Das folgende Beispiel erstellt ein Load-Sharing Mirror Volume für das Root-Volume `svm1_root` in `cluster_src`:

+
[listing]
----
cluster_src:> volume create -vserver svm1 -volume svm1_m1 -aggregate aggr_1 -size 1gb -state online -type DP
----
. link:create-replication-job-schedule-task.html["Einen Replikationsauftragszeitplan erstellen"].
. Eine Load-Sharing-Mirror-Beziehung zwischen dem SVM-Root-Volume und dem Ziel-Volume für das LSM erstellen:
+
[source, cli]
----
snapmirror create -source-path <SVM:volume> -destination-path <SVM:volume> -type LS -schedule <schedule>
----
+
Das folgende Beispiel erstellt eine Load-Sharing-Mirror-Beziehung zwischen dem Root-Volume `svm1_root` und dem Load-Sharing-Mirror-Volume `svm1_m1`:

+
[listing]
----
cluster_src::> snapmirror create -source-path svm1:svm1_root -destination-path svm1:svm1_m1 -type LS -schedule hourly
----
+
Das Attribut „type“ des Load-Sharing-Mirrors ändert sich von  `DP` zu  `LS`.

+
Weitere Informationen zu  `snapmirror create` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-create.html["ONTAP-Befehlsreferenz"^].

. Den Load-Sharing-Spiegel initialisieren:
+
[source, cli]
----
snapmirror initialize-ls-set -source-path <SVM:volume>
----
+
Das folgende Beispiel initialisiert den Load-Sharing Mirror für das Root-Volume `svm1_root`:

+
[listing]
----
cluster_src::> snapmirror initialize-ls-set -source-path svm1:svm1_root
----
+
Weitere Informationen zu  `snapmirror initialize` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-initialize.html["ONTAP-Befehlsreferenz"^].
