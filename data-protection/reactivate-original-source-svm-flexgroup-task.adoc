= Die ursprüngliche ONTAP SnapMirror Quelle-SVM für FlexGroup Volumes wird reaktiviert
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Sie können die ursprüngliche Datensicherungsbeziehung zwischen Quell-SVM und Ziel-SVM wiederherstellen, wenn keine Daten mehr vom Ziel-SVM bereitgestellt werden müssen. Um die ursprüngliche Quell-SVM bei Verwendung von FlexGroup Volumes zu reaktivieren, sind zusätzliche Schritte erforderlich, einschließlich des Löschens der ursprünglichen SVM DR-Beziehung und des Freigebens der ursprünglichen Beziehung, bevor die Beziehung umgekehrt wird. Auch die umgekehrte Beziehung muss freigegeben und die ursprüngliche Beziehung wiederhergestellt werden, bevor geplante Übertragungen gestoppt werden.

.Schritte
. Vom ursprünglichen Ziel-SVM oder vom ursprünglichen Ziel-Cluster die ursprüngliche SVM-DR-Beziehung löschen:
+
`snapmirror delete -source-path SVM: -destination-path SVM:`

+
[NOTE]
====
Sie müssen nach dem SVM-Namen in den  `-source-path` und  `-destination-path` Optionen einen Doppelpunkt (:) eingeben. Das folgende Beispiel verdeutlicht dies.

====
+
Das folgende Beispiel löscht die ursprüngliche Beziehung zwischen der ursprünglichen Quell-SVM, svm1, und der ursprünglichen Ziel-SVM, `svm_backup`:

+
[listing]
----
cluster_dst::> snapmirror delete -source-path svm1: -destination-path svm_backup:
----
. Vom ursprünglichen Quell-SVM oder vom ursprünglichen Quell-Cluster wird die ursprüngliche Beziehung freigegeben, wobei die Snapshots intakt bleiben:
+
`snapmirror release -source-path SVM: -destination-path SVM: -relationship-info-only true`

+
[NOTE]
====
Sie müssen nach dem SVM-Namen in den  `-source-path` und  `-destination-path` Optionen einen Doppelpunkt (:) eingeben. Das folgende Beispiel verdeutlicht dies.

====
+
Das folgende Beispiel gibt die ursprüngliche Beziehung zwischen der ursprünglichen Quell-SVM, svm1, und der ursprünglichen Ziel-SVM `svm_backup` frei.

+
[listing]
----
cluster_src::> snapmirror release -source-path svm1: -destination-path svm_backup: -relationship-info-only true
----
. Ausgehend von der ursprünglichen Quell-SVM oder dem ursprünglichen Quell-Cluster wird eine umgekehrte SVM DR-Beziehung mit derselben Konfiguration, Richtlinie und Einstellung zur Identitätserhaltung wie bei der ursprünglichen SVM DR-Beziehung erstellt:
+
`snapmirror create -source-path SVM: -destination-path SVM:`

+
[NOTE]
====
Sie müssen nach dem SVM-Namen in den  `-source-path` und  `-destination-path` Optionen einen Doppelpunkt (:) eingeben. Das folgende Beispiel verdeutlicht dies.

====
+
Das folgende Beispiel stellt eine Beziehung zwischen der SVM, von der Sie Daten bereitstellen, `svm_backup`, und der ursprünglichen Quell-SVM, `svm1`:

+
[listing]
----
cluster_src::> snapmirror create -source-path svm_backup: -destination-path svm1:
----
. Vom ursprünglichen Quell-SVM oder dem ursprünglichen Quell-Cluster aus wird der folgende Befehl ausgeführt, um die Datenschutzbeziehung umzukehren:
+
`snapmirror resync -source-path _SVM_: -destination-path _SVM_:`

+
[NOTE]
====
Sie müssen nach dem SVM-Namen in den  `-source-path` und  `-destination-path` Optionen einen Doppelpunkt (:) eingeben. Das folgende Beispiel verdeutlicht dies.

====
+
Obwohl für die Resynchronisierung kein Basistransfer erforderlich ist, kann sie zeitaufwändig sein. Die Resynchronisierung kann außerhalb der Spitzenzeiten durchgeführt werden.

+
[NOTE]
====
Der Befehl schlägt fehl, wenn auf der Quelle und dem Ziel kein gemeinsamer Snapshot vorhanden ist.  `snapmirror initialize` kann verwendet werden, um die Beziehung neu zu initialisieren.

====
+
Das folgende Beispiel kehrt die Beziehung zwischen der ursprünglichen Quell-SVM, `svm1`, und der SVM, von der Daten bereitgestellt werden, `svm_backup` um:

+
[listing]
----
cluster_src::> snapmirror resync -source-path svm_backup: -destination-path svm1:
----
. Wenn der Datenzugriff auf die ursprüngliche Quell-SVM wiederhergestellt werden soll, wird die ursprüngliche Ziel-SVM gestoppt, um alle derzeit mit der ursprünglichen Ziel-SVM verbundenen Clients zu trennen.
+
`vserver stop -vserver _SVM_`

+
Das folgende Beispiel stoppt die ursprüngliche Ziel-SVM, die aktuell Daten bereitstellt:

+
[listing]
----
cluster_dst::> vserver stop svm_backup
----
. Es sollte mithilfe des  `vserver show` Befehls überprüft werden, ob sich die ursprüngliche Ziel-SVM im gestoppten Zustand befindet.
+
[listing]
----
cluster_dst::> vserver show
                                  Admin      Operational Root
Vserver        Type    Subtype    State      State       Volume     Aggregate
--------       ------- ---------- ---------- ----------- ---------- ----------
svm_backup     data    default    stopped    stopped     rv         aggr1
----
. Vom ursprünglichen Quell-SVM oder dem ursprünglichen Quell-Cluster aus wird der folgende Befehl ausgeführt, um die endgültige Aktualisierung der umgekehrten Beziehung durchzuführen und alle Änderungen vom ursprünglichen Ziel-SVM auf den ursprünglichen Quell-SVM zu übertragen:
+
`snapmirror update -source-path _SVM_: -destination-path _SVM_:`

+
[NOTE]
====
Sie müssen nach dem SVM-Namen in den  `-source-path` und  `-destination-path` Optionen einen Doppelpunkt (:) eingeben. Das folgende Beispiel verdeutlicht dies.

====
+
Das folgende Beispiel aktualisiert die Beziehung zwischen der ursprünglichen Ziel-SVM, von der Sie Daten bereitstellen ,`svm_backup`, und der ursprünglichen Quell-SVM, `svm1`:

+
[listing]
----
cluster_src::> snapmirror update -source-path svm_backup: -destination-path svm1:
----
+
Weitere Informationen zu  `snapmirror update` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-update.html["ONTAP-Befehlsreferenz"^].

. Führen Sie auf der ursprünglichen Quell-SVM oder dem ursprünglichen Quell-Cluster den folgenden Befehl aus, um geplante Übertragungen für die umgekehrte Beziehung zu stoppen:
+
`snapmirror quiesce -source-path _SVM_: -destination-path _SVM_:`

+
[NOTE]
====
Sie müssen nach dem SVM-Namen in den  `-source-path` und  `-destination-path` Optionen einen Doppelpunkt (:) eingeben. Das folgende Beispiel verdeutlicht dies.

====
+
Das folgende Beispiel stoppt geplante Übertragungen zwischen der SVM, von der Sie Daten bereitstellen, `svm_backup`, und der ursprünglichen SVM, `svm1`:

+
[listing]
----
cluster_src::> snapmirror quiesce -source-path svm_backup: -destination-path svm1:
----
+
Weitere Informationen zu  `snapmirror quiesce` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-quiesce.html["ONTAP-Befehlsreferenz"^].

. Wenn das letzte Update abgeschlossen ist und die Beziehung den Beziehungsstatus „Quiesced“ anzeigt, muss der folgende Befehl auf der ursprünglichen Quell-SVM oder dem ursprünglichen Quell-Cluster ausgeführt werden, um die umgekehrte Beziehung aufzuheben:
+
`snapmirror break -source-path _SVM_: -destination-path _SVM_:`

+
[NOTE]
====
Sie müssen nach dem SVM-Namen in den  `-source-path` und  `-destination-path` Optionen einen Doppelpunkt (:) eingeben. Das folgende Beispiel verdeutlicht dies.

====
+
Das folgende Beispiel unterbricht die Beziehung zwischen dem ursprünglichen Ziel-SVM, von dem Sie Daten bereitgestellt haben, `svm_backup`, und dem ursprünglichen Quell-SVM, `svm1`:

+
[listing]
----
cluster_src::> snapmirror break -source-path svm_backup: -destination-path svm1:
----
+
Weitere Informationen zu  `snapmirror break` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-break.html["ONTAP-Befehlsreferenz"^].

. Falls die ursprüngliche Quell-SVM zuvor gestoppt wurde, starten Sie die ursprüngliche Quell-SVM vom ursprünglichen Quell-Cluster aus:
+
`vserver start -vserver _SVM_`

+
Das folgende Beispiel startet die ursprüngliche Quell-SVM:

+
[listing]
----
cluster_src::> vserver start svm1
----
. Löschen Sie von der ursprünglichen Quell-SVM oder dem ursprünglichen Quell-Cluster die umgekehrte SVM DR-Beziehung:
+
`snapmirror delete -source-path SVM: -destination-path SVM:`

+
[NOTE]
====
Sie müssen nach dem SVM-Namen in den  `-source-path` und  `-destination-path` Optionen einen Doppelpunkt (:) eingeben. Das folgende Beispiel verdeutlicht dies.

====
+
Das folgende Beispiel löscht die umgekehrte Beziehung zwischen der ursprünglichen Ziel-SVM, svm_backup, und der ursprünglichen Quell-SVM, `svm1`:

+
[listing]
----
cluster_src::> snapmirror delete -source-path svm_backup: -destination-path svm1:
----
. Vom ursprünglichen Ziel-SVM oder vom ursprünglichen Ziel-Cluster die umgekehrte Beziehung freigeben, wobei die Snapshots intakt bleiben:
+
`snapmirror release -source-path SVM: -destination-path SVM: -relationship-info-only true`

+
[NOTE]
====
Sie müssen nach dem SVM-Namen in den  `-source-path` und  `-destination-path` Optionen einen Doppelpunkt (:) eingeben. Das folgende Beispiel verdeutlicht dies.

====
+
Das folgende Beispiel löst die umgekehrte Beziehung zwischen der ursprünglichen Ziel-SVM, svm_backup, und der ursprünglichen Quell-SVM, svm1, auf:

+
[listing]
----
cluster_dst::> snapmirror release -source-path svm_backup: -destination-path svm1: -relationship-info-only true
----
. Stellen Sie vom ursprünglichen Ziel-SVM oder vom ursprünglichen Ziel-Cluster aus die ursprüngliche Beziehung wieder her. Es wird dieselbe Konfiguration, Richtlinie und Einstellung zur Identitätsbeibehaltung wie bei der ursprünglichen SVM DR-Beziehung verwendet.
+
`snapmirror create -source-path SVM: -destination-path SVM:`

+
[NOTE]
====
Sie müssen nach dem SVM-Namen in den  `-source-path` und  `-destination-path` Optionen einen Doppelpunkt (:) eingeben. Das folgende Beispiel verdeutlicht dies.

====
+
Das folgende Beispiel stellt eine Beziehung zwischen der ursprünglichen Quell-SVM, `svm1`, und der ursprünglichen Ziel-SVM, `svm_backup` her:

+
[listing]
----
cluster_dst::> snapmirror create -source-path svm1: -destination-path svm_backup:
----
. Vom ursprünglichen Ziel-SVM oder vom ursprünglichen Ziel-Cluster wird die ursprüngliche Datensicherungsbeziehung wiederhergestellt:
+
`snapmirror resync -source-path _SVM_: -destination-path _SVM_:`

+
[NOTE]
====
Sie müssen nach dem SVM-Namen in den  `-source-path` und  `-destination-path` Optionen einen Doppelpunkt (:) eingeben. Das folgende Beispiel verdeutlicht dies.

====
+
Das folgende Beispiel stellt die Beziehung zwischen der ursprünglichen Quell-SVM, `svm1`, und der ursprünglichen Ziel-SVM, `svm_backup` wieder her:

+
[listing]
----
cluster_dst::> snapmirror resync -source-path svm1: -destination-path svm_backup:
----
