= LIFs und zugehörige Netzwerkeinstellungen von ONTAP SnapMirror SVM-Replikation ausschließen
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Wenn sich Quell- und Ziel-SVMs in unterschiedlichen Subnetzen befinden, kann die  `-discard-configs network` Option des  `snapmirror policy create` Befehls verwendet werden, um LIFs und zugehörige Netzwerkeinstellungen von der SVM-Replikation auszuschließen.

.Bevor Sie beginnen
Die Quell- und Ziel-Cluster sowie die SVMs müssen miteinander verbunden sein.

Weitere Informationen finden Sie unter link:../peering/create-cluster-relationship-93-later-task.html["Erstellen einer Cluster Peer-Beziehung"] und link:../peering/create-intercluster-svm-peer-relationship-93-later-task.html["Erstellen einer SVM Intercluster Peer-Beziehung"].

.Über diese Aufgabe
Die  `-identity-preserve`Option des  `snapmirror create`Befehls muss auf  `true`gesetzt werden, wenn die SVM-Replikationsbeziehung erstellt wird.

.Schritte
. Erstellen einer Ziel-SVM:
+
`vserver create -vserver _SVM_ -subtype dp-destination`

+
Der SVM-Name muss sowohl im Quell-Cluster als auch im Ziel-Cluster eindeutig sein.

+
Das folgende Beispiel erstellt eine Ziel-SVM mit dem Namen `svm_backup`:

+
[listing]
----
cluster_dst:> vserver create -vserver svm_backup -subtype dp-destination
----
. Vom Ziel-Cluster aus eine SVM Peer-Beziehung mit dem  `vserver peer create` Befehl erstellen.
+
Weitere Informationen finden sich unter link:../peering/create-intercluster-svm-peer-relationship-93-later-task.html["Erstellen einer SVM Intercluster Peer-Beziehung"].

+
Weitere Informationen zu  `vserver peer create` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/vserver-peer-create.html["ONTAP-Befehlsreferenz"^].

. Einen Arbeitszeitplan erstellen:
+
`job schedule cron create -name _job_name_ -month _month_ -dayofweek _day_of_week_ -day _day_of_month_ -hour _hour_ -minute _minute_`

+
Für  `-month`,  `-dayofweek` und  `-hour` kann  `all` angegeben werden, damit der Job jeden Monat, an jedem Wochentag bzw. zu jeder Stunde ausgeführt wird.

+
[NOTE]
====
Der minimal unterstützte Zeitplan (RPO) für FlexVol Volumes in einer SVM SnapMirror Beziehung beträgt 15 Minuten. Der minimal unterstützte Zeitplan (RPO) für FlexGroup Volumes in einer SVM SnapMirror Beziehung beträgt 30 Minuten.

====
+
Das folgende Beispiel erstellt einen Jobplan mit dem Namen  `my_weekly`, der samstags um 3:00 Uhr morgens ausgeführt wird:

+
[listing]
----
cluster_dst::> job schedule cron create -name my_weekly -dayofweek "Saturday" -hour 3 -minute 0
----
. Erstellen einer benutzerdefinierten Replikationsrichtlinie:
+
`snapmirror policy create -vserver _SVM_ -policy _policy_ -type async-mirror|vault|mirror-vault -comment _comment_ -tries transfer_tries -transfer-priority low|normal -is-network-compression-enabled true|false -discard-configs network`

+
Das folgende Beispiel erstellt eine benutzerdefinierte Replikationsrichtlinie für SnapMirror DR, die LIFs ausschließt:

+
[listing]
----
cluster_dst::> snapmirror policy create -vserver svm1 -policy DR_exclude_LIFs -type async-mirror -discard-configs network
----
+
Das folgende Beispiel erstellt eine benutzerdefinierte Replikationsrichtlinie für die einheitliche Replikation, die LIFs ausschließt:

+
[listing]
----
cluster_dst::> snapmirror policy create -vserver svm1 -policy unified_exclude_LIFs -type mirror-vault -discard-configs network
----
+
[NOTE]
====
Es empfiehlt sich, dieselbe benutzerdefinierte SnapMirror-Richtlinie auf dem Quell-Cluster für zukünftige Failover- und Failback-Szenarien zu erstellen.

====
+
Weitere Informationen zu  `snapmirror policy create` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-policy-create.html["ONTAP-Befehlsreferenz"^].

. Von der Ziel-SVM oder dem Ziel-Cluster aus den folgenden Befehl ausführen, um eine Replikationsbeziehung zu erstellen:
+
`snapmirror create -source-path _SVM_: -destination-path _SVM_: -type DP|XDP -schedule _schedule_ -policy _policy_ -identity-preserve true|false -discard-configs true|false`

+
[NOTE]
====
Sie müssen nach dem SVM-Namen in den  `-source-path` und  `-destination-path` Optionen einen Doppelpunkt (:) eingeben. Die folgenden Beispiele veranschaulichen dies.

====
+
Das folgende Beispiel erstellt eine SnapMirror DR-Beziehung, die LIFs ausschließt:

+
[listing]
----
cluster_dst::> snapmirror create -source-path svm1: -destination-path svm_backup: -type XDP -schedule my_weekly -policy DR_exclude_LIFs -identity-preserve true
----
+
Das folgende Beispiel erstellt eine SnapMirror einheitliche Replikationsbeziehung, die LIFs ausschließt:

+
[listing]
----
cluster_dst::> snapmirror create -source-path svm1: -destination-path svm_backup: -type XDP -schedule my_weekly -policy unified_exclude_LIFs -identity-preserve true -discard-configs true
----
+
Weitere Informationen zu  `snapmirror create` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-create.html["ONTAP-Befehlsreferenz"^].

. Die Ziel-SVM anhalten:
+
`vserver stop`

+
`_SVM name_`

+
Das folgende Beispiel stoppt die Ziel-SVM mit dem Namen svm_backup:

+
[listing]
----
cluster_dst::> vserver stop -vserver svm_backup
----
. Vom Ziel-SVM oder Ziel-Cluster aus eine Replikationsbeziehung initialisieren:
+
`snapmirror initialize -source-path _SVM_: -destination-path _SVM_:`

+
Das folgende Beispiel initialisiert die Beziehung zwischen der Quelle, `svm1` und dem Ziel, `svm_backup`:

+
[listing]
----
cluster_dst::> snapmirror initialize -source-path svm1: -destination-path svm_backup:
----
+
Weitere Informationen zu  `snapmirror initialize` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/snapmirror-initialize.html["ONTAP-Befehlsreferenz"^].



.Nachdem Sie fertig sind
Das Netzwerk und die Protokolle auf der Ziel-SVM sind für den Datenzugriff im Katastrophenfall zu konfigurieren.
