= Zugriff auf eine schreibgeschützte LUN-Kopie aus einem ONTAP SnapVault Backup
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Sie können auf eine schreibgeschützte Kopie einer LUN aus dem letzten Snapshot in einem SnapVault Backup zugreifen. Die LUN-ID, der Pfad und die Seriennummer unterscheiden sich von der Quell-LUN und müssen zunächst zugeordnet werden. Persistente Reservierungen, LUN-Zuordnungen und Initiatorgruppen werden nicht auf das SnapVault sekundäre Volume repliziert.

.Bevor Sie beginnen
* Die SnapVault Beziehung muss initialisiert werden und der neueste Snapshot im SnapVault sekundären Volume muss die gewünschte LUN enthalten.
* Die Storage Virtual Machine (SVM), die das SnapVault Backup enthält, muss über eine oder mehrere LIFs mit dem gewünschten SAN-Protokoll verfügen, die von dem Host aus erreichbar sind, der zum Zugriff auf die LUN-Kopie verwendet wird.
* Wenn Sie planen, direkt vom SnapVault sekundären Volume auf LUN-Kopien zuzugreifen, müssen Sie Ihre Initiatorgruppen im Voraus auf der SnapVault SVM erstellen.
+
Sie können direkt auf eine LUN vom SnapVault sekundäres Volume zugreifen, ohne zuerst das Volume, das die LUN enthält, wiederherstellen oder klonen zu müssen.



.Über diese Aufgabe
Wenn dem SnapVault sekundären Volume ein neuer Snapshot hinzugefügt wird, während Sie eine LUN aus einem vorherigen Snapshot zugeordnet haben, ändert sich der Inhalt der zugeordneten LUN. Die LUN ist weiterhin mit denselben Kennungen zugeordnet, aber die Daten stammen aus dem neuen Snapshot. Wenn sich die LUN-Größe ändert, erkennen einige Hosts die Größenänderung automatisch; Windows-Hosts benötigen einen erneuten Datenträgerscan, um eine Größenänderung zu erkennen.

.Schritte
. Die verfügbaren LUNs im SnapVault sekundären Volume auflisten.
+
[source, cli]
----
lun show
----
+
In diesem Beispiel sind sowohl die ursprünglichen LUNs im primären Volume srcvolA als auch die Kopien im SnapVault sekundären Volume dstvolB zu sehen:

+
[listing]
----
cluster::> lun show

Vserver   Path                State   Mapped   Type          Size
--------  ------------------  ------  -------  --------      -------
vserverA  /vol/srcvolA/lun_A  online  mapped   windows  300.0GB
vserverA  /vol/srcvolA/lun_B  online  mapped   windows  300.0GB
vserverA  /vol/srcvolA/lun_C  online  mapped   windows  300.0GB
vserverB  /vol/dstvolB/lun_A  online  unmapped windows  300.0GB
vserverB  /vol/dstvolB/lun_B  online  unmapped windows  300.0GB
vserverB  /vol/dstvolB/lun_C  online  unmapped windows  300.0GB

6 entries were displayed.
----
+
Weitere Informationen zu  `lun show` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/lun-show.html["ONTAP-Befehlsreferenz"^].

. Falls die Initiatorgruppe für den gewünschten Host auf der SVM, die das SnapVault sekundäre Volume enthält, noch nicht existiert, sollte eine Initiatorgruppe erstellt werden.
+
[source, cli]
----
igroup create -vserver <SVM_name> -igroup <igroup_name> -protocol <protocol> -ostype <ostype> -initiator <initiator_name>
----
+
Dieser Befehl erstellt eine Initiatorgruppe für einen Windows Host, der das iSCSI Protokoll verwendet:

+
[listing]
----
cluster::> igroup create -vserver vserverB -igroup temp_igroup
  -protocol iscsi -ostype windows
  -initiator iqn.1991-05.com.microsoft:hostA
----
. Die gewünschte LUN-Kopie wird der Initiatorgruppe zugeordnet.
+
[source, cli]
----
lun mapping create -vserver <SVM_name> -path <LUN_path> -igroup <igroup_name>
----
+
[listing]
----
cluster::> lun mapping create -vserver vserverB -path /vol/dstvolB/lun_A
  -igroup temp_igroup
----
+
Weitere Informationen zu  `lun mapping create` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/lun-mapping-create.html["ONTAP-Befehlsreferenz"^].

. Den Host mit der LUN verbinden und nach Bedarf auf die Inhalte der LUN zugreifen.
