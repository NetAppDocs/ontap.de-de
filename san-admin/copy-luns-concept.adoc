= Wissenswertes vor dem Kopieren von LUNs
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Vor dem Kopieren einer LUN sind bestimmte Aspekte zu berücksichtigen.

Clusteradministratoren können eine LUN mithilfe des  `lun copy` Befehls zwischen Storage Virtual Machines (SVMs) innerhalb des Clusters kopieren. Clusteradministratoren müssen die Peering-Beziehung zwischen den Storage Virtual Machines (SVMs) mithilfe des  `vserver peer create` Befehls herstellen, bevor eine LUN-Kopie zwischen SVMs durchgeführt wird. Auf dem Quellvolume muss ausreichend Speicherplatz für einen SIS-Klon vorhanden sein.

LUNs in Snapshots können als Quell-LUNs für den  `lun copy` Befehl verwendet werden. Beim Kopieren einer LUN mit dem  `lun copy` Befehl ist die LUN-Kopie sofort für Lese- und Schreibzugriffe verfügbar. Die Quell-LUN bleibt durch das Erstellen einer LUN-Kopie unverändert. Sowohl die Quell-LUN als auch die LUN-Kopie existieren als eindeutige LUNs mit unterschiedlichen LUN-Seriennummern. Änderungen an der Quell-LUN werden nicht in der LUN-Kopie übernommen, und Änderungen an der LUN-Kopie werden nicht in der Quell-LUN übernommen. Die LUN-Zuordnung der Quell-LUN wird nicht auf die neue LUN kopiert, die LUN-Kopie muss zugeordnet werden.

Der Datenschutz durch Snapshots erfolgt auf Volume-Ebene. Wenn eine LUN auf ein anderes Volume als das Volume der Quell-LUN kopiert wird, unterliegt die Ziel-LUN dem Datenschutzschema des Ziel-Volumes. Wenn für das Ziel-Volume keine Snapshots eingerichtet sind, werden von der LUN-Kopie keine Snapshots erstellt.

Das Kopieren von LUNs ist ein unterbrechungsfreier Vorgang.

Die folgenden LUN-Typen können nicht kopiert werden:

* Eine LUN, die aus einer Datei erstellt wurde
* Eine LUN im NVFAIL Zustand
* Eine LUN, die sich in einer Lastverteilungsbeziehung befindet
* Eine Protokoll-Endpunkt-Klasse LUN


Weitere Informationen zu  `lun copy` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/search.html?q=lun+copy["ONTAP-Befehlsreferenz"^].
