== Netzwerkbetrieb

[cols="30%,70%"]
|===
| Aktualisieren | Beschreibung 


 a| 
Verwalten Sie die NFS-Versionen, die die SVM unterstützt
 a| 
Die Unterstützung für NFSv4.2 wird unabhängig von den NFSv4.1-Einstellungen verwaltet, sodass jede SVM die NFSv4.2-Funktionen je nach Bedarf aktivieren oder deaktivieren kann. Standardmäßig ist NFSv4.2 aktiviert, wenn ein neuer NFS-Service erstellt wird, sofern es nicht explizit vom Administrator deaktiviert wird.

* https://docs.netapp.com/us-en/ontap/nfs-admin/enable-disable-nfsv42-task.html["NFSv4.2 für ONTAP SVMs aktivieren oder deaktivieren"]
* https://docs.netapp.com/us-en/ontap/task_nas_enable_linux_nfs.html["Aktivieren Sie NAS-Speicher für Linux-Server mithilfe von NFS mit ONTAP System Manager"]




 a| 
xref:../concepts/path-failover-concept.html#nas-path-failover[LIF-Failover-Erweiterung]
 a| 
Die ONTAP systemdefinierte Failover-Gruppenrichtlinie erweitert die Failover-Ziele für NAS-Daten-LIFs dynamisch, sodass potenziell alle verfügbaren Ports in der Broadcast-Domäne auf jedem verfügbaren Knoten im Cluster einbezogen werden können. Sie können die systemdefinierte Richtlinie auf jede Failover-Gruppe anwenden. Vor ONTAP 9.19.1 konnten LIFs mit dieser Richtlinie nur auf einen anderen Knoten im Cluster ausweichen; wenn dieser Knoten nicht im Quorum war, fand kein Failover statt, selbst wenn andere Knoten verfügbar waren.

== San

[cols="30%,70%"]
|===
| Aktualisieren | Beschreibung 


 a| 
xref:../san-config/configure-fc-nvme-hosts-ha-pairs-reference.html[Unterstützung für FC-Konfigurationen mit direkter Anbindung]
 a| 
FC-Switches sind nicht erforderlich, um FC- und FC-NVMe-Hosts mit ONTAP-Speicher zu verbinden. Hosts können direkt an FC-Adapterports von AFF-, ASA- und FAS-Systemen angeschlossen werden. Diese direkte Verbindung ist besonders in Remote-Standorten und hybriden Umgebungen nützlich, wo Einfachheit wichtig ist.



 a| 
xref:../san-admin/multipath-automatic-lif-failover-support.html[Unterstützung für Aktiv/Aktiv Multipathing auf AFF-Systemen]
 a| 
AFF-Systeme unterstützen Aktiv/Aktiv-Multipathing in SAN-Umgebungen. In Aktiv/Aktiv-Konfigurationen sind alle Pfade innerhalb des HA-Paares, dem die LUN oder der Namespace gehört, aktiv optimiert, sodass der Host im Falle eines Storage-Failovers nicht auf den ALUA- oder ANA-Übergang der Failover-Pfade warten muss, um die E/A fortzusetzen. Dies verkürzt die Failover-Zeit und ermöglicht es SAN-Kunden, geschäftskritische Workloads auf AFF-Systemen mit derselben unterbrechungsfreien Pfadausfallsicherheit und demselben Lastausgleich auszuführen, die zuvor nur auf ASA-Systemen verfügbar waren.



 a| 
xref:../san-admin/multipath-automatic-lif-failover-support.html[Erweiterte Unterstützung für automatisches SAN-LIF-Failover]
 a| 
Automatisches LIF-Failover wird auf AFF und ASA Systemen für iSCSI-Protokoll-LIFs in Aktiv/Aktiv- und Aktiv/Aktiv-SCSI-Only-Multipath-Konfigurationen sowie für NVMe/TCP-Protokoll-LIFs in Aktiv/Aktiv-Multipath-Konfigurationen unterstützt.



 a| 
link:https://docs.netapp.com/us-en/ontap-fli/get-started/learn-about-data-migration.html["Unterstützung für den Import fremder LUNs für das iSCSI-Protokoll"^]
 a| 
Foreign LUN Import (FLI) unterstützt das iSCSI-Protokoll für die Backend-Konnektivität.

== Sicherheit

[cols="30%,70%"]
|===
| Aktualisieren | Beschreibung 


 a| 
ARP-Workload-Charakteristika aus der System Manager-Benutzeroberfläche entfernt
 a| 
System Manager zeigt die Merkmale der Arbeitslast des autonomen Ransomware-Schutzes (ARP), einschließlich Basis- und Spitzen-Statistiken, nicht mehr an. Diese Informationen wurden entfernt, da Spitzenaktivität häufig fälschlicherweise als tatsächlicher Ransomware-Angriff oder als etwas interpretiert wurde, das eine manuelle Anpassung erforderte.

Das ARP-Verhalten und die ARP-Erkennung bleiben unverändert, nur die Darstellung dieser Details im System Manager ist betroffen. Surge-Statistiken sind weiterhin über ONTAP CLI und REST API-Optionen für erweiterte Analysen verfügbar.



 a| 
xref:../anti-ransomware/use-cases-restrictions-concept.html#supported-configurations[Autonomer Ransomware-Schutz für SnapMirror synchron und SnapMirror Aktiv/Aktiv]
 a| 
ARP/AI unterstützt Volumes, die an SnapMirror synchron (NAS und SAN) und SnapMirror active sync SAN-Beziehungen auf unterstützten Plattformen teilnehmen. ARP/AI kann primäre Volumes in SnapMirror synchronen Beziehungen auf FAS-, AFF- und AFX-Systemen sowie primäre Volumes in SnapMirror active sync SAN-Beziehungen auf AFF- und ASA r2-Systemen schützen. ARP/AI läuft auf dem aktiven, schreib-lesefähigen primären Volume in der Beziehung, erstellt ausschließlich primäre ARP-Snapshots und bietet eine auf maschinellem Lernen basierende Ransomware-Erkennung.



 a| 
xref:../networking/configure-tls-offload.html[TLS-Offload-Konfiguration]
 a| 
Sie können die TLS-Auslagerung konfigurieren, um die TLS-Post-Handshake-Performance durch die Nutzung von Ressourcen auf unterstützten Ethernet-Karten zu verbessern. Diese Funktion lagert Verschlüsselung und Entschlüsselung aus, reduziert die CPU-Überlastung und verbessert die Performance.

== Storage-Effizienz

[cols="30%,70%"]
|===
| Aktualisieren | Beschreibung 


 a| 
xref:../flexcache/index.html[NAS-Performance-Verbesserung]
 a| 
ONTAP 9.19.1 stellt mehr CPU-Ressourcen für die Verarbeitung von Auflistungsanfragen kleiner NAS-Verzeichnisse bereit. Diese Verbesserung kommt Kunden zugute, deren Clients gleichzeitig Hunderte bis Tausende von Clients dasselbe kleine NAS-Verzeichnis mit einer Größe von maximal 2 MB (ca. 25.000 Dateien) auflisten.

Diese Verbesserung reduziert den Bedarf an Workarounds, wie lokalen FlexCache-Volumes, um hohe Client-Anzahlen bei Verzeichnisauflistungen zu unterstützen. Bei geringerer Client-Anzahl wird die Leistung nicht negativ beeinflusst. Diese Performance-Verbesserung ist automatisch und erfordert keine Konfiguration.

|===
