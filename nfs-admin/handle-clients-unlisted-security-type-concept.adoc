= ONTAP SVM-Zugriff für NFS-Clients mit nicht aufgeführten Sicherheitstypen verwalten
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Wenn ein Client einen Sicherheitstyp angibt, der nicht in einem Zugriffsparameter einer Exportregel aufgeführt ist, besteht die Möglichkeit, entweder den Zugriff für den Client zu verweigern oder ihn stattdessen der anonymen Benutzer-ID zuzuordnen, indem die Option  `none` im Zugriffsparameter verwendet wird.

Ein Client kann sich mit einem Sicherheitstyp melden, der nicht in einem Zugriffsparameter aufgeführt ist, weil er mit einem anderen Sicherheitstyp authentifiziert wurde oder überhaupt nicht authentifiziert wurde (Sicherheitstyp AUTH_NONE). Standardmäßig wird dem Client der Zugriff auf diese Ebene automatisch verweigert. Es besteht jedoch die Möglichkeit, die Option `none` zum Zugriffsparameter hinzuzufügen. Infolgedessen werden Clients mit einem nicht aufgeführten Sicherheitstyp stattdessen der anonymen Benutzer-ID zugeordnet. Der `-anon` Parameter bestimmt, welche Benutzer-ID diesen Clients zugewiesen wird. Die für den `-anon` Parameter angegebene Benutzer-ID muss ein gültiger Benutzer sein, der mit Berechtigungen konfiguriert ist, die als angemessen für den anonymen Benutzer gelten.

Gültige Werte für den  `-anon` Parameter liegen im Bereich von  `0` bis  `65535`.

[cols="2*"]
|===
| Benutzer-ID zugewiesen an `-anon` | Daraus resultierende Bearbeitung von Clientzugriffsanfragen 


 a| 
`0`- `65533`
 a| 
Die Clientzugriffsanfrage wird der anonymen Benutzer-ID zugeordnet und erhält Zugriff abhängig von den für diesen Benutzer konfigurierten Berechtigungen.



 a| 
`65534`
 a| 
Die Clientzugriffsanfrage wird dem Benutzer „nobody“ zugeordnet und erhält Zugriff abhängig von den für diesen Benutzer konfigurierten Berechtigungen. Dies ist die Standardeinstellung.



 a| 
`65535`
 a| 
Die Zugriffsanfrage eines beliebigen Clients wird abgelehnt, wenn sie dieser ID zugeordnet ist und der Client sich mit dem Sicherheitstyp AUTH_NONE präsentiert. Die Zugriffsanfrage von Clients mit der Benutzer-ID 0 wird abgelehnt, wenn sie dieser ID zugeordnet ist und der Client sich mit einem anderen Sicherheitstyp präsentiert.

|===
Bei Verwendung der Option `none` ist zu beachten, dass der schreibgeschützte Parameter zuerst verarbeitet wird. Folgende Richtlinien sind beim Konfigurieren von Exportregeln für Clients mit nicht aufgeführten Sicherheitstypen zu berücksichtigen:

[cols="3*"]
|===
| Schreibgeschützt umfasst `none` | Lesen/schreiben umfasst `none` | Ergebniszugriff für Clients mit nicht aufgeführten Sicherheitstypen 


 a| 
Nein
 a| 
Nein
 a| 
Verweigert



 a| 
Nein
 a| 
Ja
 a| 
Abgelehnt, da zuerst lesen verarbeitet wird



 a| 
Ja
 a| 
Nein
 a| 
Nur-Lesen als anonym



 a| 
Ja
 a| 
Ja
 a| 
Lesen/schreiben als anonymer Nutzer

|===
.Beispiele
Das folgende Beispiel zeigt eine Exportrichtlinie mit einem  `-rwrule` `any` Parameter:

Die Exportrichtlinie enthält eine Exportregel mit folgenden Parametern:

* `-protocol` `nfs3`
* `-clientmatch` `10.1.16.0/255.255.255.0`
* `-rorule` `sys,none`
* `-rwrule` `any`
* `-anon` `70`


Client #1 hat die IP-Adresse 10.1.16.207, sendet eine Zugriffsanfrage über das NFSv3-Protokoll und hat sich mit Kerberos v5 authentifiziert.

Client #2 hat die IP-Adresse 10.1.16.211, sendet eine Zugriffsanfrage mit dem NFSv3-Protokoll und hat sich mit AUTH_SYS authentifiziert.

Client #3 hat die IP-Adresse 10.1.16.234, sendet eine Zugriffsanfrage mit dem NFSv3-Protokoll und hat sich nicht authentifiziert (bedeutet Sicherheitstyp AUTH_NONE).

Das Clientzugriffsprotokoll und die IP-Adresse stimmen für alle drei Clients überein. Der Parameter „read-only“ ermöglicht Lesezugriff für Clients mit eigener Benutzerkennung, die sich mit AUTH_SYS authentifiziert haben. Der Parameter „read-only“ ermöglicht Lesezugriff als anonymer Benutzer mit der Benutzerkennung 70 für Clients, die sich mit einem anderen Sicherheitstyp authentifiziert haben. Der Parameter „read-write“ ermöglicht Lese- und Schreibzugriff für alle Sicherheitstypen, gilt in diesem Fall jedoch nur für Clients, die bereits durch die read-only-Regel gefiltert wurden.

Daher erhalten die Clients #1 und #3 Lese- und Schreibzugriff nur als anonymer Benutzer mit der Benutzer-ID 70. Client #2 erhält Lese- und Schreibzugriff mit seiner eigenen Benutzer-ID.

Das folgende Beispiel zeigt eine Exportrichtlinie mit einem  `-rwrule` `none` Parameter:

Die Exportrichtlinie enthält eine Exportregel mit folgenden Parametern:

* `-protocol` `nfs3`
* `-clientmatch` `10.1.16.0/255.255.255.0`
* `-rorule` `sys,none`
* `-rwrule` `none`
* `-anon` `70`


Client #1 hat die IP-Adresse 10.1.16.207, sendet eine Zugriffsanfrage über das NFSv3-Protokoll und hat sich mit Kerberos v5 authentifiziert.

Client #2 hat die IP-Adresse 10.1.16.211, sendet eine Zugriffsanfrage mit dem NFSv3-Protokoll und hat sich mit AUTH_SYS authentifiziert.

Client #3 hat die IP-Adresse 10.1.16.234, sendet eine Zugriffsanfrage mit dem NFSv3-Protokoll und hat sich nicht authentifiziert (bedeutet Sicherheitstyp AUTH_NONE).

Das Clientzugriffsprotokoll und die IP-Adresse stimmen für alle drei Clients überein. Der Parameter „read-only“ ermöglicht Lesezugriff für Clients mit eigener Benutzerkennung, die sich mit AUTH_SYS authentifiziert haben. Der Parameter „read-only“ ermöglicht Lesezugriff als anonymer Benutzer mit der Benutzerkennung 70 für Clients, die sich mit einem anderen Sicherheitstyp authentifiziert haben. Der Parameter „read-write“ ermöglicht Lese-/Schreibzugriff ausschließlich als anonymer Benutzer.

Daher erhalten Client #1 und Client #3 nur als anonymer Benutzer mit der Benutzer-ID 70 Lese- und Schreibzugriff. Client #2 erhält mit seiner eigenen Benutzer-ID Lesezugriff, ihm wird jedoch der Lese- und Schreibzugriff verweigert.
