= Kommunikationsprozess zwischen Knoten und externem ONTAP FPolicy Server
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Für eine optimale Planung Ihrer FPolicy Konfiguration ist es wichtig, den Kommunikationsprozess zwischen dem Knoten und dem externen FPolicy Server zu verstehen.

Jeder Knoten, der an einer Storage Virtual Machine (SVM) beteiligt ist, initiiert eine Verbindung zu einem externen FPolicy Server (FPolicy Server) über TCP/IP. Verbindungen zu den FPolicy Servern werden über Knotendaten-LIFs eingerichtet, daher kann ein beteiligter Knoten nur dann eine Verbindung herstellen, wenn er über einen betriebsbereiten Daten-LIF für die SVM verfügt.

Jeder FPolicy-Prozess auf den teilnehmenden Knoten versucht, eine Verbindung zum FPolicy Server herzustellen, wenn die Richtlinie aktiviert ist. Dabei werden die in der Richtlinienkonfiguration angegebene IP-Adresse und der Port der FPolicy External Engine verwendet.

Die Verbindung stellt einen Steuerkanal von jedem an jedem SVM beteiligten Knoten zum FPolicy Server über die Daten-LIF her. Sind auf demselben beteiligten Knoten sowohl IPv4- als auch IPv6-Daten-LIF-Adressen vorhanden, versucht FPolicy, Verbindungen sowohl für IPv4 als auch für IPv6 herzustellen. Daher muss der FPolicy Server in einem Szenario, in dem sich das SVM über mehrere Knoten erstreckt oder sowohl IPv4- als auch IPv6-Adressen vorhanden sind, nach der Aktivierung der FPolicy Policy auf dem SVM auf mehrere Anfragen zum Aufbau von Steuerkanälen aus dem Cluster vorbereitet sein.

Besteht ein Cluster beispielsweise aus drei Knoten, Node1, Node2 und Node3, und sind die SVM-Daten-LIFs nur auf Node2 und Node3 verteilt, werden Kontrollkanäle unabhängig von der Verteilung der Daten-Volumes ausschließlich von Node2 und Node3 initiiert. Angenommen, Node2 verfügt über zwei Daten-LIFs, LIF1 und LIF2, die zur SVM gehören, und die initiale Verbindung erfolgt über LIF1. Fällt LIF1 aus, versucht FPolicy, einen Kontrollkanal über LIF2 herzustellen.

[[how-fpolicy-manages-external-communication-during-lif-migration-or-failover]]
== Wie FPolicy die externe Kommunikation während einer LIF-Migration oder eines Failovers verwaltet

Daten-LIFs können auf Datenports im selben Knoten oder auf Datenports auf einem entfernten Knoten migriert werden.

Wenn ein Daten-LIF ausfällt oder migriert wird, wird eine neue Steuerkanalverbindung zum FPolicy-Server hergestellt. FPolicy kann dann SMB- und NFS-Client-Anfragen, bei denen eine Zeitüberschreitung aufgetreten ist, erneut versuchen, sodass neue Benachrichtigungen an die externen FPolicy-Server gesendet werden. Der Node weist FPolicy-Serverantworten auf ursprüngliche, aufgrund von Zeitüberschreitung abgelaufene SMB- und NFS-Anfragen zurück.



[[how-fpolicy-manages-external-communication-during-node-failover]]
== Wie FPolicy die externe Kommunikation während eines Node-Failovers verwaltet

Wenn der Clusterknoten, der die für die FPolicy-Kommunikation verwendeten Datenports hostet, ausfällt, unterbricht ONTAP die Verbindung zwischen dem FPolicy-Server und dem Knoten.

Die Auswirkungen eines Cluster-Failovers auf den FPolicy Server lassen sich abmildern, indem die Failover-Policy so konfiguriert wird, dass der für die FPolicy-Kommunikation verwendete Datenport auf einen anderen aktiven Knoten migriert wird. Nach Abschluss der Migration wird eine neue Verbindung über den neuen Datenport hergestellt.

Wenn die Failover-Richtlinie nicht für die Migration des Datenports konfiguriert ist, muss der FPolicy Server warten, bis der ausgefallene Knoten wieder verfügbar ist. Nachdem der Knoten wieder verfügbar ist, wird von diesem Knoten aus eine neue Verbindung mit einer neuen Session ID initiiert.

[NOTE]
====
Der FPolicy Server erkennt Verbindungsabbrüche mithilfe der Keep-Alive-Protokollnachricht. Die Zeitüberschreitung für das Entfernen der Session ID wird bei der Konfiguration von FPolicy festgelegt. Die Standard-Zeitüberschreitung beträgt zwei Minuten.
