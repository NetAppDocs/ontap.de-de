= Die Überwachungskonfiguration auf ONTAP SVMs planen
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Bevor Sie die Überwachung auf Storage Virtual Machines (SVMs) konfigurieren, ist es erforderlich, die verfügbaren Konfigurationsoptionen zu verstehen und die Werte zu planen, die für jede Option festgelegt werden sollen. Diese Informationen können dabei unterstützen, die Überwachungskonfiguration so zu gestalten, dass sie den geschäftlichen Anforderungen entspricht.

Es gibt bestimmte Konfigurationsparameter, die allen Audit-Konfigurationen gemeinsam sind.

Darüber hinaus gibt es bestimmte Parameter, mit denen Sie festlegen können, welche Methoden beim Rotieren der konsolidierten und konvertierten Audit-Logs verwendet werden. Bei der Konfiguration der Überwachung kann eine der folgenden drei Methoden festgelegt werden:

* Protokolle basierend auf der Protokollgröße rotieren
+
Dies ist die Standardmethode zum Rotieren von Protokolldateien.

* Protokolle gemäß einem Zeitplan rotieren
* Protokolldateien werden basierend auf der Protokolldateigröße und dem Zeitplan rotiert (je nachdem, welches Ereignis zuerst eintritt).


[NOTE]
====
Mindestens eine der Methoden zur Logrotation muss immer eingestellt sein.

====

_Protokollzielpfad_

Gibt das Verzeichnis an, in dem die konvertierten Audit-Logs gespeichert werden, typischerweise ein dediziertes Volume oder ein qtree. Der Pfad muss bereits im SVM-Namensraum vorhanden sein.

Der Pfad darf maximal 864 Zeichen lang sein und muss über Lese-/Schreibberechtigung verfügen.

Ist der Pfad ungültig, schlägt der Revisionsdatei-Konfigurationsbefehl fehl.

Wenn es sich bei der SVM um eine SVM Disaster Recovery-Quelle handelt, darf der Protokollzielpfad nicht auf dem Root-Volume liegen. Dies ist darauf zurückzuführen, dass der Inhalt des Root-Volumes nicht auf das Disaster Recovery-Ziel repliziert wird.

Es kann kein FlexCache Volume als Protokollziel verwendet werden (ONTAP 9.7 und höher).

 a| 
_Kategorien von zu prüfenden Ereignissen_

Legt die Kategorien der zu prüfenden Ereignisse fest. Die folgenden Ereigniskategorien können im Revisionsprotokoll erfasst werden:

* Dateizugriffsereignisse (sowohl SMB als auch NFSv4)
* SMB-Anmelde- und Abmeldeereignisse
* Staging-Ereignisse für zentrale Zugriffsrichtlinien
+
Zentrale Zugriffsrichtlinien-Staging-Ereignisse sind ab Windows 2012 Active Directory Domains verfügbar.

* Asynchrones Löschen
* Ereignisse der Kategorie „Dateifreigabe“
* Ereignisse bei Änderungen der Prüfungsrichtlinie
* Lokale Benutzerkontenverwaltungsereignisse
* Sicherheitsgruppen-Managementereignisse
* Ereignisse bei Änderungen der Autorisierungsrichtlinie


Standardmäßig werden der Dateizugriff und SMB-Anmelde- sowie Abmeldeereignisse protokolliert.

*Hinweis:* Bevor Sie  `cap-staging` als Ereigniskategorie angeben können, muss auf der SVM ein SMB-Server vorhanden sein. Obwohl die zentrale Zugriffsrichtlinienbereitstellung in der Überwachungskonfiguration aktiviert werden kann, ohne dass Dynamic Access Control auf dem SMB-Server aktiviert ist, werden Ereignisse zur zentralen Zugriffsrichtlinienbereitstellung nur generiert, wenn Dynamic Access Control aktiviert ist. Dynamic Access Control wird über eine SMB-Serveroption aktiviert. Standardmäßig ist sie nicht aktiviert.

Legt das Ausgabeformat der Revisionsprotokolle fest. Das Ausgabeformat kann entweder ONTAP-spezifisch `XML` oder Microsoft Windows `EVTX` Protokollformat sein. Standardmäßig ist das Ausgabeformat `EVTX`.

Legt fest, wie viele Revisionsprotokoll-Logdateien aufbewahrt werden, bevor die älteste Logdatei entfernt wird. Wenn beispielsweise der Wert `5` eingegeben wird, werden die letzten fünf Logdateien aufbewahrt.

Ein Wert von `0` zeigt an, dass alle Protokolldateien beibehalten werden. Der Standardwert ist 0.

[[parameters-used-for-determining-when-to-rotate-audit-event-logs]]
== Parameter zur Bestimmung des Rotationszeitpunkts von Revisionsprotokollen für Ereignisse

*Protokolle basierend auf der Protokollgröße rotieren*

Standardmäßig werden Revisionsprotokolle basierend auf Größe rotiert.

* Die Standard-Protokollgröße beträgt 100 MB
* Wenn Sie die Standardmethode für die Protokollrotation und die Standardprotokollgröße verwenden, ist keine spezifische Konfiguration von Parametern für die Protokollrotation erforderlich.
* Wenn die Revisionsprotokolle ausschließlich basierend auf der Protokollgröße rotiert werden sollen, kann der folgende Befehl verwendet werden, um den  `-rotate-schedule-minute`Parameter zu deaktivieren: `vserver audit modify -vserver vs0 -destination / -rotate-schedule-minute -`


Wenn Sie nicht die Standardprotokollgröße verwenden möchten, kann der  `-rotate-size` Parameter so konfiguriert werden, dass eine benutzerdefinierte Protokollgröße festgelegt wird:

[cols="40,30,10,10,10"]
|===


| Art der Information | Option | Erforderlich | Enthalten | Ihre Werte 


 a| 
_Beschränkung der Protokolldateigröße_

Legt das Größenlimit der Revisionsprotokolldatei fest.
 a| 
`-rotate-size` {`integer`[KB|MB|GB|TB|PB]}
 a| 
Nein
 a| 
 a| 

|===
*Protokolle gemäß einem Zeitplan rotieren*

Wenn Sie sich dafür entscheiden, die Revisionsprotokolle nach einem Zeitplan zu rotieren, kann die Protokollrotation mithilfe der zeitbasierten Rotationsparameter in beliebiger Kombination geplant werden.

* Bei Verwendung einer zeitbasierten Rotation ist der  `-rotate-schedule-minute` Parameter obligatorisch.
* Alle anderen zeitbasierten Rotationsparameter sind optional.
* Der Rotationsplan wird anhand aller zeitbezogenen Werte berechnet.
+
Wenn nur der  `-rotate-schedule-minute` Parameter angegeben wird, werden die Revisionsprotokoll-Dateien an allen Wochentagen, zu allen Stunden und in allen Monaten des Jahres entsprechend der angegebenen Minuten rotiert.

* Wenn Sie nur einen oder zwei zeitbasierten Rotationsparameter angeben (zum Beispiel  `-rotate-schedule-month` und  `-rotate-schedule-minutes`), erfolgt die Rotation der Protokolldateien an allen Wochentagen und zu allen Stunden auf Grundlage der angegebenen Minutenwerte, jedoch nur während der festgelegten Monate.
+
Beispielsweise kann festgelegt werden, dass das Revisionsprotokoll in den Monaten Januar, März und August jeweils montags, mittwochs und samstags um 10:30 Uhr rotiert wird.

* Wenn Sie Werte für sowohl  `-rotate-schedule-dayofweek` als auch  `-rotate-schedule-day` angeben, werden diese unabhängig voneinander betrachtet.
+
Wenn beispielsweise  `-rotate-schedule-dayofweek` als Freitag und  `-rotate-schedule-day` als 13 angegeben werden, werden die Revisionsprotokolle an jedem Freitag und am 13. Tag des angegebenen Monats rotiert, nicht nur an jedem Freitag, der auf den 13. fällt.

* Wenn die Revisionsprotokolle ausschließlich nach einem Zeitplan rotiert werden sollen, kann der folgende Befehl verwendet werden, um den  `-rotate-size` Parameter zu deaktivieren: `vserver audit modify -vserver vs0 -destination / -rotate-size -`


Die folgende Liste der verfügbaren Überwachungsparameter bietet eine Übersicht über die Werte, die für die Konfiguration eines Zeitplans für die Rotation des Revisionsprotokolls verwendet werden können:

[cols="40,30,10,10,10"]
|===


| Art der Information | Option | Erforderlich | Enthalten | Ihre Werte 


 a| 
_Protokollrotationsplan: Monat_

Bestimmt den monatlichen Zeitplan für die Rotation der Revisionsprotokolle.

Gültige Werte sind  `January` bis  `December` und  `all`. Zum Beispiel kann angegeben werden, dass das Revisionsprotokoll in den Monaten Januar, März und August rotiert wird.
 a| 
`-rotate-schedule-month` `chron_month`
 a| 
Nein
 a| 
 a| 



 a| 
_Rotationsplan für das Protokoll: Wochentag_

Legt den täglichen (Wochentag) Zeitplan für die Rotation der Revisionsprotokolle fest.

Gültige Werte sind  `Sunday` bis  `Saturday` und  `all`. Beispielsweise kann angegeben werden, dass das Revisionsprotokoll dienstags und freitags oder an allen Tagen einer Woche rotiert wird.
 a| 
`-rotate-schedule-dayofweek` `chron_dayofweek`
 a| 
Nein
 a| 
 a| 



 a| 
_Rotationsplan für das Protokoll: Tag_

Legt den Tag des Monats fest, an dem das Revisionsprotokoll rotiert wird.

Gültige Werte reichen von `1` bis `31`. Beispielsweise kann angegeben werden, dass das Revisionsprotokoll am 10. und 20. Tag eines Monats oder an allen Tagen eines Monats rotiert wird.
 a| 
`-rotate-schedule-day` `chron_dayofmonth`
 a| 
Nein
 a| 
 a| 



 a| 
_Rotationsplan für das Protokoll: Stunde_

Bestimmt den stündlichen Zeitplan für die Rotation des Revisionsprotokolls.

Gültige Werte liegen zwischen  `0` (Mitternacht) und  `23` (23:00 Uhr). Die Angabe von  `all` bewirkt, dass die Revisionsprotokolle stündlich rotiert werden. Beispielsweise kann angegeben werden, dass das Revisionsprotokoll um 6 (6:00 Uhr) und 18 (18:00 Uhr) rotiert wird.
 a| 
`-rotate-schedule-hour` `chron_hour`
 a| 
Nein
 a| 
 a| 



 a| 
_Protokollrotationsplan: Minute_

Legt den Minutenplan für die Rotation des Revisionsprotokolls fest.

Gültige Werte reichen von `0` bis `59`. Beispielsweise kann angegeben werden, dass das Revisionsprotokoll zur 30. Minute rotiert wird.
 a| 
`-rotate-schedule-minute` `chron_minute`
 a| 
Ja, wenn die zeitplanbasierte Protokollrotation konfiguriert wird; andernfalls nein.

|===
*Protokolle basierend auf Protokollgröße und Zeitplan rotieren*

Sie können die Rotation der Protokolldateien anhand der Protokolldateigröße und eines Zeitplans steuern, indem sowohl der  `-rotate-size` Parameter als auch die zeitbasierten Rotationsparameter in beliebiger Kombination gesetzt werden. Beispielsweise: Wenn  `-rotate-size` auf 10 MB und  `-rotate-schedule-minute` auf 15 gesetzt ist, werden die Protokolldateien rotiert, sobald die Protokolldateigröße 10 MB erreicht oder in der 15. Minute jeder Stunde (je nachdem, welches Ereignis zuerst eintritt).
