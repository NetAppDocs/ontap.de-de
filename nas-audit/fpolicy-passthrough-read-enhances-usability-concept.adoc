= Wie ONTAP FPolicy Passthrough-Read die Benutzerfreundlichkeit für hierarchisches Speichermanagement verbessert
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Die Passthrough-read-Funktion ermöglicht dem FPolicy Server (der als hierarchischer Storage-Management-Server (HSM) fungiert), Lesezugriff auf Offline-Dateien zu bieten, ohne die Datei vom sekundären Storage-System auf das primäres Storage-System zurückholen zu müssen.

Wenn ein FPolicy-Server für die Bereitstellung von HSM für Dateien auf einem SMB-Server konfiguriert ist, erfolgt eine richtlinienbasierte Dateimigration, bei der die Dateien offline auf sekundärem Storage abgelegt werden und auf dem primären Storage-System lediglich eine Platzhalterdatei verbleibt. Obwohl diese Platzhalterdatei für Clients wie eine normale Datei erscheint, handelt es sich tatsächlich um eine Sparse-Datei mit der gleichen Größe wie die Originaldatei. Die Sparse-Datei hat das SMB-Offline-Bit gesetzt und verweist auf die tatsächliche Datei, die auf den sekundären Storage migriert wurde.

Üblicherweise muss, wenn eine Leseanfrage für eine Offline-Datei empfangen wird, der angeforderte Inhalt zurück in das primäre Storage-System geholt und anschließend über das primäre Storage-System abgerufen werden. Die Notwendigkeit, Daten zurück in das primäre Storage-System zu holen, hat mehrere unerwünschte Auswirkungen. Zu den unerwünschten Auswirkungen gehören die erhöhte Latenz bei Clientanfragen, die durch das Zurückholen des Inhalts vor der Beantwortung der Anfrage entsteht, sowie der erhöhte Speicherplatzbedarf für zurückgeholte Dateien auf dem primären Storage-System.

FPolicy passthrough-read ermöglicht es dem HSM-Server (dem FPolicy-Server), Lesezugriff auf migrierte, offline Dateien zu gewähren, ohne die Datei vom sekundären Storage-System auf das primäre Storage-System zurückholen zu müssen. Anstatt die Dateien auf das primäre Storage-System zurückzuholen, können Leseanfragen direkt vom sekundären Storage-System bedient werden.

[NOTE]
====
Copy Offload (ODX) wird bei der FPolicy Passthrough-Read-Operation nicht unterstützt.

====
Passthrough-read verbessert die Benutzerfreundlichkeit durch folgende Vorteile:

* Leseanfragen können auch dann bearbeitet werden, wenn das primäre Storage-System nicht über genügend Speicherplatz verfügt, um die angeforderten Daten zurück in das primäre Storage-System zu übertragen.
* Besseres Kapazitäts- und Leistungsmanagement bei einem möglichen Anstieg des Datenabrufs, beispielsweise wenn ein Skript oder eine Backup-Lösung auf viele Offline-Dateien zugreifen muss.
* Leseanforderungen für Offline-Dateien in Snapshots können bedient werden.
+
Da Snapshots schreibgeschützt sind, kann der FPolicy Server die Originaldatei nicht wiederherstellen, wenn sich die Stub-Datei in einem Snapshot befindet. Die Verwendung von passthrough-read eliminiert dieses Problem.

* Es können Richtlinien eingerichtet werden, die steuern, wann Leseanfragen durch Zugriff auf die Datei im sekundären Storage bedient werden und wann die Offline-Datei in das primäre Storage-System zurückgerufen werden soll.
+
Auf dem HSM-Server kann beispielsweise eine Richtlinie erstellt werden, die festlegt, wie oft innerhalb eines bestimmten Zeitraums auf eine Offline-Datei zugegriffen werden darf, bevor diese zurück in das primäre Storage-System migriert wird. Diese Art von Richtlinie vermeidet das erneute Abrufen von Dateien, auf die selten zugegriffen wird.





[[how-read-requests-are-managed-when-fpolicy-passthrough-read-is-enabled]]
== Wie Leseanfragen verwaltet werden, wenn FPolicy passthrough-read aktiviert ist

Es ist wichtig zu verstehen, wie Leseanforderungen verwaltet werden, wenn FPolicy passthrough-read aktiviert ist, damit die Konnektivität zwischen der Storage Virtual Machine (SVM) und den FPolicy Servern optimal konfiguriert werden kann.

Wenn FPolicy passthrough-read aktiviert ist und die SVM eine Anfrage für eine Offline-Datei empfängt, sendet FPolicy eine Benachrichtigung an den FPolicy-Server (HSM-Server) über den Standardverbindungskanal.

Nach Erhalt der Benachrichtigung liest der FPolicy Server die Daten aus dem in der Benachrichtigung angegebenen Dateipfad und sendet die angeforderten Daten über die zwischen dem SVM und dem FPolicy Server eingerichtete privilegierte Passthrough-Read-Datenverbindung an die SVM.

Nach dem Senden der Daten antwortet der FPolicy Server auf die Leseanfrage mit ALLOW oder DENY. Je nachdem, ob die Leseanfrage zugelassen oder abgelehnt wird, sendet ONTAP entweder die angeforderten Informationen oder eine Fehlermeldung an den Client.
