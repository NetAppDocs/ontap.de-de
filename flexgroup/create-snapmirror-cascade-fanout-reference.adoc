= Überlegungen zur Erstellung von SnapMirror Kaskaden- und Verzweigungsbeziehungen für ONTAP FlexGroup Volumes
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Es sind Unterstützungsaspekte und Einschränkungen zu berücksichtigen, wenn SnapMirror Kaskaden- und Fanout-Beziehungen für FlexGroup Volumes erstellt werden.



[[considerations-for-creating-cascading-relationships]]
== Überlegungen zur Erstellung kaskadierender Beziehungen

* Jede Beziehung kann entweder eine Inter-Cluster- oder eine Intra-Cluster-Beziehung sein.
* Alle asynchronen Richtlinientypen, einschließlich async-mirror, mirror-vault und vault, werden für beide Beziehungen unterstützt.
* Es werden nur „MirrorAllSnapshots“- und nicht „MirrorLatest“-Richtlinien für asynchrone Spiegelung unterstützt.
* Langzeit-Snapshots werden nicht unterstützt.
+
Weitere Informationen zu link:../data-protection/long-term-retention-snapshots-concept.html["Langzeitaufbewahrungs-Snapshots"].

* Gleichzeitige Aktualisierungen von kaskadierten XDP-Beziehungen sind unterstützt.
* Unterstützt das Entfernen von A zu B und B zu C sowie das erneute Synchronisieren von A zu C oder von C zu A.
* A- und B-FlexGroup Volumes unterstützen ebenfalls Fanout, wenn auf allen Knoten ONTAP 9.9.1 oder höher ausgeführt wird.
* Wiederherstellungsvorgänge von B oder C FlexGroup Volumes werden unterstützt.
* Transfers auf FlexGroup Beziehungen werden nicht unterstützt, während das Ziel die Quelle einer Restore-Beziehung ist.
* Das Ziel einer FlexGroup Wiederherstellung kann nicht das Ziel einer anderen FlexGroup Beziehung sein.
* FlexGroup Dateiwiederherstellungsvorgänge unterliegen den gleichen Einschränkungen wie reguläre FlexGroup Wiederherstellungsvorgänge.
* Alle Knoten im Cluster, auf denen sich die B- und C-FlexGroup Volumes befinden, müssen ONTAP 9.9.1 oder höher ausführen.
* Alle Funktionen zum Erweitern und automatischen Erweitern werden unterstützt.
* In einer Kaskadenkonfiguration wie A zu B zu C ist ein Abbruchvorgang von der Quelle für die B zu C SnapMirror Beziehung nicht unterstützt, wenn A zu B und B zu C eine unterschiedliche Anzahl von SnapMirror Beziehungen aufweisen.
* System Manager unterstützt keine kaskadierenden Beziehungen, unabhängig von der ONTAP Version.
* Bei der Umwandlung einer A zu B zu C FlexVol Beziehung in eine FlexGroup Beziehung ist zunächst der B zu C Schritt umzuwandeln.
* Alle FlexGroup Kaskadenkonfigurationen für Beziehungen mit Richtlinientypen, die von REST unterstützt werden, werden auch von REST-APIs in Kaskadenkonfigurationen mit FlexGroup unterstützt.
* Wie bei FlexVol Beziehungen wird FlexGroup Kaskadierung von dem  `snapmirror protect` Befehl nicht unterstützt.




[[considerations-for-creating-fanout-relationships]]
== Überlegungen zur Erstellung von Fanout-Beziehungen

* Zwei oder mehr FlexGroup Fanout-Beziehungen werden unterstützt; zum Beispiel A nach B, A nach C, mit maximal 8 Fanout-Beinen.
* Jede Beziehung kann entweder intercluster oder intracluster sein.
* Gleichzeitige Aktualisierungen werden für die beiden Beziehungen unterstützt.
* Alle Funktionen zum Erweitern und automatischen Erweitern werden unterstützt.
* Wenn die Zweige der Beziehung eine unterschiedliche Anzahl von SnapMirror Beziehungen aufweisen, wird ein Abbruchvorgang vom Quell-System für die Beziehungen A zu B und A zu C nicht unterstützt.
* Alle Knoten im Cluster, auf denen sich die Quell- und Ziel-FlexGroup Volumes befinden, müssen ONTAP 9.9.1 oder höher ausführen.
* Alle aktuell unterstützten asynchronen Richtlinientypen für FlexGroup SnapMirror werden auch in Fanout-Beziehungen unterstützt.
* Wiederherstellungsvorgänge von Volume B auf Volume C FlexGroup Volumes sind möglich.
* Alle Fanout-Konfigurationen mit von REST unterstützten Richtlinientypen werden auch für REST-APIs in FlexGroup Fanout-Konfigurationen unterstützt.
