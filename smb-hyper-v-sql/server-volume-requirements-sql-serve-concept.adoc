= SMB-Server- und Volume-Anforderungen für SQL Server über SMB
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Bei der Erstellung von SQL Server over SMB Konfigurationen für unterbrechungsfreie Operationen sind bestimmte Anforderungen an den SMB-Server und das Volume zu berücksichtigen.



[[smb-server-requirements]]
== Anforderungen an SMB Server

* SMB 3.0 muss aktiviert sein.
+
Diese Funktion ist standardmäßig aktiviert.

* Die Standard-UNIX-Benutzeroption für den CIFS-Server muss mit einem gültigen UNIX-Benutzerkonto konfiguriert sein.
+
Die Anwendungsserver verwenden das Computerkonto beim Aufbau einer SMB-Verbindung. Da jeder SMB-Zugriff voraussetzt, dass der Windows-Benutzer erfolgreich einem UNIX-Benutzerkonto oder dem Standard-UNIX-Benutzerkonto zugeordnet wird, muss ONTAP das Computerkonto des Anwendungsservers dem Standard-UNIX-Benutzerkonto zuordnen können.

+
Darüber hinaus verwendet SQL Server einen Domänenbenutzer als SQL Server-Dienstkonto. Das Dienstkonto muss außerdem dem Standard-UNIX-Benutzer zugeordnet sein.

* Automatische Knotenweiterleitungen müssen deaktiviert sein (diese Funktion ist standardmäßig deaktiviert).
+
Wenn automatische Knotenverweise für den Zugriff auf andere Daten als SQL Server Datenbankdateien verwendet werden sollen, muss eine separate SVM für diese Daten erstellt werden.

* Dem Windows-Benutzerkonto, das für die Installation von SQL Server auf ONTAP verwendet wird, muss die SeSecurityPrivilege-Berechtigung zugewiesen werden.
+
Diese Berechtigung ist der lokalen Gruppe BUILTIN\Administrators des SMB-Servers zugewiesen.





[[volume-requirements]]
== Volumenanforderungen

* Volumes, die zur Speicherung von Dateien virtueller Maschinen verwendet werden, müssen als NTFS Security-Style-Volumes erstellt werden.
+
Um NDOs für Anwendungsserver mit permanent verfügbaren SMB-Verbindungen bereitzustellen, muss das Volume, das die Freigabe enthält, ein NTFS-Volume sein. Es muss außerdem immer ein NTFS-Volume gewesen sein. Ein Volume mit gemischtem Sicherheitsstil oder UNIX-Sicherheitsstil kann nicht in ein NTFS-Sicherheitsstil-Volume umgewandelt und direkt für NDOs über SMB-Freigaben verwendet werden. Wenn ein Volume mit gemischtem Sicherheitsstil in ein NTFS-Sicherheitsstil-Volume geändert wird und für NDOs über SMB-Freigaben genutzt werden soll, muss eine ACL manuell am Anfang des Volumes gesetzt und auf alle enthaltenen Dateien und Ordner übertragen werden. Andernfalls können Migrationen virtueller Maschinen oder Datenbankdateiexporte und -importe, bei denen Dateien auf ein anderes Volume verschoben werden, fehlschlagen, wenn entweder das Quell- oder das Zielvolume ursprünglich als Volume mit gemischtem oder UNIX-Sicherheitsstil erstellt und später in den NTFS-Sicherheitsstil geändert wurde.

* Obwohl das Volume, das die Datenbankdateien enthält, Verbindungen enthalten kann, überschreitet SQL Server beim Erstellen der Datenbankverzeichnisstruktur keine Verbindungen.
* Für erfolgreiche Sicherungsvorgänge des SnapCenter Plug-ins für Microsoft SQL Server muss auf dem Volume genügend Speicherplatz verfügbar sein.
+
Das Volume, auf dem die SQL Server Datenbankdateien gespeichert sind, muss groß genug sein, um die Datenbank Verzeichnisstruktur und alle darin enthaltenen Dateien innerhalb der Freigabe aufzunehmen.
