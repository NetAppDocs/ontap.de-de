= Wie die SMB 3.0-Funktionalität unterbrechungsfreie Vorgänge über SMB-Freigaben unterstützt
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
SMB 3.0 bietet wichtige Funktionen, die den unterbrechungsfreien Betrieb von Hyper-V und SQL Server über SMB-Freigaben ermöglichen. Dazu gehören die  `continuously-available` Freigabeeigenschaft und ein Dateihandle-Typ, der als „persistentes Handle“ bezeichnet wird, die es SMB-Clients ermöglichen, den Öffnungsstatus von Dateien wiederherzustellen und SMB-Verbindungen transparent neu aufzubauen.

Persistente Handles können SMB 3.0-fähigen Clients zugewiesen werden, die eine Verbindung zu einer Freigabe mit der Eigenschaft „continuously available“ herstellen. Wenn die SMB-Sitzung getrennt wird, behält der CIFS-Server Informationen über den Status des persistenten Handles. Der CIFS-Server blockiert während des 60-Sekunden-Zeitraums, in dem der Client die Verbindung wiederherstellen darf, andere Client-Anfragen, sodass der Client mit dem persistenten Handle das Handle nach einer Netzwerkunterbrechung zurückerlangen kann. Clients mit persistenten Handles können die Verbindung über eine der Daten-LIFs auf der Storage Virtual Machine (SVM) wiederherstellen, entweder durch Wiederverbindung über dieselbe LIF oder über eine andere LIF.

Aggregate-Verschiebung, Übernahme und Rückgabe erfolgen alle zwischen SFO-Paaren. Um das Trennen und Wiederherstellen von Sitzungen mit Dateien, die über persistente Handles verfügen, nahtlos zu verwalten, hält der Partnerknoten eine Kopie aller Sperrinformationen für persistente Handles vor. Unabhängig davon, ob das Ereignis geplant oder ungeplant ist, kann der SFO-Partner die Wiederherstellung persistenter Handles unterbrechungsfrei verwalten. Mit dieser neuen Funktionalität können SMB 3.0-Verbindungen zum CIFS-Server transparent und unterbrechungsfrei auf ein anderes, dem SVM zugewiesenes Daten-LIF übergehen, auch in Situationen, die traditionell zu Unterbrechungen geführt haben.

Obwohl die Verwendung persistenter Handles dem CIFS-Server ein transparentes Failover von SMB 3.0-Verbindungen ermöglicht, hat der Client keine Möglichkeit, die Datei-Handles dieser getrennten Handles zurückzufordern, wenn ein Ausfall dazu führt, dass die Hyper-V-Applikation auf einen anderen Knoten im Windows Server-Cluster ausweicht. In diesem Szenario können Datei-Handles im getrennten Zustand potenziell den Zugriff der Hyper-V-Applikation blockieren, wenn sie auf einem anderen Knoten neu gestartet wird. "`Failover Clustering`" ist ein Teil von SMB 3.0, der dieses Szenario adressiert, indem er einen Mechanismus zur Invalidierung veralteter, kollidierender Handles bereitstellt. Mit diesem Mechanismus kann sich ein Hyper-V-Cluster schnell erholen, wenn Hyper-V-Clusterknoten ausfallen.
