= Informationen zu ONTAP Konsistenzgruppen
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Eine Konsistenzgruppe ist eine Sammlung von Volumes, die als eine Einheit verwaltet werden. In ONTAP ermöglichen Konsistenzgruppen eine einfache Verwaltung und gewährleisten den Schutz von Anwendungsworkloads, die sich über mehrere Volumes erstrecken.

Sie können Konsistenzgruppen verwenden, um Ihre Speicherverwaltung zu vereinfachen. Stellen Sie sich vor, Sie haben eine wichtige Datenbank, die sich über zwanzig LUNs erstreckt. Die LUNs könnten einzeln verwaltet oder als ein einziger Datensatz behandelt und in einer einzigen Konsistenzgruppe organisiert werden.

Konsistenzgruppen erleichtern das Management von Anwendungsworkloads, indem sie einfach konfigurierbare lokale und Remote-Datensicherungsstrategien sowie gleichzeitige absturzkonsistente oder applikationskonsistente Snapshots einer Sammlung von Volumes zu einem bestimmten Zeitpunkt bereitstellen. Snapshots einer Konsistenzgruppe ermöglichen die Wiederherstellung eines gesamten Anwendungsworkloads.



[[learn-about-consistency-groups]]
== Informationen zu Konsistenzgruppen

Konsistenzgruppen unterstützen beliebige FlexVol volume unabhängig vom Protokoll (NAS, SAN oder NVMe) und können über die ONTAP REST API oder im System Manager unter dem Menüpunkt *Storage > Consistency Groups* verwaltet werden. Ab ONTAP 9.14.1 können Konsistenzgruppen mit der ONTAP CLI verwaltet werden.

Konsistenzgruppen können als einzelne Einheiten, als Sammlung von Volumes, oder in einer hierarchischen Beziehung existieren, die aus anderen Konsistenzgruppen besteht. Einzelne Volumes können über eine eigene volume-granulare Snapshot-Policy verfügen. Zusätzlich kann eine Konsistenzgruppen-weite Snapshot-Policy vorhanden sein. Die Konsistenzgruppe kann nur eine SnapMirror active sync Beziehung und eine gemeinsame SnapMirror Policy haben, mit der die gesamte Konsistenzgruppe wiederhergestellt werden kann.

Das folgende Diagramm veranschaulicht, wie eine einzelne Konsistenzgruppe verwendet werden kann. Die Daten für eine Anwendung, die auf `SVM1` gehostet wird, erstrecken sich über zwei Volumes: `vol1` und `vol2`. Eine Snapshot-Richtlinie für die Konsistenzgruppe erfasst alle 15 Minuten Snapshots der Daten.

image:consistency-group-single-diagram.gif["Eine einzelne Konsistenzgruppe mit ihren zugehörigen Volumes und lokaler Snapshot-Policy"]

Größere Anwendungsworkloads erfordern möglicherweise mehrere Konsistenzgruppen. In solchen Situationen können hierarchische Konsistenzgruppen erstellt werden, wobei eine einzelne Konsistenzgruppe die untergeordneten Komponenten einer übergeordneten Konsistenzgruppe bildet. Die übergeordnete Konsistenzgruppe kann bis zu fünf untergeordnete Konsistenzgruppen enthalten. Wie bei einzelnen Konsistenzgruppen kann eine Remote SnapMirror active sync Datensicherungsstrategie auf die gesamte Konfiguration der Konsistenzgruppen (übergeordnete und untergeordnete) angewendet werden, um den Anwendungsworkload wiederherzustellen.

Im folgenden Beispiel wird eine Anwendung auf `SVM1` gehostet. Der Administrator hat eine übergeordnete Konsistenzgruppe erstellt, `SVM1_app`, die zwei untergeordnete Konsistenzgruppen umfasst: `SVM1appDataCG` für die Daten und `SVM1app_logCG` für die Protokolle. Jede untergeordnete Konsistenzgruppe verfügt über eine eigene Snapshot-Strategie. Snapshots der Volumes in `SVM1appDataCG` werden alle 15 Minuten erstellt. Snapshots von `SVM1app_logCG` werden stündlich erstellt. Die übergeordnete Konsistenzgruppe `SVM1_app` verfügt über eine SnapMirror active sync-Strategie, die die Daten repliziert, um die fortlaufende Bereitstellung im Katastrophenfall sicherzustellen.

image:consistency-group-nested-diagram.gif["Eine verschachtelte Konsistenzgruppe mit ihren zugehörigen Volumes"]

Ab ONTAP 9.12.1 unterstützen Konsistenzgruppen xref:clone-task.html[Klonen] und die Änderung der Mitglieder der Konsistenzgruppe durch xref:modify-task.html[Hinzufügen oder Entfernen von Volumes] sowohl im System Manager als auch in der ONTAP REST API. Ab ONTAP 9.12.1 unterstützt die ONTAP REST API außerdem:

* Erstellen von Konsistenzgruppen mit neuen NFS- oder SMB-Volumes oder NVMe-Namespaces.
* Hinzufügen neuer oder vorhandener NFS- oder SMB-Volumes oder NVMe-Namespaces zu bestehenden Konsistenzgruppen.


Weitere Informationen zur ONTAP REST API finden Sie unter https://docs.netapp.com/us-en/ontap-automation/reference/api_reference.html#access-a-copy-of-the-ontap-rest-api-reference-documentation["ONTAP REST API Referenzdokumentation"^].



[[monitor-consistency-groups]]
== Konsistenzgruppen überwachen

Ab ONTAP 9.13.1 bieten Konsistenzgruppen Echtzeit- und historische Kapazitäts- und Leistungsüberwachung und ermöglichen Einblicke in die Performance von Anwendungen und einzelnen Konsistenzgruppen.

Die Überwachungsdaten werden alle fünf Minuten aktualisiert und bis zu einem Jahr lang gespeichert. Es lassen sich Kennzahlen für Folgendes nachverfolgen:

* Leistung: IOPS, Latenz und Durchsatz
* Kapazität: Größe, logisch verwendet, verfügbar


Überwachungsdaten sind auf der Registerkarte **Übersicht** im Konsistenzgruppen-Menü des System Managers oder über die ONTAP REST API abrufbar. Ab ONTAP 9.14.1 lassen sich Konsistenzgruppenmetriken mit der ONTAP CLI über den  `consistency-group metrics show` Befehl anzeigen. Weitere Informationen zu  `consistency-group metrics show` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/vserver-consistency-group-metrics-show.html["ONTAP-Befehlsreferenz"^].


NOTE: In ONTAP 9.13.1 können historische Metriken nur über die ONTAP REST API abgerufen werden. Ab ONTAP 9.14.1 sind historische Metriken auch im System Manager verfügbar.



[[protect-consistency-groups]]
== Konsistenzgruppen schützen

Konsistenzgruppen bieten applikationskonsistente Schutzmechanismen und gewährleisten die Konsistenz Ihrer Daten über mehrere Volumes oder LIFs hinweg. Beim Erstellen eines Snapshots einer Konsistenzgruppe wird ein „Fence“ auf der Konsistenzgruppe gesetzt. Der Fence initiiert eine Warteschlange für Ein-/Ausgabe, bis der Snapshot-Vorgang abgeschlossen ist, und stellt so die zeitpunktgenaue Konsistenz der Daten über alle Entitäten in der Konsistenzgruppe sicher. Der Fence kann während Snapshot-Erstellungsvorgängen, wie etwa einer geplanten Snapshot-Richtlinie oder der Erstellung eines Snapshots mit System Manager, zu einem vorübergehenden Anstieg der Latenz führen. Weitere Informationen zum Kontext der ONTAP REST API und ONTAP CLI befinden sich in der link:https://kb.netapp.com/on-prem/ontap/DM/REST-API["ONTAP REST API Dokumentation"^] und link:https://docs.netapp.com/us-en/ontap-cli/["ONTAP-Befehlsreferenz"^].

Konsistenzgruppen bieten Schutz durch:

* Snapshot Richtlinien
* xref:../snapmirror-active-sync/index.html[SnapMirror Active Sync]
* <<mcc>> (beginnend mit ONTAP 9.11.1)
* xref:../data-protection/snapmirror-disaster-recovery-concept.html[SnapMirror asynchron] (beginnend mit ONTAP 9.13.1)
* link:../data-protection/snapmirror-svm-replication-concept.html["SVM Disaster Recovery"] (beginnend mit ONTAP 9.14.1)


Das Erstellen einer Konsistenzgruppe aktiviert den Schutz nicht automatisch. Lokale und Remote-Datensicherungsstrategien können beim Erstellen oder nach dem Erstellen einer Konsistenzgruppe festgelegt werden.

Zum Konfigurieren des Schutzes einer Konsistenzgruppe siehe link:protect-task.html["Eine Konsistenzgruppe schützen"].

Um Fernschutz nutzen zu können, müssen die Voraussetzungen für xref:../snapmirror-active-sync/prerequisites-reference.html[SnapMirror Active Sync] erfüllt sein.


NOTE: SnapMirror active sync-Beziehungen können auf Volumes, die für den NAS-Zugriff eingebunden sind, nicht hergestellt werden.



[[multi-admin-verification-support-for-consistency-groups]]
== Unterstützung der Multi-Admin-Verifizierung für Konsistenzgruppen

Ab ONTAP 9.16.1 kann die Multi-Admin-Verifizierung (MAV) mit Konsistenzgruppen verwendet werden, um sicherzustellen, dass bestimmte Vorgänge, wie das Erstellen, Ändern oder Löschen von Konsistenzgruppen, nur nach Genehmigung durch die zuständigen Administratoren ausgeführt werden können. Dadurch wird verhindert, dass kompromittierte, böswillige oder unerfahrene Administratoren unerwünschte Änderungen an bestehenden Konfigurationen vornehmen.

link:../multi-admin-verify/index.html["Mehr erfahren"]



[[consistency-groups-in-metrocluster-configurations]]
[[mcc]]
== Konsistenzgruppen in MetroCluster Konfigurationen

Ab ONTAP 9.11.1 ist die Bereitstellung von Konsistenzgruppen mit neuen Volumes auf einem Cluster innerhalb einer MetroCluster Konfiguration möglich. Diese Volumes werden auf gespiegelten Aggregaten bereitgestellt.

Nach der Bereitstellung können Volumes, die Konsistenzgruppen zugeordnet sind, zwischen gespiegelten und ungespiegelten Aggregaten verschoben werden. Daher können Volumes, die Konsistenzgruppen zugeordnet sind, auf gespiegelten Aggregaten, ungespiegelten Aggregaten oder auf beiden liegen. Gespiegelte Aggregate, die Volumes enthalten, die Konsistenzgruppen zugeordnet sind, können in ungespiegelte Aggregate umgewandelt werden. Ebenso können ungespiegelte Aggregate, die Volumes enthalten, die Konsistenzgruppen zugeordnet sind, für die Spiegelung aktiviert werden.

Volumes und Snapshots, die Konsistenzgruppen auf gespiegelten Aggregaten zugeordnet sind, werden an den Remote-Standort (Standort B) repliziert. Die Inhalte der Volumes an Standort B gewährleisten die Schreibreihenfolge für die Konsistenzgruppe, sodass im Katastrophenfall eine Wiederherstellung von Standort B möglich ist. Auf Snapshots von Konsistenzgruppen kann mit der Konsistenzgruppe über die ONTAP REST API und den System Manager auf Clustern mit ONTAP 9.11.1 oder neuer zugegriffen werden. Ab ONTAP 9.14.1 ist auch der Zugriff auf Snapshots mit der ONTAP CLI möglich.

Wenn sich einige oder alle Volumes, die einer Konsistenzgruppe zugeordnet sind, auf nicht gespiegelten Aggregaten befinden, auf die derzeit nicht zugegriffen werden kann, verhalten sich GET- oder DELETE-Operationen auf der Konsistenzgruppe so, als wären die lokalen Volumes oder die hostenden Aggregate offline.



[[consistency-group-configurations-for-replication]]
=== Konsistenzgruppenkonfigurationen für die Replikation

Wenn auf Standort B ONTAP 9.10.1 oder eine ältere Version ausgeführt wird, werden nur die Volumes, die den auf gespiegelten Aggregaten befindlichen Konsistenzgruppen zugeordnet sind, auf Standort B repliziert. Die Konsistenzgruppen-Konfigurationen werden nur auf Standort B repliziert, wenn auf beiden Standorten ONTAP 9.11.1 oder eine neuere Version ausgeführt wird. Nach dem Upgrade von Standort B auf ONTAP 9.11.1 werden die Daten der Konsistenzgruppen auf Standort A, deren zugehörige Volumes vollständig auf gespiegelten Aggregaten liegen, auf Standort B repliziert.


NOTE: Es wird empfohlen, für gespiegelte Aggregate mindestens 20 % freien Speicherplatz vorzuhalten, um optimale Speicherleistung und Verfügbarkeit zu gewährleisten. Obwohl für nicht gespiegelte Aggregate 10 % empfohlen werden, kann das Dateisystem die zusätzlichen 10 % des Speicherplatzes nutzen, um inkrementelle Änderungen aufzufangen. Inkrementelle Änderungen erhöhen die Speicherauslastung gespiegelter Aggregate aufgrund der Redirect-on-Write-Snapshot-basierten Architektur von ONTAP. Die Nichtbeachtung dieser Best Practices kann sich negativ auf die Leistung auswirken.



[[upgrade-considerations]]
== Upgrade-Überlegungen

Beim Upgrade auf ONTAP 9.10.1 oder höher werden Konsistenzgruppen, die mit SnapMirror active sync (früher bekannt als SnapMirror Business Continuity) in ONTAP 9.8 und 9.9.1 erstellt wurden, automatisch aktualisiert und können anschließend unter *Storage > Consistency Groups* im System Manager oder über die ONTAP REST API verwaltet werden. Weitere Informationen zum Upgrade von ONTAP 9.8 oder 9.9.1 finden sich unter link:../snapmirror-active-sync/upgrade-revert-task.html["SnapMirror Überlegungen zum Upgrade und zur Rücksetzung von active sync"].

In der ONTAP REST API erstellte Konsistenzgruppen-Snapshots können über die Konsistenzgruppen-Oberfläche des System Manager und über Konsistenzgruppen-ONTAP REST API-Endpunkte verwaltet werden. Ab ONTAP 9.14.1 können Konsistenzgruppen-Snapshots auch mit der ONTAP CLI verwaltet werden.


NOTE: Snapshots, die mit den ONTAPI-Befehlen  `cg-start` und  `cg-commit` erstellt wurden, werden nicht als Konsistenzgruppe-Snapshots erkannt und können daher weder über die Konsistenzgruppe-Oberfläche des System Manager noch über die Konsistenzgruppe-Endpunkte in der ONTAP REST API verwaltet werden. Ab ONTAP 9.14.1 können diese Snapshots auf das Zielvolume gespiegelt werden, wenn eine SnapMirror asynchron Richtlinie verwendet wird. Weitere Informationen finden sich unter xref:protect-task.html#configure-snapmirror-asynchronous[Asynchrones SnapMirror konfigurieren].
