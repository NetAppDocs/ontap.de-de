= ONTAP Konsistenzgruppen schützen
:hardbreaks:
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Konsistenzgruppen bieten einfach zu verwaltenden lokalen und Remote-Schutz für SAN, NAS und NVMe-Anwendungen, die sich über mehrere Volumes erstrecken.

Das Erstellen einer Konsistenzgruppe aktiviert den Schutz nicht automatisch. Datensicherungsstrategien können zum Zeitpunkt der Erstellung oder nach dem Erstellen Ihrer Konsistenzgruppe festgelegt werden. Konsistenzgruppen können mit folgenden Methoden geschützt werden:

* Lokale Snapshots
* SnapMirror active sync (in Versionen von ONTAP vor 9.15.1 als SnapMirror Business Continuity bezeichnet)
* xref:index.html#mcc[MetroCluster (beginnend mit 9.11.1)]
* SnapMirror asynchron (ab 9.13.1)
* Asynchrone SVM Notfallwiederherstellung (ab Version 9.14.1)


Wenn Sie verschachtelte Konsistenzgruppen verwenden, können Sie unterschiedliche Datensicherungsstrategien für die übergeordnete und die untergeordnete Konsistenzgruppe festlegen.

Ab ONTAP 9.11.1 bieten Konsistenzgruppen <<two-phase,Zwei-Phasen-Konsistenzgruppen-Snapshot-Erstellung>>. Die zweiphasige Snapshot-Operation führt eine Vorprüfung durch, wodurch sichergestellt wird, dass der Snapshot erfolgreich erfasst wird.

Die Wiederherstellung kann für eine gesamte Konsistenzgruppe, eine einzelne Konsistenzgruppe in einer hierarchischen Konfiguration oder für einzelne Volumes innerhalb der Konsistenzgruppe erfolgen. Die Wiederherstellung erfolgt durch Auswahl der Konsistenzgruppe, die wiederhergestellt werden soll, Auswahl des Snapshot-Typs und anschließender Identifizierung des Snapshots, auf dem die Wiederherstellung basieren soll. Weitere Informationen zu diesem Prozess finden sich unter link:../task_dp_restore_from_vault.html["Ein Volume aus einem früheren Snapshot wiederherstellen"].



[[configure-a-local-snapshot-policy]]
== Eine lokale Snapshot-Richtlinie konfigurieren

Das Festlegen einer lokalen Snapshot Datensicherungsstrategie ermöglicht das Erstellen einer Richtlinie, die alle Volumes in einer Konsistenzgruppe umfasst.

.Über diese Aufgabe
Der minimal unterstützte Snapshot-Zeitplan für Konsistenzgruppen beträgt 30 Minuten. Dies basiert auf link:https://www.netapp.com/media/12385-tr4571.pdf["Tests für FlexGroup Volumes"^], die dieselbe Snapshot-Infrastruktur wie Konsistenzgruppen verwenden.

[role="tabbed-block"]
====
.System Manager
--
.Schritte
. *Speicher > Konsistenzgruppen* auswählen.
. Wählen Sie die von Ihnen erstellte Konsistenzgruppe im Menü „Konsistenzgruppe“ aus.
. Oben rechts auf der Übersichtsseite für die Konsistenzgruppe die Option *Bearbeiten* auswählen.
. Das Kontrollkästchen neben *Snapshots planen (lokal)* ist zu aktivieren.
. Wählen Sie eine Snapshot-Richtlinie aus. Zum Konfigurieren einer neuen, benutzerdefinierten Richtlinie siehe link:../task_dp_create_custom_data_protection_policies.html["Eine benutzerdefinierte Datensicherungsstrategie erstellen"].
. *Speichern* auswählen.
. Zurück zum Konsistenzgruppe Übersichtsmenü. In der linken Spalte unter *Snapshots (Lokal)* wird der Status als geschützt neben image:../media/icon_shield.png["Schildsymbol"] angezeigt.


--
.CLI
--
Ab ONTAP 9.14.1 lässt sich die Datensicherungsstrategie einer Konsistenzgruppe mithilfe der CLI ändern.

.Bevor Sie beginnen
* Sie müssen sich auf der Administrator-Ebene befinden, um diese Aufgabe auszuführen.
* Ab ONTAP 9.15.1 kann jeder Benutzer mit Administratorrechten diese Aufgabe ausführen. In ONTAP 9.14.1 ist es erforderlich, Cluster- oder SVM-Administrator zu sein, um diese Aufgabe auszuführen.


.Schritt
. Der folgende Befehl dient zum Festlegen oder Ändern der Datensicherungsstrategie:
+
Wenn Sie die Datensicherungsstrategie einer untergeordneten Konsistenz ändern, müssen Sie die übergeordnete Konsistenzgruppe mithilfe des  `-parent-consistency-group _parent_consistency_group_name_` Parameters identifizieren.

+
`consistency-group modify -vserver _svm_name_ -consistency-group _consistency_group_name_ -snapshot-policy _policy_name_`



--

[[create-an-on-demand-snapshot]]
== Erstellen Sie einen On-Demand-Snapshot

Falls Sie außerhalb einer regulären Datensicherungsstrategie eine Momentaufnahme Ihrer Konsistenzgruppe erstellen müssen, kann eine solche Momentaufnahme bei Bedarf erstellt werden.

[role="tabbed-block"]
====
.System Manager
--
.Schritte
. Zu *Speicher* > *Konsistenzgruppen* navigieren.
. Wählen Sie die Konsistenzgruppe aus, für die ein On-Demand-Snapshot erstellt werden soll.
. Zum Tab *Snapshot copies* wechseln und anschließend *+Add* auswählen.
. Geben Sie einen *Namen* und eine *SnapMirror Bezeichnung* an. Im Dropdown-Menü für *Konsistenz* steht *Anwendungskonsistent* oder *Absturzkonsistent* zur Auswahl.
. *Speichern* auswählen.


--
.CLI
--
Ab ONTAP 9.14.1 ist es möglich, mithilfe der CLI einen On-Demand-Snapshot einer Konsistenzgruppe zu erstellen.

.Bevor Sie beginnen
* Sie müssen sich auf der Administrator-Ebene befinden, um diese Aufgabe auszuführen.
* Ab ONTAP 9.15.1 kann jeder Benutzer mit Administratorrechten diese Aufgabe ausführen. In ONTAP 9.14.1 ist es erforderlich, Cluster- oder SVM-Administrator zu sein, um diese Aufgabe auszuführen.


.Schritt
. Snapshot erstellen:
+
Standardmäßig ist der Snapshot-Typ absturzkonsistent. Der Snapshot-Typ kann mit dem optionalen  `-type` Parameter geändert werden.

+
`consistency-group snapshot create -vserver _svm_name_ -consistency-group _consistency_group_name_ -snapshot _snapshot_name_`



--

[[create-two-phase-consistency-group-snapshots]]
== Snapshots von Zwei-Phasen-Konsistenzgruppen erstellen

Ab ONTAP 9.11.1 unterstützen Konsistenzgruppen zweiphasige Commits für die Erstellung von Konsistenzgruppen-Snapshots (CG-Snapshots), wobei vor dem Commit des Snapshots eine Vorprüfung durchgeführt wird. Diese Funktion ist nur mit der ONTAP REST API verfügbar.

Die zweiphasige Erstellung von CG-Snapshots ist nur für die Snapshot-Erstellung verfügbar, nicht für die Bereitstellung von Konsistenzgruppen oder die Wiederherstellung von Konsistenzgruppen.

Ein zweiphasiger CG-Snapshot unterteilt den Snapshot-Erstellungsprozess in zwei Phasen:

. In der ersten Phase führt die API Vorabprüfungen durch und initiiert die Snapshot-Erstellung. Die erste Phase beinhaltet einen Zeitüberschreitung Parameter, der die Zeitspanne für die erfolgreiche Speicherung des Snapshots festlegt.
. Wenn die Anfrage in Phase eins erfolgreich abgeschlossen wird, kann die zweite Phase innerhalb des festgelegten Intervalls nach der ersten Phase aufgerufen werden, wodurch der Snapshot an den entsprechenden Endpunkt übertragen wird.


.Bevor Sie beginnen
* Um die Erstellung von CG-Snapshots in zwei Phasen zu nutzen, müssen alle Knoten im Cluster mit ONTAP 9.11.1 oder höher ausgeführt werden.
* Es wird jeweils nur ein aktiver Aufruf einer Snapshot-Operation für eine Konsistenzgruppe pro Konsistenzgruppeninstanz unterstützt, unabhängig davon, ob es sich um eine ein- oder zweiphasige Operation handelt. Der Versuch, eine Snapshot-Operation aufzurufen, während eine andere bereits ausgeführt wird, führt zu einem Fehler.
* Beim Aufruf der Snapshot-Erstellung kann optional ein Zeitüberschreitung Wert zwischen 5 und 120 Sekunden festgelegt werden. Wird kein Zeitüberschreitung Wert angegeben, tritt nach dem Standardwert von 7 Sekunden eine Zeitüberschreitung auf. In der ONTAP REST API wird der Zeitüberschreitung Wert mit dem  `action_timeout` Parameter festgelegt. In der ONTAP CLI wird das  `-timeout` Flag verwendet.


.Schritte
Ein zweiphasiger Snapshot kann mit der ONTAP REST API oder, beginnend mit ONTAP 9.14.1, mit der ONTAP CLI abgeschlossen werden. Diese Operation wird im System Manager nicht unterstützt.


NOTE: Wenn Sie die Snapshot-Erstellung mit der ONTAP REST API aufrufen, muss der Snapshot mit der ONTAP REST API gespeichert werden. Wenn Sie die Snapshot-Erstellung mit der ONTAP CLI aufrufen, muss der Snapshot mit der ONTAP CLI gespeichert werden. Die Kombination verschiedener Methoden wird nicht unterstützt.

[role="tabbed-block"]
====
.CLI
--
Ab ONTAP 9.14.1 ist es möglich, mit der CLI einen zweiphasigen Snapshot zu erstellen.

.Bevor Sie beginnen
* Sie müssen sich auf der Administrator-Ebene befinden, um diese Aufgabe auszuführen.
* Ab ONTAP 9.15.1 kann jeder Benutzer mit Administratorrechten diese Aufgabe ausführen. In ONTAP 9.14.1 ist es erforderlich, Cluster- oder SVM-Administrator zu sein, um diese Aufgabe auszuführen.


.Schritte
. Snapshot initiieren:
+
`consistency-group snapshot start -vserver _svm_name_ -consistency-group _consistency_group_name_ -snapshot _snapshot_name_ [-timeout _time_in_seconds_ -write-fence {true|false}]`

. Es wird überprüft, ob die Momentaufnahme erstellt wurde:
+
`consistency-group snapshot show`

. Snapshot übernehmen:
+
`consistency-group snapshot commit _svm_name_ -consistency-group _consistency_group_name_ -snapshot _snapshot_name_`



--
.API
--
. Die Snapshot-Erstellung wird ausgelöst. Eine POST-Anfrage wird an den Konsistenzgruppe-Endpunkt mit dem  `action=start` Parameter gesendet.
+
[source, curl]
----
curl -k -X POST 'https://<IP_address>/application/consistency-groups/<cg-uuid>/snapshots?action=start&action_timeout=7' -H "accept: application/hal+json" -H "content-type: application/json" -d '
{
  "name": "<snapshot_name>",
  "consistency_type": "crash",
  "comment": "<comment>",
  "snapmirror_label": "<SnapMirror_label>"
}'
----
. Wenn die POST-Anfrage erfolgreich ist, enthält die Ausgabe eine Snapshot-UUID. Mit dieser UUID kann eine PATCH-Anfrage gesendet werden, um den Snapshot zu committen.
+
[source, curl]
----
curl -k -X PATCH 'https://<IP_address>/application/consistency-groups/<cg_uuid>/snapshots/<snapshot_id>?action=commit' -H "accept: application/hal+json" -H "content-type: application/json"

For more information about the ONTAP REST API, see link:https://docs.netapp.com/us-en/ontap-automation/reference/api_reference.html[API reference^] or the link:https://devnet.netapp.com/restapi.php[ONTAP REST API page^] at the NetApp Developer Network for a complete list of API endpoints.
----


--
====


[[set-remote-protection-for-a-consistency-group]]
== Remote-Schutz für eine Konsistenzgruppe festlegen

Konsistenzgruppen bieten Fernschutz durch SnapMirror active sync und, beginnend mit ONTAP 9.13.1, SnapMirror asynchron.



[[configure-protection-with-snapmirror-active-sync]]
=== Schutz mit SnapMirror active sync konfigurieren

Mit SnapMirror active sync kann sichergestellt werden, dass Snapshots von Konsistenzgruppen, die auf Ihrer Konsistenzgruppe erstellt wurden, an das Ziel kopiert werden. Weitere Informationen zu SnapMirror active sync oder zur Konfiguration von SnapMirror active sync über die CLI finden sich unter xref:../task_san_configure_protection_for_business_continuity.html[Schutz für die Geschäftskontinuität konfigurieren].

.Bevor Sie beginnen
* SnapMirror active sync-Beziehungen können auf Volumes, die für den NAS-Zugriff eingebunden sind, nicht hergestellt werden.
* Die Richtlinienbezeichnungen im Quell- und Ziel-Cluster müssen übereinstimmen.
* SnapMirror active sync repliziert Snapshots standardmäßig nicht, es sei denn, der vordefinierten  `AutomatedFailOver` Richtlinie wird eine Regel mit einem SnapMirror Label hinzugefügt und die Snapshots werden mit diesem Label erstellt.
+
Weitere Informationen zu diesem Prozess finden Sie unter link:../task_san_configure_protection_for_business_continuity.html["Mit SnapMirror active sync schützen"].

* xref:../data-protection/supported-deployment-config-concept.html[Kaskadierte Bereitstellungen] werden mit SnapMirror active sync nicht unterstützt.
* Ab ONTAP 9.13.1 ist es möglich, xref:modify-task.html#add-volumes-to-a-consistency-group[Volumes zu einer Konsistenzgruppe hinzufügen] unterbrechungsfrei mit einer aktiven SnapMirror active sync Beziehung zu arbeiten. Alle anderen Änderungen an einer Konsistenzgruppe erfordern das Aufheben der SnapMirror active sync Beziehung, die Modifikation der Konsistenzgruppe sowie das anschließende Wiederherstellen und Resynchronisieren der Beziehung.



TIP: Zur Konfiguration von SnapMirror Active Sync mit der CLI siehe xref:../task_san_configure_protection_for_business_continuity.html[Mit SnapMirror active sync schützen].

.Schritte für System Manager
. Stellen Sie sicher, dass Sie die link:../snapmirror-active-sync/prerequisites-reference.html["Voraussetzungen für die Verwendung von SnapMirror Active Sync"] erfüllt haben.
. *Speicher > Konsistenzgruppen* auswählen.
. Wählen Sie die von Ihnen erstellte Konsistenzgruppe im Menü „Konsistenzgruppe“ aus.
. Oben rechts auf der Übersichtsseite *Mehr* und dann *Schützen* auswählen.
. System Manager füllt die Quellseiteninformationen automatisch aus. Der passende Cluster und die passende Storage-VM für das Ziel sind auszuwählen. Wählen Sie eine Datensicherungsstrategie. Es ist sicherzustellen, dass *Beziehung initialisieren* aktiviert ist.
. *Speichern* auswählen.
. Die Konsistenzgruppe muss initialisiert und synchronisiert werden. Die erfolgreiche Synchronisierung lässt sich durch Rückkehr zum Menü *Konsistenzgruppe* bestätigen. Der Status *SnapMirror (Remote)* wird  `Protected` neben image:../media/icon_shield.png["Schildsymbol"] angezeigt.




[[configure-snapmirror-asynchronous]]
=== Asynchrones SnapMirror konfigurieren

Ab ONTAP 9.13.1 ist es möglich, SnapMirror asynchrone Datensicherung für eine einzelne Konsistenzgruppe zu konfigurieren. Ab ONTAP 9.14.1 kann SnapMirror asynchron genutzt werden, um volume-granulare Snapshots mithilfe der Konsistenzgruppe an das Ziel-Cluster zu replizieren.

.Über diese Aufgabe
Um volume-granulare Snapshots zu replizieren, muss ONTAP 9.14.1 oder höher ausgeführt werden. Für MirrorAndVault und Vault Richtlinien muss das SnapMirror Label der volume-granularen Snapshot-Richtlinie mit der SnapMirror Richtlinienregel der Konsistenzgruppe übereinstimmen. Volume-granulare Snapshots richten sich nach dem Aufbewahrungswert der SnapMirror Richtlinie der Konsistenzgruppe, der unabhängig von den Konsistenzgruppen-Snapshots berechnet wird. Wenn beispielsweise eine Richtlinie zum Aufbewahren von zwei Snapshots auf dem Ziel-Cluster besteht, können zwei volume-granulare Snapshots und zwei Konsistenzgruppen-Snapshots vorhanden sein.

Beim Resynchronisieren der SnapMirror Beziehung mit volume-granularen Snapshots können volume-granulare Snapshots mit dem  `-preserve` Flag beibehalten werden. Volume-granulare Snapshots, die neuer als Konsistenzgruppe-Snapshots sind, werden beibehalten. Wenn keine Konsistenzgruppe-Snapshot vorhanden ist, können bei der Resynchronisierung keine volume-granularen Snapshots übertragen werden.

.Bevor Sie beginnen
* SnapMirror asynchroner Schutz ist nur für eine einzelne Konsistenzgruppe verfügbar. Er wird für hierarchische Konsistenzgruppen nicht unterstützt. Zur Umwandlung einer hierarchischen Konsistenzgruppe in eine einzelne Konsistenzgruppe siehe xref:modify-geometry-task.html[Konsistenzgruppenarchitektur ändern].
* Die Richtlinienbezeichnungen im Quell- und Ziel-Cluster müssen übereinstimmen.
* Eine unterbrechungsfreie xref:modify-task.html#add-volumes-to-a-consistency-group[Volumes zu einer Konsistenzgruppe hinzufügen] ist mit einer aktiven SnapMirror asynchronen Beziehung möglich. Alle anderen Änderungen an einer Konsistenzgruppe erfordern das Aufheben der SnapMirror Beziehung, das Modifizieren der Konsistenzgruppe sowie das anschließende Wiederherstellen und Resynchronisieren der Beziehung.
* Konsistenzgruppen, die für den Schutz mit SnapMirror asynchron aktiviert sind, haben unterschiedliche Grenzwerte. Weitere Informationen finden sich unter xref:limits.html[Konsistenzgruppenlimits].
* Wenn Sie eine asynchrone SnapMirror Schutzbeziehung für mehrere einzelne Volumes konfiguriert haben, können diese Volumes in eine Konsistenzgruppe umgewandelt werden, wobei die vorhandenen Snapshots erhalten bleiben. Für eine erfolgreiche Konvertierung der Volumes gilt Folgendes:
+
** Es muss eine gemeinsame Snapshot-Kopie der Volumes vorhanden sein.
** Sie müssen die bestehende SnapMirror Beziehung unterbrechen, xref:configure-task.html[die Volumes zu einer einzigen Konsistenzgruppe hinzufügen], dann die Beziehung mithilfe des folgenden Workflows erneut synchronisieren.




.Schritte
. Im Ziel-Cluster *Speicher > Konsistenzgruppen* auswählen.
. Wählen Sie die von Ihnen erstellte Konsistenzgruppe im Menü „Konsistenzgruppe“ aus.
. Oben rechts auf der Übersichtsseite *Mehr* und dann *Schützen* auswählen.
. System Manager füllt die Quellseiteninformationen automatisch aus. Der passende Cluster und die passende Storage-VM für das Ziel sind auszuwählen. Wählen Sie eine Datensicherungsstrategie. Es ist sicherzustellen, dass *Beziehung initialisieren* aktiviert ist.
+
Bei Auswahl einer asynchronen Richtlinie besteht die Möglichkeit, den **Übertragungsplan zu überschreiben**.

+

NOTE: Der minimal unterstützte Zeitplan (Recovery Point Objective, RPO) für Konsistenzgruppen mit SnapMirror asynchron beträgt 30 Minuten.

. *Speichern* auswählen.
. Die Konsistenzgruppe muss initialisiert und synchronisiert werden. Die erfolgreiche Synchronisierung lässt sich durch Rückkehr zum Menü *Konsistenzgruppe* bestätigen. Der Status *SnapMirror (Remote)* wird  `Protected` neben image:../media/icon_shield.png["Schildsymbol"] angezeigt.




[[configure-svm-disaster-recovery]]
=== SVM-Notfallwiederherstellung konfigurieren

Ab ONTAP 9.14.1 xref:../data-protection/snapmirror-svm-replication-concept.html#[SVM Disaster Recovery] werden Konsistenzgruppen unterstützt, sodass Konsistenzgruppeninformationen vom Quell-Cluster auf den Ziel-Cluster gespiegelt werden können.

Wenn die SVM-Notfallwiederherstellung auf einer SVM aktiviert wird, die bereits eine Konsistenzgruppe enthält, sollten die SVM-Konfigurationsworkflows für xref:../task_dp_configure_storage_vm_dr.html[System Manager] oder die xref:../data-protection/replicate-entire-svm-config-task.html[ONTAP CLI] beachtet werden.

Wenn einer SVM, die sich in einer aktiven und fehlerfreien SVM-Disaster-Recovery-Beziehung befindet, eine Konsistenzgruppe hinzugefügt wird, muss die SVM-Disaster-Recovery-Beziehung vom Ziel-Cluster aus aktualisiert werden. Weitere Informationen finden sich unter xref:../data-protection/update-replication-relationship-manual-task.html[Eine Replikationsbeziehung manuell aktualisieren]. Die Beziehung muss jedes Mal aktualisiert werden, wenn die Konsistenzgruppe erweitert wird.

.Einschränkungen
* Die SVM-Notfallwiederherstellung unterstützt keine hierarchischen Konsistenzgruppen.
* Die SVM-Notfallwiederherstellung unterstützt keine Konsistenzgruppen, die mit SnapMirror asynchron geschützt sind. Die SnapMirror Beziehung muss aufgehoben werden, bevor die SVM-Notfallwiederherstellung konfiguriert wird.
* Beide Cluster müssen mit ONTAP 9.14.1 oder höher betrieben werden.
* Fan-out-Beziehungen werden für SVM-Disaster-Recovery-Konfigurationen, die Konsistenzgruppen enthalten, nicht unterstützt.
* Weitere Grenzwerte sind unter xref:limits.html[Konsistenzgruppe Grenzen] aufgeführt.




[[visualize-relationships]]
== Beziehungen visualisieren

System Manager visualisiert LUN-Zuordnungen unter dem Menüpunkt *Schutz > Beziehungen*. Bei Auswahl einer Quellbeziehung zeigt System Manager eine Visualisierung der Quell-Beziehungen an. Durch Auswahl eines Volumes ist eine detaillierte Analyse dieser Beziehungen möglich, einschließlich einer Liste der enthaltenen LUNs und der Initiatorgruppen-Beziehungen. Diese Informationen können aus der Ansicht des einzelnen Volumes als Excel-Arbeitsmappe heruntergeladen werden; der Downloadvorgang läuft im Hintergrund.
