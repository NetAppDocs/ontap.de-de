= Anforderungen und Richtlinien für die Verwendung automatischer Knotenverweise auf ONTAP SMB-Servern
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Bevor Sie die automatische Knotenweiterleitung in SMB, auch bekannt als _autolocation_, nutzen können, müssen Sie sich bestimmter Anforderungen bewusst sein, einschließlich der ONTAP Versionen, die diese Funktion unterstützen. Sie müssen außerdem über die unterstützten SMB Protokollversionen und bestimmte weitere spezielle Richtlinien informiert sein.



[[ontap-version-and-license-requirements]]
== ONTAP Versions- und Lizenzanforderungen

* Alle Knoten im Cluster müssen eine Version von ONTAP ausführen, die automatische Knotenverweise unterstützt.
* Widelinks muss auf einer SMB-Freigabe aktiviert sein, um die Autolokalisierung zu verwenden.
* CIFS muss lizenziert sein, und ein SMB-Server muss auf den SVMs vorhanden sein. Die SMB-Lizenz ist mit link:../system-admin/manage-licenses-concept.html#licenses-included-with-ontap-one["ONTAP One"] enthalten. Wenn ONTAP One nicht vorhanden ist und die Lizenz nicht installiert ist, sollte der Vertriebsmitarbeiter kontaktiert werden.




[[smb-protocol-version-requirements]]
== Anforderungen an die SMB-Protokollversion

* Für SVMs unterstützt ONTAP automatische Knotenverweise auf allen Versionen von SMB.




[[smb-client-requirements]]
== SMB-Client-Anforderungen

Alle von ONTAP unterstützten Microsoft-Clients unterstützen automatische Knotenverweise für SMB.

Die Interoperabilitätsmatrix enthält die aktuellsten Informationen darüber, welche Windows Clients ONTAP unterstützt.

link:http://mysupport.netapp.com/matrix["NetApp Interoperabilitätsmatrix-Tool"^]



[[data-lif-requirements]]
== Anforderungen an Daten-LIFs

Wenn eine Daten-LIF als potenzielle Weiterleitung für SMB-Clients verwendet werden soll, müssen Daten-LIFs erstellt werden, bei denen sowohl NFS als auch CIFS aktiviert sind.

Automatische Knotenverweise können fehlschlagen, wenn der Zielknoten Daten-LIFs enthält, die nur für das NFS-Protokoll oder nur für das SMB-Protokoll aktiviert sind.

Wird diese Anforderung nicht erfüllt, ist der Datenzugriff nicht beeinträchtigt. Der SMB-Client ordnet die Freigabe mithilfe der ursprünglichen LIF zu, die der Client für die Verbindung zur SVM verwendet hat.



[[ntlm-authentication-requirements-when-making-a-referred-smb-connection]]
== NTLM-Authentifizierungsanforderungen beim Herstellen einer weitergeleiteten SMB-Verbindung

NTLM-Authentifizierung muss in der Domäne, die den CIFS-Server enthält, sowie in den Domänen, die Clients enthalten, die automatische Knotenverweise nutzen möchten, erlaubt sein.

Bei einer Weiterleitung übermittelt der SMB-Server dem Windows-Client eine IP-Adresse. Da bei einer Verbindung über eine IP-Adresse die NTLM-Authentifizierung verwendet wird, findet für weitergeleitete Verbindungen keine Kerberos-Authentifizierung statt.

Dies geschieht, weil der Windows-Client den von Kerberos verwendeten Dienstprinzipalnamen (der die Form  `service/NetBIOS name` und  `service/FQDN` hat) nicht erstellen kann, was bedeutet, dass der Client kein Kerberos-Ticket für den Dienst anfordern kann.



[[guidelines-for-using-automatic-node-referrals-with-the-home-directory-feature]]
== Richtlinien für die Verwendung automatischer Knotenverweise mit der Home-Verzeichnis-Funktion

Wenn Freigaben mit aktivierter Home-Verzeichnis-Freigabeeigenschaft konfiguriert sind, können ein oder mehrere Home-Verzeichnis-Suchpfade für eine Home-Verzeichnis-Konfiguration eingerichtet werden. Die Suchpfade können auf Volumes verweisen, die sich auf jedem Knoten mit SVM-Volumes befinden. Clients erhalten eine Weiterleitung und stellen, sofern ein aktiver, lokaler Daten-LIF verfügbar ist, eine Verbindung über einen weitergeleiteten LIF her, der lokal zum Home-Verzeichnis des Benutzers ist.

Es gibt Richtlinien, wenn SMB 1.0-Clients auf dynamische Home-Verzeichnisse mit aktivierter automatischer Knotenweiterleitung zugreifen. Dies liegt daran, dass SMB 1.0-Clients die automatische Knotenweiterleitung benötigen, bevor sie sich authentifiziert haben, also bevor der SMB-Server den Benutzernamen kennt. Der Zugriff auf das SMB-Home-Verzeichnis funktioniert jedoch für SMB 1.0-Clients korrekt, wenn die folgenden Bedingungen erfüllt sind:

* SMB-Home-Verzeichnisse sind so konfiguriert, dass sie einfache Namen wie "`%w`" (Windows-Benutzername) oder "`%u`" (zugeordneter UNIX-Benutzername) verwenden und keine Namen im Domänenname-Stil wie "`%d\%w `" (Domänenname\Benutzername).
* Beim Erstellen von Home-Verzeichnisfreigaben werden die Namen der CIFS-Home-Verzeichnisfreigaben mit Variablen ("`%w`" oder "`%u`") und nicht mit statischen Namen wie "`HOME`" konfiguriert.


Für SMB 2.x und SMB 3.0 Clients bestehen keine besonderen Richtlinien beim Zugriff auf Home-Verzeichnisse mit automatischen Knotenverweisen.



[[guidelines-for-disabling-automatic-node-referrals-on-cifs-servers-with-existing-referred-connections]]
== Richtlinien zum Deaktivieren automatischer Knotenverweise auf CIFS-Servern mit bestehenden weitergeleiteten Verbindungen

Wenn automatische Knotenverweise deaktiviert werden, nachdem die Option aktiviert wurde, behalten Clients, die aktuell mit einem verwiesenen LIF verbunden sind, die verwiesene Verbindung bei. Da ONTAP DFS-Verweise als Mechanismus für automatische SMB-Knotenverweise verwendet, können sich Clients sogar nach Deaktivierung der Option erneut mit dem verwiesenen LIF verbinden, bis der zwischengespeicherte DFS-Verweis des Clients für die verwiesene Verbindung abläuft. Dies gilt selbst im Fall einer Rückkehr zu einer Version von ONTAP, die automatische Knotenverweise nicht unterstützt. Clients verwenden weiterhin Verweise, bis der DFS-Verweis aus dem Cache des Clients abläuft.

Autolocation nutzt automatische Knotenverweise in SMB, um die Leistung von SMB-Clients zu steigern, indem Clients an den LIF auf dem Knoten weitergeleitet werden, der das Daten-Volume einer SVM besitzt. Wenn ein SMB-Client eine Verbindung zu einer auf einer SVM gehosteten SMB-Freigabe herstellt, kann die Verbindung über einen LIF auf einem Knoten erfolgen, der nicht die angeforderten Daten besitzt und das Cluster-Interconnect-Netzwerk zur Datenübertragung verwendet. Der Client kann von schnelleren Antwortzeiten profitieren, wenn die SMB-Verbindung einen LIF auf dem Knoten verwendet, der die angeforderten Daten enthält.

ONTAP stellt diese Funktionalität mithilfe von Microsoft Distributed File System (DFS) Verweisen bereit, um SMB-Clients darüber zu informieren, dass eine angeforderte Datei oder ein Ordner im Namespace an einem anderen Ort gehostet wird. Ein Node erstellt einen Verweis, wenn festgestellt wird, dass sich auf dem Node eine SVM LIF befindet, die die Daten enthält. Weiterleitungen erfolgen auf Grundlage des Speicherorts des Stammverzeichnisses der Freigabe, über die der Client verbunden ist.

Die Weiterleitung erfolgt während der SMB-Aushandlung. Die Weiterleitung erfolgt vor dem Verbindungsaufbau. Nachdem ONTAP den SMB-Client an den Zielknoten weitergeleitet hat, wird die Verbindung hergestellt und der Client greift ab diesem Zeitpunkt über den weitergeleiteten LIF-Pfad auf die Daten zu. Dies ermöglicht den Clients einen schnelleren Zugriff auf die Daten und vermeidet zusätzliche Cluster-Kommunikation.



[[guidelines-for-using-automatic-node-referrals-with-mac-os-clients]]
== Richtlinien für die Verwendung automatischer Knotenverweise mit Mac OS Clients

Mac OS X-Clients unterstützen keine automatischen SMB-Knotenverweise, obwohl Mac OS das Distributed File System (DFS) von Microsoft unterstützt. Windows-Clients stellen vor dem Verbinden mit einer SMB-Freigabe eine DFS-Verweisanfrage. ONTAP verweist auf eine Daten-LIF auf demselben Knoten, der die angeforderten Daten hostet, was zu verbesserten Antwortzeiten für die Clients führt. Obwohl Mac OS DFS unterstützt, verhalten sich Mac OS-Clients in diesem Bereich nicht exakt wie Windows-Clients.
