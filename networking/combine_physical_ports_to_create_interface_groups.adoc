= Physische Ports können kombiniert werden, um ONTAP Schnittstellengruppen zu erstellen.
:hardbreaks:
:allow-uri-read: 
:nofooter: 
:icons: font
:linkattrs: 
:imagesdir: ../media/


[role="lead"]
Eine Schnittstellengruppe, auch Link-Aggregation-Gruppe (LAG) genannt, entsteht durch die Zusammenfassung von zwei oder mehr physischen Ports desselben Knotens zu einem einzigen logischen Port. Der logische Port bietet erhöhte Ausfallsicherheit, höhere Verfügbarkeit und Lastverteilung.



[[interface-group-types]]
== Schnittstellengruppentypen

Das Speichersystem unterstützt drei Arten von Schnittstellengruppen: Single-Mode, statischer Multimode und Dynamisches Multimode. Jede Schnittstellengruppe bietet unterschiedliche Stufen der Fehlertoleranz. Multimode-Schnittstellengruppen bieten Methoden zum Lastausgleich des Netzwerkverkehrs.



[[characteristics-of-single-mode-interface-groups]]
=== Merkmale von Einzelmodus-Schnittstellengruppen

In einer Singlemode Schnittstellengruppe ist nur eine der Schnittstellen in der Schnittstellengruppe aktiv. Die anderen Schnittstellen sind im Standby und bereit, die aktive Schnittstelle zu übernehmen, falls diese ausfällt.

Merkmale von Einzelmodus-Schnittstellengruppen:

* Im Fehlerfall überwacht der Cluster die aktive Verbindung und steuert den Failover. Da der Cluster die aktive Verbindung überwacht, ist keine Switch-Konfiguration erforderlich.
* In einer Einzelmodus-Schnittstellengruppe kann mehr als eine Schnittstelle im Standby sein.
* Wenn sich eine Single-Mode Schnittstellengruppe über mehrere Switches erstreckt, ist eine Verbindung der Switches mit einer Inter-Switch-Verbindung (ISL) erforderlich.
* Für eine Singlemode Schnittstellengruppe müssen sich die Switch-Ports im selben Broadcast-Domäne befinden.
* Link-Monitoring-ARP-Pakete mit der Quelladresse 0.0.0.0 werden über die Ports gesendet, um zu überprüfen, ob sich die Ports in derselben Broadcast-Domäne befinden.


Die folgende Abbildung zeigt ein Beispiel einer Singlemode Schnittstellengruppe. In der Abbildung gehören e0a und e1a zur a0a Singlemode Schnittstellengruppe. Wenn die aktive Schnittstelle e0a ausfällt, übernimmt die Standby-Schnittstelle e1a und hält die Verbindung zum Switch aufrecht.

image:ontap_nm_image6.png["Eine Einzelmodus Schnittstellengruppe"]


NOTE: Um die Einzelmodus-Funktionalität zu erreichen, wird stattdessen die Verwendung von Failover-Gruppen empfohlen. Durch die Verwendung einer Failover-Gruppe kann der zweite Port weiterhin für andere LIFs genutzt werden und muss nicht ungenutzt bleiben. Zusätzlich können Failover-Gruppen mehr als zwei Ports umfassen und Ports auf mehreren Nodes einschließen.



[[characteristics-of-static-multimode-interface-groups]]
=== Merkmale statischer Multimode Schnittstellengruppen

Die Implementierung der statischer Multimode Schnittstellengruppe in ONTAP entspricht IEEE 802.3ad (statisch). Jeder Switch, der Aggregate unterstützt, aber keinen Kontrollpaketaustausch zur Konfiguration eines Aggregats ermöglicht, kann mit statischer Multimode Schnittstellengruppe verwendet werden.

Statische Multimode Schnittstellengruppen entsprechen nicht IEEE 802.3ad (dynamisch), auch bekannt als Link Aggregation Control Protocol (LACP). LACP ist äquivalent zum Port Aggregation Protocol (PAgP), dem proprietären Link Aggregation Protokoll von Cisco.

Folgende Merkmale kennzeichnen eine statische Multimode Schnittstellengruppe:

* Alle Schnittstellen in der Schnittstellengruppe sind aktiv und teilen sich eine einzige MAC-Adresse.
+
** Mehrere Einzelverbindungen werden auf die Schnittstellen in der Schnittstellengruppe verteilt.
** Jede Verbindung oder Sitzung nutzt eine Schnittstelle innerhalb der Schnittstellengruppe. Beim sequenziellen Lastverteilungsschema werden alle Sitzungen paketweise auf die verfügbaren Verbindungen verteilt und sind nicht an eine bestimmte Schnittstelle der Schnittstellengruppe gebunden.


* Statische Multimode Schnittstellengruppen können sich von einem Ausfall von bis zu "n-1" Schnittstellen erholen, wobei n die Gesamtzahl der Schnittstellen ist, die die Schnittstellengruppe bilden.
* Wenn ein Port ausfällt oder getrennt wird, wird der Datenverkehr, der über die ausgefallene Verbindung lief, automatisch auf eine der verbleibenden Schnittstellen umverteilt.
* Statische Multimode Schnittstellengruppen können einen Link-Verlust erkennen, jedoch keinen Verbindungsverlust zum Client oder Switch-Fehlkonfigurationen, die sich auf Konnektivität und Performance auswirken könnten.
* Eine statische Multimode Schnittstellengruppe erfordert einen Switch, der Link-Aggregation über mehrere Switch-Ports unterstützt. Der Switch ist so konfiguriert, dass alle Ports, an die Links einer Schnittstellengruppe angeschlossen sind, Teil eines einzigen logischen Ports sind. Einige Switches unterstützen möglicherweise keine Link-Aggregation von Ports, die für Jumbo Frames konfiguriert sind. Weitere Informationen enthält die Dokumentation Ihres Switch-Herstellers.
* Mehrere Load-Balancing-Optionen stehen zur Verfügung, um den Datenverkehr auf die Schnittstellen einer statischer Multimode Schnittstellengruppe zu verteilen.


Die folgende Abbildung zeigt ein Beispiel einer statischen Multimode Schnittstellengruppe. Die Schnittstellen e0a, e1a, e2a und e3a gehören zur a1a Multimode Schnittstellengruppe. Alle vier Schnittstellen der a1a Multimode Schnittstellengruppe sind aktiv.

image:ontap_nm_image7.png["Eine dynamische Multimode-Schnittstellengruppe"]

Es gibt verschiedene Technologien, die es ermöglichen, den Datenverkehr in einer einzelnen aggregierten Verbindung auf mehrere physische Switches zu verteilen. Die zur Ermöglichung dieser Funktion verwendeten Technologien variieren je nach Netzwerkprodukt. Statische Multimode Schnittstellengruppen in ONTAP entsprechen den IEEE 802.3 Standards. Wenn eine bestimmte Multiple Switch Link-Aggregation-Technologie mit den IEEE 802.3 Standards interoperiert oder diesen entspricht, sollte sie mit ONTAP funktionieren.

Der IEEE 802.3-Standard legt fest, dass das sendende Gerät in einer aggregierten Verbindung die physikalische Schnittstelle für die Übertragung bestimmt. Daher ist ONTAP nur für die Verteilung des ausgehenden Datenverkehrs zuständig und kann nicht steuern, wie eingehende Frames ankommen. Wenn die Übertragung des eingehenden Datenverkehrs in einer aggregierten Verbindung verwaltet oder gesteuert werden soll, muss diese Übertragung auf dem direkt angeschlossenen Netzwerkgerät angepasst werden.



[[dynamic-multimode-interface-group]]
=== Dynamische Multimode Schnittstellengruppe

Dynamische Multimode-Schnittstellengruppen implementieren das Link Aggregation Control Protocol (LACP), um die Gruppenzugehörigkeit an den direkt angeschlossenen Switch zu übermitteln. LACP ermöglicht die Erkennung von Verbindungsverlusten und der Unfähigkeit des Knotens, mit dem direkt angeschlossenen Switch-Port zu kommunizieren.

Die Implementierung der Dynamisches Multimode Schnittstellengruppe in ONTAP entspricht IEEE 802.3 AD (802.1 AX). ONTAP unterstützt das Port Aggregation Protocol (PAgP) nicht, ein proprietäres Link-Aggregation-Protokoll von Cisco.

Eine Dynamische Multimode Schnittstellengruppe erfordert einen Switch, der LACP unterstützt.

ONTAP implementiert LACP im nicht konfigurierbaren aktiven Modus, der gut mit Switches funktioniert, die entweder im aktiven oder passiven Modus konfiguriert sind. ONTAP implementiert die langen und kurzen LACP-Timer (zur Verwendung mit den nicht konfigurierbaren Werten 3 Sekunden und 90 Sekunden), wie in IEEE 802.3 AD (802.1AX) spezifiziert.

Der ONTAP Algorithmus zur Lastverteilung bestimmt den Mitgliedsport, der für die Übertragung ausgehenden Datenverkehrs verwendet wird, und steuert nicht, wie eingehende Frames empfangen werden. Der Switch bestimmt das Mitglied (den einzelnen physischen Port) seiner Portkanalgruppe, das für die Übertragung verwendet wird, basierend auf dem in der Portkanalgruppe des Switch konfigurierten Lastverteilungsalgorithmus. Daher legt die Switch-Konfiguration fest, welcher Mitgliedsport (einzelner physischer Port) des Speichersystems den Datenverkehr empfängt. Weitere Informationen zur Konfiguration des Switch finden sich in der Dokumentation Ihres Switch-Anbieters.

Wenn eine einzelne Schnittstelle keine aufeinanderfolgenden LACP-Protokollpakete empfängt, wird diese einzelne Schnittstelle in der Ausgabe des Befehls "ifgrp status" als "lag_inactive" markiert. Bestehender Datenverkehr wird automatisch auf alle verbleibenden aktiven Schnittstellen umgeleitet.

Für die Verwendung von Dynamisches Multimode Schnittstellengruppen gelten die folgenden Regeln:

* Dynamische Multimode Schnittstellengruppen sollten so konfiguriert werden, dass sie die Port-basierte, IP-basierte, MAC-basierte oder Round-Robin-Lastverteilungsmethode verwenden.
* In einer Dynamischen Multimode Schnittstellengruppe müssen alle Schnittstellen aktiv sein und eine gemeinsame MAC-Adresse verwenden.

[[load-balancing-in-multimode-interface-groups]]
=== Lastausgleich in Multimode Schnittstellengruppen

Es kann sichergestellt werden, dass alle Schnittstellen einer Multimode Schnittstellengruppe gleichermaßen für ausgehenden Datenverkehr genutzt werden, indem die Methoden des IP-Adressen-, MAC-Adressen-, sequenziellen oder Port-basierten Lastausgleichs verwendet werden, um den Netzwerkverkehr gleichmäßig auf die Netzwerkports einer Multimode Schnittstellengruppe zu verteilen.

Die Methode zum Lastausgleich für eine Multimode Schnittstellengruppe kann nur bei der Erstellung der Schnittstellengruppe festgelegt werden.

*Best Practice*: Port-basiertes Load Balancing wird nach Möglichkeit empfohlen. Port-basiertes Load Balancing sollte verwendet werden, es sei denn, es liegt ein spezifischer Grund oder eine Einschränkung im Netzwerk vor, die dies verhindert.



==== Port-basierter Lastausgleich

Port-basierter Lastausgleich ist die empfohlene Methode.

Sie können den Datenverkehr auf einer Multimode Schnittstellengruppe basierend auf den Transportschicht-Ports (TCP/UDP) mithilfe der Port-basierten Lastverteilungsmethode ausgleichen.

Die Port-basierte Lastverteilungsmethode verwendet einen schnellen Hashing-Algorithmus für die Quell- und Ziel-IP-Adressen sowie die Portnummer der Transportschicht.



==== Lastverteilung von IP-Adresse und MAC-Adresse

Die Lastverteilung über IP-Adresse und MAC-Adresse sind die Methoden zur Angleichung des Datenverkehrs auf Multimode-Schnittstellengruppen.

Diese Lastverteilungsverfahren nutzen einen schnellen Hash-Algorithmus für die Quell- und Zieladressen (IP-Adresse und MAC-Adresse). Wenn das Ergebnis des Hash-Algorithmus einer Schnittstelle zugeordnet wird, die sich nicht im UP-Link-State befindet, wird die nächste aktive Schnittstelle verwendet.


NOTE: Die MAC-Adress-Lastverteilungsmethode sollte beim Erstellen von Schnittstellengruppen auf einem System, das direkt mit einem Router verbunden ist, nicht ausgewählt werden. In einer solchen Konfiguration ist bei jedem ausgehenden IP-Frame die Ziel-MAC-Adresse die MAC-Adresse des Routers. Dadurch wird nur eine Schnittstelle der Schnittstellengruppe verwendet.

Die Lastverteilung von IP-Adressen funktioniert für IPv4- und IPv6-Adressen auf die gleiche Weise.



==== Sequenzieller Lastausgleich

Sie können sequenzielles Load Balancing verwenden, um Pakete mithilfe eines Round-Robin-Algorithmus gleichmäßig auf mehrere Verbindungen zu verteilen. Die sequenzielle Option ermöglicht das Load Balancing des Datenverkehrs einer einzelnen Verbindung über mehrere Verbindungen, um den Durchsatz dieser Verbindung zu erhöhen.

Da sequenzieller Lastausgleich jedoch zu einer falschen Reihenfolge der Paketzustellung führen kann, kann dies eine extrem schlechte Leistung zur Folge haben. Daher wird sequenzieller Lastausgleich im Allgemeinen nicht empfohlen.



[[create-an-interface-group-or-lag]]
== Eine Schnittstellengruppe oder LAG erstellen

Es besteht die Möglichkeit, eine Schnittstellengruppe oder LAG (Single-Mode, statischer Multimode oder Dynamisches Multimode (LACP)) zu erstellen, um durch die Kombination der Fähigkeiten der aggregierten Netzwerkports eine einzige Schnittstelle für Clients bereitzustellen.

Die Vorgehensweise hängt von der verwendeten Schnittstelle ab, System Manager oder CLI:

[role="tabbed-block"]
====
.System Manager
--
*Mit dem System Manager ein LAG erstellen*

.Schritte
. *Netzwerk > Ethernet port > + Link Aggregation Group* auswählen, um eine LAG zu erstellen.
. Den Knoten aus der Dropdown-Liste auswählen.
. Wählen Sie aus den folgenden Optionen:
+
.. ONTAP auf *Automatische Auswahl der Broadcast-Domäne (empfohlen)*.
.. Manuelle Auswahl einer Broadcast-Domäne.


. Wählen Sie die Ports aus, die das LAG bilden.
. Modus auswählen:
+
.. Einzeln: Es wird jeweils nur ein Port verwendet.
.. Mehrere: Alle Ports können gleichzeitig verwendet werden.
.. LACP: Das LACP-Protokoll legt die Ports fest, die verwendet werden können.


. Lastverteilung auswählen:
+
.. IP-basiert
.. MAC-basiert
.. Port
.. Sequentiell


. Speichern Sie Ihre Änderungen.


--
.CLI
--
*Die CLI dient zum Erstellen einer Schnittstellengruppe*

Beim Erstellen einer Multimode Schnittstellengruppe kann eine der folgenden Lastverteilungsmethoden angegeben werden:

* `port`: Der Netzwerkverkehr wird auf Basis der Transportschicht-Ports (TCP/UDP) verteilt. Dies ist die empfohlene Methode zur Lastverteilung.
* `mac`: Der Netzwerkverkehr wird auf Basis von MAC-Adressen verteilt.
* `ip`: Der Netzwerkverkehr wird auf Basis von IP-Adressen verteilt.
* `sequential`: Der Netzwerkverkehr wird verteilt, sobald er empfangen wird.



NOTE: Die MAC-Adresse einer Schnittstellengruppe wird durch die Reihenfolge der zugrunde liegenden Ports und deren Initialisierung beim Systemstart bestimmt. Es sollte daher nicht davon ausgegangen werden, dass die MAC-Adresse der Schnittstellengruppe nach Neustarts oder ONTAP Upgrades erhalten bleibt.

.Schritt
Mit dem  `network port ifgrp create` Befehl wird eine Schnittstellengruppe erstellt.

Schnittstellengruppen müssen gemäß der Syntax `a<number><letter>` benannt werden. Beispielsweise sind a0a, a0b, a1c und a2a gültige Schnittstellengruppennamen.

Weitere Informationen zu  `network port ifgrp create` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/network-port-ifgrp-create.html["ONTAP-Befehlsreferenz"^].

Das folgende Beispiel zeigt, wie eine Schnittstellengruppe namens a0a mit der Verteilungsfunktion Port und dem Modus Multimode erstellt wird:

`network port ifgrp create -node _cluster-1-01_ -ifgrp _a0a_ -distr-func _port_ -mode _multimode_`

[[add-a-port-to-an-interface-group-or-lag]]
== Einen Port zu einer Schnittstellengruppe oder einem LAG hinzufügen

Es ist möglich, bis zu 16 physische Ports zu einer Schnittstellengruppe oder einem LAG für alle Portgeschwindigkeiten hinzuzufügen.

Die Vorgehensweise hängt von der verwendeten Schnittstelle ab, System Manager oder CLI:

[role="tabbed-block"]
====
.System Manager
--
*Mit dem System Manager einen Port zu einem LAG hinzufügen*

.Schritte
. *Netzwerk > Ethernet-Port > LAG* auswählen, um ein LAG zu bearbeiten.
. Zusätzliche Ports auf demselben Knoten können zum LAG hinzugefügt werden.
. Speichern Sie Ihre Änderungen.


--
.CLI
--
*CLI verwenden, um Ports zu einer Schnittstellengruppe hinzuzufügen*

.Schritt
Netzwerk-Ports zur Schnittstellengruppe hinzufügen:

`network port ifgrp add-port`

Das folgende Beispiel zeigt, wie Port e0c zu einer Schnittstellengruppe namens a0a hinzugefügt wird:

`network port ifgrp add-port -node _cluster-1-01_ -ifgrp _a0a_ -port _e0c_`

Ab ONTAP 9.8 werden Schnittstellengruppen automatisch etwa eine Minute nach dem Hinzufügen des ersten physischen Ports zur Schnittstellengruppe in eine geeignete Broadcast-Domäne aufgenommen. Wenn ONTAP dies nicht automatisch übernehmen soll und stattdessen die manuelle Zuordnung der ifgrp zu einer Broadcast-Domäne bevorzugt wird, ist der  `-skip-broadcast-domain-placement` Parameter als Teil des  `ifgrp add-port` Befehls anzugeben.

Weitere Informationen zu  `network port ifgrp add-port` und zu den Konfigurationseinschränkungen, die für Port-Schnittstellengruppen gelten, finden Sie im link:https://docs.netapp.com/us-en/ontap-cli/network-port-ifgrp-add-port.html["ONTAP-Befehlsreferenz"^].

[[remove-a-port-from-an-interface-group-or-lag]]
== Einen Port aus einer Schnittstellengruppe oder einem LAG entfernen

Sie können einen Port aus einer Schnittstellengruppe entfernen, die LIFs hostet, sofern es sich nicht um den letzten Port in der Schnittstellengruppe handelt. Es besteht keine Voraussetzung, dass die Schnittstellengruppe keine LIFs hosten darf oder dass die Schnittstellengruppe nicht der Home-Port eines LIFs sein darf, solange nicht der letzte Port aus der Schnittstellengruppe entfernt wird. Wenn jedoch der letzte Port entfernt wird, müssen die LIFs zuvor aus der Schnittstellengruppe migriert oder verschoben werden.

.Über diese Aufgabe
Sie können bis zu 16 Ports (physische Schnittstellen) aus einer Schnittstellengruppe oder einem LAG entfernen.

Die Vorgehensweise hängt von der verwendeten Schnittstelle ab, System Manager oder CLI:

[role="tabbed-block"]
====
.System Manager
--
*Mit dem System Manager lässt sich ein Port aus einem LAG entfernen.*

.Schritte
. *Netzwerk > Ethernet-Port > LAG* auswählen, um ein LAG zu bearbeiten.
. Wählen Sie die Ports aus, die aus dem LAG entfernt werden.
. Speichern Sie Ihre Änderungen.


--
.CLI
--
*CLI zur Entfernung von Ports aus einer Schnittstellengruppe verwenden*

.Schritt
Netzwerkports aus einer Schnittstellengruppe entfernen:

`network port ifgrp remove-port`

Weitere Informationen zu  `network port ifgrp remove-port` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/network-port-ifgrp-remove-port.html["ONTAP-Befehlsreferenz"^].

Das folgende Beispiel zeigt, wie Port e0c aus einer Schnittstellengruppe namens a0a entfernt wird:

`network port ifgrp remove-port -node _cluster-1-01_ -ifgrp _a0a_ -port _e0c_`

--

[[delete-an-interface-group-or-lag]]
== Eine Schnittstellengruppe oder LAG löschen

Schnittstellengruppen oder LAGs können gelöscht werden, wenn LIFs direkt auf den zugrunde liegenden physischen Ports konfiguriert werden sollen oder wenn entschieden wird, die Schnittstellengruppe, den LAG-Modus oder die Verteilungsfunktion zu ändern.

.Bevor Sie beginnen
* Die Schnittstellengruppe oder LAG darf keine LIF hosten.
* Die Schnittstellengruppe oder LAG darf weder der Home-Port noch das Failover-Ziel eines LIF sein.


Die Vorgehensweise hängt von der verwendeten Schnittstelle ab, System Manager oder CLI:

[role="tabbed-block"]
====
.System Manager
--
*Mit dem System Manager ein LAG löschen*

.Schritte
. *Netzwerk > Ethernet-Port > LAG* auswählen, um ein LAG zu löschen.
. Wählen Sie das LAG aus, das entfernt werden soll.
. Löschen Sie den LAG.


--
.CLI
--
*Die CLI kann zum Löschen einer Schnittstellengruppe verwendet werden*

.Schritt
Mit dem  `network port ifgrp delete` Befehl wird eine Schnittstellengruppe gelöscht.

Weitere Informationen zu  `network port ifgrp delete` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/network-port-ifgrp-delete.html["ONTAP-Befehlsreferenz"^].

Das folgende Beispiel zeigt, wie eine Schnittstellengruppe mit dem Namen a0b gelöscht wird:

`network port ifgrp delete -node _cluster-1-01_ -ifgrp _a0b_`
