= ONTAP-Port-Erreichbarkeit reparieren
:hardbreaks:
:allow-uri-read: 
:nofooter: 
:icons: font
:linkattrs: 
:imagesdir: ../media/


[role="lead"]
Broadcast-Domänen werden automatisch erstellt. Wenn jedoch ein Port neu verkabelt wird oder sich die Switch-Konfiguration ändert, muss ein Port möglicherweise einer anderen Broadcast-Domäne (neu oder bereits vorhanden) zugeordnet werden.

ONTAP kann automatisch Netzwerkverkabelungsprobleme erkennen und Lösungen dafür empfehlen, basierend auf der Layer-2-Erreichbarkeit eines Broadcast-Domänenbestandteils (ethernet ports).

Eine fehlerhafte Verkabelung während des Betriebs kann zu einer unerwarteten Zuweisung von Broadcast-Domänenports führen. Ab ONTAP 9.10.1 überprüft das Cluster automatisch Netzwerkverkabelungsprobleme, indem die Porterreichbarkeit nach der Cluster-Einrichtung oder wenn ein neuer Knoten einem bestehenden Cluster beitritt, verifiziert wird.

[role="tabbed-block"]
====
.System Manager
--
Wird ein Problem mit der Port-Erreichbarkeit festgestellt, empfiehlt der System Manager eine Reparaturmaßnahme, um das Problem zu beheben.

Nach der Einrichtung des Clusters werden Netzwerkverkabelungsprobleme im Dashboard gemeldet.

Nach dem Hinzufügen eines neuen Knotens zu einem Cluster treten auf der Seite „Knoten“ Netzwerkverkabelungsprobleme auf.

Sie können den Zustand der Netzwerkverkabelung auch im Netzwerkdiagramm einsehen. Probleme mit der Porterreichbarkeit werden im Netzwerkdiagramm durch ein rotes Fehlersymbol angezeigt.

.Cluster-Einrichtung nach dem Aufbau
Nach der Einrichtung des Clusters erscheint im Dashboard eine Meldung, wenn das System ein Problem mit der Netzwerkverkabelung erkennt.

image:auto-detect-01.png["Nachricht nach Cluster-Einrichtung"]

.Schritte
. Die Verkabelung sollte wie in der Nachricht vorgeschlagen korrigiert werden.
. Klicken Sie auf den Link, um das Dialogfeld „Broadcast-Domänen aktualisieren“ zu öffnen. Das Dialogfeld „Broadcast-Domänen aktualisieren“ wird geöffnet. image:auto-detect-02.png["Dialogfeld „Broadcast Domains aktualisieren“"]
. Die Informationen zum Port, einschließlich des Node-Namens, der Probleme, der aktuellen Broadcast-Domäne und der erwarteten Broadcast-Domäne, werden angezeigt.
. Wählen Sie die Ports aus, die Sie reparieren möchten, und klicken Sie auf *Fix*. Das System verschiebt die Ports von der aktuellen Broadcast-Domäne in die erwartete Broadcast-Domäne.


.Post-Node-Beitritt
Nach dem Hinzufügen eines neuen Knotens zu einem Cluster erscheint auf der Seite „Knoten“ eine Meldung, wenn das System ein Problem mit der Netzwerkverkabelung erkennt.

image:auto-detect-03.png["Nachricht nach dem Node-Beitritt"]

.Schritte
. Die Verkabelung sollte wie in der Nachricht vorgeschlagen korrigiert werden.
. Klicken Sie auf den Link, um das Dialogfeld „Broadcast-Domänen aktualisieren“ zu öffnen. Das Dialogfeld „Broadcast-Domänen aktualisieren“ wird geöffnet. image:auto-detect-02.png["Dialogfeld „Broadcast Domains aktualisieren“"]
. Die Informationen zum Port, einschließlich des Node-Namens, der Probleme, der aktuellen Broadcast-Domäne und der erwarteten Broadcast-Domäne, werden angezeigt.
. Wählen Sie die Ports aus, die Sie reparieren möchten, und klicken Sie auf *Fix*. Das System verschiebt die Ports von der aktuellen Broadcast-Domäne in die erwartete Broadcast-Domäne.

.CLI
--
.Bevor Sie beginnen
Sie müssen Cluster-Administrator sein, um diese Aufgabe ausführen zu können.

.Über diese Aufgabe
Ein Befehl steht zur Verfügung, um die Broadcast-Domänenkonfiguration für einen Port basierend auf der von ONTAP erkannten Layer-2-Erreichbarkeit automatisch zu reparieren.

.Schritte
. Die Switch-Konfiguration und Verkabelung überprüfen.
. Die Erreichbarkeit des Ports prüfen:
+
`network port reachability show -detail -node -port`

+
Die Befehlsausgabe enthält die Ergebnisse der Erreichbarkeitsprüfung.

+
Weitere Informationen zu  `network port reachability show` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/network-port-reachability-show.html["ONTAP-Befehlsreferenz"^].

. Der folgende Entscheidungsbaum und die Tabelle bieten eine Übersicht über die Erreichbarkeitsergebnisse und zeigen auf, ob und gegebenenfalls welche weiteren Schritte erforderlich sind.
+
image:ontap_nm_image1.png["Entscheidungsbaum zur Reparaturerreichbarkeit"]



[cols="20,80"]
|===


| Erreichbarkeitsstatus | Beschreibung 


 a| 
OK
 a| 
Der Port verfügt über Layer-2-Erreichbarkeit zu seiner zugewiesenen Broadcast-Domäne. Wenn der Erreichbarkeitsstatus „ok“ ist, aber „unerwartete Ports“ vorhanden sind, kann das Zusammenführen einer oder mehrerer Broadcast-Domänen in Betracht gezogen werden. Weitere Informationen stehen in der folgenden Zeile _Unexpected ports_.

Wenn der Erreichbarkeitsstatus „ok“ ist, aber „nicht erreichbare Ports“ vorhanden sind, kann das Aufteilen einer oder mehrerer Broadcast-Domänen in Erwägung gezogen werden. Weitere Informationen enthält die folgende Zeile _Nicht erreichbare Ports_.

Wenn der Erreichbarkeitsstatus "ok" ist und keine unerwarteten oder nicht erreichbaren Ports vorhanden sind, ist Ihre Konfiguration korrekt.



 a| 
Unerwartete Ports
 a| 
Der Port ist über Layer 2 zu seiner zugewiesenen Broadcast-Domäne erreichbar; er ist jedoch auch über Layer 2 zu mindestens einer weiteren Broadcast-Domäne erreichbar.

Die physische Konnektivität und die Switch-Konfiguration sollten überprüft werden, um festzustellen, ob sie fehlerhaft sind oder ob die dem Port zugewiesene Broadcast-Domäne mit einer oder mehreren Broadcast-Domänen zusammengeführt werden muss.

Weitere Informationen finden sich unter link:merge_broadcast_domains.html["Zusammenführen von Broadcast-Domänen"].



 a| 
Nicht erreichbare Ports
 a| 
Wenn eine einzelne Broadcast-Domäne in zwei verschiedene Erreichbarkeitsgruppen aufgeteilt wurde, kann eine Broadcast-Domäne aufgeteilt werden, um die ONTAP Konfiguration mit der physischen Netzwerktopologie zu synchronisieren.

Die Liste der nicht erreichbaren Ports definiert üblicherweise die Menge der Ports, die in eine andere Broadcast-Domäne aufgeteilt werden sollten, nachdem überprüft wurde, dass die physische und die Switch-Konfiguration korrekt sind.

Weitere Informationen finden sich unter link:split_broadcast_domains.html["Aufgeteilte Broadcast-Domänen"].



 a| 
Fehlkonfigurierte Erreichbarkeit
 a| 
Der Port hat keine Layer-2-Erreichbarkeit zu seiner zugewiesenen Broadcast-Domäne, jedoch hat der Port Layer-2-Erreichbarkeit zu einer anderen Broadcast-Domäne.

Die Port-Erreichbarkeit kann wiederhergestellt werden. Bei Ausführung des folgenden Befehls weist das System den Port der Broadcast-Domäne zu, zu der er erreichbar ist:

`network port reachability repair -node -port`



 a| 
Unerreichbarkeit
 a| 
Der Port hat keine Layer-2-Erreichbarkeit zu einer bestehenden Broadcast-Domäne.

Die Port-Erreichbarkeit kann wiederhergestellt werden. Beim Ausführen des folgenden Befehls weist das System den Port einer neu automatisch erstellten Broadcast-Domäne im Default-IPspace zu:

`network port reachability repair -node -port`

*Hinweis:* Wenn alle Mitgliedsports einer Schnittstellengruppe (ifgrp)  `no-reachability` melden, würde das Ausführen des  `network port reachability repair` Befehls auf jedem Mitgliedsport dazu führen, dass jeder aus der ifgrp entfernt und einer neuen Broadcast-Domäne zugeordnet wird, was schließlich dazu führt, dass die ifgrp selbst entfernt wird. Vor dem Ausführen des  `network port reachability repair` Befehls sollte überprüft werden, ob die erreichbare Broadcast-Domäne des Ports der erwarteten Konfiguration entsprechend der physischen Netzwerktopologie entspricht.

Weitere Informationen zu  `network port reachability repair` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/network-port-reachability-repair.html["ONTAP-Befehlsreferenz"^].



 a| 
Erreichbarkeit über mehrere Domänen hinweg
 a| 
Der Port ist über Layer 2 zu seiner zugewiesenen Broadcast-Domäne erreichbar; er ist jedoch auch über Layer 2 zu mindestens einer weiteren Broadcast-Domäne erreichbar.

Die physische Konnektivität und die Switch-Konfiguration sollten überprüft werden, um festzustellen, ob sie fehlerhaft sind oder ob die dem Port zugewiesene Broadcast-Domäne mit einer oder mehreren Broadcast-Domänen zusammengeführt werden muss.

Weitere Informationen finden sich unter link:merge_broadcast_domains.html["Zusammenführen von Broadcast-Domänen"].



 a| 
unbekannt
 a| 
Wenn der Erreichbarkeitsstatus „unbekannt“ ist, einige Minuten abwarten und den Befehl erneut ausführen.

|===
Nach der Reparatur eines Ports prüfen Sie, ob LIFs und VLANs verschoben wurden. Wenn der Port Teil einer Schnittstellengruppe war, ist es außerdem erforderlich, nachzuvollziehen, was mit dieser Schnittstellengruppe geschehen ist.

.LIFs
Wird ein Port repariert und in eine andere Broadcast-Domäne verschoben, werden alle auf dem reparierten Port konfigurierten LIFs automatisch einem neuen Heimatport zugewiesen. Dieser Heimatport wird, wenn möglich, aus derselben Broadcast-Domäne auf demselben Node ausgewählt. Alternativ wird ein Heimatport von einem anderen Node ausgewählt oder, falls keine geeigneten Heimatports existieren, wird der Heimatport entfernt.

Wird der Heimatport einer LIF auf einen anderen Node verschoben oder gelöscht, gilt die LIF als „verschoben“. Diese verschobenen LIFs lassen sich mit dem folgenden Befehl anzeigen:

`displaced-interface show`

Falls LIFs verschoben sind, besteht die Möglichkeit, entweder:

* Die Heimat der verschobenen LIF wiederherstellen:
+
`displaced-interface restore`

* Den Home-Node des LIF manuell festlegen:
+
`network interface modify -home-port -home-node`

+
Weitere Informationen zu  `network interface modify` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/network-interface-modify.html["ONTAP-Befehlsreferenz"^].

* Den Eintrag aus der Tabelle „displaced-interface“ entfernen, falls das aktuell konfigurierte Home des LIF zufriedenstellend ist:
+
`displaced-interface delete`



.VLANs
Falls der reparierte Port VLANs enthielt, werden diese VLANs automatisch gelöscht, aber auch als „verschoben“ protokolliert. Diese verschobenen VLANs können angezeigt werden:

`displaced-vlans show`

Falls verschobene VLANs vorhanden sind, müssen Sie entweder:

* Die VLANs auf einen anderen Port wiederherstellen:
+
`displaced-vlans restore`

* Den Eintrag aus der Tabelle „displaced-vlans“ entfernen:
+
`displaced-vlans delete`



.Schnittstellengruppen
Wenn der reparierte Port Teil einer Schnittstellengruppe war, wird er aus dieser Schnittstellengruppe entfernt. Wenn er der einzige zugewiesene Port der Schnittstellengruppe war, wird die Schnittstellengruppe selbst entfernt.
