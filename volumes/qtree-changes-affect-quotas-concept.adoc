= Erfahren Sie, wie sich Änderungen am ONTAP-Qtree auf Kontingente auswirken
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Wenn ein Qtree gelöscht, umbenannt oder dessen Sicherheitsstil geändert wird, können sich die von ONTAP angewendeten Quotas ändern, abhängig von den aktuell angewendeten Quotas.



[[qtree-deletions-and-tree-quotas]]
== Qtree-Löschvorgänge und Baumquoten

Wenn ein Qtree gelöscht wird, gelten alle für diesen Qtree zutreffenden Quoten, unabhängig davon, ob sie explizit oder abgeleitet sind, nicht mehr in ONTAP.

Ob die Quotenregeln bestehen bleiben, hängt davon ab, wo der Qtree gelöscht wird:

* Wenn ein Qtree mit ONTAP gelöscht wird, werden die Quota-Regeln für diesen Qtree automatisch gelöscht, einschließlich der Baum-Quota-Regeln sowie aller für diesen Qtree konfigurierten Benutzer- und Gruppenkontingent-Regeln.
* Wenn Sie einen Qtree mit Ihrem CIFS- oder NFS-Client löschen, müssen Sie alle Quota-Regeln für diesen Qtree löschen, um Fehler bei der Quota-Neuinitialisierung zu vermeiden. Wenn Sie einen neuen Qtree mit demselben Namen wie den gelöschten erstellen, werden die bestehenden Quota-Regeln erst nach der Quota-Neuinitialisierung auf den neuen Qtree angewendet.




[[how-renaming-a-qtree-affects-quotas]]
== Wie sich die Umbenennung eines Qtree auf die Quoten auswirkt

Wenn ein Qtree mit ONTAP umbenannt wird, werden die Quota-Regeln für diesen Qtree automatisch aktualisiert. Wenn ein Qtree mit dem CIFS- oder NFS-Client umbenannt wird, müssen die Quota-Regeln für diesen Qtree aktualisiert werden.


NOTE: Wenn Sie einen Qtree mithilfe Ihres CIFS- oder NFS-Client umbenennen und die Quota-Regeln für diesen Qtree nicht mit dem neuen Namen aktualisieren, bevor Sie die Quotas neu initialisieren, werden keine Quotas auf den Qtree angewendet. Explizite Quotas für den Qtree, einschließlich Baum-Quotas sowie Benutzer- oder Gruppen-Quotas für den Qtree, können in abgeleitete Quotas umgewandelt werden.



[[qtree-security-styles-and-user-quotas]]
== Qtree Sicherheitsstile und Benutzerkontingente

Zugriffskontrolllisten (ACLs) können auf qtrees mit NTFS- oder gemischten Sicherheitsstilen angewendet werden, jedoch nicht mit dem UNIX-Sicherheitsstil. Die Änderung des Sicherheitsstils eines qtrees kann Auswirkungen auf die Berechnung von Kontingenten haben. Nach jeder Änderung des Sicherheitsstils eines qtrees sollten die Kontingente neu initialisiert werden.

Wenn der Sicherheitsstil eines Qtree von NTFS oder gemischt auf UNIX geändert wird, werden alle ACLs für Dateien in diesem Qtree ignoriert und die Dateinutzung wird den UNIX-Benutzerkennungen in Rechnung gestellt.

Wenn Sie den Sicherheitsstil eines Qtrees von UNIX auf gemischt oder NTFS ändern, werden die zuvor ausgeblendeten ACLs sichtbar. Außerdem werden alle ignorierten ACLs wieder wirksam und die NFS-Benutzerinformationen werden ignoriert. Falls zuvor keine ACL vorhanden war, werden die NFS-Informationen weiterhin für die Quota-Berechnung verwendet.


NOTE: Um sicherzustellen, dass die Kontingentnutzung sowohl für UNIX- als auch für Windows-Benutzer nach der Änderung des Sicherheitsstils eines Qtrees korrekt berechnet wird, müssen die Kontingente für das Volume, das diesen Qtree enthält, neu initialisiert werden.

.Beispiel
Das folgende Beispiel zeigt, wie eine Änderung des Sicherheitsstils eines Qtrees dazu führt, dass einem anderen Benutzer die Nutzung einer Datei in diesem Qtree in Rechnung gestellt wird.

Angenommen, die NTFS-Sicherheit ist für qtree A aktiv und eine ACL weist Windows-Benutzer `corp\joe` den Besitz einer 5MB-Datei zu. Benutzer `corp\joe` wird eine Speichernutzung von 5MB für qtree A angerechnet.

Nun wird der Sicherheitsstil von qtree A von NTFS auf UNIX geändert. Nach der Neuinitialisierung der Quotas wird der Windows-Benutzer `corp\joe` nicht mehr für diese Datei belastet; stattdessen wird der UNIX-Benutzer, der der UID der Datei entspricht, für die Datei belastet. Die UID kann ein UNIX-Benutzer sein, der auf `corp\joe` abgebildet ist, oder der Root-Benutzer.
