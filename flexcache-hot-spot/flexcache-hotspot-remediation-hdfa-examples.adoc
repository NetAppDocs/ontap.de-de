= ONTAP FlexCache-Dichte ermitteln
:allow-uri-read: 


[role="lead"]
Die erste Entscheidung bei der Hotspot-Behebung besteht darin, die FlexCache-Dichte zu bestimmen. Die folgenden Beispiele beziehen sich auf Cluster mit vier Knoten. Es wird angenommen, dass die Dateianzahl gleichmäßig auf alle Komponenten jedes HDF verteilt ist. Es wird außerdem von einer gleichmäßigen Verteilung der Frontend-NAS-Verbindungen auf alle Knoten ausgegangen.

Auch wenn diese Beispiele nicht die einzigen möglichen Konfigurationen sind, ist das grundlegende Designprinzip zu verstehen: Es sollten so viele HDFs erstellt werden, wie es die Platzanforderungen und verfügbaren Ressourcen erlauben.


NOTE: HDFAs werden mit folgender Syntax dargestellt: `HDFs per HDFA x nodes per HDF x constituents per node per HDF`



[[2x2x2-hdfa-configuration]]
== 2x2x2 HDFA Konfiguration

<<Figure-1,Abbildung 1>> ist ein Beispiel für eine 2x2x2 HDFA Konfiguration: zwei HDFs, die sich jeweils über zwei Knoten erstrecken, und jeder Knoten enthält zwei zugehörige Volumes. In diesem Beispiel hat jeder Client eine 50%ige Chance auf direkten Zugriff auf die Hot File. Zwei der vier Clients haben Ost-West-Verkehr. Wichtig ist, dass nun zwei HDFs existieren, was zwei separate Caches der Hot File bedeutet. Es gibt nun zwei CPUs/Volume-Affinitäten, die den Zugriff auf die Hot File bedienen.

[[4x1x4-hdfa-configuration]]
== 4x1x4 HDFA Konfiguration

<<Figure-2,Abbildung 2>> stellt eine optimale Konfiguration dar. Dies ist ein Beispiel für eine 4x1x4 HDFA Konfiguration: vier HDFs, jeweils auf einen einzelnen Knoten beschränkt, und jeder Knoten enthält vier Bestandteile. In diesem Beispiel hat jeder Client garantiert direkten Zugriff auf einen Cache der Hot File. Da vier zwischengespeicherte Dateien auf vier verschiedenen Knoten liegen, unterstützen vier verschiedene CPUs/Volume-Affinitäten den Zugriff auf die Hot File. Zusätzlich entsteht kein Ost-West-Verkehr.
