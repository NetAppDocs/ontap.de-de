= Clients für die Verteilung von ONTAP NAS-Verbindungen konfigurieren
:allow-uri-read: 


[role="lead"]
Um Hotspotting zu beheben, sollte der Client ordnungsgemäß konfiguriert werden, damit er seinen Beitrag zur Vermeidung von CPU-Engpässen leistet.



[[linux-client-configuration]]
== Linux-Client-Konfiguration

Unabhängig davon, ob eine intra-SVM- oder inter-SVM-HDFA-Bereitstellung gewählt wurde, sollte  `autofs` unter Linux verwendet werden, um sicherzustellen, dass die Clients den Lastausgleich über die verschiedenen HDFs durchführen. Die  `autofs` Konfiguration unterscheidet sich für inter- und intra-SVM.

.Bevor Sie beginnen
Sie benötigen  `autofs` und die entsprechenden Abhängigkeiten. Hierzu bietet die Linux-Dokumentation Unterstützung.

.Über diese Aufgabe
Die beschriebenen Schritte verwenden eine Beispiel `/etc/auto_master`datei mit folgendem Eintrag:

[listing]
----
/flexcache auto_hotspot
----
Diese Konfiguration  `autofs`sucht nach einer Datei namens  `auto_hotspot`im  `/etc`Verzeichnis, sobald ein Prozess versucht, auf das  `/flexcache`Verzeichnis zuzugreifen. Der Inhalt der  `auto_hotspot`Datei legt fest, welcher NFS-Server und welcher Verbindungspfad im  `/flexcache`Verzeichnis eingebunden werden. Die beschriebenen Beispiele sind verschiedene Konfigurationen für die  `auto_hotspot`Datei.



[[intra-svm-hdfa-autofs-configuration]]
=== Intra-SVM HDFA autofs Konfiguration

Im folgenden Beispiel wird eine  `autofs`Map für das Diagramm in <<Figure-1,Abbildung 1>> erstellt. Da jeder Cache denselben Verbindungspfad hat und der Hostname  `cache` vier DNS A-Einträge besitzt, ist nur eine Zeile erforderlich:

[listing]
----
hotspot cache:/hotspot
----
Diese einfache Zeile veranlasst den NFS-Client, eine DNS-Abfrage für den Hostnamen `cache` durchzuführen. DNS ist so eingerichtet, dass die IP-Adressen im Round-Robin-Verfahren zurückgegeben werden. Dies führt zu einer gleichmäßigen Verteilung der Frontend NAS-Verbindungen. Nachdem der Client die IP-Adresse erhalten hat, wird der Verbindungspfad `/hotspot` bei `/flexcache/hotspot` eingebunden. Es kann eine Verbindung zu SVM1, SVM2, SVM3 oder SVM4 bestehen, wobei die jeweilige SVM keine Rolle spielt.

[[intra-svm-hdfa-autofs-configuration]]
=== Intra-SVM HDFA autofs Konfiguration

Im folgenden Beispiel wird eine  `autofs`Map für das Diagramm in <<Figure-2,Abbildung 2>> erstellt. Es muss sichergestellt werden, dass die NFS-Clients die IPs einbinden, die Teil der HDF-Verbindungspfad-Bereitstellung sind. Anders ausgedrückt: Es soll  `/hotspot1`nur mit der IP 192.168.0.11 eingebunden werden. Hierzu können alle vier IP/Verbindungspfad-Paare für einen lokalen Mountpunkt in der  `auto_hotspot`Map aufgeführt werden.


NOTE: Der Backslash (`\`) im folgenden Beispiel setzt den Eintrag in der nächsten Zeile fort, was das Lesen erleichtert.

[listing]
----
hotspot     cache1:/hostspot1 \
            cache2:/hostspot2 \
            cache3:/hostspot3 \
            cache4:/hostspot4
----
Wenn der Client versucht, auf `/flexcache/hotspot`, `autofs` zuzugreifen, wird ein Forward-Lookup für alle vier Hostnamen durchgeführt. Angenommen, alle vier IPs befinden sich entweder im selben Subnetz wie der Client oder in einem anderen Subnetz, `autofs` wird ein NFS-NULL-Ping an jede IP gesendet.

Dieser NULL-Ping erfordert die Verarbeitung des Pakets durch den ONTAP NFS-Service, jedoch keinen Festplattenzugriff. Der erste zurückgegebene Ping ist die IP-Adresse und der Verbindungspfad `autofs`, die für das Mounten ausgewählt werden.

[[windows-client-configuration]]
== Windows Clientkonfiguration

Bei Windows-Clients empfiehlt sich die Verwendung einer intra-SVM HDFA. Für den Lastausgleich zwischen den verschiedenen HDFs in der SVM ist es erforderlich, jeder HDF einen eindeutigen Freigabenamen zuzuweisen. Danach sind die Schritte in link:https://learn.microsoft.com/en-us/windows-server/storage/dfs-namespaces/create-a-dfs-namespace["Microsoft Dokumentation"^] zu befolgen, um mehrere DFS-Ziele für denselben Ordner zu implementieren.
