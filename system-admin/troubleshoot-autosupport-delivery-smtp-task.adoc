= Fehlerbehebung bei der ONTAP AutoSupport Nachrichtenübermittlung über SMTP
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Wenn das System AutoSupport-Nachrichten nicht über SMTP zustellen kann, lassen sich verschiedene Einstellungen überprüfen, um das Problem zu lösen.

.Bevor Sie beginnen
Die grundlegende Netzwerkverbindung und die DNS-Auflösung sollten bestätigt worden sein:

* Ihre Node Management LIF muss für den Betriebs- und Administrationsstatus aktiviert sein.
* Es muss möglich sein, einen funktionierenden Host im selben Subnetz vom Cluster-Management-LIF aus anzupingen (nicht von einem LIF auf einem der Knoten).
* Von der Cluster-Management-LIF aus muss ein funktionierender Host außerhalb des Subnetzes angepingt werden können.
* Es muss möglich sein, einen funktionierenden Host außerhalb des Subnetzes vom Cluster-Management-LIF aus mit dem Namen des Hosts (nicht der IP-Adresse) anzupingen.


.Über diese Aufgabe
Diese Schritte gelten für Fälle, in denen festgestellt wurde, dass AutoSupport die Nachricht generieren kann, aber nicht über SMTP zustellen kann.

Falls Fehler auftreten oder ein Schritt in diesem Verfahren nicht abgeschlossen werden kann, sollte das Problem ermittelt und behoben werden, bevor mit dem nächsten Schritt fortgefahren wird.

Alle Befehle werden an der ONTAP Befehlszeilenschnittstelle eingegeben, sofern nichts anderes angegeben ist.

.Schritte
. Der Status der Node-Management-LIF kann überprüft werden:
+
`*network interface show -home-node local -role node-mgmt -fields vserver,lif,status-oper,status-admin,address,role*`

+
Die Felder  `status-oper` und  `status-admin` sollten  `up` zurückgeben. Weitere Informationen zu  `up` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/up.html["ONTAP-Befehlsreferenz"^].

. Der SVM-Name, der LIF-Name und die LIF-IP-Adresse sollten für die spätere Verwendung festgehalten werden.
. Es ist sicherzustellen, dass DNS aktiviert und korrekt konfiguriert ist:
+
`*vserver services name-service dns show*`

. Alle Server, die für die Verwendung durch AutoSupport konfiguriert sind, werden angezeigt:
+
`*system node autosupport show -fields mail-hosts*`

+
Alle angezeigten Servernamen protokollieren.

. Für jeden im vorherigen Schritt angezeigten Server und `support.netapp.com`die URL ist sicherzustellen, dass der Server bzw. die URL vom Knoten aus erreichbar ist:
+
`*network traceroute -node local -destination _server_name_*`

+
Falls eine dieser Routen nicht funktioniert, kann dieselbe Route von einem funktionierenden Host im selben Subnetz wie der Cluster mit dem Dienstprogramm „traceroute“ oder „tracert“, das in den meisten Netzwerkclients von Drittanbietern enthalten ist, ausprobiert werden. Dies unterstützt bei der Feststellung, ob das Problem in der Netzwerkkonfiguration oder der Clusterkonfiguration liegt.

. Melden Sie sich bei dem als Mail-Host festgelegten Host an und stellen Sie sicher, dass er SMTP-Anfragen bearbeiten kann:
+
`*netstat -aAn|grep 25*`

+
`25` ist die SMTP-Portnummer des Listeners.

+
Es wird eine Meldung angezeigt, die in etwa folgendem Text entspricht:

+
[listing]
----
ff64878c tcp        0      0  *.25    *.*    LISTEN.
----
. Von einem anderen Host aus eine Telnet-Sitzung mit dem SMTP-Port des Mail-Hosts öffnen:
+
`*telnet _mailhost_ 25*`

+
Es wird eine Meldung angezeigt, die in etwa folgendem Text entspricht:

+
[listing]
----

220 filer.yourco.com Sendmail 4.1/SMI-4.1 ready at Thu, 30 Nov 2014 10:49:04 PST
----
. An der Telnet-Eingabeaufforderung sollte sichergestellt sein, dass eine Nachricht von Ihrem Mail-Host weitergeleitet werden kann:
+
`*HELO _domain_name_*`

+
`*MAIL FROM: _your_email_address_*`

+
`*RCPT TO: \autosupport@netapp.com*`

+
`domain_name` ist der Domänenname Ihres Netzwerks.

+
Wird eine Fehlermeldung zurückgegeben, die besagt, dass die Weiterleitung verweigert wurde, ist die Weiterleitung auf dem Mailhost nicht aktiviert. Der Systemadministrator sollte kontaktiert werden.

. An der Telnet-Eingabeaufforderung wird eine Testnachricht gesendet:
+
`*DATA*`

+
`*SUBJECT: TESTING*`
`*THIS IS A TEST*`

+
`*.*`

+
[NOTE]
====
Stellen Sie sicher, dass Sie den letzten Punkt (.) in einer eigenen Zeile eingeben. Der Punkt zeigt dem Mail-Host an, dass die Nachricht vollständig ist.

====
+
Wenn ein Fehler zurückgegeben wird, ist Ihr Mailserver nicht korrekt konfiguriert. Der Systemadministrator sollte kontaktiert werden.

. Über die ONTAP-Befehlszeilenschnittstelle eine AutoSupport-Testnachricht an eine vertrauenswürdige E-Mail-Adresse senden, auf die Sie Zugriff haben:
+
`*system node autosupport invoke -node local -type test*`

. Die Sequenznummer des Versuchs finden:
+
`*system node autosupport history show -node local -destination smtp*`

+
Die Sequenznummer des Versuchs lässt sich anhand des Zeitstempels bestimmen. Vermutlich handelt es sich um den aktuellsten Versuch.

. Der Fehler für Ihren Testnachrichtenversuch wird angezeigt:
+
`*system node autosupport history show -node local -seq-num seq_num -fields error*`

+
Wenn die Fehlermeldung  `Login denied` angezeigt wird, akzeptiert Ihr SMTP-Server keine Sendeanfragen vom Cluster-Management-LIF. Wenn keine Umstellung auf HTTPS als Transportprotokoll gewünscht ist, sollte der Netzwerkadministrator kontaktiert werden, damit die SMTP-Gateways entsprechend konfiguriert werden können, um dieses Problem zu beheben.

+
Wenn dieser Test erfolgreich ist, die gleiche Nachricht jedoch an mailto:autosupport@netapp.com nicht gesendet werden kann, sollte sichergestellt werden, dass SMTP Relay auf allen SMTP-Mail-Hosts aktiviert ist oder HTTPS als Transportprotokoll verwendet wird.

+
Falls auch die Nachricht an das lokal verwaltete E-Mail-Konto nicht zugestellt werden kann, bestätigen Sie, dass Ihre SMTP-Server so konfiguriert sind, dass sie Anhänge mit beiden dieser Eigenschaften weiterleiten:

+
** Das Suffix "`7z`"
** Der MIME-Typ "`application/x-7x-compressed`".
