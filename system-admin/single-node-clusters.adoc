= Verwaltung eines ONTAP Clusters mit einem einzelnen Knoten
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Ein Einzelknoten-Cluster ist eine spezielle Implementierung eines Clusters, der auf einem einzelnen Knoten läuft. Einzelknoten-Cluster sind nicht empfehlenswert, da sie keine Redundanz bieten. Fällt der Knoten aus, geht der Datenzugriff verloren.

[NOTE]
====
Für Fehlertoleranz und unterbrechungsfreien Betrieb wird dringend empfohlen, den Cluster mit link:../concepts/high-availability-pairs-concept.html["Hochverfügbarkeit (HA Paare)"] zu konfigurieren.

====
Wenn Sie einen Einzelknoten-Cluster konfigurieren oder aktualisieren, ist Folgendes zu beachten:

* Die Verschlüsselung des Root-Volumes wird auf Single-Node-Clustern nicht unterstützt.
* Wenn Sie Knoten entfernen, um einen Einzelknoten-Cluster zu erhalten, sollten die Cluster-Ports so modifiziert werden, dass sie den Datenverkehr bedienen, indem die Cluster-Ports in Daten-Ports umgewandelt und anschließend Daten-LIFs auf den Daten-Ports erstellt werden.
* Bei Einzelknotenclustern kann das Ziel für die Konfigurationssicherung während der Softwareinstallation festgelegt werden. Nach der Installation können diese Einstellungen mit ONTAP Befehlen geändert werden.
* Sind mehrere Hosts mit dem Knoten verbunden, kann jeder Host mit einem anderen Betriebssystem wie Windows oder Linux konfiguriert werden. Bestehen mehrere Pfade vom Host zum Controller, muss ALUA auf dem Host aktiviert sein.




[[ways-to-configure-iscsi-san-hosts-with-single-nodes]]
== Möglichkeiten zur Konfiguration von iSCSI SAN Hosts mit einzelnen Knoten

Sie können iSCSI SAN Hosts so konfigurieren, dass sie entweder direkt mit einem einzelnen Knoten oder über einen oder mehrere IP Switches verbunden werden. Der Knoten kann mehrere iSCSI Verbindungen zum Switch haben.

.Direkt angeschlossene Single-Node Konfigurationen
Bei direkt angeschlossenen Single-Node-Konfigurationen sind ein oder mehrere Hosts direkt mit dem Node verbunden.

image:scrn_en_drw_fc-302020-direct-sing-on.png["Direkt angeschlossene Single-Node Konfiguration"]

.Single-network Single-Node-Konfigurationen
In Konfigurationen mit einem einzigen Netzwerk und einem einzigen Knoten verbindet ein Switch einen einzelnen Knoten mit einem oder mehreren Hosts. Da nur ein einziger Switch verwendet wird, ist diese Konfiguration nicht vollständig redundant.

image:r-oc-set-iscsi-singlenetwork-singlenode.png["Single-network Single-Node-Konfiguration"]

.Multi-Network Single-Node-Konfigurationen
In Multi-Netzwerk-Einzelknotenkonfigurationen verbinden zwei oder mehr Switches einen einzelnen Knoten mit einem oder mehreren Hosts. Da mehrere Switches vorhanden sind, ist diese Konfiguration vollständig redundant.

image:scrn-en-drw-iscsi-multinw-singlen.png["Multi-Network Single-Node-Konfiguration"]



[[ways-to-configure-fc-and-fc-nvme-san-hosts-with-single-nodes]]
== Möglichkeiten zur Konfiguration von FC- und FC-NVMe SAN-Hosts mit einzelnen Knoten

FC- und FC-NVMe SAN-Hosts können über ein oder mehrere Fabrics mit einzelnen Knoten konfiguriert werden. N-Port ID Virtualisierung (NPIV) ist erforderlich und muss auf allen FC-Switches im Fabric aktiviert sein. FC- oder FC-NVMe SAN-Hosts können nicht ohne FC-Switch direkt an einzelne Knoten angebunden werden.

.Single-fabric Single-Node-Konfigurationen
In Single Fabric Single Node Konfigurationen verbindet ein einzelner Switch einen Knoten mit einem oder mehreren Hosts. Da nur ein einziger Switch verwendet wird, ist diese Konfiguration nicht vollständig redundant.

Bei Konfigurationen mit einem einzigen Fabric und einem einzelnen Knoten ist Multipathing-Software nicht erforderlich, wenn Sie nur einen einzigen Pfad vom Host zum Knoten haben.

.Multifabric Einzelknoten-Konfigurationen
In Multifabric Single-Node-Konfigurationen verbinden zwei oder mehr Switches einen einzelnen Knoten mit einem oder mehreren Hosts. Zur Vereinfachung zeigt die folgende Abbildung eine Multifabric Single-Node-Konfiguration mit nur zwei Fabrics, jedoch sind in jeder Multifabric-Konfiguration zwei oder mehr Fabrics möglich. In dieser Abbildung ist der Storage Controller im oberen Chassis montiert, und das untere Chassis kann leer sein oder, wie in diesem Beispiel, ein IOMX-Modul enthalten.

Die FC-Zielports (0a, 0c, 0b, 0d) in den Abbildungen sind Beispiele. Die tatsächlichen Portnummern variieren je nach Modell Ihres Speicherknotens und ob Erweiterungsadapter verwendet werden.

image:scrn_en_drw_fc-62xx-multi-singlecontroller.png["Multifabric Einzelknotenkonfiguration mit nur zwei Fabrics"]

[[ontap-upgrade-for-single-node-cluster]]
== ONTAP Upgrade für Einzelknoten-Cluster

Sie können die ONTAP CLI verwenden, um ein automatisiertes Update eines Einzelknotenclusters durchzuführen. Einzelknotencluster weisen keine Redundanz auf, das bedeutet, dass Updates immer unterbrechungsfrei sind. Unterbrechende Upgrades können nicht mit System Manager durchgeführt werden.

.Bevor Sie beginnen
Das Upgrade-link:../upgrade/prepare.html["Vorbereitung"]Schritte müssen abgeschlossen werden.

.Schritte
. Das vorherige ONTAP Softwarepaket löschen:
+
[source, cli]
----
cluster image package delete -version <previous_package_version>
----
. Das gewünschte ONTAP Softwarepaket herunterladen:
+
[source, cli]
----
cluster image package get -url location
----
+
[listing]
----
cluster1::> cluster image package get -url http://www.example.com/software/9.7/image.tgz

Package download completed.
Package processing completed.
----
. Es sollte sichergestellt sein, dass das Softwarepaket im Cluster-Paket-Repository verfügbar ist:
+
[source, cli]
----
cluster image package show-repository
----
+
[listing]
----
cluster1::> cluster image package show-repository
Package Version  Package Build Time
---------------- ------------------
9.7              M/DD/YYYY 10:32:15
----
. Es sollte überprüft werden, ob der Cluster für das Upgrade bereit ist:
+
[source, cli]
----
cluster image validate -version <package_version_number>
----
+
[listing]
----
cluster1::> cluster image validate -version 9.7

WARNING: There are additional manual upgrade validation checks that must be performed after these automated validation checks have completed...
----
. Fortschritt der Validierung überwachen:
+
[source, cli]
----
cluster image show-update-progress
----
. Alle im Rahmen der Validierung identifizierten erforderlichen Aktionen müssen abgeschlossen werden.
. Optional kann eine Kostenschätzung für ein Software-Upgrade erstellt werden:
+
[source, cli]
----
cluster image update -version <package_version_number> -estimate-only
----
+
Die Kostenschätzung für das Software-Upgrade enthält Details zu jeder zu aktualisierenden Komponente und die geschätzte Dauer des Upgrades.

. Das Software-Upgrade ausführen:
+
[source, cli]
----
cluster image update -version <package_version_number>
----
+

NOTE: Wenn ein Problem auftritt, wird das Update angehalten und Sie werden aufgefordert, eine Korrekturmaßnahme zu ergreifen. Mit dem Befehl cluster image show-update-progress lassen sich Details zu etwaigen Problemen und dem Fortschritt des Updates anzeigen. Nach Behebung des Problems lässt sich das Update mit dem Befehl cluster image resume-update fortsetzen.

. Fortschritt der Clusteraktualisierung anzeigen:
+
[source, cli]
----
cluster image show-update-progress
----
+
Der Knoten wird im Rahmen des Updates neu gestartet und ist während des Neustarts nicht erreichbar.

. Benachrichtigung auslösen:
+
[source, cli]
----
autosupport invoke -node * -type all -message "Finishing_Upgrade"
----
+
Wenn Ihr Cluster nicht für den Versand von Nachrichten konfiguriert ist, wird eine Kopie der Benachrichtigung lokal gespeichert.
