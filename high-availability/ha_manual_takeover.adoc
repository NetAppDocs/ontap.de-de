= ONTAP manuelle Takeover-Befehle
:hardbreaks:
:allow-uri-read: 
:nofooter: 
:icons: font
:linkattrs: 
:imagesdir: ../media/


[role="lead"]
Sie können eine Übernahme manuell durchführen, wenn beim Partner Wartungsarbeiten erforderlich sind und in anderen ähnlichen Situationen. Je nach Zustand des Partners variiert der Befehl, den Sie zur Übernahme verwenden.

|===


| Wenn Sie möchten... | Verwenden Sie diesen Befehl... 


| Den Partnerknoten übernehmen | `storage failover takeover` 


| Der Fortschritt der Übernahme kann verfolgt werden, während die Aggregate des Partners auf den Knoten verschoben werden, der die Übernahme durchführt. | `storage failover show‑takeover` 


| Der Storage-Failover-Status für alle Knoten im Cluster wird angezeigt. | `storage failover show` 


| Den Partnerknoten ohne Migration der LIFs übernehmen | `storage failover takeover ‑skip‑lif‑migration‑before‑takeover true` 


| Den Partnerknoten auch bei einer Festplatteninkompatibilität übernehmen | `storage failover takeover -allow-disk-inventory-mismatch true` 


| Den Partnerknoten auch bei einer ONTAP Versionsabweichung übernehmen. *Hinweis:* Diese Option wird nur während des unterbrechungsfreien ONTAP Upgrade-Prozesses verwendet. | `storage failover takeover ‑option allow‑version‑mismatch` 


| Den Partnerknoten übernehmen, ohne eine Aggregate-Umverteilung durchzuführen | `storage failover takeover ‑bypass‑optimization true` 


| Den Partnerknoten übernehmen, bevor dieser seine Speicherressourcen ordnungsgemäß schließen kann | `storage failover takeover ‑option immediate` 
|===
[NOTE]
====
Bevor Sie den storage failover-Befehl mit der Option "immediate" ausführen, müssen die Data-LIFs mithilfe des folgenden Befehls auf einen anderen Node migriert werden: `network interface migrate-all -node _node_`

Weitere Informationen zu  `network interface migrate-all` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/network-interface-migrate-all.html["ONTAP-Befehlsreferenz"^].

Wenn Sie den  `storage failover takeover ‑option immediate` Befehl angeben, ohne vorher die Daten-LIFs zu migrieren, wird die Daten-LIF-Migration vom Knoten erheblich verzögert, selbst wenn die  `skip‑lif‑migration‑before‑takeover` Option nicht angegeben wird.

Wenn die Option „sofort“ angegeben wird, wird die ausgehandelte Übernahmeoptimierung umgangen, selbst wenn die Option „bypass‑optimization“ auf _false_ gesetzt ist.

====


[[moving-epsilon-for-certain-manually-initiated-takeovers]]
== Verschiebung von Epsilon für bestimmte manuell initiierte Takeovers

Sie sollten Epsilon verschieben, wenn Sie davon ausgehen, dass manuell eingeleitete Übernahmen dazu führen könnten, dass Ihr Speichersystem nur einen unerwarteten Knotenausfall von einem clusterweiten Quorumverlust entfernt ist.

.Über diese Aufgabe
Für geplante Wartungsarbeiten muss einer der Knoten in einem HA-Paar übernommen werden. Das Cluster-weite Quorum muss aufrechterhalten werden, um ungeplante Unterbrechungen der Client-Daten für die verbleibenden Knoten zu verhindern. In einigen Fällen kann die Übernahme dazu führen, dass der Cluster nur noch einen unerwarteten Knotenausfall vom clusterweiten Verlust des Quorums entfernt ist.

Dies kann auftreten, wenn der zu übernehmende Knoten Epsilon enthält oder wenn der Knoten mit Epsilon nicht fehlerfrei ist. Zur Aufrechterhaltung eines widerstandsfähigeren Clusters kann Epsilon auf einen fehlerfreien Knoten übertragen werden, der nicht übernommen wird. Typischerweise wäre dies der HA Partner.

Nur gesunde und teilnahmeberechtigte Knoten nehmen an der Quorumsabstimmung teil. Um ein clusterweites Quorum aufrechtzuerhalten, sind mehr als N/2 Stimmen erforderlich (wobei N die Summe der gesunden, teilnahmeberechtigten, online befindlichen Knoten darstellt). In Clustern mit einer geraden Anzahl online befindlicher Knoten verleiht Epsilon dem Knoten, dem es zugeordnet ist, zusätzliches Stimmgewicht zur Aufrechterhaltung des Quorums.


NOTE: Obwohl die Abstimmung zur Clusterbildung mithilfe des  `cluster modify ‑eligibility false` Befehls geändert werden kann, sollte dies nur in Situationen wie der Wiederherstellung der Knotenkonfiguration oder bei längerer Wartung eines Knotens erfolgen. Wenn ein Knoten als nicht berechtigt festgelegt wird, stellt er keine SAN-Daten mehr bereit, bis er wieder als berechtigt markiert und neu gestartet wird. Auch der NAS-Datenzugriff auf den Knoten kann beeinträchtigt sein, wenn der Knoten nicht berechtigt ist.

.Schritte
. Den Clusterstatus überprüfen und bestätigen, dass Epsilon von einem fehlerfreien Knoten gehalten wird, der nicht übernommen wird:
+
.. Zur erweiterten Berechtigungsstufe wechseln und bestätigen, dass fortgefahren werden soll, wenn die Aufforderung für den erweiterten Modus erscheint (*>):
+
`set -privilege advanced`

.. Ermitteln Sie, welcher Knoten Epsilon enthält:
+
`cluster show`

+
Im folgenden Beispiel enthält Node1 das Epsilon:

+
|===


| Node | Systemzustand | Teilnahmeberechtigung | Epsilon 


 a| 
Node1 Node2
 a| 
wahr wahr
 a| 
wahr wahr
 a| 
wahr falsch

|===
+
Falls der Knoten, den Sie übernehmen möchten, kein Epsilon enthält, ist mit Schritt 4 fortzufahren.

+
Weitere Informationen zu  `cluster show` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/cluster-show.html["ONTAP-Befehlsreferenz"^].



. Epsilon vom Knoten entfernen, der übernommen werden soll:
+
`cluster modify -node Node1 -epsilon false`

. Epsilon dem Partnerknoten (in diesem Beispiel Node2) zuweisen:
+
`cluster modify -node Node2 -epsilon true`

. Die Übernahme wird durchgeführt:
+
`storage failover takeover -ofnode _node_name_`

. Zurück zur Administrator-Privilegstufe:
+
`set -privilege admin`
