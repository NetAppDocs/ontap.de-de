= ONTAP Knoten in Zwei-Node-Clustern anhalten oder neu starten, ohne eine Übernahme einzuleiten
:hardbreaks:
:allow-uri-read: 
:nofooter: 
:icons: font
:linkattrs: 
:imagesdir: ../media/


[role="lead"]
Ein Halt oder Neustart eines Knotens in einem Zwei-Node-Cluster ohne Einleitung einer Übernahme erfolgt, wenn bestimmte Hardware-Wartungsarbeiten an einem Knoten oder einem Shelf durchgeführt werden und die Ausfallzeit durch das Weiterbetreiben des Partnerknotens begrenzt werden soll oder wenn Probleme eine manuelle Übernahme verhindern und die Aggregate des Partnerknotens weiterhin in Betrieb bleiben und Daten bereitstellen sollen. Zusätzlich kann es vorkommen, dass der technische Support im Rahmen der Fehlerbehebung diese Vorgehensweise empfiehlt.

.Über diese Aufgabe
* Bevor Sie die Übernahme verhindern (mithilfe des  `-inhibit-takeover true` Parameters), deaktivieren Sie die Cluster HA.


[CAUTION]
====
* In einem Zwei-Node-Cluster stellt Cluster HA sicher, dass der Ausfall eines Knotens den Cluster nicht deaktiviert. Wenn Cluster HA jedoch nicht deaktiviert wird, bevor der  `-inhibit-takeover true` Parameter verwendet wird, stellen beide Knoten die Datenbereitstellung ein.
* Wenn versucht wird, einen Node anzuhalten oder neu zu starten, bevor Cluster HA deaktiviert wird, gibt ONTAP eine Warnung aus und weist darauf hin, dass Cluster HA deaktiviert werden muss.


====
* Sie migrieren LIFs (logische Schnittstellen) zu dem Partner-Node, der online bleiben soll.
* Wenn sich auf dem Knoten, den Sie anhalten oder neu starten, Aggregate befinden, die Sie behalten möchten, werden diese auf den Knoten verschoben, der online bleiben soll.


.Schritte
. Überprüfen Sie, ob beide Knoten fehlerfrei sind:
+
[source, cli]
----
cluster show
----
+
Für beide Knoten `true` erscheint es in der `Health` Spalte.

+
[listing]
----
cluster::> cluster show
Node         Health  Eligibility
------------ ------- ------------
node1        true     true
node2        true     true
----
+
Weitere Informationen zu  `cluster show` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/cluster-show.html["ONTAP-Befehlsreferenz"^].

. Migrieren Sie alle LIFs von dem Knoten, den Sie anhalten oder neu starten möchten, auf den Partnerknoten:
+
[source, cli]
----
network interface migrate-all -node <node_name>
----
+
Weitere Informationen zu  `network interface migrate-all` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/network-interface-migrate-all.html["ONTAP-Befehlsreferenz"^].

. Wenn sich auf dem Knoten Aggregate befinden, die Sie beim Anhalten oder Neustarten des Knotens online halten möchten, verschieben Sie diese auf den Partnerknoten.
+
.. Zeigen Sie die Aggregate auf dem Knoten, den Sie anhalten oder neu starten werden, an:
+
[source, cli]
----
storage aggregates show -node <node_name>
----
+
Beispielsweise ist node1 der Knoten, der angehalten oder neu gestartet wird:

+
[listing]
----
cluster::> storage aggregates show -node node1
Aggregate  Size  Available  Used%  State  #Vols   Nodes   RAID  Status
---------  ----  ---------  -----  -----  -----   -----   ----  ------
aggr0_node_1_0
           744.9GB   32.68GB   96% online       2 node1    raid_dp,
                                                                normal
aggr1       2.91TB    2.62TB   10% online       8 node1    raid_dp,
                                                                normal
aggr2
            4.36TB    3.74TB   14% online      12 node1    raid_dp,
                                                                normal
test2_aggr  2.18TB    2.18TB    0% online       7 node1    raid_dp,
                                                                normal
4 entries were displayed.
----
.. Verschieben Sie die Aggregate auf den Partnerknoten:
+
[source, cli]
----
storage aggregate relocation start -node <node_name> -destination <node_name> -aggregate-list <aggregate_name>
----
+
Beispielsweise werden die Aggregate aggr1, aggr2 und test2_aggr von node1 auf node2 verschoben:

+
`storage aggregate relocation start -node _node1_ -destination _node2_ -aggregate-list _aggr1_,_aggr2_,_test2_aggr_`



. Cluster-HA deaktivieren:
+
[source, cli]
----
cluster ha modify -configured false
----
+
Wenn Cluster-HA deaktiviert ist, wird Epsilon automatisch auf Knoten 1 verschoben. Die Rückgabe bestätigt, dass HA deaktiviert ist: `Notice: HA is disabled`

+

NOTE: Dieser Vorgang deaktiviert das Storage-Failover nicht.

. Wenn der anzuhaltende oder neu zu startende Knoten Knoten 1 ist, verschieben Sie Epsilon zu Knoten 2:
+
.. Stellen Sie die Berechtigungsstufe auf „Erweitert“ ein:
+
[source, cli]
----
set -privilege advanced
----
.. Überprüfen Sie, ob Knoten 2 intakt und für Epsilon geeignet ist:
+
[source, cli]
----
cluster show
----
.. Entfernen Sie Epsilon von Knoten 1:
+
[source, cli]
----
cluster modify -node <node1_name> -epsilon false
----
.. Weisen Sie Knoten 2 Epsilon zu:
+
[source, cli]
----
cluster modify -node <node2_name> -epsilon true
----
.. Überprüfen Sie, ob Epsilon auf Knoten 2 ist:
+
[source, cli]
----
cluster show
----


. Anhalten oder Neustarten und Übernahme des Zielknotens verhindern:
+
[cols="1h,1h"]
|===


 a| 
Den Node anhalten, ohne eine Übernahme einzuleiten:
 a| 
[source, cli]
----
system node halt -node _node_name_ -inhibit-takeover true
----


 a| 
Starten Sie den Knoten neu, ohne die Übernahme einzuleiten:
 a| 
[source, cli]
----
system node reboot -node _node_name_ -inhibit-takeover true
----
|===
+

NOTE: In der Befehlsausgabe erscheint eine Warnung, ob Sie fortfahren möchten, geben Sie  `y` ein.

. Überprüfen Sie, ob sich der noch online befindliche Knoten in einem fehlerfreien Zustand befindet (während der Partner ausgefallen ist):
+
[source, cli]
----
cluster show
----
+
Für den Online-Knoten, `true` erscheint in der `Health` Spalte.

+

NOTE: In der Ausgabe des Befehls wird eine Warnung angezeigt, dass Cluster-HA nicht konfiguriert ist. Sie können die Warnung ignorieren.

. Führen Sie die Aktionen aus, die ein Anhalten oder einen Neustart des Knotens erfordern.
. Starten Sie den offline geschalteten Node über die LOADER-Eingabeaufforderung:
+
[source, cli]
----
boot_ontap
----
. Überprüfen Sie, ob beide Knoten fehlerfrei sind:
+
[source, cli]
----
cluster show
----
+
Für beide Knoten `true` erscheint es in der `Health` Spalte.

+

NOTE: In der Ausgabe des Befehls wird eine Warnung angezeigt, dass Cluster-HA nicht konfiguriert ist. Die Warnung kann zu diesem Zeitpunkt ignoriert werden.

. Cluster-HA wieder aktivieren:
+
[source, cli]
----
cluster ha modify -configured true
----
. Wenn Sie zuvor in diesem Verfahren Aggregate auf den Partnerknoten verschoben haben, verschieben Sie sie zurück auf ihren Heimatknoten.
+
[source, cli]
----
storage aggregate relocation start -node <node_name> -destination <node_name> -aggregate-list <aggregate_name>
----
+
Beispielsweise werden die Aggregate aggr1, aggr2 und test2_aggr vom Knoten node2 zum Knoten node1 verschoben:
`storage aggregate relocation start -node _node2_ -destination _node1_ -aggregate-list _aggr1_,_aggr2_,_test2_aggr_`

. LIFs auf ihre Home-Ports zurückführen:
+
.. LIFs anzeigen, die nicht zu Hause sind:
+
[source, cli]
----
network interface show -is-home false
----
+
Weitere Informationen zu  `network interface show` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/network-interface-show.html["ONTAP-Befehlsreferenz"^].

.. Falls es nicht-Home-LIFs gibt, die nicht vom ausgefallenen Node migriert wurden, sollte vor dem Zurücksetzen überprüft werden, ob eine sichere Verschiebung möglich ist.
.. Wenn es sicher ist, tun Sie dies und setzen Sie alle LIFs auf ihre Ausgangsposition zurück.
+
[source, cli]
----
 network interface revert *
----
+
Weitere Informationen zu  `network interface revert` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/network-interface-revert.html["ONTAP-Befehlsreferenz"^].
