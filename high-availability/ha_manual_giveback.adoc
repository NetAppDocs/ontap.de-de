= ONTAP manuelle Rückgabebefehle
:hardbreaks:
:allow-uri-read: 
:nofooter: 
:icons: font
:linkattrs: 
:imagesdir: ../media/


[role="lead"]
Es ist möglich, eine normale Rückgabe, eine Rückgabe mit Beendigung von Prozessen auf dem Partnerknoten oder eine erzwungene Rückgabe durchzuführen.


NOTE: Vor der Durchführung einer Rückgabe müssen die defekten Laufwerke im übernommenen System wie in link:../disks-aggregates/index.html["Festplatten- und Aggregatverwaltung"] beschrieben entfernt werden.



[[if-giveback-is-interrupted]]
== Wenn die Rückgabe unterbrochen wird

Wenn der Übernahmeknoten während des Rückgabeprozesses einen Ausfall oder einen Stromausfall erleidet, stoppt dieser Prozess und der Übernahmeknoten kehrt in den Übernahmemodus zurück, bis der Ausfall behoben oder die Stromversorgung wiederhergestellt ist.

Dies hängt jedoch von der Phase der Rückgabe ab, in der der Fehler aufgetreten ist. Wenn beim Knoten während des teilweisen Rückgabestatus (nachdem das Root-Aggregat zurückgegeben wurde) ein Fehler oder ein Stromausfall auftritt, kehrt er nicht in den Übernahmemodus zurück. Stattdessen kehrt der Knoten in den Modus der teilweisen Rückgabe zurück. Tritt dies auf, wird der Vorgang abgeschlossen, indem die Rückgabeoperation wiederholt wird.



[[if-giveback-is-vetoed]]
== Wenn das Giveback abgelehnt wird

Wird die Rückgabe abgelehnt, müssen die EMS-Meldungen geprüft werden, um die Ursache zu ermitteln. Abhängig vom Grund oder von den Gründen kann entschieden werden, ob die Ablehnungen gefahrlos außer Kraft gesetzt werden können.

Der  `storage failover show-giveback` Befehl zeigt den Fortschritt der Rückgabe an und gibt an, welches Subsystem die Rückgabe gegebenenfalls abgelehnt hat. Weiche Vetos können überschrieben werden, harte Vetos hingegen nicht, selbst wenn sie erzwungen werden. Die folgenden Tabellen fassen die weichen Vetos zusammen, die nicht überschrieben werden sollten, sowie empfohlene Alternativen.

Die EMS-Details zu etwaigen Giveback-Vetos lassen sich mit dem folgenden Befehl anzeigen:

`event log show -node * -event gb*`

Weitere Informationen zu  `event log show` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/event-log-show.html["ONTAP-Befehlsreferenz"^].



[[giveback-of-the-root-aggregate]]
== Rückgabe des Root-Aggregats

Diese Vetos gelten nicht für Root-Aggregat-Umsiedlungsmaßnahmen:

[cols="35,65"]
|===


| Vetoing-Subsystemmodul | Problemumgehung 


 a| 
vfiler_low_level
 a| 
Die SMB-Sitzungen, die das Veto verursachen, müssen beendet werden, oder die SMB-Anwendung, die die offenen Sitzungen hergestellt hat, muss heruntergefahren werden.

Das Übergehen dieses Vetos könnte dazu führen, dass die Anwendung, die SMB verwendet, abrupt die Verbindung trennt und Daten verliert.



 a| 
Datenträgerprüfung
 a| 
Alle ausgefallenen oder umgangenen Datenträger sollten vor dem Rückgabeversuch entfernt werden. Wenn Datenträger bereinigt werden, sollte abgewartet werden, bis der Vorgang abgeschlossen ist.

Das Aufheben dieses Vetos kann zu einem Ausfall führen, wenn Aggregate oder Volumes aufgrund von Reservierungskonflikten oder nicht zugänglichen Festplatten offline gehen.

|===


[[giveback-of-the-sfo-aggregates]]
== Rückgabe der SFO Aggregate

Diese Vetos gelten nicht für Root-Aggregat-Umsiedlungsmaßnahmen:

[cols="35,65"]
|===


| Vetoing-Subsystemmodul | Problemumgehung 


 a| 
Lock Manager
 a| 
Die SMB-Anwendungen, die geöffnete Dateien haben, sollten ordnungsgemäß heruntergefahren werden, oder diese Volumes in ein anderes Aggregat verschoben werden.

Das Außerkraftsetzen dieses Vetos führt zum Verlust des SMB-Sperrstatus, was zu Unterbrechungen und Datenverlust führt.



 a| 
Lock Manager NDO
 a| 
Warten Sie, bis die Sperren gespiegelt sind.

Das Übergehen dieses Vetos führt zu Unterbrechungen bei Microsoft Hyper-V-virtuellen Maschinen.



| RAID  a| 
Die EMS-Meldungen geben Aufschluss über die Ursache des Vetos:

Falls das Veto auf nvfile zurückzuführen ist, sollten die offline befindlichen Volumes und Aggregate online geschaltet werden.

Wenn Datenträger-Hinzufügen- oder Datenträgereigentumsübertragungs-Operationen im Gange sind, abwarten, bis diese abgeschlossen sind.

Falls das Veto auf einen Konflikt mit einem Aggregatnamen oder einer UUID zurückzuführen ist, sollte das Problem behoben werden.

Wenn das Veto auf eine Spiegelsynchronisierung, eine Spiegelverifizierung oder Offline-Festplatten zurückzuführen ist, kann das Veto überschrieben werden und der Vorgang startet nach der Rückgabe erneut.



| Datenträgerinventar  a| 
Fehlerbehebung zur Identifizierung und Behebung der Problemursache.

Der Zielknoten kann möglicherweise die zu einem migrierten Aggregat gehörenden Datenträger nicht sehen.

Nicht zugängliche Datenträger können zu nicht zugänglichen Aggregaten oder Volumes führen.



| Volume-Move-Vorgang  a| 
Fehlerbehebung zur Identifizierung und Behebung der Problemursache.

Dieses Veto verhindert, dass der Vorgang zum Verschieben des Volumes während der wichtigen Umstellungsphase abgebrochen wird. Wenn der Vorgang während der Umstellungsphase abgebrochen wird, könnte das Volume nicht mehr erreichbar sein.

|===


[[commands-for-performing-a-manual-giveback]]
=== Befehle zur Durchführung einer manuellen Rückgabe

Sie können auf einem Node in einem HA-Paar manuell eine Rückgabe initiieren, um den Speicher nach Abschluss der Wartung oder der Behebung von Problemen, die die Übernahme verursacht haben, an den ursprünglichen Eigentümer zurückzugeben.

|===


| Wenn Sie möchten... | Verwenden Sie diesen Befehl... 


 a| 
Speicher an einen Partnerknoten zurückgeben
| `storage failover giveback ‑ofnode _nodename_` 


 a| 
Speicherplatz kann auch dann zurückgegeben werden, wenn sich der Partner nicht im Modus „Warten auf Rückgabe“ befindet.
 a| 
`storage failover giveback ‑ofnode _nodename_`
`‑require‑partner‑waiting false`

Diese Option sollte nur verwendet werden, wenn ein längerer Clientausfall akzeptabel ist.



| Speicher auch dann zurückgeben, wenn Prozesse den Giveback-Vorgang verhindern (Giveback erzwingen)  a| 
`storage failover giveback ‑ofnode _nodename_`
`‑override‑vetoes true`

Die Nutzung dieser Option kann unter Umständen zu längeren Client-Ausfällen führen oder dazu, dass Aggregate und Volumes nach der Rückmeldung nicht online gehen.



| Nur die CFO-Aggregate (das Root-Aggregat) zurückgeben  a| 
`storage failover giveback ‑ofnode _nodename_`

`‑only‑cfo‑aggregates true`



| Der Fortschritt der Rückgabe kann nach Ausführung des Befehls „giveback“ überwacht werden. | `storage failover show‑giveback` 
|===
