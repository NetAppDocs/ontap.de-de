= Unterstützte Umgebungsvariablen für ONTAP NDMP
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Umgebungsvariablen werden verwendet, um Informationen über einen Backup- oder Wiederherstellungsvorgang zwischen einer NDMP-fähigen Backup-Anwendung und einem Speichersystem auszutauschen.

Wenn ein Benutzer beispielsweise angibt, dass eine Backup-Anwendung ein Backup erstellen soll `/vserver1/vol1/dir1`, setzt die Backup-Anwendung die Umgebungsvariable FILESYSTEM auf `/vserver1/vol1/dir1`. Ebenso setzt die Backup-Anwendung die Umgebungsvariable LEVEL auf 1 (eins), wenn ein Benutzer angibt, dass es sich um ein Backup der Stufe 1 handeln soll.

[NOTE]
====
Das Festlegen und Prüfen von Umgebungsvariablen erfolgt in der Regel transparent für Backup-Administratoren; das heißt, die Backup-Anwendung legt sie automatisch fest.

====
Ein Backup-Administrator legt Umgebungsvariablen selten fest; es kann jedoch erforderlich sein, den Wert einer Umgebungsvariablen gegenüber dem von der Backup-Anwendung festgelegten Wert zu ändern, um ein Funktions- oder Leistungsproblem zu charakterisieren oder zu umgehen. Beispielsweise könnte ein Administrator die Generierung des Dateiverlaufs vorübergehend deaktivieren, um festzustellen, ob die Verarbeitung von Dateiverlaufsinformationen durch die Backup-Anwendung zu Leistungsproblemen oder Funktionsproblemen beiträgt.

Viele Backup-Anwendungen bieten die Möglichkeit, Umgebungsvariablen zu überschreiben oder zu ändern oder zusätzliche Umgebungsvariablen anzugeben. Informationen dazu enthält die Dokumentation der jeweiligen Backup-Anwendung.



[[environment-variables-supported-by-ontap]]
== Von ONTAP unterstützte Umgebungsvariablen

ONTAP unterstützt Umgebungsvariablen, die jeweils einen zugeordneten Standardwert haben. Diese Standardwerte können jedoch manuell geändert werden.

Wenn Sie die von der Backup-Anwendung festgelegten Werte manuell ändern, kann sich die Anwendung unvorhersehbar verhalten. Dies liegt daran, dass die Backup- oder Wiederherstellungsvorgänge möglicherweise nicht das tun, was die Backup-Anwendung erwartet. In manchen Fällen kann eine gezielte Änderung jedoch helfen, Probleme zu identifizieren oder zu umgehen.

Die folgenden Tabellen listen die Umgebungsvariablen auf, deren Verhalten für dump und SMTape identisch ist, sowie diejenigen Variablen, die nur für dump beziehungsweise nur für SMTape unterstützt werden. Diese Tabellen enthalten außerdem Beschreibungen, wie die von ONTAP unterstützten Umgebungsvariablen funktionieren, falls diese verwendet werden:

[NOTE]
====
In den meisten Fällen akzeptieren Variablen, die den Wert  `Y` haben, auch  `T` und  `N` akzeptieren auch  `F`.

====

[[environment-variables-supported-for-dump]]
== Unterstützte Umgebungsvariablen für Dump

|===
| Umgebungsvariable | Gültige Werte | Standard | Beschreibung 


 a| 
ACL_START
 a| 
`return_only`
 a| 
`none`
 a| 
Die von der Backup-Operation erzeugte Variable ACL_START ist ein Offset-Wert, der von einer Direct Access Restore- oder einer wiederaufnehmbaren NDMP Backup-Operation verwendet wird.

Der Offset-Wert ist der Byte-Offset in der Dump-Datei, an dem die ACL-Daten (Pass V) beginnen und am Ende einer Sicherung zurückgegeben werden. Damit eine Wiederherstellung per Direktzugriff die gesicherten Daten korrekt wiederherstellt, muss der Wert ACL_START beim Start der Wiederherstellung an die Wiederherstellungsoperation übergeben werden. Eine NDMP wiederaufnehmbare Sicherungsoperation verwendet den Wert ACL_START, um der Backup-Anwendung mitzuteilen, wo der nicht wiederaufnehmbare Teil des Sicherungsdatenstroms beginnt.



 a| 
BASISDATUM
 a| 
`0`, `-1` oder `DUMP_DATE` Wert
 a| 
`-1`
 a| 
Legt das Startdatum für inkrementelle Backups fest.

Wenn auf  `-1` gesetzt, ist der inkrementelle BASE_DATE-Spezifizierer deaktiviert. Wenn auf  `0` bei einer Stufe-0-Sicherung gesetzt, sind inkrementelle Sicherungen aktiviert. Nach der ersten Sicherung wird der Wert der DUMP_DATE-Variable aus der vorherigen inkrementellen Sicherung der BASE_DATE-Variable zugewiesen.

Diese Variablen stellen eine Alternative zu den auf LEVEL/UPDATE basierenden inkrementellen Backups dar.



 a| 
DIREKT
 a| 
`Y` oder `N`
 a| 
`N`
 a| 
Legt fest, dass bei einer Wiederherstellung direkt zu der Stelle auf dem Band vorgespult wird, an der sich die Dateidaten befinden, anstatt das gesamte Band zu durchsuchen.

Für die Wiederherstellung per Direktzugriff muss die Backup-Anwendung Positionsinformationen bereitstellen. Wenn diese Variable auf  `Y` gesetzt ist, gibt die Backup-Anwendung die Datei- oder Verzeichnisnamen und die Positionsinformationen an.



 a| 
DMP_NAME
 a| 
`string`
 a| 
`none`
 a| 
Gibt den Namen für ein Backup mit mehreren Unterverzeichnissen an.

Diese Variable ist für mehrere Teilbaum-Sicherungen obligatorisch.



 a| 
DUMP_DATE
 a| 
`return_value`
 a| 
`none`
 a| 
Diese Variable wird nicht direkt geändert. Sie wird vom Backup erstellt, wenn die Variable BASE_DATE auf einen anderen Wert als  `-1` gesetzt ist.

Die Variable DUMP_DATE wird berechnet, indem dem vom Backup-Programm ermittelten 32-Bit-Zeitwert der 32-Bit-Levelwert vorangestellt wird. Der Levelwert wird gegenüber dem zuletzt an die Variable BASE_DATE übergebenen Wert inkrementiert. Der resultierende Wert wird als BASE_DATE Wert bei einem nachfolgenden inkrementellen Backup verwendet.



 a| 
ENHANCED_DAR_ENABLED
 a| 
`Y` oder `N`
 a| 
`N`
 a| 
Gibt an, ob die erweiterte DAR-Funktionalität aktiviert ist. Die erweiterte DAR-Funktionalität unterstützt Verzeichnis-DAR und DAR von Dateien mit NT Streams. Sie bietet Leistungsverbesserungen.

Erweiterte DAR während der Wiederherstellung ist nur möglich, wenn die folgenden Bedingungen erfüllt sind:

* ONTAP unterstützt erweitertes DAR.
* Die Dateiversionshistorie ist während des Backups aktiviert (HIST=Y).
* Die  `ndmpd.offset_map.enable`Option ist auf `on` eingestellt.
* Die Variable ENHANCED_DAR_ENABLED ist während der Wiederherstellung auf  `Y` gesetzt.




 a| 
AUSSCHLIESSEN
 a| 
`pattern_string`
 a| 
`none`
 a| 
Gibt Dateien oder Verzeichnisse an, die beim Sichern von Daten ausgeschlossen werden.

Die Ausschlussliste ist eine durch Kommas getrennte Liste von Datei- oder Verzeichnisnamen. Wenn der Name einer Datei oder eines Verzeichnisses mit einem der Namen in der Liste übereinstimmt, wird es vom Backup ausgeschlossen.

Für die Angabe von Namen in der Ausschlussliste gelten folgende Regeln:

* Es muss der genaue Name der Datei oder des Verzeichnisses verwendet werden.
* Das Sternchen (*), ein Platzhalterzeichen, muss entweder das erste oder das letzte Zeichen der Zeichenkette sein.
+
Jede Zeichenkette darf bis zu zwei Sternchen enthalten.

* Einem Komma in einem Datei- oder Verzeichnisnamen muss ein Backslash vorangestellt werden.
* Die Ausschlussliste kann bis zu 32 Namen enthalten.


[NOTE]
====
Dateien oder Verzeichnisse, die für den Ausschluss vom Backup angegeben wurden, werden nicht ausgeschlossen, wenn NON_QUOTA_TREE `Y` gleichzeitig gesetzt wird.

====


 a| 
EXTRAKT
 a| 
`Y`, `N`, oder `E`
 a| 
`N`
 a| 
Gibt an, dass Teilstrukturen eines gesicherten Datensatzes wiederhergestellt werden.

Die Backup-Anwendung gibt die Namen der zu extrahierenden Unterverzeichnisse an. Wenn eine angegebene Datei mit einem Verzeichnis übereinstimmt, dessen Inhalt gesichert wurde, wird das Verzeichnis rekursiv extrahiert.

Um eine Datei, ein Verzeichnis oder einen Qtree während der Wiederherstellung ohne Verwendung von DAR umzubenennen, muss die Umgebungsvariable EXTRACT auf  `E` gesetzt werden.



 a| 
EXTRACT_ACL
 a| 
`Y` oder `N`
 a| 
`Y`
 a| 
Gibt an, dass die ACLs aus der gesicherten Datei bei einem Wiederherstellungsvorgang wiederhergestellt werden.

Standardmäßig werden beim Wiederherstellen von Daten auch die ACLs wiederhergestellt, mit Ausnahme von DARs (DIRECT=Y).



 a| 
ERZWINGEN
 a| 
`Y` oder `N`
 a| 
`N`
 a| 
Legt fest, ob bei der Wiederherstellungsoperation der Speicherplatz und die Inode-Verfügbarkeit auf dem Zielvolume überprüft werden müssen.

Das Setzen dieser Variablen auf `Y` bewirkt, dass beim Wiederherstellungsvorgang die Prüfungen auf Speicherplatz und Inode-Verfügbarkeit im Zielpfad übersprungen werden.

Wenn auf dem Zielvolume nicht genügend Speicherplatz oder Inodes verfügbar sind, stellt der Wiederherstellungsvorgang so viele Daten wieder her, wie es der verfügbare Speicherplatz und die Inode-Verfügbarkeit des Zielvolumes zulassen. Der Wiederherstellungsvorgang stoppt, wenn kein Speicherplatz oder keine Inodes mehr verfügbar sind.



 a| 
GESCHICHTE
 a| 
`Y` oder `N`
 a| 
`N`
 a| 
Legt fest, dass Dateiverlaufsinformationen an die Backup-Anwendung gesendet werden.

Die meisten kommerziellen Backup-Anwendungen setzen die HIST-Variable auf `Y`. Um die Geschwindigkeit eines Backup-Vorgangs zu erhöhen oder ein Problem mit der Dateiverlaufserfassung zu beheben, kann diese Variable auf `N`gesetzt werden.

[NOTE]
====
Die Variable HIST sollte nicht auf  `Y` gesetzt werden, wenn die Sicherungsanwendung keine Dateiverlaufsfunktion unterstützt.

====


 a| 
IGNORE_CTIME
 a| 
`Y` oder `N`
 a| 
`N`
 a| 
Gibt an, dass eine Datei nicht inkrementell gesichert wird, wenn sich seit der letzten inkrementellen Sicherung nur ihr ctime-Wert geändert hat.

Manche Anwendungen, wie beispielsweise Virenscanner, ändern den ctime-Wert einer Datei innerhalb des Inodes, selbst wenn die Datei oder ihre Attribute unverändert sind. Dadurch kann es vorkommen, dass bei einer inkrementellen Sicherung auch unveränderte Dateien gesichert werden. Die  `IGNORE_CTIME` Variable sollte nur dann angegeben werden, wenn inkrementelle Sicherungen aufgrund der Änderung des ctime-Werts unzumutbar viel Zeit oder Speicherplatz beanspruchen.

[NOTE]
====
Der  `NDMP dump`Befehl setzt  `IGNORE_CTIME`standardmäßig auf  `false`. Die Einstellung auf  `true`kann zu folgendem Datenverlust führen:

. Wenn  `IGNORE_CTIME` auf true gesetzt ist und ein inkrementelles Volume-Level  `ndmpcopy` verwendet wird, führt dies zum Löschen von Dateien, die auf der Quelle zwischen Qtrees verschoben werden.
. Wenn  `IGNORE_CTIME` auf „true“ gesetzt wird, während ein inkrementelles Backup auf Volume-Ebene durchgeführt wird, führt dies zum Löschen von Dateien, die während der inkrementellen Wiederherstellung über Qtrees auf dem Quellsystem verschoben werden.


Um dieses Problem zu vermeiden, muss  `IGNORE_CTIME` während der Volume-Ebene  `NDMP dumps` oder  `ndmpcopy` auf „false“ gesetzt werden.

====


 a| 
IGNORE_QTREES
 a| 
`Y` oder `N`
 a| 
`N`
 a| 
Gibt an, dass beim Wiederherstellungsvorgang keine Qtree-Informationen aus gesicherten Qtrees wiederhergestellt werden.



 a| 
EBENE
 a| 
`0`-`31`
 a| 
`0`
 a| 
Legt die Sicherungsebene fest.

Stufe 0 kopiert den gesamten Datensatz. Inkrementelle Sicherungsstufen, die durch Werte über 0 angegeben werden, kopieren alle Dateien (neue oder geänderte) seit der letzten inkrementellen Sicherung. Beispielsweise sichert Stufe 1 neue oder geänderte Dateien seit der Sicherung der Stufe 0, Stufe 2 neue oder geänderte Dateien seit der Sicherung der Stufe 1 und so weiter.



 a| 
LISTE
 a| 
`Y` oder `N`
 a| 
`N`
 a| 
Listet die Namen der gesicherten Dateien und die Inode-Nummern auf, ohne die Daten tatsächlich wiederherzustellen.



 a| 
LIST_QTREES
 a| 
`Y` oder `N`
 a| 
`N`
 a| 
Listet die gesicherten Qtrees auf, ohne die Daten tatsächlich wiederherzustellen.



 a| 
MULTI_SUBTREE_NAMES
 a| 
`string`
 a| 
`none`
 a| 
Gibt an, dass es sich bei der Sicherung um eine Sicherung mehrerer Unterbäume handelt.

Mehrere Teilbäume werden in der Zeichenkette angegeben, die eine durch Zeilenumbrüche getrennte, mit einem Nullbyte abgeschlossene Liste von Teilbaumnamen darstellt. Teilbäume werden durch Pfadnamen relativ zu ihrem gemeinsamen Stammverzeichnis angegeben, das als letztes Element der Liste angegeben werden muss.

Wenn Sie diese Variable verwenden, müssen Sie auch die Variable DMP_NAME verwenden.



 a| 
NDMP_UNICODE_ FH
 a| 
`Y` oder `N`
 a| 
`N`
 a| 
Gibt an, dass zusätzlich zum NFS-Namen der Datei auch ein Unicode-Name in die Dateiverlaufsinformationen aufgenommen ist.

Diese Option wird von den meisten Backup-Anwendungen nicht verwendet und sollte nur gesetzt werden, wenn die Backup-Anwendung für den Empfang dieser zusätzlichen Dateinamen ausgelegt ist. Die Variable HIST muss ebenfalls gesetzt werden.



 a| 
NO_ACLS
 a| 
`Y` oder `N`
 a| 
`N`
 a| 
Legt fest, dass ACLs beim Sichern von Daten nicht kopiert werden.



 a| 
NON_QUOTA_TREE
 a| 
`Y` oder `N`
 a| 
`N`
 a| 
Legt fest, dass Dateien und Verzeichnisse in Qtrees beim Sichern von Daten ignoriert werden.

Wenn auf  `Y` gesetzt, werden Elemente in Qtrees des durch die Variable FILESYSTEM angegebenen Datensatzes nicht gesichert. Diese Variable hat nur eine Auswirkung, wenn die Variable FILESYSTEM ein gesamtes Volume angibt. Die Variable NON_QUOTA_TREE funktioniert nur bei einem Backup auf Level 0 und ist wirkungslos, wenn die Variable MULTI_SUBTREE_NAMES angegeben ist.

[NOTE]
====
Dateien oder Verzeichnisse, die für den Ausschluss vom Backup angegeben wurden, werden nicht ausgeschlossen, wenn NON_QUOTA_TREE `Y` gleichzeitig gesetzt wird.

====


 a| 
NOWRITE
 a| 
`Y` oder `N`
 a| 
`N`
 a| 
Gibt an, dass beim Wiederherstellungsvorgang keine Daten auf die Festplatte geschrieben werden.

Diese Variable wird zum Debugging verwendet.



 a| 
REKURSIV
 a| 
`Y` oder `N`
 a| 
`Y`
 a| 
Legt fest, dass Verzeichniseinträge während einer DAR Wiederherstellung erweitert werden.

Die Umgebungsvariablen DIRECT und ENHANCED_DAR_ENABLED müssen ebenfalls aktiviert sein (auf  `Y` gesetzt). Wenn die Variable RECURSIVE deaktiviert ist (auf  `N` gesetzt), werden nur die Berechtigungen und ACLs für alle Verzeichnisse im ursprünglichen Quellpfad vom Tape wiederhergestellt, nicht aber deren Inhalt. Wenn die Variable RECURSIVE auf  `N` gesetzt ist oder die Variable RECOVER_FULL_PATHS auf  `Y` gesetzt ist, muss der Wiederherstellungspfad mit dem ursprünglichen Pfad enden.

[NOTE]
====
Wenn die Variable RECURSIVE deaktiviert ist und mehr als ein Wiederherstellungspfad vorhanden ist, müssen alle Wiederherstellungspfade im längsten der Wiederherstellungspfade enthalten sein. Andernfalls wird eine Fehlermeldung angezeigt.

====
Beispielsweise sind die folgenden Wiederherstellungspfade gültig, da sich alle Wiederherstellungspfade innerhalb von  `foo/dir1/deepdir/myfile` befinden:

* `/foo`
* `/foo/dir`
* `/foo/dir1/deepdir`
* `/foo/dir1/deepdir/myfile`


Folgende Wiederherstellungspfade sind ungültig:

* `/foo`
* `/foo/dir`
* `/foo/dir1/myfile`
* `/foo/dir2`
* `/foo/dir2/myfile`




 a| 
RECOVER_FULL_PATHS
 a| 
`Y` oder `N`
 a| 
`N`
 a| 
Gibt an, dass nach der DAR die Berechtigungen und ACLs des vollständigen Wiederherstellungspfads wiederhergestellt werden.

DIRECT und ENHANCED_DAR_ENABLED müssen ebenfalls aktiviert sein (auf  `Y` gesetzt). Wenn RECOVER_FULL_PATHS auf  `Y` gesetzt ist, muss der Wiederherstellungspfad mit dem ursprünglichen Pfad enden. Falls Verzeichnisse bereits auf dem Zielvolume vorhanden sind, werden deren Berechtigungen und ACLs nicht vom Tape wiederhergestellt.



 a| 
AKTUALISIEREN
 a| 
`Y` oder `N`
 a| 
`Y`
 a| 
Aktualisiert die Metadateninformationen, um LEVEL basierte inkrementelle Backups zu ermöglichen.

|===


[[environment-variables-supported-for-smtape]]
== Für SMTape unterstützte Umgebungsvariablen

|===
| Umgebungsvariable | Gültige Werte | Standard | Beschreibung 


 a| 
BASISDATUM
 a| 
`DUMP_DATE`
 a| 
`-1`
 a| 
Legt das Startdatum für inkrementelle Backups fest.

 `BASE_DATE` ist eine Zeichenkettendarstellung der Referenz-Snapshot-Kennungen. Mit der  `BASE_DATE`Zeichenkette wird der Referenz-Snapshot von SMTape gefunden.

 `BASE_DATE` ist für Basissicherungen nicht erforderlich. Für eine inkrementelle Sicherung wird der Wert der  `DUMP_DATE`Variablen aus der vorherigen Basissicherung oder inkrementellen Sicherung der  `BASE_DATE`Variablen zugewiesen.

Die Backup-Anwendung weist den  `DUMP_DATE` Wert aus einem vorherigen SMTape Baseline- oder inkrementellen Backup zu.



 a| 
DUMP_DATE
 a| 
`return_value`
 a| 
`none`
 a| 
Am Ende eines SMTape-Backups enthält DUMP_DATE eine Zeichenfolgenkennung, die den für dieses Backup verwendeten Snapshot identifiziert. Dieser Snapshot kann als Referenz-Snapshot für ein nachfolgendes inkrementelles Backup verwendet werden.

Der resultierende Wert von DUMP_DATE wird als BASE_DATE Wert für nachfolgende inkrementelle Backups verwendet.



 a| 
SMTAPE_BACKUP_SET_ID
 a| 
`string`
 a| 
`none`
 a| 
Identifiziert die Abfolge der inkrementellen Backups, die mit dem Baseline-Backup verknüpft sind.

Die Backup-Set-ID ist eine 128-Bit- eindeutige ID, die während eines Basis-Backups generiert wird. Die Backup-Anwendung weist diese ID während eines inkrementellen Backups der  `SMTAPE_BACKUP_SET_ID` Variable als Eingabe zu.



 a| 
SMTAPE_SNAPSHOT_NAME
 a| 
Jeder gültige Snapshot, der im Volume verfügbar ist
 a| 
`Invalid`
 a| 
Wenn die Variable SMTAPE_SNAPSHOT_NAME auf einen Snapshot gesetzt ist, werden dieser Snapshot und seine älteren Snapshots auf Tape-Backup gesichert.

Für das inkrementelle Tape-Backup gibt diese Variable den inkrementellen Snapshot an. Die Variable BASE_DATE stellt den Baseline-Snapshot bereit.



 a| 
SMTAPE_DELETE_SNAPSHOT
 a| 
`Y` oder `N`
 a| 
`N`
 a| 
Für einen von SMTape automatisch erstellten Snapshot, wenn die Variable SMTAPE_DELETE_SNAPSHOT auf  `Y` gesetzt ist, löscht SMTape diesen Snapshot nach Abschluss des Backup-Vorgangs. Ein von der Backup-Anwendung erstellter Snapshot wird jedoch nicht gelöscht.



 a| 
SMTAPE_BREAK_MIRROR
 a| 
`Y` oder `N`
 a| 
`N`
 a| 
Wenn die Variable SMTAPE_BREAK_MIRROR auf  `Y` gesetzt ist, wird das Volume vom Typ  `DP` nach einer erfolgreichen Wiederherstellung in ein  `RW` Volume geändert.

|===
