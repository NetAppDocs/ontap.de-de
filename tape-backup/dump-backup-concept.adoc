= Wie eine Dump-Sicherung mit ONTAP NDMP funktioniert
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Eine Dump-Sicherung schreibt Dateisystemdaten mithilfe eines vordefinierten Prozesses von der Festplatte auf Band. Es ist möglich, ein Volume, einen Qtree oder einen Teilbaum zu sichern, der weder ein vollständiges Volume noch ein vollständiger Qtree ist.

Die folgende Tabelle beschreibt den Prozess, den ONTAP verwendet, um das durch den Dump-Pfad angegebene Objekt zu sichern:

[cols="1,4"]
|===
| Stufe | Aktion 


 a| 
1
 a| 
Bei weniger als vollständigen Volume- oder vollständigen Qtree-Backups durchsucht ONTAP die Verzeichnisse, um die zu sichernden Dateien zu identifizieren. Beim Backup eines gesamten Volumes oder Qtrees kombiniert ONTAP diese Phase mit Phase 2.



 a| 
2
 a| 
Bei einer vollständigen Volume- oder vollständigen Qtree-Sicherung identifiziert ONTAP die zu sichernden Verzeichnisse in den Volumes oder Qtrees.



 a| 
3
 a| 
ONTAP schreibt die Verzeichnisse auf Band.



 a| 
4
 a| 
ONTAP schreibt die Dateien auf Band.



 a| 
5
 a| 
ONTAP schreibt die ACL-Informationen (falls zutreffend) auf Tape.

|===
Die Dump-Sicherung verwendet eine Snapshot-Kopie Ihrer Daten für das Backup. Daher ist es nicht erforderlich, das Volume vor dem Start des Backups offline zu nehmen.

Die Speicherabbildsicherung benennt jeden Snapshot, den sie erstellt, als  `snapshot_for_backup.n`, wobei  `n` eine ganze Zahl ist, die bei 0 beginnt. Jedes Mal, wenn die Speicherabbildsicherung einen Snapshot erstellt, wird die Zahl um 1 erhöht. Die Zahl wird nach einem Neustart des Speichersystems auf 0 zurückgesetzt. Nach Abschluss der Sicherungsoperation löscht die Speicherabbildsicherung diesen Snapshot.

Wenn ONTAP mehrere Dump-Backups gleichzeitig durchführt, erstellt die Dump-Engine mehrere Snapshots. Wenn ONTAP zwei Dump-Backups gleichzeitig ausführt, finden sich die folgenden Snapshots in den Volumes, von denen Daten gesichert werden: `snapshot_for_backup.0` und `snapshot_for_backup.1`.

[NOTE]
====
Beim Backup von einem Snapshot erstellt die Dump-Engine keinen zusätzlichen Snapshot.

====


[[types-of-data-that-the-dump-engine-backs-up]]
== Datentypen, die die Dump-Engine sichert

Die Dump-Engine ermöglicht es, Daten auf Band zu sichern, um sich gegen Katastrophen oder Controller-Ausfälle abzusichern. Zusätzlich zur Sicherung von Datenobjekten wie Dateien, Verzeichnissen, Qtrees oder ganzen Volumes kann die Dump-Engine viele Arten von Informationen zu jeder Datei sichern. Kenntnisse über die Datentypen, die die Dump-Engine sichern kann, sowie die zu berücksichtigenden Einschränkungen können bei der Planung des Ansatzes zur Notfallwiederherstellung helfen.

Zusätzlich zur Datensicherung in Dateien kann die Dump-Engine gegebenenfalls folgende Informationen zu jeder Datei sichern:

* UNIX-GID, Besitzer-UID und Dateiberechtigungen
* UNIX-Zugriffs-, Erstellungs- und Änderungszeit
* Dateityp
* Dateigröße
* DOS-Name, DOS-Attribute und Erstellungszeit
* Zugriffskontrolllisten (ACLs) mit 1.024 Zugriffskontrolleinträgen (ACEs)
* Qtree-Informationen
* Junction-Pfade


Junction-Pfade werden als symbolische Links gesichert.

* LUN und LUN-Klone
+
Sie können ein gesamtes LUN-Objekt sichern; jedoch kann keine einzelne Datei innerhalb des LUN-Objekts gesichert werden. Ebenso kann ein gesamtes LUN-Objekt wiederhergestellt werden, jedoch keine einzelne Datei innerhalb des LUN-Objekts.

+
[NOTE]
====
Die Dump-Engine sichert LUN-Klone als unabhängige LUNs.

====
* VM-ausgerichtete Dateien
+
Backup von VM-ausgerichteten Dateien wird in Versionen vor Data ONTAP 8.1.2 nicht unterstützt.



[NOTE]
====
Wenn ein durch Snapshot gesicherter LUN-Klon von Data ONTAP im 7-Mode auf ONTAP migriert wird, wird er zu einer inkonsistenten LUN. Der Dump-Engine sichert inkonsistente LUNs nicht.

====
Bei der Wiederherstellung von Daten auf einem Volume ist die Client-E/A für die wiederherzustellenden LUNs eingeschränkt. Die LUN-Einschränkung wird erst aufgehoben, wenn der Dump-Wiederherstellungsvorgang abgeschlossen ist. Ebenso ist während eines SnapMirror Single File- oder LUN-Wiederherstellungsvorgangs die Client-E/A sowohl für die wiederherzustellenden Dateien als auch für die wiederherzustellenden LUNs eingeschränkt. Diese Einschränkung wird erst aufgehoben, wenn der Single File- oder LUN-Wiederherstellungsvorgang abgeschlossen ist. Wenn eine Dump-Sicherung auf einem Volume durchgeführt wird, auf dem gerade ein Dump-Wiederherstellungs- oder SnapMirror Single File- oder LUN-Wiederherstellungsvorgang läuft, werden die Dateien oder LUNs mit Client-E/A-Einschränkung nicht in die Sicherung aufgenommen. Diese Dateien oder LUNs werden in einer nachfolgenden Sicherung berücksichtigt, wenn die Client-E/A-Einschränkung aufgehoben ist.

[NOTE]
====
Eine LUN, die unter Data ONTAP 8.3 läuft und auf Tape gesichert wurde, kann nur auf 8.3 und spätere Versionen wiederhergestellt werden, nicht jedoch auf eine frühere Version. Wenn die LUN auf eine frühere Version wiederhergestellt wird, wird die LUN als Datei wiederhergestellt.

====
Wenn ein SnapVault sekundäres Volume oder ein Volume SnapMirror Ziel auf Band gesichert wird, werden nur die Daten auf dem Volume gesichert. Die zugehörigen Metadaten werden nicht gesichert. Daher werden beim Wiederherstellen des Volumes nur die Daten auf diesem Volume wiederhergestellt. Informationen über die Volume SnapMirror Beziehungen sind in der Sicherung nicht enthalten und werden daher nicht wiederhergestellt.

Wenn eine Datei gesichert wird, die nur Windows NT-Berechtigungen besitzt, und sie in einem UNIX-artigen Qtree oder Volume wiederhergestellt wird, erhält die Datei die standardmäßigen UNIX-Berechtigungen für diesen Qtree oder dieses Volume.

Wenn eine Datei, die nur UNIX-Berechtigungen hat, in einem NTFS-artigen Qtree oder Volume wiederhergestellt wird, erhält die Datei die standardmäßigen Windows-Berechtigungen für diesen Qtree oder dieses Volume.

Andere Dumps und Wiederherstellungen erhalten die Berechtigungen.

Sie können VM-ausgerichtete Dateien und die  `vm-align-sector` Option sichern. Weitere Informationen zu VM-ausgerichteten Dateien finden sich unter link:../volumes/index.html["Logische Speicherverwaltung"].
