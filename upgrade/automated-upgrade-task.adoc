= Das ONTAP Image mit automatisiertem, unterbrechungsfreiem ONTAP Upgrade installieren
:hardbreaks:
:toclevels: 1
:allow-uri-read: 
:toclevels: 1
:nofooter: 
:icons: font
:linkattrs: 
:imagesdir: ../media/


[role="lead"]
Wenn ein automatisiertes Upgrade durchgeführt wird, installiert ONTAP automatisch das Ziel-ONTAP-Image auf jedem Knoten, überprüft, ob der Cluster erfolgreich aktualisiert werden kann, und führt dann abhängig von der Anzahl der Knoten im Cluster entweder ein xref:concept_upgrade_methods.html[Batch- oder Rolling-Upgrade] im Hintergrund aus.

Wenn Ihre Konfiguration dies unterstützt, empfiehlt sich die Verwendung von System Manager für ein automatisiertes Upgrade. Wenn Ihre Konfiguration kein automatisiertes Upgrade mit System Manager unterstützt, kann die ONTAP Befehlszeilenschnittstelle (CLI) für ein automatisiertes Upgrade verwendet werden.


NOTE: Wenn Sie auf ONTAP 9.15.1 oder höher über die NetApp aktualisieren, befolgen Sie die link:https://docs.netapp.com/us-en/console-software-updates/get-started/software-updates.html["Upgrade-Verfahren in der NetApp Console Dokumentation"^].


IMPORTANT: Die Änderung der Einstellung der  `storage failover modify-auto-giveback` Befehlsoption vor dem Start eines automatischen unterbrechungsfreien Upgrades (ANDU) hat keine Auswirkungen auf den Upgrade-Prozess. Der ANDU-Prozess ignoriert während der für das Update erforderlichen Übernahme/Rückgabe jeglichen voreingestellten Wert dieser Option. Beispielsweise hat das Setzen von  `-autogiveback` auf „false“ vor Beginn von ANDU keine Unterbrechung des automatischen Upgrades vor der Rückgabe zur Folge. Weitere Informationen zu  `storage failover modify-auto-giveback` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/search.html?q=storage+failover+modify-auto-giveback["ONTAP-Befehlsreferenz"^].

.Bevor Sie beginnen
* Sie sollten link:prepare.html["Vorbereitung auf Ihr Upgrade"].
* Das sollten Sie link:download-software-image.html["Das ONTAP Software-Image herunterladen"] für Ihre Ziel-ONTAP-Version tun.
+
Wenn Sie ein link:../upgrade/concept_upgrade_paths.html#types-of-upgrade-paths["direktes Multi-Hop-Upgrade"] durchführen, müssen Sie beide für Ihre spezifische link:../upgrade/concept_upgrade_paths.html#supported-upgrade-paths["Upgrade-Pfad"] erforderlichen ONTAP Images herunterladen.

* Für jedes HA-Paar sollte jeder Knoten über einen oder mehrere Ports in derselben Broadcast-Domäne verfügen.
+
Wenn Ihr ONTAP Cluster 8 oder mehr Knoten umfasst, wird die Batch-Upgrade-Methode im automatischen unterbrechungsfreien Upgrade verwendet, um die Daten-LIF-Migration vor der SFO-Übernahme präventiv zu erzwingen. Wie LIFs während eines Batch-Upgrades migriert werden, hängt von Ihrer ONTAP Version ab.

+
[cols="2"]
|===
| Wenn Sie ONTAP ausführen... | LIFs werden migriert... 


 a| 
** 9.15.1 oder höher
** 9.14.1P5
** 9.13.1P10
** 9.12.1P13
** 9.11.1P16, P17
** 9.10.1P19

| Zu einem Knoten in der anderen Batch-Gruppe. Falls die Migration zur anderen Batch-Gruppe fehlschlägt, werden die LIFs zum HA-Partner des Knotens in derselben Batch-Gruppe migriert. 


| 9.8 bis 9.14.1 | Zu einem Knoten in der anderen Batch-Gruppe. Wenn die Netzwerk-Broadcast-Domäne die LIF-Migration zur anderen Batch-Gruppe nicht zulässt, schlägt die LIF-Migration fehl und ANDU pausiert. 


| 9.7 oder früher | An den HA-Partner des zu aktualisierenden Knotens. Falls der Partner keine Ports in derselben Broadcast-Domäne besitzt, schlägt die LIF-Migration fehl und ANDU pausiert. 
|===
* Wenn ONTAP in einer MetroCluster FC-Konfiguration aktualisiert wird, sollte der Cluster für die automatische ungeplante Umschaltung aktiviert sein.
* Wenn Sie nicht vorhaben, den Fortschritt des Upgrade-Prozesses zu überwachen, sollten Sie link:../error-messages/configure-ems-notifications-sm-task.html["EMS-Benachrichtigungen über Fehler anfordern, die möglicherweise manuelles Eingreifen erfordern"].
* Bei einem Einzelknoten-Cluster ist der im link:../system-admin/single-node-clusters.html["automatisiertes, unterbrechendes Upgrade"] Prozess beschriebene Ablauf zu beachten.
+
Upgrades von Einzelknoten-Clustern sind mit Störungen verbunden.



.Schritte
[role="tabbed-block"]
====
.System Manager
--
. Das ONTAP Zielabbild validieren:
+

NOTE: Wenn Sie eine MetroCluster Konfiguration aktualisieren, sollte Cluster A validiert und anschließend der Validierungsprozess auf Cluster B wiederholt werden.

+
.. Je nachdem, welche ONTAP Version verwendet wird, ist einer der folgenden Schritte auszuführen:
+
|===


| Wenn Sie ausführen... | Folgendes ist zu tun... 


| ONTAP 9.8 oder höher  a| 
*Cluster > Übersicht* auswählen.



| ONTAP 9.5, 9.6 und 9.7  a| 
*Konfiguration* > *Cluster* > *Aktualisieren* auswählen.



| ONTAP 9.4 oder früher  a| 
*Konfiguration* > *Cluster Update* auswählen.

|===
.. In der rechten Ecke des *Übersicht*-Bereichs auf image:icon_kabob.gif["Symbol für Menüoptionen"] klicken.
.. *ONTAP Update* auswählen.
.. Auf der Registerkarte *Cluster Update* ein neues Image hinzufügen oder ein verfügbares Image auswählen.
+
|===


| Wenn Sie möchten... | Dann... 


 a| 
Ein neues Software-Image aus einem lokalen Ordner hinzufügen

Sie sollten bereits link:download-software-image.html["das Image heruntergeladen"] eine Verbindung zum lokalen Client hergestellt haben.
 a| 
... Unter *Verfügbare Software-Images* auf *Von lokal hinzufügen* klicken.
... Navigieren Sie zu dem Speicherort, an dem Sie das Software-Image gespeichert haben, wählen Sie das Image aus und klicken Sie anschließend auf *Open*.




 a| 
Ein neues Software-Image von einem HTTP- oder FTP-Server hinzufügen
 a| 
... *Vom Server hinzufügen* auswählen.
... Im Dialogfeld *Neues Software-Image hinzufügen* ist die URL des HTTP- oder FTP-Servers einzugeben, auf den das ONTAP Software-Image von der NetApp Support Site heruntergeladen wurde.
+
Für anonymes FTP müssen Sie die URL im Format \ftp://anonymous@ftpserver angeben.

... Klicken Sie auf *Add*.




 a| 
Wählen Sie ein verfügbares Bild
 a| 
Eines der aufgelisteten Bilder kann ausgewählt werden.

|===
.. Wählen Sie *Validieren*, um die Validierungsprüfungen vor dem Upgrade auszuführen.
+
Werden während der Validierung Fehler oder Warnungen gefunden, werden diese zusammen mit einer Liste von Korrekturmaßnahmen angezeigt. Alle Fehler müssen behoben werden, bevor mit dem Upgrade fortgefahren werden kann. Es gilt als Best Practice, auch Warnungen zu beheben.

+
Ab ONTAP 9.19.1 werden Fehler und Warnungen separat angezeigt, damit Sie Probleme, die vor dem Upgrade behoben werden müssen, leichter erkennen können.



. Klicken Sie auf *Weiter*.
. Auf *Aktualisieren* klicken.
+
Die Validierung wird erneut durchgeführt. Alle verbleibenden Fehler oder Warnungen werden zusammen mit einer Liste der Korrekturmaßnahmen angezeigt. Fehler müssen behoben werden, bevor mit dem Upgrade fortgefahren werden kann. Wenn die Validierung mit Warnungen abgeschlossen wird, können die Warnungen behoben oder *Mit Warnungen aktualisieren* ausgewählt werden.

+

NOTE: Standardmäßig verwendet ONTAP die link:concept_upgrade_methods.html["Batch-Upgrade-Prozess"] zum Upgrade von Clustern mit acht oder mehr Knoten. Ab ONTAP 9.10.1 besteht die Möglichkeit, bei Bedarf *Jedes HA-Paar einzeln aktualisieren* auszuwählen, um die Standardeinstellung zu überschreiben und das Cluster mithilfe des Rolling-Upgrade-Prozesses jeweils um ein HA-Paar zu aktualisieren.

+
Bei MetroCluster Konfigurationen mit mehr als 2 Knoten startet der ONTAP Upgrade-Prozess gleichzeitig auf den HA-Paaren an beiden Standorten. Bei einer 2-Knoten MetroCluster Konfiguration wird das Upgrade zuerst an dem Standort gestartet, an dem das Upgrade nicht initiiert wird. Das Upgrade am verbleibenden Standort beginnt, nachdem das erste Upgrade vollständig abgeschlossen ist.

. Wenn Ihr Upgrade aufgrund eines Fehlers pausiert, auf die Fehlermeldung klicken, um die Details anzuzeigen, dann den Fehler beheben und link:resume-upgrade-after-andu-error.html["Das Upgrade fortsetzen"].


.Nachdem Sie fertig sind
Nach erfolgreichem Abschluss des Upgrades startet der Knoten neu, und Sie werden zur Anmeldeseite des System Manager weitergeleitet. Falls der Neustart des Knotens längere Zeit in Anspruch nimmt, empfiehlt sich ein Aktualisieren des Browsers.

.CLI
--
. Die ONTAP Zielsoftware-Abbilddatei validieren
+

NOTE: Wenn Sie eine MetroCluster Konfiguration aktualisieren, sollten zuerst die folgenden Schritte auf Cluster A und anschließend die gleichen Schritte auf Cluster B ausgeführt werden.

+
.. Das vorherige ONTAP Softwarepaket löschen:
+
[source, cli]
----
cluster image package delete -version <previous_ONTAP_Version>
----
.. Das Ziel-ONTAP Software-Image wird in das Cluster-Paket-Repository geladen:
+
[source, cli]
----
cluster image package get -url location
----
+
[listing]
----
cluster1::> cluster image package get -url http://www.example.com/software/9.13.1/image.tgz

Package download completed.
Package processing completed.
----
+
Wenn Sie ein link:../upgrade/concept_upgrade_paths.html#types-of-upgrade-paths["direktes Multi-Hop-Upgrade"] durchführen, muss auch das Softwarepaket für die für Ihr Upgrade erforderliche Zwischenversion von ONTAP geladen werden. Wenn beispielsweise ein Upgrade von 9.8 auf 9.13.1 erfolgt, muss das Softwarepaket für ONTAP 9.12.1 geladen werden, und anschließend wird mit demselben Befehl das Softwarepaket für 9.13.1 geladen.

.. Es sollte sichergestellt sein, dass das Softwarepaket im Cluster-Paket-Repository verfügbar ist:
+
[source, cli]
----
cluster image package show-repository
----
+
[listing]
----
cluster1::> cluster image package show-repository
Package Version  Package Build Time
---------------- ------------------
9.13.1              MM/DD/YYYY 10:32:15
----
.. Die automatisierten Vorabprüfungen für das Upgrade werden ausgeführt:
+
[source, cli]
----
cluster image validate -version <package_version_number>
----
+
Wenn Sie ein link:../upgrade/concept_upgrade_paths.html#types-of-upgrade-paths["direktes Multi-Hop-Upgrade"] durchführen, benötigen Sie für die Verifizierung lediglich das Ziel-ONTAP-Paket. Das Zwischen-Upgrade-Image muss nicht separat validiert werden. Beispielsweise wird bei einem Upgrade von 9.8 auf 9.13.1 das 9.13.1-Paket für die Verifizierung verwendet. Das 9.12.1-Paket muss nicht separat validiert werden.

+
[listing]
----
cluster1::> cluster image validate -version 9.13.1

WARNING: There are additional manual upgrade validation checks that must be performed after these automated validation checks have completed...
----
.. Fortschritt der Validierung überwachen:
+
[source, cli]
----
cluster image show-update-progress
----
.. Alle im Rahmen der Validierung identifizierten erforderlichen Aktionen müssen abgeschlossen werden.
+
Es wird eine Liste aller automatisierten Vorabprüfungen für das Upgrade angezeigt, zusammen mit allen Fehlern und Warnungen, die vor Beginn des Upgrade-Prozesses behoben werden sollten. Alle Fehler müssen behoben werden, bevor mit dem Upgrade fortgefahren werden kann. Es gilt als Best Practice, auch Warnungen zu beheben.

+
Ab ONTAP 9.19.1 werden Fehler und Warnungen separat angezeigt, damit Sie Probleme, die vor dem Upgrade behoben werden müssen, leichter erkennen können.

.. Wenn eine MetroCluster Konfiguration aktualisiert wird, sind die oben genannten Schritte auf Cluster B zu wiederholen.


. Kostenschätzung für ein Software-Upgrade erstellen:
+
[source, cli]
----
cluster image update -version <package_version_number> -estimate-only
----
+

NOTE: Wenn Sie eine MetroCluster Konfiguration aktualisieren, kann dieser Befehl entweder auf Cluster A oder auf Cluster B ausgeführt werden. Es ist nicht erforderlich, ihn auf beiden Clustern auszuführen.

+
Die Kostenschätzung für das Software-Upgrade enthält Details zu jeder zu aktualisierenden Komponente sowie die geschätzte Dauer des Upgrades.

. Das Software-Upgrade ausführen:
+
[source, cli]
----
cluster image update -version <package_version_number>
----
+
** Wenn Sie ein link:../upgrade/concept_upgrade_paths.html#types-of-upgrade-paths["direktes Multi-Hop-Upgrade"] durchführen, verwenden Sie die Ziel ONTAP Version für die package_version_number. Wenn Sie beispielsweise von ONTAP 9.8 auf 9.13.1 aktualisieren, wird 9.13.1 als package_version_number verwendet.
** Standardmäßig verwendet ONTAP die link:concept_upgrade_methods.html["Batch-Upgrade-Prozess"] zum Upgrade von Clustern mit acht oder mehr Knoten. Falls gewünscht, kann der  `-force-rolling` Parameter verwendet werden, um den Standardprozess zu überschreiben und das Cluster nacheinander, jeweils ein Knoten, mit dem Rolling-Upgrade-Prozess zu aktualisieren.
** Nach jedem Übernahme- und Rückgabevorgang wartet das Upgrade 8 Minuten, damit sich Clientanwendungen von der während der Übernahme und Rückgabe auftretenden E/A-Pause erholen können. Wenn Ihre Umgebung mehr oder weniger Zeit für die Stabilisierung der Clients benötigt, kann mit dem  `-stabilize-minutes` Parameter eine andere Dauer für die Stabilisierung festgelegt werden.
** Bei MetroCluster Konfigurationen mit 4 oder mehr Knoten startet das automatisierte Upgrade gleichzeitig auf den HA-Paaren an beiden Standorten. Bei einer 2-Node-MetroCluster Konfiguration beginnt das Upgrade an dem Standort, an dem das Upgrade nicht initiiert wurde. Das Upgrade am verbleibenden Standort beginnt, nachdem das erste Upgrade vollständig abgeschlossen ist.


+
[listing]
----
cluster1::> cluster image update -version 9.13.1

Starting validation for this update. Please wait..

It can take several minutes to complete validation...

WARNING: There are additional manual upgrade validation checks...

Pre-update Check      Status     Error-Action
--------------------- ---------- --------------------------------------------
...
20 entries were displayed

Would you like to proceed with update ? {y|n}: y
Starting update...

cluster-1::>
----
. Fortschritt der Clusteraktualisierung anzeigen:
+
[source, cli]
----
cluster image show-update-progress
----
+
Wenn Sie eine 4-Node- oder 8-Node MetroCluster Konfiguration aktualisieren, zeigt der  `cluster image show-update-progress` Befehl nur den Fortschritt für den Knoten an, auf dem der Befehl ausgeführt wird. Der Befehl muss auf jedem Knoten ausgeführt werden, um den Fortschritt der einzelnen Knoten anzuzeigen.

. Es sollte überprüft werden, ob das Upgrade auf jedem Knoten erfolgreich abgeschlossen wurde.
+
[source, cli]
----
cluster image show-update-progress
----
+
[listing]
----
cluster1::> cluster image show-update-progress

                                             Estimated         Elapsed
Update Phase         Status                   Duration        Duration
-------------------- ----------------- --------------- ---------------
Pre-update checks    completed                00:10:00        00:02:07
Data ONTAP updates   completed                01:31:00        01:39:00
Post-update checks   completed                00:10:00        00:02:00
3 entries were displayed.

Updated nodes: node0, node1.
----
. Eine AutoSupport-Benachrichtigung auslösen:
+
[source, cli]
----
autosupport invoke -node * -type all -message "Finishing_NDU"
----
+
Wenn Ihr Cluster nicht für den Versand von AutoSupport-Nachrichten konfiguriert ist, wird eine Kopie der Benachrichtigung lokal gespeichert.

. Wenn ein Upgrade einer 2-Node MetroCluster FC Konfiguration durchgeführt wird, sollte überprüft werden, ob der Cluster für die automatische ungeplante Umschaltung aktiviert ist.
+

NOTE: Wenn Sie eine Standardkonfiguration, eine MetroCluster IP-Konfiguration oder eine MetroCluster FC-Konfiguration mit mehr als 2 Knoten aktualisieren, ist dieser Schritt nicht erforderlich.

+
.. Es kann überprüft werden, ob die automatische ungeplante Umschaltung aktiviert ist:
+
[source, cli]
----
metrocluster show
----
+
Wenn die automatische ungeplante Umschaltung aktiviert ist, erscheint die folgende Meldung in der Befehlsausgabe:

+
....
AUSO Failure Domain    auso-on-cluster-disaster
....
.. Wenn die Anweisung nicht in der Ausgabe erscheint, sollte die automatische ungeplante Umschaltung aktiviert werden:
+
[source, cli]
----
metrocluster modify -auto-switchover-failure-domain auso-on-cluster-disaster
----
.. Es sollte überprüft werden, ob die automatische ungeplante Umschaltung aktiviert ist:
+
[source, cli]
----
metrocluster show
----

[[resume-ontap-software-upgrade-after-an-error-in-the-automated-upgrade-process]]
== Fortsetzung des ONTAP Software Upgrades nach einem Fehler im automatisierten Upgrade-Prozess

Wenn ein automatisiertes ONTAP Software-Upgrade aufgrund eines Fehlers pausiert, sollte der Fehler behoben und das Upgrade anschließend fortgesetzt werden. Nach der Fehlerbehebung besteht die Möglichkeit, den automatisierten Upgrade-Prozess fortzusetzen oder den Upgrade-Prozess manuell abzuschließen. Wenn die Fortsetzung des automatisierten Upgrades gewählt wird, dürfen keine der Upgrade-Schritte manuell durchgeführt werden.

.Schritte
[role="tabbed-block"]
====
.System Manager
--
. Je nachdem, welche ONTAP Version verwendet wird, ist einer der folgenden Schritte auszuführen:
+
|===


| Wenn Sie ausführen... | Dann... 


 a| 
ONTAP 9.8 oder höher
 a| 
*Cluster* > *Übersicht* auswählen



 a| 
ONTAP 9.7, 9.6 oder 9.5
 a| 
*Konfiguration* > *Cluster* > *Aktualisieren* auswählen.



 a| 
ONTAP 9.4 oder früher
 a| 
** *Konfiguration* > *Cluster Update* auswählen.
** In der rechten Ecke des *Übersicht*-Bereichs auf die drei blauen vertikalen Punkte klicken und *ONTAP Update* auswählen.


|===
. Das automatisierte Upgrade kann fortgesetzt oder abgebrochen und manuell weitergeführt werden.
+
|===


| Wenn Sie möchten... | Dann... 


 a| 
Automatisierte Aktualisierung fortsetzen
 a| 
Klicken Sie auf *Fortsetzen*.



 a| 
Das automatisierte Upgrade abbrechen und manuell fortsetzen
 a| 
Auf *Abbrechen* klicken.

|===

----


 a| 
Das automatisierte Upgrade abbrechen und manuell fortsetzen
 a| 
[source, cli]

----
