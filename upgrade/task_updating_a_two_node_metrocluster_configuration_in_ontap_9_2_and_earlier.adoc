= Manuelles unterbrechungsfreies Upgrade einer Zwei-Node-Cluster MetroCluster Konfiguration in ONTAP 9.2 oder früher
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Die Vorgehensweise zum Upgrade einer Zwei-Node-MetroCluster Konfiguration variiert je nach Ihrer ONTAP-Version. Bei Verwendung von ONTAP 9.2 oder einer früheren Version empfiehlt sich dieses Verfahren für ein manuelles unterbrechungsfreies Upgrade, das das Initiieren eines ausgehandelten Switchovers, das Aktualisieren des Clusters am „`failed`“-Standort, das Initiieren des Switchbacks und anschließend das Wiederholen des Prozesses am Cluster des anderen Standorts umfasst.

Wenn Sie eine Zwei-Knoten-MetroCluster-Konfiguration mit ONTAP 9.3 oder höher verwenden, befolgen Sie die CLI-Schritte in link:automated-upgrade-task.html["Das ONTAP Image mit automatisiertem, unterbrechungsfreiem ONTAP Upgrade installieren"].

.Schritte
. Die Berechtigungsstufe wird auf „Erweitert“ gesetzt, wobei bei entsprechender Aufforderung *y* einzugeben ist, um fortzufahren:
+
[source, cli]
----
set -privilege advanced
----
+
Die erweiterte Eingabeaufforderung (`*>`) erscheint.

. Auf dem zu aktualisierenden Cluster wird das neue ONTAP Software-Image als Standard installiert:
+
[source, cli]
----
system node image update -package package_location -setdefault true -replace-package true
----
+
[listing]
----
cluster_B::*> system node image update -package http://www.example.com/NewImage.tgz -setdefault true -replace-package true
----
. Es sollte überprüft werden, ob das Zielsoftware-Image als Standardimage festgelegt ist:
+
[source, cli]
----
system node image show
----
+
Das folgende Beispiel zeigt, dass  `NewImage` als Standardbild festgelegt ist:

+
[listing]
----
cluster_B::*> system node image show
                    Is      Is                           Install
Node     Image      Default Current Version              Date
-------- -------    ------- ------- -------------------- -------------------
node_B_1
         OldImage  false   true    X.X.X                MM/DD/YYYY TIME
         NewImage  true    false   Y.Y.Y                MM/DD/YYYY TIME
2 entries were displayed.
----
. Falls das Zielsoftware-Image nicht als Standard-Image festgelegt ist, sollte es geändert werden:
+
[source, cli]
----
system image modify {-node * -iscurrent false} -isdefault true
----
. Es wird sichergestellt, dass sich alle Cluster-SVMs in einem fehlerfreien Zustand befinden:
+
[source, cli]
----
metrocluster vserver show
----
. Auf dem Cluster, der nicht aktualisiert wird, einen ausgehandelten Switchover initiieren:
+
[source, cli]
----
metrocluster switchover
----
+
Der Vorgang kann einige Minuten dauern. Mit dem Befehl `metrocluster operation show` kann überprüft werden, ob die Umschaltung abgeschlossen ist.

+
Im folgenden Beispiel wird ein ausgehandelter Switchover auf dem Remote-Cluster („cluster_A“) durchgeführt. Dies führt dazu, dass der lokale Cluster („cluster_B“) angehalten wird, sodass er aktualisiert werden kann.

+
[listing]
----
cluster_A::> metrocluster switchover

Warning: negotiated switchover is about to start. It will stop all the data
         Vservers on cluster "cluster_B" and
         automatically re-start them on cluster
         "cluster_A". It will finally gracefully shutdown
         cluster "cluster_B".
Do you want to continue? {y|n}: y
----
. Es wird sichergestellt, dass sich alle Cluster-SVMs in einem fehlerfreien Zustand befinden:
+
[source, cli]
----
metrocluster vserver show
----
. Die Datenaggregate auf dem "`überlebenden`" Cluster neu synchronisieren:
+
[source, cli]
----
metrocluster heal -phase aggregates
----
+
Nach dem Upgrade der MetroCluster IP-Konfigurationen auf ONTAP 9.5 oder höher befinden sich die Aggregate für kurze Zeit in einem degradierten Zustand, bevor sie sich neu synchronisieren und wieder in einen gespiegelten Zustand zurückkehren.

+
[listing]
----
cluster_A::> metrocluster heal -phase aggregates
[Job 130] Job succeeded: Heal Aggregates is successful.
----
. Es sollte überprüft werden, ob der Heilungsvorgang erfolgreich abgeschlossen wurde:
+
[source, cli]
----
metrocluster operation show
----
+
[listing]
----
cluster_A::> metrocluster operation show
Operation: heal-aggregates
State: successful
Start Time: MM/DD/YYYY TIME
End Time: MM/DD/YYYY TIME
Errors: -
----
. Die Root-Aggregate auf dem "`surviving`" Cluster resynchronisieren:
+
[source, cli]
----
metrocluster heal -phase root-aggregates
----
+
[listing]
----
cluster_A::> metrocluster heal -phase root-aggregates
[Job 131] Job succeeded: Heal Root Aggregates is successful.
----
. Es sollte überprüft werden, ob der Heilungsvorgang erfolgreich abgeschlossen wurde:
+
[source, cli]
----
metrocluster operation show
----
+
[listing]
----
cluster_A::> metrocluster operation show
Operation: heal-root-aggregates
State: successful
Start Time: MM/DD/YYYY TIME
End Time: MM/DD/YYYY TIME
Errors: -
----
. Im angehaltenen Cluster wird der Node über die LOADER-Eingabeaufforderung gebootet:
+
[source, cli]
----
boot_ontap
----
. Warten Sie, bis der Bootvorgang abgeschlossen ist, und anschließend überprüfen, ob sich alle Cluster SVMs in einem fehlerfreien Zustand befinden:
+
[source, cli]
----
metrocluster vserver show
----
. Einen Switchback vom "`überlebenden`" Cluster durchführen:
+
[source, cli]
----
metrocluster switchback
----
. Es wird überprüft, ob der Rückwechsel erfolgreich abgeschlossen wurde:
+
[source, cli]
----
metrocluster operation show
----
+
[listing]
----
cluster_A::> metrocluster operation show
Operation: switchback
State: successful
Start Time: MM/DD/YYYY TIME
End Time: MM/DD/YYYY TIME
Errors: -
----
. Es wird sichergestellt, dass sich alle Cluster-SVMs in einem fehlerfreien Zustand befinden:
+
[source, cli]
----
metrocluster vserver show
----
. Alle vorherigen Schritte werden auf dem anderen Cluster wiederholt.
. Es sollte überprüft werden, ob die MetroCluster Konfiguration einwandfrei ist:
+
.. Konfiguration prüfen:
+
[source, cli]
----
metrocluster check run
----
+
[listing]
----
cluster_A::> metrocluster check run
Last Checked On: MM/DD/YYYY TIME
Component           Result
------------------- ---------
nodes               ok
lifs                ok
config-replication  ok
aggregates          ok
4 entries were displayed.

Command completed. Use the "metrocluster check show -instance"
command or sub-commands in "metrocluster check" directory for
detailed results.
To check if the nodes are ready to do a switchover or switchback
operation, run "metrocluster switchover -simulate" or "metrocluster
switchback -simulate", respectively.
----
.. Für detailliertere Ergebnisse kann der Befehl metrocluster check run verwendet werden:
+
[source, cli]
----
metrocluster check aggregate show
----
+
[source, cli]
----
metrocluster check config-replication show
----
+
[source, cli]
----
metrocluster check lif show
----
+
[source, cli]
----
metrocluster check node show
----
.. Stellen Sie die Berechtigungsstufe auf „Erweitert“ ein:
+
[source, cli]
----
set -privilege advanced
----
.. Den Umschaltvorgang simulieren:
+
[source, cli]
----
metrocluster switchover -simulate
----
.. Die Ergebnisse der Umschaltsimulation werden angezeigt:
+
[source, cli]
----
metrocluster operation show
----
+
[listing]
----
cluster_A::*> metrocluster operation show
    Operation: switchover
        State: successful
   Start time: MM/DD/YYYY TIME
     End time: MM/DD/YYYY TIME
       Errors: -
----
.. Zurück zur Administrator-Privilegstufe:
+
[source, cli]
----
set -privilege admin
----
.. Diese Teilschritte sind auch auf dem anderen Cluster durchzuführen.




.Nachdem Sie fertig sind
Beliebige Aktionen können ausgeführt werden link:task_what_to_do_after_upgrade.html["Aufgaben nach dem Upgrade"].
