= Cluster nach ONTAP Upgrade überprüfen
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Nach dem Upgrade von ONTAP sollten die Clusterversion, der Clusterzustand und der Speicherzustand überprüft werden. Für MetroCluster FC Konfigurationen ist außerdem zu prüfen, ob der Cluster für die automatische ungeplante Umschaltung aktiviert ist.



[[verify-cluster-version]]
== Clusterversion überprüfen

Nachdem alle HA-Paare aktualisiert wurden, kann mit dem Befehl version überprüft werden, ob auf allen Knoten die Zielversion ausgeführt wird.

Die Clusterversion ist die niedrigste Version von ONTAP, die auf einem beliebigen Knoten im Cluster ausgeführt wird. Wenn die Clusterversion nicht der Ziel-ONTAP-Version entspricht, kann Ihr Cluster aktualisiert werden.

. Wechsel zum erweiterten Berechtigungslevel:
+
[source, cli]
----
set -privilege advanced
----
. Es sollte sichergestellt werden, dass die Clusterversion der Ziel-ONTAP-Version entspricht:
+
[source, cli]
----
system node image show -version
----
. Wenn die Clusterversion nicht der Zielversion ONTAP entspricht, ist der Aktualisierungsstatus aller Knoten zu überprüfen:
+
[source, cli]
----
system node upgrade-revert show
----




[[verify-cluster-health]]
== Clusterzustand überprüfen

Nach der Aktualisierung eines Clusters ist zu überprüfen, ob die Knoten fehlerfrei funktionieren und zur Teilnahme am Cluster berechtigt sind und ob sich der Cluster im Quorum befindet.

. Es sollte sichergestellt werden, dass die Knoten im Cluster online sind und zur Teilnahme am Cluster berechtigt sind:
+
[source, cli]
----
cluster show
----
+
[listing]
----
cluster1::> cluster show
Node                  Health  Eligibility
--------------------- ------- ------------
node0                 true    true
node1                 true    true
----
+
Wenn ein Knoten fehlerhaft oder nicht geeignet ist, sollten die EMS-Protokolle auf Fehler überprüft und Korrekturmaßnahme ergriffen werden.

. Die Konfigurationsdetails für jeden RDB-Prozess werden überprüft.
+
** Die relationalen Datenbank-Epochen und die Datenbank-Epochen sollten für jeden Knoten übereinstimmen.
** Das primäre Quorum pro Ring sollte für alle Knoten gleich sein.
+
Beachten Sie, dass jeder Ring ein anderes Quorum-Primary haben kann.

+
|===


| Um diesen RDB-Prozess anzuzeigen... | Geben Sie diesen Befehl ein... 


 a| 
Verwaltungsanwendung
 a| 
`cluster ring show -unitname mgmt`



 a| 
Volume-Standortdatenbank
 a| 
`cluster ring show -unitname vldb`



 a| 
Manager für virtuelle Schnittstellen
 a| 
`cluster ring show -unitname vifmgr`



 a| 
SAN Management-Daemon
 a| 
`cluster ring show -unitname bcomd`

|===
+
Weitere Informationen zu  `cluster ring show` finden sich in der link:https://docs.netapp.com/us-en/ontap-cli/cluster-ring-show.html["ONTAP-Befehlsreferenz"^].

+
Dieses Beispiel zeigt den Volume Location Database-Prozess:



+
[listing]
----
cluster1::*> cluster ring show -unitname vldb
Node      UnitName Epoch    DB Epoch DB Trnxs Master    Online
--------- -------- -------- -------- -------- --------- ---------
node0     vldb     154      154      14847    node0     master
node1     vldb     154      154      14847    node0     secondary
node2     vldb     154      154      14847    node0     secondary
node3     vldb     154      154      14847    node0     secondary
4 entries were displayed.
----
. Wenn Sie in einer SAN-Umgebung arbeiten, überprüfen Sie, ob sich jeder Knoten in einem SAN-Quorum befindet:
+
[source, cli]
----
cluster kernel-service show
----
+
[listing]
----
cluster1::*> cluster kernel-service show
Master            Cluster           Quorum        Availability  Operational
Node              Node              Status        Status        Status
----------------- ----------------- ------------- ------------- -------------
cluster1-01       cluster1-01       in-quorum     true          operational
                  cluster1-02       in-quorum     true          operational
2 entries were displayed.
----
. Die Berechtigungsstufe wird auf Administrator zurückgesetzt:
+
[source, cli]
----
set -privilege admin
----

[[verify-automatic-unplanned-switchover-is-enabled-metrocluster-fc-configurations-only]]
== Überprüfen, ob die automatische ungeplante Umschaltung aktiviert ist (MetroCluster FC-Konfigurationen nur)

Wenn Ihr Cluster in einer MetroCluster FC Konfiguration betrieben wird, sollte nach dem Upgrade von ONTAP überprüft werden, ob der automatische ungeplante Switchover aktiviert ist.

Wenn Sie eine MetroCluster IP-Konfiguration verwenden, kann dieses Verfahren übersprungen werden.

.Schritte
. Es kann überprüft werden, ob die automatische ungeplante Umschaltung aktiviert ist:
+
[source, cli]
----
metrocluster show
----
+
Wenn die automatische ungeplante Umschaltung aktiviert ist, erscheint die folgende Meldung in der Befehlsausgabe:

+
[listing]
----
AUSO Failure Domain  auso-on-cluster-disaster
----
. Falls die Meldung nicht angezeigt wird, kann eine automatische ungeplante Umschaltung aktiviert werden:
+
[source, cli]
----
metrocluster modify -auto-switchover-failure-domain auso-on-cluster-disaster
----
. Es sollte überprüft werden, ob eine automatische ungeplante Umschaltung aktiviert ist:
+
[source, cli]
----
metrocluster show
----
